      Metadatenservice in OpenStack Neutron und Nova

      • by Ulysses Kanigel ● 2. Dezember 2016

      F. Warum sollten Sie den Nova/Neutron-Metadatenservice verwenden?

      A. Der Nova/Neutron-Metadatenservice ist eine Möglichkeit, Instanzen über das Netz Anpassungen hinzuzufügen. Sie können beispielsweise SSH-Schlüssel, Kennwörter, Hostnamen und angepasste Scripts hinzufügen. Der Metadatenservice wird unter 169.254.169.254 ausgeführt und übergibt die empfangenen Befehle über einen Proxy an Nova.

      F. Was benötige ich in meinem Gastbetriebssystem, um den Metadatenservice nutzen zu können?

      Im Gastbetriebssystem muss ein Softwarepaket mit dem Namen cloud-init ausgeführt werden, damit Metadaten beim Booten automatisch arbeiten können. Während des Bootvorgangs ruft cloud-init den Metadatenservice auf, um die Instanzen basierend auf den Metadaten, die der Benutzer während der Erstellung der Instanz übergibt, anzupassen.

      F. Warum sollten Sie den Nova/Neutron-Metadatenservice NICHT verwenden?

      A. Weil die Implementierung des Service sehr komplex ist und manchmal fehlschlägt. Neutron führt den Metadatenservice nicht wirklich aus, sondern vielmehr einen Proxy für die Metadaten. Eigner des eigentlichen Metadatenservice ist Nova. Im Folgenden sehen Sie eine Abbildung zur Funktionsweise des Service:

      (Abbildung von Matt Dorn)

      Wenn Sie wirklich wissen möchten, wie kompliziert es werden kann, sehen Sie sich die Informationen unter https://www.suse.com/communities/blog/vms-get-access-metadata-neutron/ an. Probleme können überall im Datenfluss auftreten. Meistens wird ein Fehler wie der folgende in nova console-log aufgezeichnet:

      2016-11-30 17:09:33,599 - url_helper.py[WARNING]: Calling 'http://169.254.169.254/2009-04-04/meta-data/instance-id'
failed [50/120s]: request error [HTTPConnectionPool(host='169.254.169.254', port=80): Read timed out.
(read timeout=50.0)]

      **oder **

      checking http://169.254.169.254/2009-04-04/instance-id
failed 1/20: up 1.13. request failed
...
failed 20/20: up 229.58. request failed
failed to read iid from metadata. tried 20

      Wenn Sie sehen, dass versucht wird, eine Verbindung zu einer IP-Adresse wie 92.168.0.2 und nicht zu 169.254.169.254 herzustellen, bedeutet dies, dass versucht wird, Metadaten vom DHCP-Agenten abzurufen, wenn das zulässige Zeitlimit für 169.254.169.254 überschritten wird.

      Q. Was kann ich bei einer Zeitlimitüberschreitung tun?

      A. Versuchen Sie, innerhalb der Instanz mit curl http://169.254.169.254/latest/meta-data/ manuell auf den Metadatenservice zuzugreifen. Um den Fehler näher einzugrenzen, versuchen Sie, Ihre Sicherheitsgruppenregeln und Ihre SELinux-Einstellungen vorübergehend zu lockern. Sie können auch ein anderes Image ausprobieren, vorzugsweise eines der von IBM offiziellen bereitgestellten Images.

      Wenn das nicht funktioniert, öffnen Sie ein Support-Ticket und fordern Sie einen Neustart dieser Services auf beiden Controllerknoten (Netz) an.

      • neutron-metadata-agent (auf beiden Netzknoten)
      • neutron-l3-agent (auf beiden Netzknoten)
      • neutron-dhcp-agent (auf beiden Netzknoten)
      • neutron-linuxbridge-agent (auf dem Rechenknoten, auf dem sich die Instanz befindet)

      Nachdem diese Agenten erneut gestartet wurde, muss der Techniker nach blockierten Neutron-Ports suchen und die Blockierungen ggf. beseitigen. Meistens wird das Problem durch einen Neustart von neutron-metadata-agent mit anschließendem Neustart der Instanz behoben.

      F. Gibt es eine einfachere und zuverlässigere Alternative?

      A. Ja! Die Alternative ist ConfigDrive. Wenn Sie ConfigDrive verwenden, fällt die Komplexität mit dem Einsatz des Neutron-Metadatenagenten weg. ConfigDrive bringt Ihrer Instanz Ihre Daten näher, indem sie auf einer virtuellen CD-ROM/ISO gespeichert werden, die während der Ausführung von cloud_init angehängt und wieder abgehängt wird. Beispiel:

      nova boot --config-drive true --image my-image-name --flavor my-flavor myinstance

      Weitere Informationen finden Sie unter http://docs.openstack.org/user-guide/cli-config-drive.html.

      F. Welche Nachteile hat die Verwendung von ConfigDrive?

      A. Der Hauptnachteil ist der, dass mit ConfigDrive aufgrund eines Programmfehlers in libvirt beim Kopieren einer schreibgeschützten Platte keine Livemigration zugelassen wird. Es gibt einen Patch unter https://bugs.launchpad.net/nova/+bug/1246201/comments/65, der dieses Problem behebt, aber nur, wenn die blockbasierte Migration verwendet wird. Da die blockbasierte Migration im aktuellen Release noch nicht unterstützt wird, wurde dieser Patch nicht hinzugefügt.

      Beachten Sie auch, dass die Livemigration selbst nur in einem einzigen seltenen Fall von Supportmitarbeitern vorgenommen werden kann und zwar bei der Evakuierung eines Hosts, wenn eine geplante Wartung durchgeführt wird, die die Kundenworkload ansonsten beeinträchtigen würde. Selbst in diesem einen Anwendungsfall wird sie nur für Instanzen unterstützt, die über den Datenträger gebootet werden.

      F. Was ist der Unterschied zwischen Metadaten und Benutzerdaten?

      A. Instanzdaten werden hauptsächlich von Nova-Instanzen verwendet. Gewöhnlich gehören dazu Instanzhostnamen, SSH-Schlüssel usw.

      Benutzerdaten werden hauptsächlich von Heat verwendet. Gewöhnlich gehören dazu die Rootkennwörter von Instanzen und Startscripts.

      F. Ich möchte Benutzerdaten mit Heat verwenden, aber meine Daten überschreiten das 16-KB-Limit. Was soll ich tun?

      A. Verwenden Sie eine Include-Anweisung in Ihren Benutzerdaten, um das Script abzurufen und die Größenbeschränkung zu umgehen:

      #include
http://example.com/yourscript.txt

      F. Welches sind die 4 Typen von Metadaten und wie kann ich Sie verwenden?

      Die vier Typen sind meta-data, user-data, vendor-data und network-data. Weitere Informationen zu diesen Typen und zu deren Verwendung finden Sie unter http://www.madorn.com/openstack-metadata-types.html#.WEHS63eZMl4.
