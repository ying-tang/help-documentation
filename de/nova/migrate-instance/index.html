      Instanz in eine neue Cloud migrieren

      • by Jason Kennedy ● 5. Oktober 2015

      • Tags:

        • nova

        • Nova

        • migrieren

        • Migration in eine neue Cloud

      Gelegentlich ist es aufgrund von Upgrades, Rechenzentrumsumzügen oder anderen unvorhergesehenen Ereignissen erforderlich, Ihre Instanzen aus einer OpenStack-Cloud in eine andere zu migrieren.

      In diesem Beispiel erstellen Sie eine Momentaufnahme einer Instanz (einer einfachen, auf der nur nginx ausgeführt wird), laden diese auf Ihre lokale Maschine herunter und laden sie dann wieder in die neue Cloud hoch. Lassen Sie uns zuerst einen Blick auf Ihre Instanz werfen:

      Sie sehen, dass nginx innerhalb der Instanz ausgeführt wird. Sehen Sie sich nun die Instanz selbst an. (Anmerkung: Die Instanz hat den Namen snap_test und so kann grep speziell für die Instanz ausgeführt werden):

      Nun können Sie die externe IP-Adresse der Instanz sehen. Rufen Sie diese auf, um sicherzustellen, dass nginx betriebsbereit ist:

      Nachdem Sie nun wissen, dass die Instanz betriebsbereit ist, können Sie sie in die neue Cloud verschieben.

      Aber zuerst noch eine Anmerkung zum Bash-Befehl ‘source’. Sie müssen die Berechtigungsnachweise Ihrer vorhandenen Cloud mit dem Bash-Befehl ‘source’ in den aktuellen Kontext übernommen haben, um die für die Momentaufnahmeerstellung und den Umzug erforderlichen Cloudbefehle ausführen zu können. In diesem Fall wurde python-openstackclient auf einem Mac installiert und stackrc über die Option Download OpenStack RC File auf der rechten Seite aus der Cloud heruntergeladen, wie in der folgenden Abbildung gezeigt wird:

      Stellen Sie sicher, dass Sie den Bash-Befehl ‘source’ mit Ihrer stackrc-Datei als Argument ausführen:

      Nun erstellen Sie eine Momentaufnahme Ihrer Instanz. Sie kennen die Instanz-ID aus dem vorherigen Nova-Befehl. Gehen Sie also wie folgt vor:

      Sie können den Prozess auch in Horizon verfolgen. Dort sollten Sie sehen, dass zuerst eine Momentaufnahme Ihrer Instanz erstellt und die Instanz dann hochgeladen wird. Anschließend sollten Sie Ihre Momentaufnahme in der Ausgabe des nova-Parameters image-list sehen:

      Wenn diese Aktion abgeschlossen ist (was je nach Größe des Image eine Weile dauern kann) und wenn der Status ACTIVE für das Image angezeigt wird, können Sie das Image auf Ihre lokale Maschine herunterladen:

      Nach Abschluss des Downloads ist das Image lokal gespeichert. Laden Sie das Image sofort in die neue Cloud hoch. Zuerst müssen Sie den Bash-Befehl ‘source’ mit der Datei stackrc aus dieser Cloud als Argument ausführen, um sicherzustellen, dass Sie mit der richtigen Cloud interagieren. Laden Sie anschließend die erstellte Momentaufnahme hoch:

      Ein Hinweis zum Befehl image-create bei der Verwendung von Glance. Im vorherigen Befehl wurde Folgendes ausgeführt:

      --container-format: Containerformat des Image. Die zulässigen Formate sind ami, ari, aki, bare und ovf.

      --disk-format: Plattenformat des Image. Die zulässigen Formate sind ami, ari, aki, vhd, vmdk, raw, qcow2, vdi und iso. (Beachten Sie, dass OpenStack intern qcow2 für alle Images verwendet.)

      --name Name Ihres Image.

      --file Lokale Datei, die ein Plattenimage enthält, das während der Erstellung hochgeladen werden soll. Alternativ können Images über stdin an den Client übergeben werden.

      Booten Sie anschließend eine aus der Momentaufnahme erstellte Instanz:

      Im vorherigen Befehl (nova boot --image snap_migration --nic net-id=ba0fdd03-72b5-41eb-bb67-fef437fd6cb4 --security-groups jksec --flavor m1.tiny snap_migration_test) wurde die --nic net-id verwendet, die durch Ausführung von nova net-list und Extrahieren der ID des internen Netzes abgerufen wurde. Die --security-groups-Variable jksec stammt aus der Ausführung des Befehls nova secgroup-list. Beide Befehle wurden über die API ausgeführt. Beide Befehle sind auch in der Horizon-Anzeige verfügbar.

      Nach dem Start der Instanz müssen Sie ihr eine externe IP-Adresse zuweisen, damit nginx extern erreichbar ist. Stellen Sie sicher, dass Sie Port 80 Zugang zur neuen Cloudsicherheitsgruppe gewähren, damit die Instanz auf diese Weise erreichbar ist.

      Testen Sie die migrierte Instanz abschließend, um sicherzustellen, dass nginx betriebsbereit ist und keine Probleme auftreten (denken Sie daran, dass Sie sich noch nicht an der Instanz angemeldet haben, bisher war es eine reine Drag-and-drop-Prozedur):
