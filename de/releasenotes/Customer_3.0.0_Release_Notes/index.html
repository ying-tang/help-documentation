      IBM Bluemix Private Cloud Customer

      Releaseinformationen 3.0.0

      30. Juni 2016

      Dieses Dokument enthält eine grundlegende technische Beschreibung der neuesten IBM Bluemix Private Cloud-Angebote, einschließlich der technischen Daten für die Angebote Dedicated und Local. Links zu ausführlichen Dokumentationen finden Sie in den einzelnen Abschnitten.

      Zusammenfassung des aktuellen Release

      In Release 3.0.0 von IBM Bluemix Private Cloud wurden zahlreiche Aktualisierungen an der zugrundeliegenden OpenStack-Plattform vorgenommen, auf der jede Bluemix Private Cloud basiert. Außerdem wurden Infrastruktur- und Feature-Upgrades in Bluemix Private Cloud und Bluemix Private Cloud Local vorgenommen. Herauszuheben ist hierbei der Umzug auf das Release OpenStack Mitaka, das viele neue Funktionen enthält. Es wurden verschiedene allgemeine Produktverbesserungen eingeführt, wie z. B. Leistungsbenchmarks und eine aktualisierte Rally-Testsuite, die einen reibungsloseren Betrieb und eine einfachere Bereitstellung gewährleisten sollen. Darüber hinaus wurden in diesem Release auch zahlreiche Verbesserungen an der Dokumentation vorgenommen.

      IBM Bluemix Private Cloud Version 3.0.0 enthält die folgenden neuen Features:

      1. Unterstützung von Openstack Mitaka
      2. K2K-föderierte Identität in mehreren Bluemix Private Cloud-Instanzen
      3. Aktualisierte Dokumentationswebsite, die jetzt über Box Panel zugänglich ist
      4. Regelmäßig vorgenommene Aktualisierungen für IBM Bluemix Private Cloud-Images

      Features von IBM Bluemix Private Cloud 3.0.0 (in IBM Cloud-Rechenzentren):

      1. Unterstützung für VLAN-Spanning in IBM Cloud-Rechenzentren und Konnektivität zu Bare-Metal-Kundenhardware
      2. Vyatta mit 10-GB-Netzbetrieb

      Features von IBM Bluemix Private Cloud Local 3.0.0:

      1. Hybridblockspeicherknoten
      2. Neue Rechenknotentypen
      3. IBM Bluemix Private Cloud Local mit HIPAA-Unterstützung
      4. Unterstützung für IBM Bluemix Private Cloud Local auf der Bluemix Local System-Plattform (Betaversion)

      Bekannte Einschränkungen dieses Release

      Weitere Benutzerdokumentationen finden Sie auf der Hauptseite der Kundenhilfe, die jederzeit online unter help.blueboxcloud.com verfügbar ist.

      Es ist eine allgemeine Produktübersicht für IBM Bluemix Private Cloud Version 2.0 und höher verfügbar.

      IBM Bluemix Private Cloud 3.0.0 enthält die folgenden neuen Features:

      Unterstützung für Openstack Mitaka: Basierend auf dem Umzug auf OpenStack Mitaka sind mehrere neue Features verfügbar.

      • Es ist jetzt möglich, Gruppen für automatische Skalierung und LBaaS (v2) mit Heat-Vorlagen und dem Horizon-Dashboard einzurichten.
      • Das Mitaka-Release führt den neuen Alert-Service aodh ein, der in jeder Umgebung installiert wird, in der Ceilometer aktiviert ist.
      • Weitere Informationen finden Sie in den Releaseinformationen zu OpenStack Mitaka.

      Spezielle OpenStack-Services, die für jede Instanz der IBM Bluemix Private Cloud Version 3.0.0 im Mitaka-Release aktiviert werden

      • Block Storage API & Extensions (Cinder v2.0)
      • Compute API & Extensions (Nova v2.1)
      • Dashboard (Horizon ) API N/A
      • Identity API & Extensions (Keystone v3.0)
      • Image Service API (Glance v2.2)
      • Networking API & Extensions (Neutron v2.0, Neutron LBaaS v2)
      • Object Storage API & Extensions (Swift v1.0)
      • Orchestration API (Heat v1.0)
      • Telemetry API (Ceilometer v2.0) (plus aodh)

      K2K-Föderation in mehreren IBM Bluemix Private Cloud-Instanzen: Die K2K-Föderation (Keystone-to-Keystone) ermöglicht es Ihnen, sich mit Ihren Benutzerberechtigungsnachweisen, die in einem Keystone-Identitätsprovider gespeichert sind, bei mehreren IBM Bluemix Private-Cloud-Instanzen anzumelden. Zur K2K-Konfiguration wurde ein spezieller Artikel bereitgestellt.

      Aktualisierte Dokumentationswebsite, die jetzt über Box Panel zugänglich ist: Die Website mit der Kundendokumentation wurde aktualisiert und neu gestaltet. Die Website ist jetzt unter help.blueboxcloud.com verfügbar und ohne großen Aufwand über die Box Panel-Schnittstelle zugänglich. Weitere Informationen finden Sie im Box Panel-Benutzerhandbuch.

      Regelmäßig vorgenommene Aktualisierungen für IBM Bluemix Private Cloud-Images: Die regelmäßig vorgenommenen Aktualisierungen für die unterstützten IBM Bluemix Private Cloud-Images werden über die Box Panel-Schnittstelle bereitgestellt. Die Kundenreleaseinformationen zu den einzelnen Imageaktualisierungen wurden hier veröffentlicht. Außerdem wurde ein Benutzerhandbuch für die Arbeit mit Cloud-Images geschrieben.

      IBM Bluemix Private Cloud 3.0.0 (in IBM Cloud-Rechenzentren):

      Unterstützung für VLAN-Spanning in IBM Cloud-Rechenzentren und Verbindungen zu Bare-Metal-Hardware, die von Kunden verwaltet wird: Dieses Feature ist für alle IBM Bluemix Private Cloud-Kunden bestimmt, für die die Leistung von Bare-Metal-Servern eine strikte Voraussetzung ist, die aber auch Zugriff auf sichere Übertragungen über das private Netz der IBM Cloud-Rechenzentren zwischen ihrer Bare-Metal-Infrastruktur und einem OpenStack-Netz mit Cloudinstanzen benötigen. Anmerkung: Zur Verwendung dieses Features muss sich die Bare-Metal-Hardware in einem separaten, kundeneigenen und kundenverwalteten SoftLayer-Konto befinden.

      Vyatta mit 10-GB-Netzberieb: Dieses Feature ist für Kunden bestimmt, die hohe Bandbreitenanforderungen für ausgehenden und eingehenden Clouddatenverkehr benötigen.

      IBM Bluemix Private Cloud Local 3.0.0:

      Neue Rechenknotentypen: Erweiterungsrechenknoten erfüllen Ihre Anforderungen in Bezug auf schnelleren ephemeren E/A-Speicher, höhere Arbeitsspeicherkapazität und um Größenordnungen mehr ephemeren Speicher. Erweiterungsknoten bieten eine bessere Abdeckung der verschiedenen Instanztypen und Workloads. Ab diese Release werden Rechentypen zur einfacheren Zuordnung von Workloads zum entsprechenden Rechentyp in Verfügbarkeitszonen gruppiert. Sie können die folgenden Rechenknotenerweiterungen zu Ihrer IBM Bluemix Private Cloud Local-Instanz hinzufügen:

      • Sie können spezielle Erweiterungsrechenknoten auswählen: IO oder HIGH STORAGE (neu in diesem Release)
      • Sie können weitere Standard- oder Unternehmensrechenknoten auswählen: STANDARD COMPUTE oder ENTERPRISE COMPUTE (bereits vorher verfügbar)
      • Sie können POWER-Rechenknoten (Habañero) auswählen: Diese Knoten unterstützen insbesondere IBM Watson-Anwendungen und sonstige.

      Hybridblockspeicherknoten: Hybridblockspeicherknoten bieten Blockspeicher zu geringeren Kosten und höherer Dichte. Es wurde ein Hybrid-Ceph-Speicherknotentyp hinzugefügt, der Blockspeicher zu geringeren Kosten und mit höherer Dichte bietet. Bei identischem Software-Stack und identischer Referenzarchitektur bietet er im Vergleich mit SSD für denselben Preis das Vierfache an Kapazität. Beachten Sie, dass nur ein einziger Typ von Blockspeicher pro Cluster verwendet werden kann: SSD-Blockspeicher und Hybridblockspeicher können in demselben Cluster nicht gemischt verwendet werden.

      • Sie können Blockspeicher auswählen: SSD oder HYBRID

      Cloud mit HIPAA-Unterstützung: Eine Cloud, in der HIPAA aktiviert ist, hat dieselben Konfigurations- und Erweiterungsoptionen wie jedes andere IBM Bluemix Private Cloud-Angebot, aber wir aktivieren über 50 weitere Steuerelemente für die Informationssicherheit, die unsere Kunden unterstützen, die persönliche Gesundheitsdaten speichern müssen. Aufgrund dieser erhöhten Informationssicherheit muss eine Cloud bereits während der Erstellung für die HIPAA-Unterstützung aktiviert werden. Es ist nicht möglich, ein Upgrade einer Cloud auf das HIPAA-Protokoll durchzuführen. Das Downgrade einer Bluemix Private Cloud Local-Instanz HIPAA-Unterstützung ist ebenfalls nicht möglich. Die vorhandene HIPAA-Cloud muss gemäß HIPAA-Protokoll bereinigt werden, und Sie müssen eine neue Local-Cloud mit Controllern ohne HIPAA-Unterstützung bestellen.

      • Sie können die HIPAA-Aktivierung jetzt für IBM Bluemix Private Cloud Local auswählen.

      Unterstützung für Bluemix Private Cloud on Bluemix Local System (Beta-Release): Bluemix Local ist ein vorhandenes IBM System für das Hardware-Management (und wurde früher unter dem Namen PureApp vertrieben). Mit dem Beta-Release wird dieses System zusätzlich zur bereits verfügbaren Cisco-Hardware als Plattform für IBM Bluemix Private Cloud Local bereitgestellt.

      Bekannte Einschränkungen dieses Release

      Horizon-Dashboard und LBaaS v2:

      • Im LBaaS v2-Horizon-Dashboard können Sie keine HTTPS-Listener erstellen.
      • Die gemeinsame Poolnutzung ist über die Horizon-Schnittstelle nicht möglich.
      • Die Gewichtung eines Members kann über Horizon nicht auf 0 gesetzt werden.
      • L7-Content-Switching ist beim Standardtreiber nicht verfügbar. (HaproxyOnHostPluginDriver)
      • TLS-Beendigung wird nicht unterstützt.

      Offene Probleme in der OpenStack-Community bezüglich LBaaS v2:

      • LBaaS VIP ist standardmäßig die Standardsicherheitsgruppe zugeordnet. Eine Ausweichlösung ist, die erforderlichen Ports in der Standardgruppe zu öffnen. https://bugs.launchpad.net/neutron/+bug/1295424
      • Beim Löschen von LBaaSv2 wird der LBaaS-Namespace nicht gelöscht. https://bugs.launchpad.net/neutron/+bug/1495430
      • Der Status der Statusüberwachung wird nicht angezeigt, unabhängig davon, ob sie erfolgreich war oder fehlgeschlagen ist. https://bugs.launchpad.net/neutron/+bug/1464229
      • Der PING-Monitor sendet HTTP-Anforderungen. Wenn die LBaaS v2-Statusüberwachung für Ping konfiguriert ist, werden HTTP-Anforderungen gesendet. https://bugs.launchpad.net/neutron/+bug/1463816
      • Die LBaaSv2-Statusüberwachung kann nicht gelöscht werden, wenn der zugehörige Listener gelöscht wurde. Eine Ausweichlösung ist, die verwaiste Statusüberwachung über die Befehlszeile zu löschen. https://bugs.launchpad.net/neutron/+bug/1571097
      • lbaas-pool kann gelöscht werden, wenn lbaas-hm noch zugeordnet ist. In diesem Fall kann selbst der Administrator die verwaiste Statusüberwachung nur in der Datenbank direkt löschen. https://bugs.launchpad.net/neutron/+bug/1526964

      Horizon-Dashboard und Projekte:

      • Im Horizon-Dashboard wird die Seite zum Ändern von Kontingenten dem cloud_admin im Modalfenster zur Projektbearbeitung nicht angezeigt. cloud_admin kann die Kontingente jedoch über die OpenStack-CLI oder -APIs ändern.
      • Wenn ein Benutzer mit der Rolle cloud_admin in der Anzeige Identify im Horizon-Dashboard ein neues Projekt erstellt, wird ein Fehlernachrichtenfenster angezeigt (Unable to set project quotas). Das Projekt wird jedoch mit dem Standardkontingent erstellt. Ein Benutzer mit der Rolle cloud_admin kann das Kontingent anschließend über die OpenStack-CLI oder -API aktualisieren.
      • Wenn ein Benutzer mit der Rolle cloud_admin in der Anzeige Identify im Horizon-Dashboard ein vorhandenes Projekt ändert, wird ein Fehlernachrichtenfenster angezeigt (Unable to modify project). Das Projekt wird jedoch aktualisiert. Das Fehlernachrichtenfenster wird angezeigt, weil die Projektkontingente nicht definiert werden konnten.

      Horizon-Dashboard und Administratoranzeige:

      • Wenn ein Benutzer mit der Rolle cloud_admin in der Administratoranzeige des Horizon-Dashboards auf Hypervisors klickt, wird ein Fehlernachrichtenfenster angezeigt: “Unable to get Nova services list.” Dieses Verhalten wird erwartet, weil ein Benutzer mit der Rolle cloud_admin keine Operationen für diese Elemente ausführen darf.

      Weitere bekannte Einschränkungen:

      • Neutron-Befehl RBAC-create: Der CLI-Befehl neutron rbac-create gibt den Fehlerstatuscode “500” zurück, sofern --target-tenant nicht bei der Ersterstellung angegeben wird.
      • Beim Tests in einer Leistungstestumgebung mit 30 Knoten und Hybrid-Ceph-Konfiguration (HDD + SSD bcache) wurde eine Leistungseinschränkung beim Blockspeicher beobachtet.
      • K2K-Föderation (Keystone-to-Keystone): Derzeit tritt ein Problem bei der Erstellung der Trust-for-Trust-Delegierung (Vertrauen gegen Vertrauen) durch einen föderierten Benutzer auf. Dieses Problem kann umgangen werden, indem dem föderierten Benutzer explizit die Rolle heat_stack_owner zugewiesen wird.
