      Q. Wie kann ich feststellen, wann meine virtuelle Maschine zuletzt neu gestartet wurde?

      A. Sie können den folgenden Befehl verwenden:

      zgrep BOOT_IMAGE /var/log/kern.log*

      Dieser Befehl zeigt an, wann der Host (VM) zuletzt gebootet wurde. Anhand dieser Zeitmarke können Sie dann in den Protokollen nach Hinweisen auf Probleme suchen, die Sie derzeit beheben.

      F. Was soll ich tun, wenn meine Instanz blockiert und nicht mehr reagiert?

      A. Wir empfehlen Ihnen, in der Konsole der virtuellen Maschine nachzusehen, ob sie vollständig blockiert ist oder ob es Anzeichen für eine Kernelpanik oder einen anderen Fehler gibt. Sie können die URL der Konsole wie folgt abrufen:

      nova get-vnc-console VM-UUID novnc

      Notieren Sie die Konsolenzeit, sofern vorhanden.

      Erstellen Sie einen Screenshot der Konsole, wenn sie blockiert ist und hilfreiche Informationen enthält. Wenn Sie nach einem Warmstart Absturzspeicherauszüge auf der virtuellen Maschine aktiviert haben, können Sie im Verzeichnis /var/crash nach vmcore- (Red Hat) oder .crash-Dateien (Ubuntu) suchen und den Speicherauszug analysieren. Sehen Sie sich auch die SAR-Statistiken (Systemaktivitäten) um die Zeit der Blockierung herum an.

      Starten Sie die Maschine erneut, wenn diese so schnell wie möglich wieder aktiv sein muss, oder besser starten Sie eine neue Maschine, die auf demselben Image basiert. Wenn weitere Untersuchungen der Infrastruktur erforderlich sind, teilen Sie uns, sofern möglich, die Uhrzeit der Blockierung in UTC vor dem Warmstart und die UUID der Instanz mit, und wir prüfen, ob Informationen in den Protokollen zu finden sind.

      Wenn Sie zuverlässig in den Einzelbenutzermodus wechseln müssen, z. B., um eine Dateisystemprüfung durchzuführen oder um SSH-Verzeichnisberechtigungen zu korrigieren, könnten wir die Konsole im Safari-Browser verwenden. Sie müssen die Tastenkombination “send Strg/Alt/Entf” drücken, um die VM erneut zu starten, während Sie in der Konsole angemeldet sind, sodass Sie sie in der GRUB-Ladeanzeige abfangen können. Wenn Sie in das GRUB-Bootmenü wechseln müssen, geben Sie ein e ein (für Bearbeiten) ein. Gehen Sie für den Einzelbenutzermodus in CentOS beispielsweise folgendermaßen vor:

      1. Blättern Sie nach unten und löschen Sie den größten Teil am Ende der Zeile (ab console), sodass nur noch console=tty übrig bleibt.
      2. Nach der Bearbeitung sollte die Zeile ähnlich wie die folgende aussehen: kernel /boot/vmlinuz-2.6.32-431.el6.x86_64 ro root=UUID=8a057bba-2787-46a7-9436-fd6dc8a3aa85 rd_NO_LUKS rd_NO_LVM LANG=en_US.UTF-8 rd_NO_MD crashkernel=auto rd_NO_DM SYSFONT=latarcyrheb-sun16 KEYBOARDTYPE=pc KEYTABLE=us console=tty notsc single
      3. Sie können auch versuchen, rw init=/bin/bash anstelle von ro in der Zeile hinzuzufügen.
      4. Wenn es den Anschein hat, dass der Bootprozess gestoppt ist, drücken Sie mehrmals die Eingabetaste.

      F. Was ist, wenn ich Wiederherstellungsoperationen für meine Instanz ausführen muss?

      A.

      1. Beschaffen Sie sich ein CD-ISO-Image für die Wiederherstellung, z. B. von http://www.system-rescue-cd.org: 
        wget https://dronedata.dl.sourceforge.net/project/systemrescuecd/sysresccd-x86/4.9.6/systemrescuecd-x86-4.9.6.iso
      2. Installieren Sie isohybrid aus dem sysutils-Paket: 
        apt-get install syslinux [oder yum install syslinux]
      3. Fügen Sie dem ISO-Abbild den Master-Bootsatz hinzu, um aus dem Abbild eine bootfähige Festplatte zu machen: 
        isohybrid systemrescuecd-x86-4.9.6.iso
      4. Upload the new hybrid rescue CD to Glance: 
        openstack image create --file systemrescuecd-x86-4.9.6.iso --public sysrescuecd-4.9.6.iso
      5. Versetzen Sie die wiederherzustellende Instanz in den Wiederherstellungsmodus: 
        nova rescue brokeninstance --image sysrescuecd-4.9.6.iso
      6. Melden Sie sich für die Ausführung der erforderlichen Wiederherstellungsoperation über die VNC-Konsole in Horizon bei der Instanz an. Möglicherweise müssen Sie 1-2 Minuten warten, bis die Wiederherstellungs-CD in der Konsole sichtbar wird. Laden Sie die Konsolenanzeige nach Ablauf der Wartezeit erneut.

      (/img/de/systemrescuefromvnc.png)

      Ab hier können Sie Tools wie die folgenden verwenden:

      blkid
fsck /dev/vdb1
mount -t ntfs /dev/xxx /mnt/windows -o ro
ntfs-3g /dev/sda1 /mnt/windows
      1. Wenn Sie mit der Wiederherstellung der Instanz zufrieden sind, beenden Sie den Wiederherstellungsmodus und starten Sie dann die Instanz von der normalen Bootplatte mit dem folgenden Befehl erneut: 
        nova unrescue brokeninstance

      F. Warum kann ich die Größe meiner Instanz nicht von m1.tiny in m1.medium ändern?

      Es ist möglich, dass die Instanz zu einer Zeit erstellt wurde, als die m1.tiny-Variante eine größere ephemere Platte hatte. Dies wird vom Größenänderungsprozess geprüft und die Größenänderung schlägt fehl, wenn dieser Wert nicht gefunden wird.

      Möglicherweise müssen Sie ein Support-Ticket öffnen, um mithilfe des Support zu verifizieren, ob es sich hierbei um die Fehlerursache handelt. Es handelt sich um diesen Fehler, wenn das Support-Team einen Fehler ähnlich dem Folgenden in den Protokollen findet:

      Exception during message handling: Resize error: Unable to resize disk down.

      Alternative Optionen für die weitere Vorgehensweise ohne Öffnen eines Tickets sind das Setzen der m1.medium-Variante auf dieselbe größere ephemere Plattengröße wie zuvor und die im folgenden Artikel beschriebene Ausweichlösung für die Erstellung einer neuen Instanz aus einer Momentaufnahme: /help-documentation/gettingstarted/commontech/Instance_Resize/

      F. Worauf sind blockierte Ports in Neutron zurückzuführen?

      Portblockierungen können in Neutron bei der Instanziierung von VMs mit Heat auftreten. Es wird vermutet, dass dieses Problem mit einem Fehler in OpenStack Neutron (Mitaka-Release) zusammenhängt, der auf eine Konkurrenzbedingung in Neutron zurückzuführen ist. Deshalb wird dieses Problem als bekannte Einschränkung behandelt. Diese Bedingung kann auch zu Metadatenfehlern mit Portblockierungen führen.

      Es gibt mehrere Ausweichlösungen:

      Ausweichlösung 1. Sofern möglich, können Instanzen ohne Heat bereitgestellt werden. Durch eine manuelle Bereitstellung scheint diese Bedingung umgangen werden zu können.

      Ausweichlösung 2. Eine andere Ausweichlösung ist die Einführung einer Verzögerung in Heat unter Verwendung von WaitCondition aus der API. Die Verwendung von WaitCondition ist in diesem Szenario jedoch unter Umständen nicht ideal.

      • WaitCondition arbeitet mit Kombination mit WaitConditionHandler. Dieser Handler erstellt eine URL (bzw. einen URI), die auf eine bestimmte Textzeichenfolge hin überwacht wird. Das Vorhandensein dieser Zeichenfolge bedeutet den Erfolg einer bereitgestellten VM und das Fehlen dieser Zeichenfolge einen Fehler. Nach der Instanziierung einer VM kann diese angewiesen werden, die URL (bzw. den URI) mit einem PUT-API-Aufruf mit der entsprechenden Textzeichenfolge zu füllen.

      • Derzeit gibt es keinen Mechanismus in unserer OpenStack-Bereitstellung, der dafür sorgt, dass ein L3-Router diesen API-Aufruf absetzt. Unser Entwicklungsteam schlägt die Verwendung von ‘depends_on’ vor. (Sehen Sie sich dazu die folgende Option 3 an.)

      Ausweichlösung 3. Die beste Ausweichlösung ist die Verwendung von depends_on. Sie sorgt für ein solches Szenario in der Orchestrierung, dass die Bereitstellung einer VM von der Existenz anderer definierbarer Ressourcen abhängig ist (depends on), z. B. eines L3-Routers und eines L3-Netzes.

      Im Folgenden sehen Sie ein kurzes Beispiel für depends_on:

      add_router_interface:
   type: OS::Neutron::RouterInterface
   properties:
     router: { get_resource: router}
     subnet: { get_resource: subnet}


server1:
    type: OS::Nova::Server
    depends on: add_router_interface
    properties:
      image: { get_param: image }

      Weitere Details zu expliziten Abhängigkeiten finden Sie unter dem folgenden Link:

      • https://wiki.openstack.org/wiki/Heat/Blueprints/hot-software-config-spec

      F. Welche Ports müssen für OpenStack offen sein?

      Sie müssen mindestens die Ports 67/68 für DHCP und den Port 80 für Metadaten öffnen. Wenn Sie möchten, dass VMs auf OpenStack-API-Services zugreifen können, schließen Sie außerdem die hier dokumentierten Ports hinzu:

      /help-documentation/gettingstarted/commontech/FAQ/#which-portsurls-etc-are-publicly-accessible-on-my-bluemix-private-cloud-and-for-what-purpose

      Wenn nicht die mindestens erforderlichen Ports geöffnet sind, können Probleme mit cloud_init auftreten, sodass Ihre Instanzen unter Umständen nicht erfolgreich gestartet werden.

      F. Was deutet der folgende Fehler? Unable to retrieve volume limit information.”

      Häufig erscheint beim Laden einer Seite im Horizon-Dashboard ein rotes Popup-Fenster, in dem “Error: Unable to retrieve volume limit information.” angezeigt wird. Was bedeutet das?

      Dieser Fehler ist harmlos, tritt aber auf, wenn kein Cinder-Blockspeicher in der Bereitstellung enthalten ist.

      Dieser Fehler kann behoben werden, indem der Cinder-Endpunkt aus dem Keystone-Service entfernt wird. Dazu müssen Sie möglicherweise Ihre Automationskonfigurationsdateien aktualisieren, die dann vom Support-Team geprüft werden müssen. Öffnen Sie ein Support-Ticket, wenn Sie Hilfe bei der Behebung dieses Fehlers benötigen.

      Im Folgenden sehen Sie ein Beispiel für die Schritte, die ein Support-Techniker ausführen muss:

      # openstack endpoint list | grep cinder 
| 19a90853883049ed92cf4af82a24c0db | RegionOne | cinderv2     | volumev2       | True    | name  | https://... | 
| 2c3e8d0a15304788a9dc1806917b61ee | RegionOne | cinderv2     | volumev2       | True    | other     | https://... | 
| 39434faf90484a208b61a517926042b2 | RegionOne | cinderv2     | volumev2       | True    | name2    | https://... | 

# openstack endpoint delete 19a90853883049ed92cf4af82a24c0db 
# openstack endpoint delete 2c3e8d0a15304788a9dc1806917b61ee 
# openstack endpoint delete 39434faf90484a208b61a517926042b2 

# openstack service list | grep cinder 
| ef7a982aaf224c62a09c6e813ac66141 | cinderv2   | volumev2       | 

# openstack service delete ef7a982aaf224c62a09c6e813ac66141

      Prüfen, ob die Automation korrekt ist:

      $ grep -A2 ^cinder ../all.yml 
cinder:  
  enabled: False 
cinderv2:  
  enabled: False

      F. Warum funktioniert der Zugriff auf 25 für abgehende E-Mails nicht?

      A. SoftLayer lässt in neuen Konten keine abgehenden Verbindungen über TCP-Port 25 (SMTP) mehr zu. Weitere Informationen finden Sie im folgenden Artikel:

      https://knowledgelayer.softlayer.com/procedure/outbound-email-port-25
