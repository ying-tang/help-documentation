      OpenStack-Netzkonzepte

      • by Ying Tang ● 25. Mai 2016

      • Tags:

        • Openstack

        • OpenStack

        • Konzepte

      Dieses Dokument enthält eine grundlegende Einführung in einige gängige Netzbetriebskonzepte. Diese Konzepte begegnen Ihnen bei der Arbeit mit IBM Bluemix Private Cloud.

      In diesem Dokument verwendete Terminologie

      Konzept Beschreibung
      L2 Layer 2 (Ebene 2), der ISO-7-Ebenen-Internetarchitektur, in der Pakete weitergeleitet werden.
      VLAN Virtuelles LAN, im Grunde ein Teilnetz.
      IP Internetprotokoll, verwendet als Abkürzung für IP-Adresse, gibt das Ziel eines Pakets im Netz an.
      IPv4 Internetprotokoll, Version 4, Internetadresse im Format xxx.xxx.xxx, standardmäßig mit der Erweiterung /xx am Ende, die das Teilnetz mit dem Protokoll CIDR bezeichnet.
      CIDR Classless Inter-Domain Routing bzw. Supernetting ist eine Methode, die Nutzungsdauer von IPv4-Adressen zu verlängern. Diese kann in einem globalen Pool gering sein und muss daher normalerweise gekauft werden. Weitere Informationen zu CIDR finden Sie unter http://searchnetworking.techtarget.com/definition/CIDR. Unter dem Link http://www.subnet-calculator.com/cidr.php finden Sie eine CIDR-Teilnetzgrößenberechnungsfunktion, mit der Sie feststellen können, wie viele IP-Adressen Sie aus einem bestimmten Bereich erhalten werden. (Beachten Sie, dass einige dieser IP-Adressen vorab zugeordnet und für Ihre Cloud nicht verfügbar sind.)
      IPv6 Internetprotokoll, Version 6, neueres Protokoll für Internetadressen, stellt viele Netzadressen bereit, kann die Komplexität der Netzbetriebsarchitektur erhöhen.
      NIC Netzschnittstellenkarte, eine Hardwarekomponente, mit der Server mit einem Netz verbunden werden können. Außerdem sind virtuelle Netzschnittstellenkarten (Virtual NICs oder auch vNICs), die im Wesentlichen Software sind, die sich wie eine Netzschnittstellenkarte verhalten, bei der Cloudnutzung gängig. Der Begriff “Netzschnittstelle” bezieht sich gewöhnlich auf eine virtuelle Netzschnittstelle, sofern im Kontext nicht ausdrücklich angegeben ist, dass es sich um Hardware handelt.
      MAC Eine individuelle Kennung, ähnlich einer Hardwareseriennummer, die jeden Host in einem Netz eindeutig identifiziert.

      Grundlegende Netzbetriebskonzepte

      Konzept Beschreibung
      Netz Ein isoliertes Layer-2-Segment (L2), ähnlich einem VLAN im regulären Netzbetrieb. Ein Netz kann ein oder mehrere Teilnetze enthalten.
      Teilnetz Ein Bereich von IPv4-Adressen.
      Port Ein Verbindungspunkt zum Anschluss eines einzelnen Gerätes an ein virtuelles Netz, wie zum Beispiel die Netzschnittstellenkarte eines virtuellen Servers. Der Begriff “Port” beschreibt außerdem die zugehörige Netzkonfiguration, wie zum Beispiel die MAC- und IP-Adressen, die an diesem Port verwendet werden.
      Feste IP-Adresse Die IP-Adresse, die einer Instanz beim Booten zugeordnet wird, wie zum Beispiel eine IP-Netzadresse für eine Instanz, die Zugriff auf das öffentliche Internet erfordert.
      Variable IP-Adresse Die IP-Adresse, die einer Instanz auf Anforderung zugeordnet wird. Sie stammt aus einem Pool und hat häufig das Format IPv6.
      Kontingent Ein Kontingent beschränkt die Anzahl verfügbarer Ressourcen wie Compute- oder Plattenzuordnungen für Ihre Cloud.
      Sicherheitsgruppen Ein für Ihr Projekt spezifischer Satz von Netzfilterregeln, die auf die Netzinteraktionen einer Instanz angewendet werden. Dabei werden die IP-Adressen angegeben, die Pakete an eine andere IP-Adresse oder einen anderen Port senden bzw. nicht senden können.
      Sicherheitsgruppenregel Eine Liste der Protokoll-, Port- und IP-Adressspezifikationen, die festlegen, wie Datenverkehr zu und von einer bestimmten Instanz im Netz fließen kann. Hierbei handelt es sich um die Spezifikationen einer Sicherheitsgruppe.

      Cloudnetztypen

      Im Allgemeinen enthält Ihre Cloud ein oder mehrere interne und externe Netze. Die Instanzen in Ihrer Cloud kommunizieren untereinander über die internen Netze und mit Quellen, die sich außerhalb Ihrer privaten Cloud befinden, über die externen Netze.

      Es gibt viele Begriffe für die Beschreibung interner und externer Netze. Interne Netze können beispielsweise als “Tenantnetze” bezeichnet werden. Externe Netze, über die mit dem öffentlichen Internet kommuniziert wird, werden manchmal als “Providernetze” bezeichnet.

      Interne Netze

      Ein internes Netz kommuniziert innerhalb Ihrer Cloud. Es ermöglicht die Konnektivität zwischen allen virtuellen Netzschnittstellen (NICs), die mit ihm verbunden sind. Jede interne Netzschnittstelle ist einer virtuellen Maschnine (VM) zugeordnet. Einer einzelnen virtuellen Maschine können mehrere Netzschnittstellen zugeordnet sein. Sie kann also mit vielen Netzen kommunizieren.

      Sie können die Kommunikation zwischen mehreren virtuellen Maschinen untereinander ermöglichen, indem Sie sie mit demselben internen Netz verbinden. Interne Netze sind einem einzelnen Projekt (Tenant) zugeordnet und nur für dieses Projekt sichtbar und zugänglich.

      Externe Netze

      Externe Netze können innerhalb und außerhalb Ihrer Cloud “kommunizieren”. Sie können das öffentliche Internet, das externe Netz Ihrer Cloud und das Netz des IBM Cloud-Rechenzentrums einschließen, das den Router des IBM Cloud-Rechenzentrums enthält.

      • Das “öffentliche Netz” bzw. “externe Netz” Ihrer Cloud ermöglicht den Zugriff aus dem öffentlichen Internet und auf das öffentliche Internet.
      • Das private SoftLayer-Netz dient dem Zugriff auf andere Services in SoftLayer, einschließlich anderer Clouds, mit denen Sie kommunizieren möchten. Weitere Informationen zum Zugriff auf das private Netz von SoftLayer finden Sie unter Release Notes.
      • Das SoftLayer-Routernetz stellt eine Verbindung zum SoftLayer Backend Core-Router (BCR) her. Dadurch können Sie auf Ressourcen außerhalb Ihrer privaten Cloud-Umgebung, Ressourcen, die sich in einem Rechenzentrum an Ihrem Standort (Blue Box Local) oder in einer Ihrer anderen Cloud-Umgebungen, in einem SoftLayer-Rechenzentrum oder an einem anderen Standort befinden (Blue Box Dedicated bzw. Ihre eigene traditionelle Cloud).
      • Öffentliche Netze und private SoftLayer-Netze sind physische Netze.

      Verbindung zwischen virtuellen Maschinen in verschiedenen privaten Teilnetzen

      Angenommen Sie verfügen über zwei private Netze. Sie haben virtuelle Maschinen in jedem Teilnetz der Netze erstellt und die erforderlichen Sicherheitsgruppen hinzugefügt, damit eingehender Datenverkehr empfangen werden kann. Wie können Sie Konnektivität zwischen virtuellen Maschinen in verschiedenen privaten Teilnetzen erreichen? Angenommen, Ihre Netze heißen operations und internal. Die Kommunikation zwischen den beiden Netzen kann auf verschiedene Weisen erfolgen:

      • Verbinden Sie das Netz operations mit dem Standardrouter. Diese Verbindung lässt Datenverkehr zwischen dem Standardnetz internal und dem Netz operations zu. Sie können nun eine Instanz im Netz internal implementieren, eine öffentliche variable IP-Adresse zuordnen und über diese Instanz eine Verbindung zur Zielinstanz aufbauen.
      • Eine ähnliche Vorgehensweise ist, einen neuen internen Router zwischen den Netzen internal und operations zu erstellen. Verwenden Sie eine Instanz mit einer öffentlichen IP-Adresse im Netz internal, um den Zugriff auf Ihre virtuelle Maschine im Netz operations zu gewähren. Durch das Erstellen eines internen Router stellen Sie sicher, dass virtuelle Maschinen im Netz operations keine externe Verbindung zum Internet haben können.
      • Eine weitere Möglichkeit ist das Erstellen einer Instanz mit mehreren Netzen. Erstellen Sie eine Instanz, die mit den Netzen internal und operations verbunden ist. Ordnen Sie ihr eine öffentliche variable IP-Adresse zu. Ermöglichen Sie mit dieser virtuellen Maschine den Zugriff auf Instanzen im Netz operations über eine zweite private Schnittstelle.

      Sicherheitsvorkehrungen für Systeme mit öffentlichen variablen IP-Adressen

      Empfehlungen für den Start:

      • Konfigurieren Sie SSHD so, dass ausschließlich Schlüsselauthentifizierung zulässig ist, und inaktivieren Sie den SSH-Rootzugriff.
      • Verwenden Sie keine Standardsystembenutzernamen für SSH (Ubuntu, Deploy u.ä.).
      • Verwenden Sie einen Systemservice wie fail2ban oder denyhosts, um potenzielle Brute-Force-Attacken abzuwehren. Beide Tools sind in den meisten gängigen Linux-Distributionen bereits im Paket enthalten.
      • Verwenden Sie Sicherheitsgruppenregeln, um den SSH-Zugriff auf eine begrenzte Anzahl IP-Adressen zu beschränken.
      • Aktualisieren Sie das System auf Ihrer öffentlichen Instanz regelmäßig.
      • Führen Sie keine überflüssigen Services (wie z. B. HTTP, FTP, SMB usw.) auf Ihrer öffentlichen Instanz aus.
