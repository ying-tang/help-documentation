      Größe von Instanzen ändern

      Feature: Instanzen vergrößern

      Beschreibung: Ab Release 1.3 von Bluemix Private Cloud wird das Vergrößern von Instanzen unterstützt. Diese Operation wird nicht live durchgeführt. Zum Durchführen der Operation wird die Instanz gestoppt und anschließend erneut gestartet. Die Größenänderung kann auf demselben Host durchgeführt werden, was in kleinen Umgebungen hilfreich ist.

      Wenn Sie die Größe einer Instanz ändern, können Sie deren Variante in eine größere Variante ändern. Eine Instanz kann nur in eine größere Variante geändert werden. Bei dieser Änderung kommt es zu einer kurzen Betriebsunterbrechung, während die Instanz gestoppt, auf eine größere Variante migriert und dann erneut gestartet wird. Bei dieser Größenänderung kann die Instanz auch in einen neuen Hypervisor verlagert werden.

      Aktualisieren Sie nicht die Zielvariante (auf die die Instanz migriert wird), während die Größenänderung vorgenommen wird, da die Operation ansonsten fehlschlagen und die Instanz in einen Fehlerstatus versetzt werden kann.

      Das Ändern der Größe großer Instanzen kann schwierig sein, weil es zu Fehlern kommen kann, wenn nicht genügend Plattenspeicherplatz für das neue Plattenimage verfügbar ist.

      Anmerkung: Es kann zu einer kurzen Verzögerung kommen, bis der Netzbetrieb wiederaufgenommen wird, nachdem eine Instanz vergrößert oder migriert wurde und sich im Status VERIFY_RESIZE befindet.

      Weitere Informationen zum Durchführen einer Größenänderung finden Sie unter http://docs.openstack.org/user-guide/enduser/cli_change_the_size_of_your_server.html.

      Fehlerbehebung

      Wenn die Größenänderungsoperation für große Instanzen sehr lange dauert (in einigen Protokollen wurde beispielsweise gesehen, dass eine Operation um 09:30 gestartet wurde und um 14:41 ein Authentifizierungsfehler aufgetreten ist), kann das von Nova für die Durchführung dieser Operation verwendete Authentifizierungstoken während dieses Zeitraums ablaufen.

      Es ist wahrscheinlich, dass beim Einleiten einer neuen Größenänderungsoperation derselbe Fehler erneut auftritt. In diesem Fall wird ein anderer Ansatz empfohlen. Wenn in Ihrem Cluster Blockspeicher aktiviert ist, empfiehlt es sich, wie folgt vorzugehen:

      1. Fahren Sie die Ziel-VM herunter.
      2. Erstellen Sie mit nova image-create eine Momentaufnahme der Ziel-VM.
      3. Erstellen Sie aus dieser Momentaufnahme mit cinder create --image-id <SNAPSHOT-UUID> einen Datenträger.
      4. Löschen Sie die Momentaufnahme mit dem OpenStack-Befehl image delete oder mit glance image-delete.
      5. Booten Sie mit nova boot --boot-volume eine neue VM mit dem Datenträger.

      Wenn Sie dieselben IP-Adressen beibehalten möchten, können Sie wie folgt vorgehen:

      1. Rufen Sie die Port-UUID der mit dem Datenträger gebooteten VM mit neutron port-list | grep <IP-Adresse der neuen VM> ab (die Port-UUID ist in der ersten Spalte aufgeführt).

      2. Entfernen Sie mit nova interface-detach den Port aus der alten VM.

      3. Aktualisieren Sie mit neutron port-update --fixed-ip ip_address=<IP-Adresse der alten VM> <Port-UUID> den Port in der neuen VM so, dass die IP-Adresse der alten VM verwendet wird.

      Wenn Sie mit dieser Umstellung zufrieden sind, können Sie die alte VM mit nova delete bereinigen.

      Dieser Prozess ist komplizierter, als Sie es sich wahrscheinlich wünschen. Die Umstellung auf den Blockspeicher als Gast-Back-End hat jedoch mehrere Vorteile:

      • Die Änderung der Größe von Varianten ist sehr viel schneller, weil die Änderung der Plattengröße übersprungen wird.
      • Die Migration auf einen neuen Hypervisor ist bei der Verwendung von Blockspeicher sehr schnell (der neue Hypervisor stellt einfach eine Verbindung zum Blockspeichercluster her).
      • Sie können die Platte Ihrer VM mit dem Befehl cinder extend während des Betriebs erweitern.

      Wenn Sie keinen Blockspeicher haben und Ihre Instanz in eine kleinere oder größere Variante ändern möchten, können Sie eine neue Instanz mit der gewünschten Größenvariante erstellen und dann eine Resynchronisation vornehmen, um die benötigten Daten aus der alten Instanz in die neue Instanz zu übertragen.

      Warum lässt OpenStack keine Änderung der Instanz in eine kleinere Variante zu?

      Ausführliche Informationen hierzu finden Sie unter https://blueprints.launchpad.net/nova/+spec/no-downward-resize. Kurz gesagt: Die Änderung der Instanz in eine kleinere Variante könnte sich auf die Datenintegrität auswirken und wird deshalb nicht zugelassen. Eine sichere Ausweichlösung ist die Erstellung einer neuen VM mit einer kleineren Variante und eine anschließende Resynchronisation, um die benötigten Daten aus der größeren Instanz in die kleinere Instanz zu übertragen.

      Wenn die Größe einer über einen Datenträger gebooteten Instanz geändert wird, ändert sich die Variante, aber die Größe bleibt dieselbe. Warum?

      Dieses Verhalten ist erwartet. Die Plattengröße der Variante ist nicht die Plattengröße der Rootplatte, wenn die Instanz über einen Datenträger gebootet wird. Weiterführende Informationen zu diesem OpenStack-Programmfehler finden Sie unter https://bugs.launchpad.net/nova/+bug/1412993.

      Informationen zum Ändern der Plattengröße und der Datenträgergröße finden Sie unter https://docs.openstack.org/user-guide/dashboard-manage-volumes.html#edit-a-volume. Wenn diese Methode aus irgendeinem Grund nicht funktioniert, können Sie auch versuchen, die unter https://mcwhirter.com.au/craige/blog/2014/Resizing_a_Root_Volume_for_an_Openstack_Instance/ beschriebenen Schritte auszuführen.
