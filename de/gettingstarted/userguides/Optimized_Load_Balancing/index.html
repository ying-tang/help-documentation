      Cloud 101 -- Optimierter Lastausgleich

      • by Dustin Lundquist and Leslie Lundquist ● 17. Mai 2017

      • Tags:

        • load balancing

        • Lastausgleich

        • HA

        • ARP-Responder

        • MAC-Umzug

      Bei der Ausführung Ihrer Workloads in einer Cloud können Sie die Last auf zwei oder mehr VM-Hosts verteilen. Ein Grund kann Redundanz sein, z. B., wenn Sie eine geschäftskritische Datenbankanwendung ausführen. Ein anderer Grund könnte sein, dass Sie Ihre Workload auf mehrere Maschinen skalieren möchten, damit mehrere Clients bedient werden können. Da einzelne VMs ausfallen können können (und das werden sie tun), geht es beim Design und beim Betrieb einer Cloud nicht nur um die automatische Bereitstellung von Services, sondern immer auch um die Berücksichtigung möglicher Ausfälle.

      Die meisten Cloudkunden wünschen sich hoch verfügbare Services (auch HA-Services genannt). Damit Ihre Workload in hoch verfügbarer Weise ausgeführt wird, können Sie sie auf zwei (oder mehr) vollständig voneinander getrennten Maschinen (Hardware) oder sogar noch besser in zwei separaten Verfügbarkeitszonen (Rechenzentren) hosten, sodass der Service beim Ausfall einer Maschine auf der anderen Maschine weiter ausgeführt werden kann. Wenn eine Maschine ausfällt, findet ein “Failover” auf die andere Maschine statt, d. h., dass der Service aktiv bleiben kann, also ein HA-Service ist, der nahezu unterbrechungsfrei verfügbar ist.

      Weitere wichtige Informationen zur Hochverfügbarkeit

      Hochverfügbarkeit bedeutet keine garantierte durchgängige hundertprozentige Verfügbarkeit, sondern nur “mehr Verfügbarkeit”, als es mit einem einzigen Server möglich ist. Wenn Sie beispielsweise zwei Server haben und jeder Server nur zu 70 % zuverlässig ist, was bedeutet, dass es zu jedem Zeitpunkt eine Chance von 70 % gibt, dass Anforderungen erfolgreich verarbeitet werden können, dann wären diese beiden Server in einer idealen HA-Konfiguration zu 91 % zuverlässig. Mathematisch? Es gibt eine Chance von 70 %, dass der erste Server die Anforderung erfolgreich verarbeitet, und eine weitere Chance (von 70 % der verbleibenden 30 %), dass der zweite Server die Anforderung erfolgreich verarbeitet, was 91 % entspricht. Vielleicht sollten Sie angesichts dieser Zuverlässigkeitsstufe das gesamte Setup trotzdem verwerfen oder sich zumindest einige zuverlässigere Server anschaffen. Allerdings ist eine “bessere” Verfügbarkeit immerhin besser.

      Hochverfügbarkeit und Lastausgleich

      Wenn Sie eine intensive Workload haben, steht gewöhnlich der Load Balander (LB) an erster Stelle auf der “Wunschliste” für Hochverfügbarkeit, damit Sie eine zentrale Stelle, ein bekanntes Ziel, haben, über die Ihre Anwendung immer erreichbar ist, selbst bei einem Failover. Wenn Sie beispielsweise Ihre Datenbank im Hochverfügbarkeitsmodus ausführen möchten, müssen Sie herausfinden, wie Sie die Datenbank auf zwei VMs replizieren, und die beiden VMs dann hinter einem Load Balander platzieren, um den Netzverkehr gleichmäßiger auf die VMs zu verteilen. In der Cloud benötigen Sie keine Garantie, dass jedes Mal dieselbe VM erreicht wird. Sie müssen lediglich wissen, dass Ihre Datenbank immer erreichbar und konsistent ist.

      In einer Cloudumgebung ist die Verwendung des Load Balancer des Cloud-Providers hierfür gewöhnlich die beste Wahl. Wenn Sie versuchen, Ihr eigenes LB-Schema zu erstellen, wird dieses wahrscheinlich unnötig komplex, weil Overlay- und SDN-Netze schlussendlich auf niederer Ebene mit den HA-IP-Schemas des Cloud-Providers interagieren. Wenn Sie den Load Balancer des Service-Providers verwenden, machen Sie es zu “seinem” Problem und deshalb müssen Sie die Mechanismen in seinem zugrundeliegenden Netz nicht sehen. Amazon Cloud verwendet beispielsweise Elastic Load Balancer (ELB) und das ist eine gute Lösung. Die Kunden müssen keinen Lastausgleich für sich selbst durchführen.

      Trotzdem gibt es drei effektive Methoden für den Lastausgleich in einer Cloudumgebung, falls Sie in die Situation kommen sollten, selbst für den Lastausgleich sorgen zu müssen. Jede Methode hat ihre Stärken und ihre Schwächen.

      Lastausgleich definiert: drei grundlegende Methoden

      Im Wesentlichen entspricht der Lastausgleich der Verwaltung (oder Änderung) der Zuordnung zwischen dem Namen eines Service (z. B. des Datenbankservice) und der Erreichbarkeit des Hosts, der diesen Service bereitstellt. Angenommen, der Server ist über das Ethernet erreichbar und Sie setzen die Anforderung über DNS ab. In diesem Fall können Sie die Zuordnung auf drei grundlegende Arten ändern. Man könnte auch sagen, Sie fügen an drei grundlegenden Positionen auf dem Pfad eine “Unterbrechung” ein. Sie können Folgendes ändern:

      • Zuordnung zwischen dem DNS-Namen und der IP-Adresse
      • Zuordnung zwischen der IP-Adresse und der Ethernet-MAC-Adresse
      • Zuordnung zwischen der Ethernet-MAC-Adresse und dem Switch-Port

      [Apropos: Verwenden Sie DNS nicht, wenn Sie eine direkte Verbindung zu einer bestimmten IP-Adresse herstellen möchten. So ist es einfacher, Ihre Services zu verwalten.]

      In der folgenden Tabelle sind die drei Optionen zusammengefasst. Wenn Sie diese Tabelle auf Anhieb verstehen, müssen Sie den Rest des Artikels wahrscheinlich nicht lesen. Für Benutzer, die die Tabelle nicht verstehen, folgt im Anschluss eine Erklärung.

      Option 1: Zuordnung von DNS-Namen zu IP-Adresse ändern

      Global Load Balancing (GLB) und Round-Robin-DNS sind Beispiele für die erste Option. GLB verweist auf einen intelligenten DNS-Resolver, der die Serververfügbarkeit, die Serverlast und/oder den Client und die geografische Nähe eines Service bei der Auswahl der zurückzugebenden DNS-Ressourcendatensätze berücksichtigt.

      Round-Robin-DNS ist ein einfacherer Ansatz, bei dem der DNS-Resolver die Reihenfolge der Ressourcendatensätze so ändert, dass nachfolgende Anforderungen reihum (RR, Round-Robin) verwendet werden. Angenommen, der Client probiert die erste RR-Adresse seiner bevorzugten Adressfamilie (IPv4 oder IPv6) aus. Falls er innerhalb des Zeitfensters (gewöhnlich 2 Sekunden) keine Verbindung herstellen kann, kann der Client versuchen, eine Verbindung zu einer der anderen Adressen herzustellen, die vom Resolver zurückgegeben werden.

      Die GLB-DNS-Methode findet fast immer “außerhalb” der anderen Methoden statt und kann mit diesen anderen Methoden kombiniert werden. Sie kann sicher in den meisten anderen Cloudlösungen verwendet werden, z. B., wenn Sie Rechenzentren haben, die in der ganzen Welt operieren. Sie behandelt dieselben Lastausgleichsprobleme wie andere Methoden, aber in einem ganz anderen Maßstab.

      Option 2: Zuordnung der IP-Adresse zur Ethernet-MAC-Adresse ändern

      Diese Option wird manchmal auch als “ARP-Umzug” bezeichnet. Beim ARP-Umzug hat jeder Host in der Gruppe eine eigene MAC-Adresse, aber jeder Host antwortet für diese gemeinsam genutzte HA-IP-Adresse. Die gemeinsam genutzte HA-IP-Adresse ist eine neue über die Gratuitous-ARP-Anforderung zugeordnete MAC-Adresse und die Aktualisierung muss an jeden anderen Host im lokalen Teilnetz gesendet werden. (Dies ist gewöhnlich ein VLAN und gewöhnlich, aber nicht immer eine 1:1-Zuordnung von Hosts zu IP-Adressen.) In diesem Fall sendet ein Host, der Daten von außerhalb des Teilnetzes sendet, die Daten über den Router. Somit ist es der Router, der die aktualisierte MAC-Adresse (nächster Hop) für diese IP-Adresse erfahren muss.

      Das Problem beim ARP-Umzug besteht darin, dass es zu einem partiellen Failover kommen kann, was nicht wünschenswert ist, wenn der Datenverkehr im Netz sehr hoch ist. In diesem Zustand werden einige Anforderungen an jeden HA-Server gesendet, weil einige Hosts die Anforderungen an das alte Ziel senden. Dies kann passieren, wenn die Aktualisierung nicht weitergegeben wird (sie kann gelöscht werden, wenn einige Verbindungen im Teilnetz sehr hohen Datenverkehr aufweisen.) Da jeder Host auf die Gratuitous-ARP-Aktualisierung antwortet, können beim Durchführen der Aktualisierungen die Ressourcen auf jedem Host aufgebraucht werden.

      Beachten Sie, dass die Methode mit dem ARP-Umzug an der L2/L3-Grenze des OSI-Modells operiert.

      Option 3: Zuordnung der Ethernet-MAC-Adresse zum Switch-Port ändern

      Bei der Option “MAC-Umzug” kommt auf dem aktiven Knoten einer HA-Gruppe eine virtuelle MAC-Adresse hinzu. Der “aktive” Host empfängt Frames für die gemeinsam genutzte MAC-Adresse sowie die Frames, die für seine lokale MAC-Adresse bestimmt sind.

      Die Methode “MAC-Umzug” ist ziemlich elegant, wenn Sie ein kleines L2-Netz mit wenigen Switches haben, die über das Spanning-Tree-Protokoll usw. (d. h. mit redundanten Verbindungen) verbunden sind. In dieser Situation sind die Techniken für den MAC-Umzug im Allgemeinen zuverlässiger und vorhersehbarer. (Aber an diesem Punkt führen Sie ihn auf der Hardware aus.) Wenn Sie ihn auf der Hardware eines anderen Providers ausführen möchten, verweisen Sie diesen auf diesen Artikel und fragen Sie nach, was empfohlen wird. (Wenn der Provider es nicht weiß, sollten Sie einen anderen suchen!)

      Beachten Sie, dass das MAC-Umzugsmodell an der L2/L1-Grenze des OSI-Modells operiert.

      Verwendung des Gratuitous-ARP-Signal in zwei der Methoden

      Verwirrend bei der Unterscheidung der letzten beiden Ansätze ist, dass sowohl der MAC-Umzug als auch der ARP-Umzug Gratuitous-ARP-Antworten für seine LB-Funktion verwendet. Tatsächlich verwenden sie dasselbe Signal für verschiedene Zwecke. Beim Empfang des Gratuitous-ARP-Signals wird jeweils eine andere Aktion ausgeführt. (Der Begriff “Gratuitous” bedeutet gratis und wird verwendet, weil das Signal von niemandem angefordert wird, sondern einfach per Broadcast verteilt wird.)

      MAC-Umzug

      Beim MAC-Umzug wird Gratuitous ARP als Signalisierungsmethode verwendet, aber die Hosts, die ein Gratuitous-ARP-Signal empfangen, kennen diese neuen Informationen bereits, es wird lediglich eine Aktualisierung des Felds “age to zero” vorgenommen. Die Hosts enthalten diese Informationen bereits im ihren Switchtabellen und deshalb wird bei der Aktualisierung nur die Alterungszeit zurückgesetzt, wodurch aus dem vorhandenen Eintrag wieder ein “neuer Eintrag” wird.

      (Apropos. Alle Einträge für Switches und Hosts (in der ARP-Tabelle und in der MAC-Adresstabelle) altern und werden zu löschbaren Einträgen. Die Einträge bleiben je nach Anbieter zwischen 5 Minuten und 4 Stunden bestehen.)

      Zusammenfassend für die MAC-Umzugsmethode gesagt, der Host ignoriert die Nachricht einfach oder er setzt den Alterungswert in der Switchtabelle zurück.

      ARP-Umzug

      Bei der ARP-Umzugsmethode mit diesem Beispiel sagen die Switches jedoch “Hmm, dieser Frame stammt von einem anderen Port als dem, den ich kenne” und führt deshalb die Aktualisierung durch, d. h., der Switch weiß, dass die Quellenadresse für diesen Frame an einem anderen Port als erwartet ankommt, und aktualisiert deshalb seine eigene MAC-Adresstabelle entsprechend.

      Warum sind der ARP-Umzug und der MAC-Umzug in SDN nicht geeignet?

      Diese Methoden sind aus allen Standardgründen mit SDN (Software-Defined Networking) nicht besonders skalierbar: potenzielle Leistungseinbußen, potenzielle Sicherheitsprobleme, potenzielle Verzögerungen bei der Weitergabe, die das Failover verlangsamen, und potenzielle Split-Brain-Situationen.

      Leistung und Sicherheit

      Wenn Sie controllerbasiertes SDN verwenden und nur im L2-Netzbetrieb arbeiten, ist ein MAC-Umzug ein Ereignis, das sich seinen ganzen Wert nach oben bis zur Steuerebene bahnen muss. Dann werden die Pfade neu berechnet und wieder in die Switches übertragen.

      Diese Aktivität ist rechenintensiv und kann als Denial-of-Service-Attacke verwendet werden. Sie ist im Allgemeinen abgeregelt oder sollte es zumindest sein.

      Verzögerungen bei der Weitergabe und Grenzfälle

      Anders als in traditionellen Distributed Learning-Netzen (wie z. B. normalen Ethernet-Switch-Netzen, die miteinander verbunden sind) braucht der lokale Switch für das Lernen dieser Informationen länger. In einem traditionellen Netz werden die neuen Informationen wie eine sich immer weiter ausdehnende kreisförmige Welle weitergegeben. Diese Art der Informationsweitergabe (= die sich ausdehnende kreisförmige Welle) leitet einen neuen Status ein. Manche betrachten diese Informationsweitergabe eher wie eine Ausbreitungswelle, deren Ausdehnungsrichtung mehr linear als kreisförmig ist.

      Die Verzögerung bei der Informationsweitergabe (“Wellenverzögerung” in dem Bild) führt zu Grenzfällen, bei denen einige der Switches die neue Zieladresse für einen HA-Failover bereits kennen, jedoch nicht alle. Diese Grenzfälle sind leichter absehbar und einfacher zu verwalten, wenn aktualisierte Informationen wie eine Welle weitergegeben werden/sich wellenförmig ausbreiten. Es ist beispielsweise sehr viel weniger wahrscheinlich, dass zwischen zwei Switches ein Schleifenkreislauf entsteht, bei dem jeder Switch annimmt, der andere Switch muss aktualisiert werden. (Diese Aktivität würde allein schon die gesamte verfügbare Bandbreite der Verbindungen zwischen diesen Switches verbrauchen.)

      Schlussendlich kann das SDN L3-Netze enthalten, muss es aber nicht. Wenn SDN L3-Netze enthält, wirkt sich der ARP-Umzug ähnlich auf die von SDN gesteuerten Router aus. Insbesondere die Verarbeitung von Aktualisierungen dauert länger. Da es gewöhnlich weniger Router als Switches in der L2-Fabric eines bestimmten Teilnetzes gibt, ist die Situation normalerweise gar nicht so schlecht. Das Hauptproblem ist die Minimierung der für die Aktualisierung erforderlichen Zeit, die natürlich der für ein Failover erforderlichen Zeit entspricht.

      SDN mit aktiviertem ARP-Responder

      Einige SDNs haben einen ARP-Responder (z. B. in OpenStack Neutron) und können deshalb mit alten oder inkonsistenten Informationen antworten. Diese Situation betont die normalen Nachteile bei der ARP-Umzugsmethode, die potenziell zu einer Split-Brain-Funktion führen können. Die Anwendung dieser Methode wäre in OpenStack Neutron schädlich, wenn Sie beispielsweise ein Master-Master-Datenbanksetup haben. Glücklicherweise ist der ARP-Responder in Neutron optional und kann deshalb ggf. inaktiviert werden.

      Weitere Informationen zum Lastausgleich in Neutron finden Sie in einem zugehörigen einführenden Artikel, und hier ist ein Artikel, in dem verschiedene Einschränkungen, denen Sie begegnen können,, beschrieben sind.

      Degenerativer Fall: ARP-Lastausgleich

      ARP-Lastausgleich ist ein Fall, in dem Sie fortlaufend Gratuitous-ARP-Pakete senden, sodass Sie in einem zweiten untergeordneten Zeitfenster Tausende und Abertausende von Paketen senden, die verarbeitet werden müssen. Dabei erhöht sich die normale Last um das Hundert- oder Tausendfache. In einer normalen Netzsituation ist der ARP- oder MAC-Umzug ein seltenes Ereignis. Es wird beispielsweise ein ARP-Paket gesendet, wenn das System aktiviert wird, dann möglicherweise ein zweites 4 Stunden später usw. Dies ist kein gewöhnlicher Designfall für SDN. Der Lastausgleich wird in diesem Fall durch einen konstanten Strom von Aktualisierungen erreicht, der bis zur Steuerebene aufsteigen und dann an jeden Host verteilt werden muss, wodurch zahlreiche Systemressourcen verbraucht werden, um eine Funktion auszuführen, die sehr viel effizienter ausgeführt werden könnte.

      Einige namentlich hier nicht genannten kommerziellen Cloud-Provider verwenden ARP-Lastausgleich als normale Methode, um den Lastausgleich für ihre Kunden durchzuführen. Diese Provider täten gut daran, ihre Techniken zu überdenken, weil sie sich selbst allen zuvor aufgelisteten potenziellen Problemen aussetzen, insbesondere Leistungs- und Sicherheitsproblemen.

      Weitere Implementierungen dieser 3 grundlegenden LB-Methoden

      Es sind andere Protokolle vorhanden, die Load Balancer verwenden können, um eine Verbindung zum Netz herzustellen. Sie können beispielsweise ein Netzrouting-Protokoll sprechen oder als Switch in einer gebündelten Multi-Ethernet-Switch-Verbindung (LAG oder Portkanal) auftreten.

      Diese Implementierungen erfordern eine erweiterte Netzkonfiguration und können im Allgemeinen spezifischer gestaltet werden. In bestimmten Situationen kann diese Methode eine bessere Wahl als die zuvor beschriebenen Methoden sein. (Sie funktionieren nicht überall, aber ein Cloud-Provider kann eine dieser Methoden verwenden, um eine Verbindung zu ihrem LB-Service herzustellen.)

      Bestimmte Services wie der keepalived-Service können ebenfalls in diese Kategorie eingeordnet werden.

      Experten wie Provider für verwaltetes Hosting können diese komplexen Alternativen vermutlich erfolgreich einsetzen. Vor der Übernahme von Blue Box durch IBM wurde beispielsweise OSPF verwendet, um die gemeinsam genutzte IP-Adresse in der Netzinfrastruktur für den Blue Box-Blocks-LB zu veröffentlichen.

      Zusammenfassung: Es sind zwar andere, erweiterte Implementierungen der drei grundlegenden Methoden, die in diesem Artikel beschrieben werden, vorhanden, aber die drei beschriebenen Methoden sind die grundlegenden Methoden, die jede Implementierung für den Lastausgleich in einer Cloudumgebung verwenden sollte.
