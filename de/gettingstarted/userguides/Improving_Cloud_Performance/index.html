      Einführung in OpenStack II -- Cloudleistung verbessern

      • by Dustin Lundquist and Leslie Lundquist ● 21. November 2016

      • Tags:

        • openstack

        • OpenStack

        • sr-iov

        • virtio

        • vhost

      Neuveröffentlichung von https://admin.blogs.prd.ibm.event.ibm.com/blogs/bluemix/2016/11/building-openstack-ii-improving-cloud-performance/

      In diesem Artikel, dem zweiten in der Serie, behandeln wir weitere Cloudgrundlagen. Sie lernen mehr über QEMU und noch viel mehr darüber, wie Sie die Gesamtleistung bei Workloadverarbeitung in Ihrer Cloud verbessern können.

      Die Verbesserung der Leistung in Ihrer Cloud setzt im Wesentlichen E/A-Effizienzsteigerungen beim Zugriff auf Netz- und Speichergeräte voraus. Indem Sie die E/A-Virtualisierung so optimieren, dass die E/A-Geräteschnittstellen zwischen den Gast-VMs und dem Host effizienter arbeiten, können Sie die Leistung verbessern und die Flexibilität für die Eingabe/Ausgabe erhöhen. Damit verbessert sich die Leistung der Cloud insgesamt, weil mehr Gesamtprozessorkapazitäten für Ihre Workload verfügbar sind.

      Verbesserung der E/A-Leistung durch Virtualisierung

      Denken Sie daran, dass QEMU das Basiselement einer Cloud-VM ist. Da es sich bei QEMU jedoch um einen Emulator handelt, ist es sinnvoll, sich so wenig wie möglich auf QEMU zu verlassen, wenn Geschwindigkeit im Vordergrund steht. Obwohl durch die Verwendung von KVM mit QEMU gewisse Verbesserungen bei der Prozessor- und Hauptspeicherleistung erzielt werden können, bleibt ein Engpass bei der E/A-Leistung.

      Es gibt zwei Pfade zur Verbesserung der E/A-Leistung durch Virtualisierung: einen Softwarepfad und einen Hardwarepfad. In diesem Blog werden beide Pfade untersucht. Auf jedem Pfad wird QEMU mit jedem Schritt mehr ausgeschaltet. Warum? Weniger Emulation bedeutet bessere Leistung.

      Der Softwarepfad beinhaltet eine effizientere Virtualisierung der “traditionellen” Geräte und deren Treiber. Beim Hardwarepfad werden Geräte direkter mit den VMs verknüpft und unter Umständen sogar spezielle Hardwarekomponenten eingesetzt, die das Feature Single Root IO Virtualization (SR-IOV) implementieren. Hierbei handelt es sich im Wesentlichen um einen hardwaregestützten Pfad für eine schnelle Paarung von VMs mit virtualisierten Geräten, die bei Bedarf instanziiert werden können.

      Softwarepfad: QEMU, ‘virtio’ und ‘vhost’

      Wie in einem vorherigen Blog beschrieben, wurde QEMU als Emulator entwickelt, der selbst “historische” Betriebssysteme wie Windows NT und OS/2 emulieren kann. Er emuliert jede Hardware, die das Betriebssystem erwartet. Deshalb sind die Standardgeräte, die bereitgestellt werden, “gängige” physische Hardwaregeräte. Die Methode, die das Betriebssystem für die Kommunikation mit diesen Geräten verwendet, wurde jedoch nicht mit Virtualisierung im Hinterkopf entwickelt!

      Ältere E/A-Geräte sind gewöhnlich keine Universalprozessoren und sie sind langsamer. Deshalb brachte die Optimierung der E/A-Leistung in den alten “Hardwarezeiten” nur wenig Vorteile. Physische Platten brauchen beispielsweise für den Zugriff auf die Position auf einer Platte sehr viel Zeit (4 ms). Die Vorteile der Optimierung der Kommunikation zwischen dem Treiber und dem Plattencontroller wären somit nur marginal. In einer modernen Cloud mit VMs als Servern können durch die Verbesserung der E/A-Leistung hingegen erhebliche Vorteile erzielt werden und dies mit klar verständlichen Methoden.

      Warum hilft die E/A-Virtualisierung?

      Für eine effiziente Ein-/Ausgabe benötigt Ihr Cloudsystem (1) einen Treiber für die Gast-VM, (2) ein zusätzliches Back-End in der Host-Software (z. B. QEMU), das das E/A-Gerät unterstützt, und (3) die Schnittstellen, die die Host-Software für die Kommunikation mit dem zugrunde liegenden E/A-Gerät verwendet.

      Wie Sie in der Abbildung sehen, sind verschiedene Komponenten an der Verarbeitung einer E/A-Anforderung in einer VM beteiligt. Jeder Übergang zwischen diesen Komponenten bedeutet Leistungseinbußen. Um zu veranschaulichen, wie schnell sich die E/A-Leistung verschlechtern kann, sehen Sie im Folgenden eine schrittweise Aufgliederung des Prozesses:

      • Zuerst setzt das Programm, das in der VM ausgeführt wird, seine E/A-Anforderung über die “normale” Schnittstelle an das Betriebssystem der VM ab. (Wenn die auf einer VM ausgeführte Software eine andere Methode verwendet, ist die Virtualisierung für den Benutzer nicht transparent.)

      • Als Nächstes verwendet das Betriebssystem der VM den Treiber für das von QEMU bereitgestellte Gerät.

      • Anschließend führt der Treiber verschiedene Operationen aus, die von KVM abgefangen und zur Verarbeitung an QEMU übergeben werden.

      • QEMU verwendet dann eine ähnliche Schnittstelle, um eine E/A-Anforderung an das Hostbetriebssystem abzusetzen.

      • Das Hostbetriebssystem verwendet daraufhin den Treiber für die physische Hardware, um die angeforderte E/A-Operation auszuführen.

      Wie kann dieser Prozess beschleunigt werden? Es gibt zwei gute Methoden. Zuerst könnten Sie die Emulation eines bestimmten Geräts effizienter virtualisieren. Diese Option wird gewöhnlich als “paravirtualisierter Treiber” bezeichnet und ist das beste Beispiel für das Feature virtio, das in OpenStack Nova integriert wurde. Als Zweites könnten Sie für weitere Leistungsverbesserungen einen weiteren Übergang verhindern, indem Sie die Geräteemulation vollständig aus der VM entfernen. Das heißt, Sie entfernen die QEMU-Geräteemulation aus der Gast-VM und beschränken sie auf das Hostbetriebssystem.

      Optimierung 1. Paravirtualisierte Treiber: Die virtio-Methode

      Virtio ist eine Familie virtualisierter Geräte, die von QEMU zusammen mit den entsprechenden Treibern im Betriebssystem einer VM bereitgestellt werden. Diese Treiber optimieren die Schnittstelle zwischen dem Treiber der Gast-VM und QEMU, indem moderne, effiziente Datenstrukturen wie Ringpuffer verwendet werden, anstatt die Schnittstelle eines vorhandenen Hardwaregeräts sklavisch zu emulieren. Mit virtio kommuniziert die VM effizienter mit QEMU und verbessert dadurch die E/A-Leistung.

      Der Nachteil von virtio ist, dass Gast-VM-Images diese schnelleren virtio-Treiber enthalten müssen, und damit funktionieren einige ältere Images (z. B. Windows 95 oder 98) unter Umständen nicht mit virtio. Glücklicherweise enthalten die meisten Betriebssysteme die virtio-Treiber seit 2013 standardmäßig.

      Optimierung 2: Vhost (Removing the QEMU userspace device model)

      Für die nächste Leistungsoptimierung müssen Sie die virtio-Geräteemulationsfunktion vollständig aus der VM entfernen und in den Hostbetriebssystemkernel verschieben. Dadurch werden die Leistungseinbußen, die mit den Übergängen von KVM in QEMU und in den Host-Kernel zurück verbunden sind, vermieden.

      Der Hardwarepfad: PCI-Durchgriff und SR-IOV

      Der PCI-Durchgriff ist ein generischer Mechanismus, der auf jedes PIC-Gerät angewendet werden kann. Er ist eine Alternative zur Softwarevirtualisierung von Geräten und Treibern, die in den vorherigen Abschnitten beschrieben wurde. Im Gegensatz zu virtio und vhost wird der Gast-VM beim PCI-Durchgriff ein Hardware-E/A-Gerät direkt zur Verfügung gestellt. Damit werden QEMU und der Host-Kernel vollständig aus dem E/A-Pfad entfernt und die Leistungseinbußen, die mit diesen Übergängen für die Softwareemulation verbunden sind, vermieden.

      Der PCI-Durchgriff ist in vielen Anwendungsfällen eine gute Alternative für das Erreichen einer besseren Leistung.

      • Manche Personen verwenden diese Durchgriffstechnik beispielsweise für Spiele, indem sie den PCI-Durchgriff für Ihre Videokarte einrichten. Beachten Sie, dass in diesem Fall die Virtualisierung nicht für die Verbesserung der Spieleleistung verwendet wird, sondern für die Ausführung von Spielen für andere Betriebssysteme. Die Virtualisierung moderner Spiele ist ohne PCI-Durchgriff nicht wirklich möglich.

      • Der wahrscheinlich gängigste Typ von PCI-Durchgriffsgeräten ist die Netzkarte. Sie können den Netzkartendurchgriff beispielsweise mit vhost_net durchführen. Dabei wird QEMU aus dem Netzdatenpfad entfernt, aber vhost_net stützt sich weiterhin auf den intakten Netzstack des Host-Kernels.

      • Es muss erwähnt werden, dass sich der PCI-Durchgriff im Allgemeinen für Hostbusadapter (HBA) und Platten nicht eignet, weil die meisten Server nur einen einzigen Plattencontroller haben, der mit allen Platten des Servers verbunden ist. Damit könnte die Kapazität dieser Platten nicht von den VMs in einer Cloud gemeinsam genutzt werden, weil alle Platten zu einer einzigen VM gehören. Im nächsten Abschnitt wird beschrieben, welche Lösung HBAs, in denen SR-IOV aktiviert für diesen Fall haben.

      • Sie könnten sich eine esoterische Anwendung des PCI-Durchgriffs für einen HBA vorstellen, bei der eine Cloud über USB gebootet wird und die gesamte Plattenspeicherkapazität dann einer einzigen VM zugeordnet wird, die als Back-End-Datenbank dient.

      Es ist wichtig, daran zu denken, dass der PCI-Durchgriff eine dedizierte E/A-Gerät für jede Gast-VM erfordert. Diese Voraussetzung dedizierter Hardware für jede VM verringert die Flexibilität, die die Virtualisierung mit sich bringt, und erhöht die Kosten erheblich. Glücklicherweise sind einige neuere Hardwaretypen verfügbar, die die Kosten durch eine integrierte, hardwaregestützte Präsentation virtualisierter PCI-Geräte verringert. Bei dieser Technik handelt es sich um SR-IOV.

      SR-IOV

      Neuere High-End-Netzschnittstellen und -Plattencontroller enthalten SR-IOV (Single Root I/O Virtualization). Diese Hardwaregeräte können sich im Wesentlichen selbst als mehrere PCI-Geräte präsentieren, die jeweils einer VM zugeordnet werden können. In einer typischen Bereitstellung gibt es ein Hauptgerät (übergeordnetes Gerät oder übergeordnete Funktionseinheit) und mindestens ein logisches Gerät, das als virtuelle Funktionseinheit bezeichnet wird. Durch die Konfiguration der übergeordneten Funktionseinheit können zusätzliche virtuelle Funktionseinheiten zusammen mit den zugehörigen Ressourcen erstellt werden. Diese virtuellen Funktionseinheiten können dann über den PCI-Durchgriff einzelnen VMs zugeordnet werden.

      Potenziell können Sie den PCI-Durchgriff für zwei E/A-Geräte pro Gast-VM verwenden: eine Netzschnittstelle und einen Plattencontroller. Mit zwei SR-IOV-Geräten könnten Sie beispielsweise Verbesserungen bei der E/A-Leistung für das Netz und für den Speicher erzielen.

      Zusammenfassung

      Sie können die Workloadleistung Ihrer Cloud mit den in diesem Artikel beschriebenen Pfaden - Softwarepfad oder Hardwarepfad - verbessern. Es gibt einen optimierten Softwarepfad mit virtio und vhost und einen hardwaregestützten optimierten Pfad, bei dem PCI-Durchgriff und SR-IOV verwendet werden.
