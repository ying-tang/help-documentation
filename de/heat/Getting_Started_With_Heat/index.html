      Erste Schritte mit Heat-Vorlagen

      Als bewährtes Verfahren sollte Ihr Team die Verwendung des OpenStack-Orchestrierungsservice (Heat) in Erwägung ziehen, der mit jeder IBM Bluemix Private Cloud-Instanz bereitgestellt wird. Mit Heat können Sie alle Ressourcen verfahrensorientiert in einer Textdatei definieren und diese Datei dann für mehrere Clouds ausführen, um das jeweils selbe Ergebnis zu erzielen (dies wird als Stack bezeichnet).

      Sie können Heat verwenden, um einen Stack einzurichten, der jede OpenStack-Ressoure enthält, die Sie sich wünschen.

      Angenommen, Sie möchten mehrere Stacks mit denselben Sicherheitsgruppendefinitionen einrichten. In den Zeilen 82-110 des folgenden Beispielscripts sehen Sie, wie Sicherheitsgruppen in Heat definiert werden. Damit können Sie Sicherheitsgruppen zwar per se nicht “kopieren”, aber Sie können einen identischen Stack (mit beispielsweise Sicherheitsgruppenressourcen) in so vielen Projekten oder verschiedenen Clouds wie gewünscht erstellen. Da die Eingabetextdatei immer dieselbe ist, sind der erstellte Stack und die darin enthaltenen Ressourcen identisch. Sie können sogar mehrere identische Stacks unter demselben Projekt starten.

      Von OpenStack erstellte einführende Dokumentationen finden Sie hier und hier.

      Ihre Heat-Vorlage kann so viele (oder so wenige) Ressourcen enthalten, wie Sie wünschen. Nachdem Sie eine Vorlage erstellt haben, können Sie daraus mit Befehlen wie den folgenden so viele Stacks wie gewünscht erstellen:

      openstack stack create -t <Vorlagendatei> <Stackname>

openstack stack show <Stackname>

      Sie können auch vorhandene Stacks “aktualisieren”, indem Sie den Inhalt der Datei ändern und dann einen Aktualisierungsbefehl ausführen:

      openstack stack update -t <Vorlagendatei> <Stackname>

      Diese Informationen sind hoffentlich hilfreich für Sie. Auf jeden Fall beschreiben Sie eine sichere Methode für die Lösung jedes Multi-Cloud-Problems.

      Benutzerrollen in Heat-Stacks

      In OpenStack können Sie einem Benutzer viele verschiedene Rollen zuweisen. Einem “normalen” Benutzer kann beispielsweise nur die Rolle Member (Mitglied) zugewiesen werden. Ein Administrator kann die Rolle cloud_admin (Cloudadministrator) haben.

      Mithilfe dieser Rollen kann der Zugriff auf einige API-Aufrufe definiert und beschränkt werden. Die Benutzer mit der zugewiesenen Rolle Member können beispielsweise das Kennwort für einen anderen Benutzer nicht ändern, aber ein Benutzer mit der zugewiesenen Rolle cloud_admin schon.

      Der OpenStack-Orchestrierungsservice Heat unterstützt einen eigenen Satz zusätzlicher Rollen.

      Der Heat-Service weist Benutzern, die er während der Stackbereitstellung erstellt, automatisch die Rolle heat_stack_user zu. Diese Rolle hat eingeschränkten Zugriff auf einige API-Aufrufe, da von Heat in einem Stack erstellte Benutzer nur für diesen Stack bestimmt sind.

      Wenn ein Benutzer in der Lage sein soll, verschiedene Stacks zu verwalten (z. B. Ressourcen in vielen aktiven Stacks zu löschen und hinzuzufügen), müssen Sie diesem Benutzer die Rolle heat_stack_owner zuweisen. Die Rolle heat_stack_owner hat sehr viel weniger Einschränkungen in Bezug auf die absetzbaren API-Aufrufe an Heat.

      Leider stehen die Rollen heat_stack_user und heat_stack_owner miteinander in Konflikt. Wenn Sie also einen Benutzer mit der Rolle heat_stack_owner haben, dürfen Sie diesem Benutzer nicht die Rolle heat_stack_user zuweisen, um diesen Konflikt zu vermeiden.

      Die verschiedenen Rollen haben den Zweck, die Verwendung von “Keystone Trusts” als Methode für verzögerte Authentifizierung zu vereinfachen. Wenn Sie mehr über dieses Thema erfahren möchten, lesen Sie den folgenden Blogbeitrag: http://hardysteven.blogspot.com/2014/04/heat-auth-model-updates-part-1-trusts.html
