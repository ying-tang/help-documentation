      Benutzer und Projekte verwalten

      Basiskonzepte

      Benutzer: Ein Benutzer ist eine Einzelperson oder ein Service mit Anmeldeberechtigungsnachweisen für die OpenStack-Umgebung. Ein Benutzer hat eine oder mehrere Rollen und ist Mitglied eines oder mehrerer Projekte.

      Projekte: Projekte sind Organisationseinheiten in der Cloud und werden auch Tenants oder Konten genannt. Jeder Benutzer ist Mitglied eines oder mehrerer Projekte. Innerhalb eines Projekts erstellt und verwaltet ein Benutzer Ressourcen wie Instanzen und Datenträger. Administratoren können die zulässige Anzahl der Ressourcen (Instanzen, virtuelle CPUs, Datenträger usw.), die in einem Projekt erstellt werden können, einschränken.

      Rollen: Eine Rolle ist eine Gruppe von Berechtigungen, die einem Benutzer die Ausführung einer bestimmten Gruppe von Operationen ermöglicht.

      Gruppen: Gruppen fungieren als Verbindung zwischen Benutzern und Rollen. Ein Benutzer ist Mitglied einer oder mehrerer Gruppen und eine Gruppe hat eine oder mehrere zugeordnete Rollen.

      Ihre Installation von IBM Bluemix Private Cloud enthält vier vordefinierte Rollen: cloud_admin, project_admin, _member_ und heat_stack_owner. Eine fünfte Rolle, heat_stack_user, wird automatisch zugeordnet.

      |Rolle| Beschreibung |:———|:———– | cloud_admin | Diese Rolle ermöglicht die Zugriffssteuerung auf Cloudebene. Sie können unabhängig von Ihrem Projekt API-Ausführungaufgaben ausführen. Diese Rolle kann Kontingente, Gruppen, Benutzer und Projekte erstellen und verwalten und administrative Datenträgeraktionen ausführen.| project_admin | Diese Rolle ermöglicht die Zugriffssteuerung auf Projektebene. Der Benutzer kann Benutzer und Projekte innerhalb des angegebenen Projekts verwalten. | _member_ | Mit dieser Rolle können Benutzer die Ressourcen (wie Instanzen und Datenträger) verwenden, die dem Projekt zugeordnet sind. | heat_stack_owner | Mit dieser Rolle können Benutzer Heat-Stacks im Projekt erstellen. Diese Rolle muss manuell zugewiesen werden und darf nie einem Benutzer zugewiesen werden, dem bereits die Rolle heat_stack_user zugewiesen wurde. | heat_stack_user | Diese Rolle wird von Heat automatisch jedem erstellten Benutzer zugewiesen. Diese Rolle hat keinen API-Zugriff und darf keinem Benutzer explizit zugewiesen werden. Die Rolle cloud_admin ist auf Cloud- und Projektverwaltungsaktionen beschränkt. Nur die Rolle admin hat mehr Berechtigungen, darunter Verwaltungsaktionen für Services, Domänen, Rollen, Netze und Identitätsprovider. Nur IBM hat die Rolle admin. Dem Kunden wird diese Rolle nicht zugewiesen.

      Mit Ihrer Installation von IBM Bluemix Private Cloud erhalten Sie einen Benutzer mit Administratorrechten (cloud_admin), der weitere Projekte und Benutzer erstellen kann. Der Benutzer cloud_admin kann anderen Benutzern die Rolle cloud_admin oder Rollen mit weniger Berechtigungen zuteilen.

      Voraussetzungen

      Diese Rollenstruktur erfordert, dass die Keystone-API V3 ausgeführt wird. Die Verwendung der Keystone-API V2 führt dazu, dass die Rollen cloud_admin und project_admin Berechtigungen verlieren und bei Benutzern mit der Rolle _member_ unerwartetes Berechtigungsverhalten auftritt.

      Das Ausführen von API-Aufrufen über das Horizon-Dashboard erfordert keine speziellen Aktionen. Lediglich das Ausführen von Befehlszeilenaufrufen erfordert Änderungen der RC-Datei (Run Command), je nachdem, welche Version von IBM Bluemix Private Cloud Sie verwenden.

      • In IBM Bluemix Private Cloud Version 2.1.1 aktivieren die über das Dashboard erstellten RC-Dateien Keystone V2.
      • Der Wechsel zu V3 erfordert in Version 2.1.1 Änderungen an der RC-Standarddatei. Der Wert für OS_AUTH_URL muss von v2.0 in v3 geändert werden. Außerdem muss OS_IDENTITY_API_VERSION erstellt und als Wert 3 festgelegt werden. Nachfolgend sehen Sie eine RC-Beispieldatei.

      Anmerkung: Seit dem letzten Update von IBM Bluemix Private Cloud (Version 3.0.0) können Sie mit dem Horizon-Dashboard eine Identity-V3-RC-Datei generieren und herunterladen, die die oben hervorgehobenen Änderungen nicht erfordert. Dennoch ist wegen eines nicht behobenen vorgelagerten Fehlers eine einzige Änderung zu machen. Sie müssen in der von Horizon generierten Datei eine Zeile hinzufügen:

      Außerdem MÜSSEN Keystone-Funktionen mit der neuen Befehlszeilenschnittstelle (CLI) python-openstackclient ausgeführt werden. Um diesen Client zu installieren, führen Sie den Befehl sudo pip install python-openstackclient aus. (Ein bekanntes Problem ist, dass Befehle der Keystone-Befehlszeilenschnittstelle unter V3 unerwartetes Verhalten zeigen.)

      Keystone V3-Gruppenfunktion verwenden

      Die Version 3 der Keystone-API mit IBM Bluemix Private Cloud führt das Konzept der Gruppen ein. Sie können einfach Gruppen verwenden, um mehrere Zuweisungen gleichzeitig zu machen.

      Angenommen, Sie haben ein Support-Team, das Zugriff auf jedes Projekt in Ihrer Umgebung benötigt. Anstatt jedes Mitglied jedem Projekt zuzuordnen, können Sie die Gruppe Support erstellen, dieser die Rolle _member_ für alle Ihre Projekte zuweisen und anschließend alle Mitarbeiter des Support-Teams zur Gruppe hinzufügen.

      Ein weiterer gängiger Anwendungsfall ist das Erstellen der Gruppe cloud_admin, der die Rolle cloud_admin für jedes Projekt in der Umgebung zugewiesen wird. Ein Benutzer mit der Rolle cloud_admin kann so einfach die Berechtigungen eines Benutzer erweitern, indem er ihn vorübergehend zur Gruppe cloud_admin hinzufügt (beispielsweise bei Urlaubsvertretung).

      Die Unterschiede der Vorgehensweisen werden in den folgenden zwei Abbildungen veranschaulicht.

      Diese Abbildung zeigt ein Beispiel einer konventionellen Rollenzuweisung, wie sie die Keystone-V2-API unterstützt. Der Benutzer cloud_admin hat die Rolle cloud_admin für Projekt 1, 2 und 3. Benutzer 1 hat die Rolle project_admin für Projekt 2. Benutzer 2 und Support-Benutzer 1 haben die Rolle _member_ für Projekt 2. Support-Benutzer 1 hat außerdem die Rolle _member_ für Projekt 3.

      Folgen der Inaktivierung von Projekten

      Benutzer mit der Rolle cloud_admin können Projekte aktivieren und inaktivieren. Die Inaktivierung eines Projekts hat folgende Konsequenzen:

      • Benutzer können im Dashboard auf der Registerkarte “Project” in der Liste der aktuellen Projekte nicht mehr auf das Projekt zugreifen.
      • Benutzer, die nur Mitglied des inaktivierten Projekts sind, können sich nicht mehr anmelden.
      • Für das inaktivierte Projekt können keine Instanzen gestartet werden. Instanzen, die bereits ausgeführt werden, werden nicht automatisch beendet. Sie müssen manuell gestoppt werden.
      • Sie können das Projekt jedoch jederzeit erneut aktivieren, da die Daten eines inaktivierten Projekts nicht gelöscht werden.

      Dashboard verwenden

      Im nächsten Abschnitt finden Sie Informationen zur Verwendung des Dashboards zum Hinzufügen, Aktualisieren, Ändern und Löschen eines Projekts oder eines Benutzerkontos.

      Projekt hinzufügen

      1. Öffnen Sie in der Navigationsleiste das Fenster “Identity” und klicken Sie auf Projects.
      2. Klicken Sie auf Create project.
      3. Geben Sie auf der Registerkarte “Project Information” im Fenster “Create Project” den Namen und eine Beschreibung des Projekts ein. Standardmäßig ist das Projekt aktiviert.
      4. Fügen Sie auf der Registerkarte “Project Members” Mitglieder zum Projekt hinzu und weisen Sie ihnen die entsprechenden Rollen zu.
      5. Fügen Sie auf der Registerkarte “Project Groups” Benutzergruppen zum Projekt hinzu und weisen Sie ihnen die entsprechenden Rollen zu.
      6. Bearbeiten Sie auf der Registerkarte “Quota” die Kontingentwerte.
      7. Klicken Sie auf Create Project. Das Projekt wird nun einschließlich der zugewiesenen ID in der Projektliste angezeigt.

      Projekt aktualisieren

      Sie können ein Projekt aktualisieren, um dessen Namen oder Beschreibung zu ändern oder um es zu aktivieren bzw. vorübergehend zu inaktivieren.

      1. Öffnen Sie in der Navigationsleiste das Fenster “Identity” und klicken Sie auf Projecs.
      2. Wählen Sie das Projekt aus, das Sie aktualisieren möchten.
      3. Wählen Sie in der Dropdown-Liste unter der Spalte “Actions” Edit Project aus.
      4. Im Fenster “Edit Project” können Sie ein Projekt aktualisieren, um dessen Namen oder Beschreibung zu ändern oder um es zu aktivieren bzw. vorübergehend zu inaktivieren. Standardmäßig ist das Projekt aktiviert. Um es vorübergehend zu inaktivieren, wählen Sie das Kontrollkästchen Enabled ab. Um ein inaktiviertes Projekt zu aktivieren, wählen Sie das Kontrollkästchen Enabled aus.
      5. Klicken Sie auf Save.

      Benutzerzuordnungen für ein Projekt ändern

      Wenn Sie die Rolle cloud_admin haben, können Sie jedem Projekt Rollen zuweisen. Wenn Sie die Rolle project_admin haben, können Sie Zuweisungen zu den Projekten ändern, für die Sie die Rolle project_admin haben.

      1. Öffnen Sie in der Navigationsleiste das Fenster “Identity” und klicken Sie auf Projects.
      2. Klicken Sie für das zu ändernde Projekt auf die Schaltfläche Manage Members.

      Im Fenster “Edit Project” sehen Sie die folgenden Benutzerlisten:

      • All Users: Benutzer, die dem aktuellen Projekt zugewiesen werden können.
      • Project Members: Benutzer, die dem aktuellen Projekt zugewiesen sind. Außerdem können Sie auf der Registerkarte “Project Groups” Gruppen zuweisen.

      Projekte löschen

      1. Öffnen Sie in der Navigationsleiste das Fenster “Identity” und klicken Sie auf Projects.
      2. Wählen Sie die Projekte aus, die Sie löschen möchten.
      3. Klicken Sie auf Delete Projects, um den Löschvorgang zu bestätigen.

      Benutzerkonto erstellen

      Wenn Ihnen die Rolle cloud_admin oder project_admin zugewiesen ist, können Sie neue Benutzerkonten erstellen. Wenn Sie ein Benutzerkonto erstellen, müssen Sie das Konto einem primären Projekt zuordnen. Außerdem haben Sie die Option, das Konto weiteren Projekten zuzuordnen.

      1. Öffnen Sie in der Navigationsleiste das Fenster “Identity” und klicken Sie auf Users.
      2. Klicken Sie auf Create User.
      3. Geben Sie im Fenster Create User einen Benutzernamen, eine E-Mail-Adresse und ein vorläufiges Kennwort für den Benutzer ein. Bestätigen Sie das Kennwort. Wählen Sie in der Liste mit den primären Projekten ein Projekt aus. Wählen Sie in der Rollenliste eine Rolle für den Benutzer aus. Der Standardwert ist _member_.
      4. Klicken Sie auf Create User, um die Änderungen zu bestätigen. Das Dashboard weist dem Benutzer eine ID zu und der Benutzer wird in der Kategorie “Users” angezeigt.

      Benutzer inaktivieren bzw. aktivieren

      Wenn Ihnen die Rolle cloud_admin oder project_admin zugewiesen ist, können Sie Benutzerkonten aktivieren und inaktivieren. Wenn Sie ein Benutzerkonto inaktivieren, kann sich der Benutzer nicht mehr anmelden. Sie können den Benutzer jedoch jederzeit erneut aktivieren, da die Benutzerdaten nicht gelöscht werden.

      1. Öffnen Sie in der Navigationsleiste das Fenster “Identity” und klicken Sie auf Users.
      2. Suchen Sie den Benutzer, der inaktiviert werden soll, oder aktivieren Sie die Benutzerliste.
      3. Wählen Sie in der Dropdown-Liste unter der Spalte “Actions” Disable User oder Enable User aus. In der Spalte “Enabled” wird der Wert in “Yes” bzw. “No” geändert.
