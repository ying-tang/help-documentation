      Was ist SAML-Föderation?

      Security Assertion Markup Language (SAML) ist ein Protokoll, das es einem Benutzer ermöglicht, Single Sign-On (SSO) zusammen mit einem Identitätsprovider (IDP) und einem Serviceprovider (SP) zu verwenden. Wenn sich ein Benutzer anmeldet, werden seine Berechtigungsnachweise an den IDP übergeben. Der IDP überprüft die Berechtigungsnachweise und dann wird eine SAML-Zusicherung mit den Attributen des Benutzers erstellt. Diese Zusicherung wird dann an den SP übergeben und der SP prüft, ob diese vom IDP gesendet wurde. Da der SP Zusicherungen vom IDP anerkennt, muss der Benutzer seine Berechtigungsnachweise nie direkt an den SP senden.

      Keystone tritt als SP auf und authentifiziert Benutzer durch den Empfang von Zusicherungen von einem IDP. In dieser Dokumentation wird der fiktive SAML-IDP Company XYZ IDP verwendet.

      Beachten Sie, dass alle Verwaltungsbefehle auf dieser Seite voraussetzen, dass der Benutzer cloud_admin-Zugriff hat.

      IDP und Keystone integrieren

      Ein Bluemix Private Cloud-Operator muss zusammen mit einem Integrationstechniker des Company XYZ IDP beide Systeme für die Zusammenarbeit einrichten.

      Der Bluemix Private Cloud-Operator benötigt die folgenden Informationen:

      • URL-Endpunkt der IDP-Metadaten (momentan wird der Abruf von Metadaten nur über eine URL und nicht über eine Datei unterstützt)
      • IDP-Entitäts-ID aus den IDP-Metadaten
      • Benutzerattribute - Hierbei handelt es sich um die SAML-Benutzerattribute, die vom IDP übergeben werden. Sie finden diese in der IDP-Dokumentation oder in IDP-serverseitigen Protokollen.
      • Standard-/Anfangszuordnung - Die Dokumentation zu Zuordnungen finden Sie unter http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination.

      Der Integrationstechniker des Company XYZ IDP benötigt unter Umständen die folgenden Informationen vom Bluemix Private Cloud-Operator:

      • URL der SP-Metadaten
      • SP-Umleitungs-URL (wird auch als Callback-URL bezeichnet; eine in einer Whitelist aufgeführte URL, zu der der IDP die Umleitung zulässt)

      Der Bluemix Private Cloud-Operator muss zusammen mit dem Integrationstechniker die Anfangs- bzw. Standardzuordnung erstellen. Diese Zuordnung legt fest, wie die fernen Benutzer im IDP lokalen Gruppen im SP zugeordnet werden. Aus dem verbleibenden Teil dieses Dokuments wird verständlich, was eine Zuordnung ist und wie sie verwendet wird. Der Integrationstechniker benötigt ebenfalls cloud_admin-Zugriff. Hierfür kann der Benutzer cloud_admin oder ein Benutzer mit der gleichnamigen Rolle verwendet werden.

      Die Benutzerattribute müssen mit dem Zuordnungsnamen angegeben werden, z. B.:

      attributes:
- name: http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress
  id: email
- name: http://schemas.xmlsoap.org/claims/Group
  id: group

      Der angegebene Name entspricht dem Namen in der SAML-Zusicherung. Die ID ist die ID, die in den Keystone-Zuordnungen verwendet wird.

      Nachdem der Bluemix Private Cloud-Operator die Integration des Company XYZ IDP mit Keystone durchgeführt hat, wurde eine Zuordnung mapping-for-company-xyz-idp (oder ein ähnlicher Zuordnungsname mit dem IDP-Namen) erstellt.

      Im Folgenden sind die Schritte für die Verwaltung der Zuordnung aufgelistet:

      1. Gruppen verwalten (föderierte Benutzer haben dieselben Zugriffsrechte wie die Gruppe) 2. Zuordnungen verwalten

      Gruppen verwalten

      Bei der Keystone-Föderation werden IDP-Benutzer in eine Gruppe im Service-Provider eingebunden. Der Gruppe werden Zugriffsrechte erteilt, indem dieser Gruppe eine Rolle in einem Projekt zugewiesen wird. Die folgenden Rollen sind verfügbar:

      • project_admin - Die Administratorrolle für das Projekt.
      • cloud_admin - Die Administratorrolle für die Cloudinstanz.
      • _member_ - Ein Mitglied eines Projekts.
      • heat_stack_owner - Ein Eigner für Heat-Stacks.

      Gruppen auflisten:

      $ openstack group list

      Gruppe erstellen:

      $ openstack group create --help
$ openstack group create <Name der neuen Gruppe>

      Rollen auflisten:

      $ openstack role list

      Rollenzuweisungen auflisten:

      $ openstack role assignment list --name

      Projekte auflisten:

      $ openstack project list

      Hilfe zum Suchen und Hinzufügen von Rollenzuweisungen anzeigen:

      $ openstack role add --help

      Rollenzuweisung hinzufügen, um einer Gruppe eine Rolle im Projekt zuzuordnen:

      $ openstack role  add <Rolle> --project <project> --group <Gruppe>

      Föderierte Benutzer werden Gruppen im Service-Provider zugeordnet.

      Im Folgenden ist die Reihenfolge der Operationen für die Erteilung des Zugriffs für Gruppen beschrieben:

      1. Zuerst erstellen oder bearbeiten Sie die Gruppe. 2. Anschließend erstellen oder bearbeiten Sie die Rollenzuweisungen für Gruppen in Projekten.

      Beispiel 1. Gruppe company_xyz_admin_group erstellen

      $ source cloud_adminrc # Ihre OpenStack-Berechtigungsnachweisdatei
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      Als Nächstes erstellen Sie die Rollenzuweisung, sofern diese noch nicht existiert. Die folgenden Befehle fügen die Rolle ‘cloud_admin’ für die Gruppe ‘cloud_admin’ für das Projekt “demo” hinzu.

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      Nach der Ausführung des Befehls openstack role assignment list sollen Sie eine Ausgabe wie die folgende sehen:

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      Beispiel 2. Gruppe mit Mitgliederzugriff erstellen

      $ source cloud_adminrc # Ihre OpenStack-Berechtigungsnachweisdatei
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      Nach der Ausführung des Befehls openstack role assignment list sollen Sie eine Ausgabe wie die folgende sehen:

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      Zuordnungen verwalten

      Im Folgenden wird Company XYZ IDP verwendet und es ist bereits eine Zuordnung vorhanden. Ein Benutzer mit cloud_admin-Zugriff (Benutzer oder Rolle) kann die Zuordnung bearbeiten. Der Benutzer kann keine Zuordnungen erstellen oder löschen.

      Zuordnungen können in Horizon über Identity Dashboard -> Federation -> Mappings -> Edit bearbeitet werden.

      Der Befehl openstack mapping list zeigt eine Liste mit Zuordnungen an. In der Liste sollte die Zuordnung mapping-for-company-xyz-idp aufgeführt sein.

      Der Befehl openstack mapping show mapping-for-company-xyz-idp -f json zeigt die Details der angegebenen Zuordnung an. Die Details sollten eine json-Liste mit den Regeln für die Zuordnung enthalten.

      Der Befehl openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json aktualisiert die Zuordnung und legt sie fest.

      Die Zuordnungsdatei sollte das folgende Format haben:

      [
   {
      "local": [
        <Gruppe>
      ],
      "remote": [
        <Bedingung>
      ]
   }
]

      Die Regeldatei sollte eine Liste mit den Regeln enthalten, die die fernen Benutzer einer lokalen Gruppe zuordnen. Der Abschnitt local enthält eine Regel, die die Gruppe angibt, der die fernen Benutzer zugeordnet werden sollen. Die lokale Gruppe ist erforderlich.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      Der Abschnitt remote enthält Regeln, die angeben, welchen fernen Benutzern der Zugriff erteilt werden soll.

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      Die möglichen Benutzerattributtypen, die in den fernen Regeln angegeben werden können, sind im Folgenden aufgelistet:

      Die möglichen Aktionen zum Filtern der Einträge nach fernen Attributen sind im Folgenden aufgelistet:

      empty
any_ony_of
not_any_of
blacklist
whitelist

      Zum Festlegen des Benutzernamens können Sie {0} im Abschnitt local verwenden, um anzugeben, dass ein Feld aus dem Abschnitt remote verwendet werden soll. Anschließend können Sie dem neuen föderierten Benutzernamen weitere Informationen hinzufügen. Sie können den Abschnitt remote im Abschnitt local per Index (“{i}” referenzieren, wobei i für den Index des fernen Felds steht):

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "email"
      },
               ...
    ]
  }
]

      Horizon verwenden

      Das OpenStack-Dashboard im Identitätsprovider enthält ein Dropdown-Menü mit dem Namen Authenticate with SSO.

      • In dem Dropdown-Menü sollte ein Eintrag für Company XYZ IDP angezeigt werden. Wählen Sie den IDP aus und klicken Sie auf Connect.
      • Daraufhin werden Sie auf die IDP-Anmeldeseite umgeleitet.
      • Nach erfolgreicher Authentifizierung beim IDP werden Sie auf die Horizon-Seite umgeleitet.

      CLI verwenden

      Anmerkung: Der CLI-Client (Command-Line Interface, Befehlszeilenschnittstelle) funktioniert nur mit IDPs, die die folgenden Bedingungen erfüllen: 1. Der IDP unterstützt das SAML-ECP-Protokoll. 2. Der IDP unterstützt die Basisauthentifizierung in einer GET-Anforderung.

      Die CLI bietet zwei Typen von Plug-ins: 1. v3unscopedsaml (Token ohne Bereichsangabe verwenden) 2. v3scopedsaml (Token zum ausführen bereichsorientierter Aktionen verwenden)

      **1. Nicht bereichsorientiertes Beispiel: **

      Erstellen Sie eine Datei, die Sie in Ihrer Umgebung im aktuellen Kontext ausführen können, um SAML zu verwenden:

      # This file is called companyxyz_saml_rc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=CompanyXYZPassword #replace this
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 #replace this
export OS_USERNAME=CompanyXYZUser #replace this
export OS_IDENTITY_PROVIDER=company-xyz-idp #replace this
export OS_PROTOCOL=saml2
export OS_AUTH_TYPE=v3unscopedsaml
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP #replace this

      Nachdem Sie die Datei erstellt und die Optionen bearbeitet haben, können Sie den folgenden Befehl ausführen, um ein Token ohne Bereichsangabe abzurufen und die ID zu erfassen:

      $ source companyxyz_saml_rc
$ openstack token issue

      2. Bereichsorientiertes Beispiel:

      Da Sie jetzt ein Token ohne Bereichsangabe verwenden, können Sie es einem Projekt zuordnen und die Server in diesem Projekt auflisten.

      openstack server list --os-auth-type v3scopedsaml --os-token <ID_des_bereichsorientierten_Token> --os-project-id <Projekt-ID>

      Weitere Hinweise

      Momentan existiert ein Problem mit föderierten Benutzern und mit dem Trustor/Trustee-Feature in Keystone, das verhindert, das der föderierte Heat-Benutzer seine heat_stack_owner-Rolle delegiert.

      Sie müssen eine Ausweichlösung für die Delegierung der Vertrauensstellung verwenden und die heat_stack_owner-Rolle dem Benutzer direkt zuordnen. Dies ist der Benutzer, der während der föderierten Anmeldung erstellt wird. Die Domäne des Benutzers ist föderiert und könnte den Wert None (Federation=None) haben.

      $ openstack user list

      Der föderierte Benutzer muss die Domäne None haben: openstack user show <föderierter Benutzer>

      Jetzt benötigen Sie die Benutzer-ID:

      $ openstack role add heat_stack_owner --user <ID des föderierten Benutzers> --project <Projekt>

      Ein weiteres bekanntes Problem ist, dass ein föderierter Benutzer mit der Option zum Ändern seines Kennworts auf der Einstellungsseite dargestellt wird. Ein föderierter Benutzer ist aber nicht in der Lage, sein Kennwort zu ändern.
