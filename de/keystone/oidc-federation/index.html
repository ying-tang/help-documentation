      Was ist Keystone-Föderation?

      Die Keystone-Föderation ermöglicht Ihnen, Identitäten eines Identitätsproviders (IDP) in einem Service-Provider (SP) zu verwenden. Die unterstützten Protokolle und Formate sind OpenIDConnect (OIDC) und Security Assertion Markup Language (SAML).

      Der IDP speichert und verwaltet die Berechtigungsnachweise des Benutzers und sendet so genannte Ansprüche (oder Claims) oder Zusicherungen an den SP. Dies ermöglicht dem Kunden, seine Unternehmensberechtigungsnachweise für die Authentifizierung bei Bluemix Private Cloud zu verwenden, ohne sein Kennwort an Bluemix Private Cloud senden zu müssen. Keystone ist gewöhnlich der SP, da Keystone Identitäten aus externen Ressourcen konsumiert.

      Was ist OpenID Connect (OIDC)?

      OpenID ist ein Protokoll für die Authentifizierung. OAuth ist ein Protokoll für die Berechtigung. OpenID Connect ist beides.

      OpenID Connect ist eine einfache Identitätsschicht oberhalb des Protokolls OAuth 2.0. Mit OpenID Connect können Clients die Identität eines Endbenutzers basierend auf der von einem Berechtigungsserver durchgeführten Authentifizierung verifizieren. Außerdem können die Clients in einer interoperablen und REST-ähnlichen Weise Basisprofilinformationen zum Endbenutzer anfordern.

      In diesem Dokument wird ein Mechanismus beschrieben, mit dem eine OpenID Connect-Relying-Party den OpenID-Provider des Endbenutzers erkennen und die Informationen abrufen kann, die für die Interaktion erforderlich sind, wie z. B. die OAuth 2.0-Endpunktpositionen.

      Wie funktioniert das OpenID-Protokoll?

      1. Die Relying Party sendet die Anforderung zum Authentifizieren des Endbenutzers an den OpenID-Provider.
      2. Der OpenID-Provider authentifiziert den Benutzer.
      3. Der OpenID-Provider sendet das ID-Token und das Zugriffstoken an die Relying Party.
      4. Die Relying Party sendet eine Anforderung mit dem vom OpenID-Provider empfangenen Zugriffstoken an den Endpunkt für Benutzerinformationen.
      5. Der Endpunkt für Benutzerinformationen gibt die Ansprüche zurück.

      OIDC einrichten

      Ein Blue Box-Operator muss mit einem IDP-Integrationstechniker des Kundenunternehmens (z. B. Company XYZ) zusammenarbeiten, um OIDC einzurichten.

      Der Bluemix Private Cloud-Operator benötigt die folgenden Informationen:

      • URL für berechtigte Umleitungen/Callbacks
      • Client-ID und geheimer Clientschlüssel
      • Benutzeranspruch - OIDC-Anspruch
      • Bekannte/Konfiguration-URL, die alle erforderlichen Informationen für den Zugriff auf den Provider bereitstellt
      • Standard-/Anfangszuordnung - Die Dokumentation zu Zuordnungen finden Sie unter http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination.

      Der Bluemix Private Cloud-Operator muss zusammen mit dem Integrationstechniker die Anfangs- bzw. Standardzuordnung erstellen. Diese Zuordnung legt fest, wie die fernen Benutzer im IDP lokalen Gruppen im SP zugeordnet werden. Aus dem verbleibenden Teil dieses Dokuments wird verständlich, was eine Zuordnung ist und wie sie verwendet wird.

      Nachdem der Bluemix Private Cloud-Operator die Integration des Company XYZ IDP mit Keystone durchgeführt hat, wurde eine Zuordnung mapping-for-company-xyz-idp (oder ein ähnlicher Zuordnungsname mit dem IDP-Namen) erstellt.

      Im Folgenden sind die Schritte für die Verwaltung der Zuordnung aufgelistet:

      1. Gruppen verwalten (föderierte Benutzer haben dieselben Zugriffsrechte wie die Gruppe)

      2. Zuordnungen verwalten

      Gruppen verwalten

      Bei der Keystone-Föderation werden IDP-Benutzer in eine Gruppe im Service-Provider eingebunden. Der Gruppe werden Zugriffsrechte erteilt, indem dieser Gruppe eine Rolle in einem Projekt zugewiesen wird. Die folgenden Rollen sind verfügbar:

      • project_admin - Die Administratorrolle für das Projekt.
      • cloud_admin - Die Administratorrolle für die Cloudinstanz.
      • _member_ - Ein Mitglied eines Projekts.
      • heat_stack_owner - Ein Eigner für Heat-Stacks.

      Gruppen auflisten:

      $ openstack group list

      Gruppe erstellen:

      $ openstack group create --help
$ openstack group create <Name der neuen Gruppe>

      Rollen auflisten:

      $ openstack role list

      Rollenzuweisungen auflisten:

      $ openstack role assignment list --name

      Projekte auflisten:

      $ openstack project list

      Hilfe zum Suchen und Hinzufügen von Rollenzuweisungen anzeigen:

      $ openstack role add --help

      Rollenzuweisung hinzufügen, um einer Gruppe eine Rolle im Projekt zuzuordnen:

      $ openstack role  add <Rolle> --project <project> --group <Gruppe>

      Föderierte Benutzer werden Gruppen im Service-Provider zugeordnet. Im Folgenden ist die Reihenfolge der Operationen für die Erteilung des Zugriffs für Gruppen beschrieben:

      1. Zuerst erstellen oder bearbeiten Sie die Gruppe.

      2. Anschließend erstellen oder bearbeiten Sie die Rollenzuweisungen für Gruppen in Projekten.

      Beispiel 1. Gruppe company_xyz_admin_group erstellen

      $ source cloud_adminrc # Ihre OpenStack-Berechtigungsnachweisdatei
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      Als Nächstes erstellen Sie die Rollenzuweisung, sofern diese noch nicht existiert.

      Die folgenden Befehle fügen die Rolle ‘cloud_admin’ für die Gruppe ‘cloud_admin’ für das Projekt “demo” hinzu.

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      Nach der Ausführung des Befehls openstack role assignment list sollen Sie eine Ausgabe wie die folgende sehen:

+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      Beispiel 2. Gruppe mit Mitgliederzugriff erstellen

      $ source cloud_adminrc # Ihre OpenStack-Berechtigungsnachweisdatei
$ openstack group list
$ openstack group create company_xyz_member_group --or-show

$ openstack role add _member_ --group company_xyz_member_group --project demo

| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      Zuordnungen verwalten

      Im Folgenden wird das Beispiel Company XYZ IDP verwendet und angenommen, dass bereits eine Zuordnung vorhanden ist. Ein Benutzer mit cloud_admin-Zugriff (Benutzer oder Rolle) kann die Zuordnung bearbeiten. Der Benutzer kann keine Zuordnungen erstellen oder löschen.

      Zuordnungen können in Horizon über Identity Dashboard -> Federation -> Mappings -> Edit bearbeitet werden.

      Der Befehl openstack mapping list zeigt eine Liste mit Zuordnungen an. In der Liste sollte die Zuordnung mapping-for-company-xyz-idp aufgeführt sein.

      Der Befehl openstack mapping show mapping-for-company-xyz-idp -f json zeigt die Details der angegebenen Zuordnung an. Die Details sollten eine json-Liste mit den Regeln für die Zuordnung enthalten.

      Der Befehl openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json aktualisiert die Zuordnung und legt sie fest.

      Die Zuordnungsdatei sollte das folgende Format haben:

      [
   {
      "local": [
        <Gruppe>
      ],
      "remote": [
        <Bedingung>
      ]
   }
]

      Die Regeldatei sollte eine Liste mit den Regeln enthalten, die die fernen Benutzer einer lokalen Gruppe zuordnen. Der Abschnitt local enthält eine Regel, die die Gruppe angibt, der die fernen Benutzer zugeordnet werden sollen. Die lokale Gruppe ist erforderlich.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      Der Abschnitt remote enthält Regeln, die angeben, welchen fernen Benutzern der Zugriff erteilt werden soll.

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      Die möglichen Benutzerattributtypen, die in den fernen Regeln angegeben werden können, sind im Folgenden aufgelistet:

      Die möglichen Aktionen zum Filtern der Einträge nach fernen Attributen sind im Folgenden aufgelistet:

      empty
any_ony_of
not_any_of
blacklist
whitelist

      Zum Festlegen des Benutzernamens können Sie {0} im Abschnitt local verwenden, um anzugeben, dass ein Feld aus dem Abschnitt remote verwendet werden soll. Anschließend können Sie dem neuen föderierten Benutzernamen weitere Informationen hinzufügen. Sie können den Abschnitt remote im Abschnitt local per Index (“{i}” referenzieren, wobei i für den Index des fernen Felds steht):

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "HTTP_OIDC_CLAIM_EMAIL"
      },
               ...
    ]
  }
]

      Horizon verwenden

      Das OpenStack-Dashboard im Identitätsprovider enthält ein Dropdown-Menü mit dem Namen Authenticate with SSO.

      1. In dem Dropdown-Menü sollte ein Eintrag für den IdP angezeigt werden. Wählen Sie den IDP aus und klicken Sie auf Connect.

      2. Daraufhin werden Sie auf die IDP-Anmeldeseite umgeleitet.

      3. Nach erfolgreicher Authentifizierung beim IDP werden Sie auf die Horizon-Seite umgeleitet.

      Beispiel 4. API für den Berechtigungscodeablauf verwenden

      Bei dieser Datei handelt es sich um oidc-authflow.py.

      from keystoneauth1.identity.v3 import oidc
from keystoneauth1 import session
from requests_oauthlib import OAuth2Session
from keystoneauth1.identity.v3 import Token
from novaclient import client
import os

def main():

    auth_url = os.environ.get('OS_AUTH_URL')
    identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
    protocol = 'oidc'
    redirect_uri = os.environ.get('OS_REDIRECT_URI')
    scope = ['openid','profile','email']
    client_id = os.environ.get('OS_CLIENT_ID')
    client_secret = os.environ.get('OS_CLIENT_SECRET')
    discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')
    access_token_endpoint = os.environ.get('OS_TOKEN_ENDPOINT')
    authorization_endpoint = os.environ.get('OS_AUTHORIZATION_ENDPOINT')
    project_id = os.environ.get('OS_PROJECT_ID')

      OAuth-CLI verwenden

      Berechtigungscodeablauf für die Befehlszeile

      Voraussetzung: Version von python-openstackclient > 3.0.0 Überprüfen Sie die Version mit dem folgenden Befehl: pip freeze | grep openstackclient

      Schritte zur Verwendung des OpenStack-Client mit Berechtigungscode:

      1. Erstellen Sie eine Datei mit dem Namen oidcrc.

      2. Fügen Sie den folgenden Inhalt in die Datei ein. Aktualisieren Sie die Parameter basierend auf Ihrer Umgebung.

        OIDC stackrc

         export OS_TOKEN_ENDPOINT=https://<replace>/oauth/token
 export OS_CLIENT_SECRET=<geheimer Clientschlüssel>
 export OS_CLIENT_ID=<Client-ID>
 export OS_DISCOVERY_ENDPOINT=https://<ersetzen>/.well-known/openid-configuration
 export OS_IDENTITY_PROVIDER=auth0
 export OS_REDIRECT_URI=<Umleitungs-URI>
 export OS_IDENTITY_API_VERSION=3
 export OS_DOMAIN=Default
 export OS_PROTOCOL=oidc
 export OS_AUTH_URL=https://<FQDN>:5000/v3
 export OS_USERNAME=<Benutzername>
 export OS_AUTHORIZATION_ENDPOINT=https://<ersetzen>/authorize

      3. Fordern Sie einen Berechtigungscode von Ihrem Identitätsmanager an.

      4. Der folgende Befehl ruft ein Keystone-Token ab, das Sie bis zu dessen Ablauf für die Ausführung jedes Befehls verwenden können:

          openstack --os-auth-type v3oidcauthcode --os-code <Berechtigungscode> --os-project-name <Projektname> --os-project-domain-name Default token issue

      Beispiel:

      root@allinone-multiple:~# openstack --os-auth-type v3oidcauthcode --os-code 6LxfzO7GvCjmmcRa --os-project-name admin --os-project-domain-name Default token issue

      openstack --os-auth-type v3token --os-token <Token-ID aus vorheriger Anforderung> --os-project-name <Projektname> --os-project-domain-name Default user list

      Weitere Hinweise

      Momentan existiert ein Problem mit föderierten Benutzern und mit dem Trustor/Trustee-Feature in Keystone, das verhindert, das der föderierte Heat-Benutzer seine heat_stack_owner-Rolle delegiert.

      Sie müssen eine Ausweichlösung für die Delegierung der Vertrauensstellung verwenden und die heat_stack_owner-Rolle dem Benutzer direkt zuordnen. Dies ist der Benutzer, der während der föderierten Anmeldung erstellt wird. Die Domäne des Benutzers ist föderiert und könnte den Wert None (Federation=None) haben.

      $ openstack user list

      Der föderierte Benutzer muss die Domäne None haben: openstack user show *föderierter Benutzer*

      Jetzt benötigen Sie die Benutzer-ID:

      $ openstack role add heat_stack_owner --user *ID des föderierten Benutzers* --project *Projekt*

      Ein weiteres bekanntes Problem ist, dass ein föderierter Benutzer mit der Option zum Ändern seines Kennworts auf der Einstellungsseite dargestellt wird. Ein föderierter Benutzer ist aber nicht in der Lage, sein Kennwort zu ändern.
