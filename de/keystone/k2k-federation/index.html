      Was ist K2K-Föderation?

      K2K-Föderation (Keystone to Keystone) ist eine Methode, Berechtigungsnachweise einer Keystone-Instanz für die Anmeldung an mehreren Instanzen von IBM Bluemix Private Cloud zu verwenden. Eine Instanz von Bluemix Private Cloud wird als Identity Provider bezeichnet. Andere Cloudinstanzen werden als Service-Provider bezeichnet. Der Identity Provider speichert die Benutzerberechtigungsnachweise und sichert den Service-Providern zu, dass der Benutzer tatsächlich ein gültiger Benutzer ist. Die Benutzer-ID und das Kennwort werden nie an die Service-Provider gesendet, sondern lediglich die Zusicherungen des Identity Providers.

      Wie funktioniert K2K?

      Der Benutzer meldet sich am Identity Provider an und erhält eine signierte SAML-Zusicherung von diesem. Mit dieser SAML-Zusicherung authentifiziert er sich bei den Service-Providern. Der Anmeldeprozess zur Handhabung des SAML-Ablaufs wird vom K2K-Authentifizierungs-Plug-in abgewickelt (siehe keystoneauth1). Berechtigungsnachweise werden nur im Identity Provider verwaltet und von Service-Providern über Zusicherungen akzeptiert.

      Wenn mehrere Instanzen von Bluemix Private Cloud mit dem K2K-Feature konfiguriert wurden, wird eine Cloudinstanz als Identity Provider bezeichnet und die anderen als Service-Provider. Service-Provider verwenden Zuordnungen, um Benutzerattribute aus dem Identity Provider auszuwählen, die für die Zuordnung einer lokalen Gruppe verwendet werden. Wenn sich der Benutzer über die K2K-Föderation anmeldet, werden ihm die Rollen, die der Gruppe zugeordnet sind, zugewiesen.

      Standardmäßig wird die Zuordnung mapping-for-k2k-federation für die Berechtigungsnachweise für cloud_admin auf dem Identity Provider erstellt, wodurch diese der lokalen Gruppe cloud_admin zugeordnet werden. Diese Gruppe hat durch die Rolle cloud_admin nun Zugriff auf das Demoprojekt.

      Im Folgenden finden Sie den Ablauf zum Einrichten der K2K-Föderation:

      1. Verwalten Sie zunächst Gruppen auf dem Service-Provider (wobei föderierte Benutzer dieselben Zugriffsrechte haben wie die Gruppe).

      2. Verwalten Sie anschließend Zuordnungen auf dem Service-Provider (wobei ferne Identity-Provider-Benutzer lokalen Service-Provider-Gruppen zugeordnet werden).

      Gruppen verwalten

      K2K-Föderation funktioniert durch das Föderieren von Identity-Provider-Benutzern mit einer Gruppe auf dem Service-Provider. Die Gruppe erhält Zugriff, indem Sie ihr eine Rolle für ein Projekt zuweisen. Die folgenden Rollen sind verfügbar:

      • project_admin - Die Administratorrolle für das Projekt.
      • cloud_admin - Die Administratorrolle für die Cloudinstanz.
      • _member_ - Ein Mitglied des Projekts.
      • heat_stack_owner - Ein Heat-Stack-Eigner.

      Eine Quellendatei mit den Berechtigungsnachweisen ist erforderlich, um die Gruppenbefehle verwenden zu können. Sie müssen eine Datei erstellen, die die Berechtigungsnachweise enthält, und die Tags in spitzen Klammern < > im folgenden Beispiel durch die entsprechenden Informationen ersetzen.

      #This file is cloud_adminrc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=<cloud_admin password> #Hier Kennwort von cloud_admin einfügen
export OS_AUTH_URL=https://<vollständig qualifizierter Domänenname des Service-Providers>:5000/v3
export OS_USERNAME=cloud_admin
export OS_TENANT_NAME=demo
export OS_CACERT=/opt/stack/ssl/openstack.crt
export OS_NO_CACHE=True

      Nachdem die Quellendatei mit den Berechtigungsnachweisen erstellt wurde, können Sie die Datei als Quelle verwenden und sämtliche nachstehende OpenStack-Befehle für die Verwaltung von Gruppen und Rollen verwenden. Anmerkung: Diese Befehle werden auf dem Service-Provider ausgeführt.

      Berechtigungsnachweise in Umgebung als Quelle verwenden:

      $ source cloud_adminrc

      Gruppen auflisten:

      $ openstack group list

      Gruppe erstellen:

      $ openstack group create --help
$ openstack group create <Neuer_Gruppenname>

      Rollen auflisten:

      $ openstack role list

      Rollenzuordnungen auflisten:

      $ openstack role assignment list --name

      Projekte auflisten:

      $ openstack project list

      Hilfe zum Hinzufügen einer Rollenzuordnung anzeigen:

      $ openstack role add --help

      Rollenzuordnung zu einer Gruppe hinzufügen, um die Rolle einem Projekt zuzuordnen:

      $ openstack role  add <Rolle> --project <Projekt> --group <Gruppe>

      Föderierte Benutzer werden Gruppen auf dem Service-Provider zugeordnet. Die Gruppe cloud_admin ist standardmäßig vorhanden, wenn K2K-Föderation aktiviert ist. Diese Gruppe hat die Rolle cloud_admin für das Projekt “demo”.

      Hier finden Sie die Vorgehensweise, um Gruppen Zugriff zu erteilen:

      1. Zunächst erstellen bzw. bearbeiten Sie die Gruppe.
      2. Anschließend erstellen bzw. bearbeiten Sie die Rollenzuordnungen von Gruppen zu Projekten.

      Beispiel 1: Gruppe cloud_admin erstellen

      Mit den nachfolgenden Befehlen legen Sie eine Datei mit den Authentifizierungsinformationen für den Service-Provider-Benutzer cloud_admin als Quelle fest.

      $ source cloud_adminrc
$ openstack group list
$ openstack group create cloud_admin --or-show
$ openstack role assignment list --name --group cloud_admin

      Erstellen Sie jetzt die Rollenzuordnung, falls diese noch nicht vorhanden ist. Mit den nachfolgenden Befehlen fügen Sie die Rolle “cloud_admin” der Gruppe “cloud_admin” des Projekts “demo” hinzu.

      $ openstack role add cloud_admin --group cloud_admin --project demo
$ openstack role assignment list --name --group cloud_admin

      Nach Ausführung des Befehls openstack role assignment list wird eine ähnliche Ausgabe wie die folgende angezeigt:

      # +-------------+------+---------------------+--------------+--------+-----------+
# | Role        | User | Group               | Project      | Domain | Inherited |
# +-------------+------+---------------------+--------------+--------+-----------+
# | cloud_admin |      | cloud_admin@Default | demo@Default |        | False     |
# +-------------+------+---------------------+--------------+--------+-----------+

      Beispiel 2:Gruppe mit Mitgliedszugriff erstellen

      Mit den nachfolgenden Befehlen legen Sie eine Datei mit den Authentifizierungsinformationen für den Service-Provider-Benutzer cloud_admin als Quelle fest.

      $ source cloud_adminrc
$ openstack group list
$ openstack group create demo_member_group --or-show
$ openstack role assignment list --name --group demo_member_group
$ openstack role add _member_ --group demo_member_group --project demo
$ openstack role assignment list --name --group demo_member_group

      Nach Eingabe des Befehls openstack role assignment list wird eine ähnliche Ausgabe wie die folgende angezeigt:

      # +----------+------+---------------------------+--------------+--------+-----------+
# | Role     | User | Group                     | Project      | Domain | Inherited |
# +----------+------+---------------------------+--------------+--------+-----------+
# | _member_ |      | demo_member_group@Default | demo@Default |        | False     |
# +----------+------+---------------------------+--------------+--------+-----------+

      Zuordnungen verwalten

      Der Service-Provider-Benutzer cloud_admin hat Zugriff zum Lesen und Festlegen der Zuordnungen für die K2K-Föderation. Beachten Sie, dass der Service-Provider-Benutzer cloud_admin keine bestehenden Zuordnungen löschen oder neue Zuordnungen erstellen kann.

      Eine Quellendatei mit den V3-Berechtigungsnachweisen ist erforderlich, um die Zuordnungsbefehle verwenden zu können. Sie müssen eine Datei erstellen, die die Berechtigungsnachweise enthält, und die Tags in spitzen Klammern < > im folgenden Beispiel durch die entsprechenden Informationen ersetzen.

      Weitere Informationen zum Verwalten von Zuordnungen finden Sie in der Dokumentation zu OpenStack unter den folgenden Links: http://docs.openstack.org/developer/keystone/mapping_combinations.html http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combinations

      Wenn sich ein Benutzer über die K2K-Föderation am Service-Provider mit seinem Identity-Provider-Konto anmeldet, wird lokal auf dem Service-Provider ein Benutzerkonto erstellt. Beachten Sie, dass das während der Föderation erstellte Benutzerkonto ephemer ist. Es kann zu einem späteren Zeitpunkt ohne Warnung gelöscht werden.

      In der Standardzuordnung ist der Benutzer cloud_admin auf dem Identity Provider föderiert. Daher wird in der föderierten Domäne ein lokaler Benutzer cloud_admin erstellt. Dieser Benutzer unterscheidet sich von dem Benutzer cloud_admin in der Standarddomäne. Der ephemere Benutzer cloud_admin in der föderierten Domäne kann zu einem späteren Zeitpunkt gelöscht werden.

      Sie benötigen die Quellendatei aus dem vorherigen Abschnitt, um die folgenden Zuordnungsbefehle ausführen zu können:

      $ source cloud_adminrc
$ openstack mapping list
$ openstack mapping show mapping-for-k2k-federation
$ openstack mapping show mapping-for-k2k-federation --format json
$ openstack mapping show --help
$ openstack mapping set mapping-for-k2k-federation --rules <Zuordnungsdatei>

      Der Befehl openstack mapping list zeigt eine Zuordnungsliste an. In dieser sollte die Zuordnung mapping-for-k2k-federation zu finden sein.

      Der Befehl openstack mapping show mapping-for-k2k-federation zeigt die Details zur angegebenen Zuordnung. Sie sollten eine json-Liste der Zuordnungsregeln enthalten.

      Der Befehl openstack mapping set mapping-for-k2k-federation --rules newmappings.json zeigt, wie Sie die Zuordnung mapping-for-k2k-federation aktualisieren und einstellen können.

      Die Zuordnungsdatei sollte das folgende Format haben:

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      Die Regeldatei muss eine Liste der Regeln enthalten, die die fernen Benutzer einer Gruppe auf dem Service-Provider zuordnen. Der Abschnitt “local” enthält eine Regel, die die Gruppe angibt, der die fernen Benutzer zugeordnet werden sollen. Die lokale Gruppenregel muss angegeben sein.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "cloud_admin"
  }
}

      Der Abschnitt “remote” schließt Regeln ein, die angeben, welchen fernen Benutzern Zugriff gewährt werden soll.

      {
    "type": "openstack_user",
    "any_one_of": [
        "cloud_admin"]
}

      Die möglichen Benutzerattributtypen, die in den Remote-Regeln verwendet werden können, sind:

      openstack_user
openstack_user_domain
openstack_roles
openstack_project
openstack_project_domain

      Die möglichen Aktionen zum Filtern nach Remote-Attributen sind:

      empty
any_ony_of
not_any_of
blacklist
whitelist

      Beim Konfigurieren des Benutzernamens kann der Wert {0} im Abschnitt “local” verwendet werden, um anzugeben, dass ein Feld aus dem fernen Abschnitt verwendet werden soll. Sie können zusätzliche Informationen zum neuen föderierten Benutzernamen hinzufügen. Für die K2K-Föderation ist es wichtig, dass Informationen hinzugefügt werden, die angeben, welche Cloud Sie verwenden. Der Abschnitt “remote” kann im Abschnitt “local” durch den Index referenziert werden (“{i}”, wobei i der Index des Felds “remote” ist):

      [
  {
    "local": [
      {
        "user": {
          "name": "my_cloud/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
               ...
    ]
  }
]

      Im Folgenden werden einige Beispielszenarios aufgeführt:

      Beispiel 1: Den Benutzer cloud_admin der Gruppe cloud_admin zuordnen und den Benutzernamen so einrichten, dass er in der föderierten Domäne my_cloud/cloud_admin heißt. Hier finden Sie die Datei mit den Standardzuordnungen:

      [
  {
    "local": [
      {
        "user": {
          "name": "my_cloud/{0}"
        }
      },
      {
        "group": {

            "name": "Default"
          },

        }

        "type": "openstack_user"
      },

        "any_one_of": [
          "cloud_admin"
        ]
      }
    ]
  }
]

      Beispiel 2: Zuordnung von Mitgliedern

      Im folgenden Beispiel wird gezeigt, wie Benutzer mit der Rolle _member_ im Projekt “demo” auf dem Identity Provider einer Gruppe (demo_member_group aus Gruppenbeispiel 2) auf dem Service-Provider zugeordnet werden. Angenommen demo_member_group ist eine Gruppe auf dem Service-Provider, der die Rolle _member_ für das Projekt “demo” auf dem Service-Provider zugewiesen ist.

      Eine Reihe von Bash-Befehlen müssen ausgeführt werden, um die folgende Zeichenfolge in einer Datei zu verknüpfen und anschließend die Zuordnung festzulegen.

      $ cat <<EOF > federated_member_mapping.json
[
  {
    "local": [
      {
        "user": {
          "name": "my_cloud/{0}"
        }
      },
      {
        "group": {
          "domain": {
            "name": "Default"
          },
          "name": "demo_member_group"
        }
      }
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
      {
        "type": "openstack_project",
        "any_one_of": [
          "demo"
        ]
      },
      {
        "type": "openstack_roles",
        "any_one_of": [
          "_member_"
        ]
      }
    ]
  }
]
EOF
$ openstack mapping set mapping-for-k2k-federation --rules federated_member_mapping.json

      {
        "type": "openstack_roles",
        "any_one_of": [
          "_member_"
        ]
      }
    ]
  }
]

      Horizon verwenden

      Das OpenStack-Dashboard auf dem Identity Provider sollte ein Dropdown-Menü mit dem Namen Authenticate with Keystone to Keystone Federation haben.

      • Wenn Sie die Option Identity Provider auswählen, wird keine Föderation verwendet. Der Benutzer wird in der OpenStack-Cloud des Identity Providers angemeldet.
      • Wenn Sie die Option Service-Provider auswählen, wird die Föderation verwendet. Die Benutzerberechtigungsnachweise des Identity Providers werden mit dem Service-Provider föderiert.
      • Um zwischen Providern zu wechseln, muss sich der Benutzer abmelden und erneut einen Provider auswählen.

      Bitte beachten Sie, dass der Horizon-Anmeldebildschirm für K2K für IBM Bluemix Private Cloud 3.0.0 kein OpenStack-Standardfeature ist und daher in späteren Releases geändert werden kann.

      Python-API-Bibliotheken verwenden

      Zunächst erstellen Sie eine Quellendatei, die Sie verwenden, um sich mit dem K2K-Authentifizierungs-Plug-in zu authentifizieren. Das Token OS_SERVICE_PROVIDER verweist auf die Service-Provider-ID, die Sie durch Ausführen des folgenden Befehls für den Identity Provider abrufen können:

      $ openstack server provider list

      Sie erstellen eine Quellendatei, mit der das Python-Script Instanzen auf dem Service-Provider mithilfe der Berechtigungsnachweise von cloud_admin aus dem Identity Provider auflistet.

      Diese Quellendatei (federationrc) enthält Authentifizierungsnachweise für die Anmeldung am Identity Provider und die Zielinformationen des Service-Providers.

      # This file is federationrc
export OS_USERNAME=cloud_admin
export OS_PASSWORD=<cloud admin password>
export OS_AUTH_URL=https://k2kf-idp.open-test.ibmcloud.com:5000/v3
export OS_PROJECT_NAME=demo
export OS_DOMAIN_NAME=Default
export OS_SERVICE_PROVIDER=<service provider id>

      Führen Sie die folgenden Befehle auf dem Identity Provider aus, um die Liste der Service-Provider anzuzeigen:

      $ openstack service provider list

      Der folgende Code verwendet das Authentifizierungs-Plug-in von Keystone-to-Keystone, um ein Token ohne Bereichsangabe abzurufen. Dieses Token ohne Bereichsangabe wird verwendet, um ein bereichsorientiertes Token zu erhalten, dessen Gültigkeitsbereich eines der Projekte ist, auf die der föderierte Benutzer Zugriff hat. Wenn Sie ein bereichsorientiertes Token haben, können Sie eine Liste der Serverinstanzen auf dem Service-Provider anzeigen.

      Datei list_servers.py erstellen:

      Um das Python-Script auszuführen, führen Sie die folgenden Befehle aus:

      $ source federationrc
$ chmod 700 list_servers.py
$ ./list_servers.py

      Weitere Zuordnungen und Gruppen

      Beispiel 1: Heat-Benutzer

      Momentan besteht ein Problem mit föderierten Benutzern und dem Verwaltungsfeature in Keystone, das verhindert, dass der föderierte Heat-Benutzer seine Rolle heat_stack_owner delegieren kann.

      Föderierte Heat-Gruppe erstellen

      Verwenden Sie eine Datei mit den Authentifizierungsinformationen für den Service-Provider-Benutzer cloud_admin als Quelle: source cloud_adminrc

      Im folgenden Beispiel wird die Rolle heat_stack_owner und NICHT heat_stack_user verwendet:

      $ openstack group list
$ openstack group create heat_stack_owner_group --or-show
$ openstack role assignment list --name --group heat_stack_owner_group
$ openstack role add heat_stack_owner --group heat_stack_owner_group --project demo
$ openstack role assignment list --name --group heat_stack_owner_group
# +------------------+------+--------------------------------+--------------+--------+-----------+
# | Role             | User | Group                          | Project      | Domain | Inherited |
# +------------------+------+--------------------------------+--------------+--------+-----------+
# | heat_stack_owner |      | heat_stack_owner_group@Default | demo@Default |        | False     |
# +------------------+------+--------------------------------+--------------+--------+-----------+

      Föderierte Heat-Zuordnungen erstellen

      $ cat <<EOF > federated_heat_stack_owner_mapping.json
[
  {
    "local": [
      {
        "user": {
          "name": my_cloud/"{0}"
        }
      },
      {
        "group": {
          "domain": {
            "name": "Default"
          },
          "name": "heat_stack_owner_group"
        }
      }
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
      {
        "type": "openstack_roles",
        "any_one_of": [
          "heat_stack_owner"
        ]
      }
    ]
  }
]
EOF
$ openstack mapping set mapping-for-k2k-federation --rules federated_heat_stack_owner_mapping.json

      Sie umgehen den Fehler bei der Delegierung der Verwaltung, indem Sie dem Benutzer direkt die Rolle heat_stack_owner zuweisen. Der Benutzer ist der, der bei der föderierten Anmeldung erstellt wurde. Die Benutzerdomäne ist föderiert und kann keinen Wert enthalten (Federation=None).

      $ openstack user list

      Für den föderierten Benutzer muss für Domäne NONE angegeben sein: openstack user show <federated user>

      Benutzer-ID anfordern:

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      Ein bekanntes Problem ist, dass einem föderierten Benutzer auf der Einstellungsseite die Option angezeigt wird, das Kennwort zu ändern. Das Kennwort kann jedoch nicht geändert werden.
