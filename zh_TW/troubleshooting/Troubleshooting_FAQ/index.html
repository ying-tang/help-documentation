      問：如何得知我的虛擬機器前次重新啟動的時間？

      答：您可以使用此指令：

      zgrep BOOT_IMAGE /var/log/kern.log*

      以查看主機 (VM) 前次啟動的時間，然後使用該時間戳記來查看日誌，檢查是否有與您可能要進行疑難排解的問題相關的任何線索。

      問：如果我的執行實例已凍結並停止回應，我該怎麼辦？

      答：建議您檢查虛擬機器的主控台，看看它是否完全凍結，或者是否有核心危急或其他錯誤的證明。您可以透過下列指令取得主控台的 URL：

      nova get-vnc-console VM-UUID novnc

      請注意主控台的時間（如果有的話）。

      擷取主控台的畫面（如果它已凍結且包含有用的資訊）。重新啟動之後，如果您已在虛擬機器上啟用當機傾出，您可以查閱 /var/crash 中的 vmcore（在 RedHat 上）或 .crash（在 Ubuntu 上）檔案，然後從那裡分析傾出。另外，也請檢查凍結時的系統活動 (sar) 統計資料。

      將機器重新開機，如果您需要它，請儘快備份，或最好是根據相同的映像檔來操作新機器。如果您需要在基礎架構上完成進一步調查，在重新開機之前，讓儘量讓我們知道凍結時的世界標準時間 (UTC) 及執行實例的 UUID，我們看看能否在日誌中找到任何蛛絲馬跡。

      如果您需要可靠地進入單一使用者模式（例如，為了執行檔案系統檢查，或修正 SSH 目錄權限），我們已成功在 Safari 瀏覽器中使用全螢幕主控台。您需要按 “send ctrl/alt/del” 按鈕，以便在登入主控台時將 VM 重新開機，這樣您才可以在 GRUB 載入畫面上擷取到它。當您進入 GRUB 開機功能表時，請按 e 來進行編輯。對於 CentOS 中的單一使用者模式，例如：

      1. 向下捲動，然後消除大部分的行尾（從主控台開始），只留下 console=tty。
      2. 編輯之後，這一行會類似這樣：kernel /boot/vmlinuz-2.6.32-431.el6.x86_64 ro root=UUID=8a057bba-2787-46a7-9436-fd6dc8a3aa85 rd_NO_LUKS rd_NO_LVM LANG=en_US.UTF-8 rd_NO_MD crashkernel=auto rd_NO_DM SYSFONT=latarcyrheb-sun16 KEYBOARDTYPE=pc KEYTABLE=us console=tty notsc single
      3. 您也可以嘗試新增 rw init=/bin/bash 來代替該行的 ro 部分。
      4. 如果開機處理程序似乎停止，請按 Enter 鍵幾次。

      問：如果需要在我的執行實例上執行救援作業，我該怎麼做？

      答：

      1. 取得救援 CD ISO，例如從 http://www.system-rescue-cd.org 取得： 
        wget https://dronedata.dl.sourceforge.net/project/systemrescuecd/sysresccd-x86/4.9.6/systemrescuecd-x86-4.9.6.iso
      2. 從 sysutils 套件安裝 isohybrid： 
        apt-get install syslinux [or yum install syslinux]
      3. 將主要開機記錄 (MBR) 加到 ISO，使其成為可開機硬碟 
        isohybrid systemrescuecd-x86-4.9.6.iso
      4. 將新的混合式救援 CD 上傳到 Glance： 
        openstack image create --file systemrescuecd-x86-4.9.6.iso --public sysrescuecd-4.9.6.iso
      5. 讓需要救援的執行實例處於救援模式： 
        nova rescue brokeninstance --image sysrescuecd-4.9.6.iso
      6. 透過 Horizon 中的 VNC 主控台登入到執行實例，並執行您需要的任何救援作業。您可能需要等待 1 到 2 分鐘，才能在主控台中看到救援 CD。看到之後，請重新載入主控台畫面。

      (/img/zh_TW/systemrescuefromvnc.png)

      您可以在此處使用諸如下列的工具：

      blkid
fsck /dev/vdb1
mount -t ntfs /dev/xxx /mnt/windows -o ro
ntfs-3g /dev/sda1 /mnt/windows
      1. 如果您對執行實例的修正情況感到滿意，請讓它離開救援模式，然後使用以下指令，透過正常開機磁碟再次啟動它： 
        nova unrescue brokeninstance

      問：為何我無法將我的執行實例從 m1.tiny 重新調整大小為 m1.medium？

      該執行實例實際上可能是在 m1.tiny 虛擬硬體範本具有較大的暫時磁碟時建立的。調整大小程序會檢查這個項目，如果找不到該值，則無法調整大小。

      您可能需要開啟支援問題單，以協助驗證這是否是您失敗的原因。支援團隊會在日誌中看到類似下面的錯誤：

      訊息處理期間發生異常狀況：調整大小錯誤：無法將磁碟大小往下調整。

      或者，有一些不錯的選項可以讓您繼續進行而不必開啟問題單，亦即，將 m1.medium 虛擬硬體範本設為跟之前的暫時磁碟一樣大小，或是遵循這裡的疑難排解暫行解決方法，從 Snapshot 中建立新執行實例：/help-documentation/gettingstarted/commontech/Instance_Resize/

      問：是什麼導致 Neutron 中的埠停滯？

      當使用 Heat 將 VM 實例化時，Neutron 中可能發生埠停滯的情況。我們相信這個問題應該與 OpenStack Neutron（Mitaka 版本）中的問題報告有關，因為 Neutron 中有競爭狀況。因此，我們將它視為一項已知限制。此狀況可能同時導致 meta 資料失敗及埠停滯。

      有數個暫行解決方法：

      暫行解決方法 #1. 可能的話，不使用 Heat 部署執行實例。手動部署似乎可避開此狀況。

      暫行解決方法 #2. 另一個暫行解決方法是使用 API 中的 WaitCondition，在 Heat 中產生延遲。不過，WaitCondition 在此實務中並非理想選項。

      • WaitCondition 可以與 WaitConditionHandler 一起使用。這個處理程式會建立 URL/URI，然後監視其中的文字字串。此字串如果出現，表示部署 VM 成功，如果沒有出現此字串，則表示失敗。當 VM 實例化之後，可透過 PUT API 呼叫，指示它在 URL/URI 中移入適當的文字字串。

      • 目前我們的 OpenStack 部署並沒有針對 L3 路由器發出 API 呼叫的機制。我們的工程師團隊建議使用 ‘depends_on’。（請參閱下面的選項 3）。

      暫行解決方法 #3. 最佳的暫行解決方法是使用 depends_on。它會在編配中建立一個實務，讓 VM 的部署「取決於」其他可定義資源的存在性：例如，L3 路由器和網路。

      簡短的 depends_on 範例：

      add_router_interface:
   type: OS::Neutron::RouterInterface
   properties:
     router: { get_resource: router}
     subnet: { get_resource: subnet}


server1:
    type: OS::Nova::Server
    depends on: add_router_interface
    properties:
      image: { get_param: image }

      「明確相依關係」的進一步詳細資料可在下列鏈結中找到：

      • https://wiki.openstack.org/wiki/Heat/Blueprints/hot-software-config-spec

      問：需要為 OpenStack 開啟哪些埠？

      您至少需要開啟用於 DHCP 的埠 67/68，以及用於 meta 資料的埠 80。此外，如果您希望 VM 能夠存取 OpenStack API 服務，請包括這裡所記載的埠：

      /help-documentation/gettingstarted/commontech/FAQ/#which-portsurls-etc-are-publicly-accessible-on-my-bluemix-private-cloud-and-for-what-purpose

      如果未開啟最少需要的一組埠，則使用 cloud_init 時可能會發生問題，您的執行實例可能無法順利啟動。

      問：此錯誤代表什麼意思？「無法擷取磁區限制資訊。」

      通常當我在 Horizon 儀表板中載入頁面時，會看到一個紅色蹦現畫面，指出「錯誤：無法擷取磁區限制資訊。」這是什麼意思？

      此錯誤無傷大雅，它是因為您的部署中沒有任何 Cinder 區塊儲存空間才會發生。

      透過 Keystone 服務移除 Cinder 端點，即可解決此錯誤。這可能需要更新自動化配置檔，且支援團隊需要加以檢查。如果您需要我們協助您解決此錯誤訊息，請開啟支援問題單。

      以下的範例說明支援技術人員需要遵循的步驟：

      # openstack endpoint list | grep cinder
| 19a90853883049ed92cf4af82a24c0db | RegionOne | cinderv2     | volumev2       | True    | name  | https://... | 
| 2c3e8d0a15304788a9dc1806917b61ee | RegionOne | cinderv2     | volumev2       | True    | other     | https://... | 
| 39434faf90484a208b61a517926042b2 | RegionOne | cinderv2     | volumev2       | True    | name2    | https://... | 

# openstack endpoint delete 19a90853883049ed92cf4af82a24c0db
# openstack endpoint delete 2c3e8d0a15304788a9dc1806917b61ee
# openstack endpoint delete 39434faf90484a208b61a517926042b2

# openstack service list | grep cinder
| ef7a982aaf224c62a09c6e813ac66141 | cinderv2   | volumev2       |

# openstack service delete ef7a982aaf224c62a09c6e813ac66141
