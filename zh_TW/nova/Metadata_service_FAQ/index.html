      OpenStack Neutron 和 Nova 中的 Meta 資料服務

      問：您為何想要使用 Nova/Neutron meta 資料服務？

      答：Nova/Neutron meta 資料服務是透過網路將自訂作業注入執行實例中的一種方式。例如，您可能想要新增 SSH 金鑰、密碼、主機名稱及自訂 Script。meta 資料服務是在 169.254.169.254 上執行，它會將接收到的指令交給 Nova。

      問：我的訪客 OS 中需要具備什麼項目才能使用 meta 資料服務？

      訪客 OS 必須執行一個稱為 cloud-init 的套裝軟體，這樣 meta 資料才能在啟動時自動運作。啟動期間，cloud-init 會根據使用者在執行實例建立期間所傳遞的 meta 資料，呼叫 meta 資料服務來自訂執行實例。

      問：您為何「不」想要使用 Nova/Neutron meta 資料服務？

      答：因為它是以複雜方式實作，有時會失敗。Neutron 實際上並未執行 meta 資料服務。相反地，它是執行 meta 資料的代理。實際的 meta 資料服務是由 Nova 擁有。您可以查看下列顯示服務運作方式的圖表：

      （映像檔貢獻者：Matt Dorn）

      如果您真的想要瞭解它有多複雜，請參閱 https://www.suse.com/communities/blog/vms-get-access-metadata-neutron/。在資料流程的任意位置都有可能會發生問題。在大部分情況下，您會在 console-log nova 中看到類似下面內容的錯誤：

      2016-11-30 17:09:33,599 - url_helper.py[WARNING]: Calling 'http://169.254.169.254/2009-04-04/meta-data/instance-id'
failed [50/120s]: request error [HTTPConnectionPool(host='169.254.169.254', port=80): Read timed out.
(read timeout=50.0)]

      或

      checking http://169.254.169.254/2009-04-04/instance-id
failed 1/20: up 1.13. request failed
...
failed 20/20: up 229.58. request failed
failed to read iid from metadata. tried 20

      如果您看到它嘗試連接的 IP 是 192.168.0.2 而非 169.254.169.254，這表示如果 169.254.169.254 即將逾時，它會嘗試從 DHCP 代理程式取得 meta 資料。

      問：當它逾時的時候該怎麼辦？

      答：從該執行實例內，嘗試使用 curl http://169.254.169.254/latest/meta-data/ 來手動存取 meta 資料服務。為了進一步縮小範圍，請嘗試暫時放寬安全群組規則和 SELinux 設定。您也可以嘗試使用不同的映像檔，最好是 IBM 提供的官方映像檔之一。

      如果這樣沒用，請開啟支援問題單，然後要求在您的兩個控制器（網路）節點上重新啟動這些服務。

      • neutron-metadata-agent（在兩個網路節點上）
      • neutron-l3-agent（在兩個網路節點上）
      • neutron-dhcp-agent（在兩個網路節點上）
      • neutron-linuxbridge-agent（在執行實例所在的計算節點上）

      這些代理程式重新啟動之後，技術人員還應該掃描是否有停滯的 Neutron 埠，如果有看到則加以修正。在大部分情況下，在 neutron-metadata-agent 重新啟動後接著重新啟動執行實例，即可解決該問題。

      問：是否有更簡單、更可靠的替代方案？

      答：有！那就是 ConfigDrive。它可以消除運用 Neutron meta 資料代理程式的一切複雜性。ConfigDrive 把您的資料放置在 cloud_init 期間所裝載及卸載的虛擬 CD-ROM/ISO 上，使資料更加接近您的執行實例。請如下所示使用它：

      nova boot --config-drive true --image my-image-name --flavor my-flavor myinstance

      進一步瞭解：http://docs.openstack.org/user-guide/cli-config-drive.html

      問：使用 ConfigDrive 時需要注意些什麼？

      答：主要需注意的是，使用 ConfigDrive 時禁止即時移轉，因為複製唯讀磁碟的 libvirt 有錯誤。https://bugs.launchpad.net/nova/+bug/1246201/comments/65 有修補程式可解決此問題，但僅限於使用區塊移轉時。由於現行版本尚未支援區塊移轉，所以我們不會新增此修補程式。

      題外話：另請注意，即時移轉本身僅限於在一個罕見情況中支援技術人員使用：即將進行計劃性維護而退出主機時（這可能會中斷客戶工作負載）。即使在這一個使用案例中，也只有從磁區啟動的執行實例才支援它。

      問：meta 資料與使用者資料之間有何差異？

      答：執行實例 meta 資料主要是由 Nova 執行實例使用，它通常包括執行實例主機名稱、SSH 金鑰等等。

      使用者資料主要是由 Heat 使用，它通常包括執行實例 root 密碼和啟動 Script。

      問：我想要在 Heat 中利用使用者資料，但我的資料長度超過 16K 的限制。我該怎麼辦？

      答：在使用者資料中使用 include 來提取 Script 及略過大小限制：

      #include
http://example.com/yourscript.txt

      問：meta 資料有哪 4 種類型，以及如何使用它們？

      這 4 種類型為：meta 資料、使用者資料、供應商資料 及網路資料。進一步瞭解這些類型及如何使用它們：http://www.madorn.com/openstack-metadata-types.html#.WEHS63eZMl4

      問：在何處可以找到 OpenStack Meta 資料服務的相關資訊？

      答：請觀看 Matt Dorn 在奧斯汀 OpenStack 高峰會中的卓越視訊研討會：http://www.madorn.com/openstack-metadata-service.html

      視訊的直接鏈結：https://www.youtube.com/watch?v=YGUG8vU5KuQ
