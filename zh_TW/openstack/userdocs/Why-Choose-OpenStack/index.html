      為什麼選擇 OpenStack？

      • by Leslie Lundquist ● April 14, 2016

      • Tags:

        • OpenStack

        • OpenStack

        • 參照

      OpenStack™ 是一種大範圍的開放程式碼軟體平台，可用來提供「基礎架構即服務 (IaaS)」。OpenStack 的開發始於 2010 年，由 Rackspace 和 NASA 共同努力進行。OpenStack 可以管理資料中心內的大型或小型計算、儲存及網路資源儲存區，並透過集中化儀表板來提供和調整這些資源。

      歷程和願景：OpenStack 是有史以來成長最快速的開放程式碼專案，現在已經有兩千萬行以上的程式碼。由「OpenStack 基金會」控管，以促進其開發、散佈及採用。OpenStack 專案包含超過 34,000 位個人貢獻者及超過 550 家參與公司，他們全都是本「基金會」的成員。參與「基金會」的個人成員資格為免費。公司成員資格則有兩種付費等級：白金級和金級。「基金會」由「董事會」管理，而「董事會」則由個人和參與公司的代表組成。

      授權與發行：OpenStack 軟體排定每年兩次發行（四月/五月、十月/十一月），得在符合「Apache 授權條款」下使用。版本代號按字母順序排序。最新版本稱為 “Ocata”。同時，下一個版本正在積極開發中。一年兩次發行的同時，OpenStack 社群會召開「OpenStack 高峰會」以規劃未來的發行，並公佈成就或使用案例。

      硬體需求：OpenStack 使用標準或商用硬體，不需要使用特殊化的供應商硬體。若客戶想要使用特定的企業級硬體，則可以使用驅動程式。 如需這些驅動程式的清單，請造訪 OpenStack 網站的「市場」。

      軟體專案方法（大帳篷）：從 Liberty 版本開始，OpenStack 社群採用「大帳篷 (The Big Tent)」作為 OpenStack 軟體開發的隱喻。專案由社群成員提出並由「大帳篷」保護，當社群成員決定一起合作時即可取得資源。 沒有任何專案會在起始階段就被正式批准或正式拒絕。

      OpenStack 社群：OpenStack 最大的力量即在於其活躍的社群。來自 177 個國家、超過 34,000 位個人貢獻者是此社群的一部分，另外還有超過 70 個全球使用者群組。OpenStack 使用者群組會定期聚會。會議是作為傳播意識的場所、用來促進技術討論，同時充分顯示社群之間的情誼。如果您是 OpenStack 新手，則參與這些群組、向其他使用者學習是最簡單的開始使用方式。

      堆疊的市場趨勢和狀態：OpenStack 逐漸變成私有雲部署的選擇平台。預期市場收益在 2016 年可達 13 億美元，到 2018 年可超過 30 億美元。 採用 OpenStack 的前三大驅動因素是可以在 OpenStack IaaS 平台之上迅速革新 (DevOps)、可以自動化雲端部署以方便向上調整，以及 OpenStack 是開放式技術的事實。軟體開發和測試是最常用的工作負載，接著是基礎架構服務。企業工作負載正逐步增加，特別是在 OpenStack API 方面，已經成為「基礎架構即服務 (IaaS)」的標準。 從更寬廣的觀點來看，不但學術及研究、電信、金融、媒體界的 OpenStack 採用案例正逐漸增加，更遍及其他領域。

      OpenStack 說明文件：若是 OpenStack 新手，可以使用適用不同平台（Ubuntu、RHEL、Debian、SUSE）的安裝手冊。每次發行時均會更新這些手冊。OpenStack 專案針對系統管理者、雲端管理者、一般使用者、設計師及雲端開發人員，提供詳盡的線上文件。 同時也提供關於一些進階主題的說明文件，如安全和高可用性。這些進階文件會定期更新，只是不會與 OpenStack 發行同步。

      詢問 OpenStack：如果您有關於 OpenStack 的特定問題，可以在詢問 OpenStack 網站上請專家指導，該網站是以熱門網站 StackOverflow 為模型所建立、集群眾之力的問答網站。

      OpenStack 市場：OpenStack 客戶有許多選擇：服務提供者、供應商、系統整合者、發行套件、訓練者、顧問等等。為了協助您做出明智的決策，「OpenStack 市場」提供一個中央儲存庫，儲存與 OpenStack 供應商、產品、訓練、顧問及其他服務相關的資訊與評論。

      OpenStack：概觀，依元件說明

      OpenStack 平台是一種服務集合，可分成不同的功能元件。少數幾個核心元件共同提供資料中心內的計算、物件和區塊儲存空間及網路作業功能，並透過集中化儀表板來管理所有這些資源。這些核心服務已透過一組支援服務整合在一起，本文件的後續章節會加以說明。下圖說明 OpenStack 元件之間的關係。

      圖 4. OpenStack 高階架構，依元件說明

      OpenStack 基礎架構 (IaaS) 服務：核心專案

      OpenStack 社群最先開發的專案通稱為「核心」。它們已在正式作業環境中經歷數年的使用和測試，仍然是部署最多的 OpenStack 元件。這些元件都在相同的開放式平台和 API 上標準化，驅動公有和私有雲的廣域網路。此外，針對基礎架構的傳統資本支出，「核心專案」還提供了策略性、可行的取代方案。

      「OpenStack 核心基礎架構 (IaaS)」專案如下：

      • Nova（計算）
      • Cinder（區塊儲存空間）
      • Swift（物件儲存空間）
      • Neutron（網路）
      • Glance（映像檔）
      • Keystone（身分）

      事實上，這些服務（Swift 除外）最初都是 Nova 的一部分，也就是計算服務，是最早期的 OpenStack 專案（連同 Swift 一起）。 每個專案都在日後從 Nova 分割出來（Glance 及 Cinder）或重新實作 (Neutron)。Glance、Cinder 及 Swift 一起構成了 OpenStack 的資料處理和儲存功能。Neutron 將 OpenStack 的網路功能合併成可以輕鬆部署的形式。Ironic 協助自動化硬體佈建，以容許完整的 IaaS 功能和延伸。

      Nova：Nova 是 OpenStack 的計算服務。提供隨需應變佈建和管理虛擬機器。可支援多個 Hypervisor，包括 KVM、XenServer®、VMware ESXi 及 Microsoft Hyper-V。同時支援 Linux 容器，如 LXC。

      Glance：Glance 提供 API 可存取服務來處理磁碟和伺服器映像檔，包括探索、登錄和遞送。Glance 可以利用各種不同的後端設定來儲存映像檔，包括 OpenStack Swift。Glance 服務可包容多種映像檔格式，包括「原始」、VHD、VMDK 及 VDI。

      Cinder：Cinder 是 OpenStack 的區塊儲存空間模組，提供用於虛擬執行實例的虛擬儲存空間。它可讓您建立裝置、將裝置連接至虛擬伺服器，以及從虛擬伺服器分離裝置。Cinder 已完全整合 OpenStack 的 Nova 計算服務，可以從 Horizon 儀表板加以管理，Cinder 也支援透過特定驅動程式整合企業儲存平台，例如 Nimble®、Solidfire® 或 EMC®。另外還支援 Snapshot。

      Swift：Swift 處理分散式、非結構化資料。它提供一個非常好的平台，用來儲存映像檔、影片、虛擬機器映像檔和保存檔。Swift 提供符合成本效益、API 可存取的、分散式、備援、可調整的儲存空間，非常適合用於備份、保存及資料保留。

      Neutron：Neutron 提供外掛式 API 驅動平台，用來管理網路和 IP 位址。Neutron 支援多種網路模型（Flat、VLAN、VXLAN）、靜態 IP，以及 DHCP。它可以利用 SDN 平台來使用進階網路功能，例如 OpenFlow。

      Keystone：Keystone 提供鑑別和授權功能。它整合了現有的後端目錄服務，例如 LDAP。Keystone 提供集中式使用者目錄，並對映至使用者獲授權使用的服務。

      基於所有實用目的，在任何 OpenStack 部署中，Keystone都是必要元件。每一個 OpenStack 部署都需要某種形式的鑑別服務，而 Keystone 是最常見的使用案例。（在某些極端的測試案例中，可能會有不作業鑑別。）例如，雖然不常使用，但 Swift 還是有自己的鑑別，但在無法使用 Keystone 的測試部署中除外。

      OpenStack 會將這些計算、儲存空間、網路作業及佈建元件模組彼此整合（同時整合其他支援服務），藉以提供統一、可調整的「基礎架構即服務 (IaaS)」使用者體驗。請注意，也可以透過一致的 RESTful API 存取這些相同的 OpenStack 服務；您可以選擇使用 API 介面，建立自己的使用者體驗 (UX)。

      OpenStack 支援 IaaS 服務

      「OpenStack 支援服務」可延伸 OpenStack 核心 IaaS 的功能及特性。這些支援服務包括安全服務、使用情形監視及自動化部署服務。下列項目符號依據概略的部署頻率列出下列額外的 OpenStack 服務，以及前一節所列出的基本 OpenStack IaaS 服務。

      支援的 IaaS 服務包括：

      • Ceilometer（遙測）
      • Heat（編配）
      • Horizon（儀表板）
      • Ironic（裸機佈建）

      Ceilometer、Horizon 及 Heat 共同將所有 OpenStack 元件上的一般使用和功能資訊合併在一起，讓 OpenStack 雲端環境變得更豐富。（附註：在最新的 OpenStack 版本中，部分程式庫程式碼 Oslo 也因為是這些服務所需要的一般程式庫而會自動部署。）

      Ceilometer：Ceilometer 聚集 OpenStack 服務中的使用和效能資料。

      Heat：Heat 提供以範本為基礎的功能，說明和自動化相對複雜的基礎架構部署。例如，如果您需要使用當成應用程式伺服器執行的執行實例、儲存在 Trove 中的資料庫，以及負載平衡器來部署簡式 Web 應用程式，則要以應用程式執行實例數的參數建立 Heat 範本。與 Ceilometer 遙測服務一起使用時，Heat 可提供自動化調整功能，在上述範例中，這些功能可在必要時部署更多的執行實例。

      Horizon：Horizon 提供集中化圖形使用者介面，可讓您檢視、佈建及管理雲端資源。協力廠商服務（例如，計費和監視）可以和 Horizon 儀表板介面輕易地整合在一起。OpenStack 仍然可在沒有 Horizon 的情況下完全運作，並使用基礎 OpenStack API 存取相同的使用資訊。

      Ironic：Ironic 可讓您自動佈建及管理實體機器。通常，公司會花錢購買或建置此服務，但會被視為計算服務類型。

      OpenStack 應用程式支援

      本節所含的每一個 OpenStack 專案均用作特定應用程式的封套。這些模組最適合用來簡化雲端以因應特定目的：資料庫存取、海量資料分析或共用資訊。尤其是，這些 OpenStack 模組將啟動一些新的 Nova 執行實例，而這些執行實例已經在執行您所需要的特定應用程式。這些 OpenStack 專案可以在容器（例如 Docker 容器）內順利執行，特別是因為在大部分情況下，使用者無法直接存取這些執行實例：

      • Trove（資料庫）
      • Sahara（資料處理—Hadoop）
      • Manila（檔案—CIFS）

      Trove：Trove 在 OpenStack 雲端基礎架構上提供「資料庫即服務 (DBaaS)」功能。

      Sahara：Sahara 提供可調整的資料處理堆疊及其相關聯的管理介面。

      Manila：Manila 提供共用檔案系統服務。它要與 Samba（Windows 檔案共用的 Linux 實作）及其他搭配使用。

      持續改善：「大帳篷」OpenStack 服務中的其他專案會持續不斷地醞釀、實作及改善，以符合前述已廣泛部署的 OpenStack 模組（整合的 IaaS、支援的 IaaS 及 PaaS）所建立的品質限制。目前在「大帳篷」下醞釀的一些專案包括「部署服務」(TripleO)、「DNS 即服務」(Designate)、「金鑰管理」(Barbican)、「負載平衡即服務」(Octavia)、「防火牆即服務」、「網路功能虛擬化 (NFV)」及其他。

      OpenStack 雲端遞送方法

      您可能會問：現在我知道這些了，要如何實際取得「OpenStack 雲端」？OpenStack 軟體目前以三種主要方式交付給客戶：

      1. 以套件集合（稱為「發行套件」）：OpenStack 網站提供標準、可下載的 OpenStack 發行套件。除了預設的 OpenStack 發行套件外，許多供應商會提供含有一些加值服務的專利發行套件。這些發行套件可協助使用 OpenStack 執行安裝和升級程序，這些程序原本是很費力的。然而，發行套件並非特別用來協助簡化 OpenStack 雲端的作業。因為可順利使用發行套件的客戶，通常都有豐富的 OpenStack 使用經驗。發行套件是由許多供應商所建立，供應商數量多達 25 個，包括 Mirantis、Red Hat、HP、Platform 9 及其他。如需發行套件供應商的清單，請造訪「OpenStack 市場」，此清單也是可協助您部署 OpenStack 的顧問清單。

      2. 即服務：OpenStack 即服務表示您向資深雲端操作員（例如 IBM 或 Rackspace）依約購買（私有）雲端服務。雲端操作員會支援其用戶端，所以您可以免費「使用」OpenStack，而不是在 OpenStack 上工作。這是使用 OpenStack 雲端即服務 (IaaS) 的主要好處：它可讓您將重點放在您自己的商業應用程式上，而不是在維護雲端上。

      3. 標準 OpenStack 發行套件：標準 OpenStack 發行套件將 OpenStack 服務組合成可供下載的套件。標準套件適用於數種平台，包括 Ubuntu®、CentOS、Debian 及其他。OpenStack 軟體會持續不斷更新，您可以從一般稱為「幹線」的位置下載。這些持續的更新項目並不穩定；只有一年兩次發行的才是穩定版本。最新的穩定版本也一定會同時提供下載，可以從幹線連同工作版本一起取得。

      發行套件

      協力廠商發行套件：參與的供應商可提供其自訂的 OpenStack 發行套件，其中包含供應商所提供的某些優點。自訂發行套件可能會含有加值服務、專利服務，也可能是與其他供應商供應項目整合得特別好。如需詳盡的協力廠商發行套件清單，請參閱「OpenStack 市場」。最熱門的協力廠商發行套件為 Mirantis OpenStack、Red Hat Enterprise Linux OpenStack Platform、Rackspace® Private Cloud Software 及 Ubuntu OpenStack。

      OpenStack 雲端即服務

      少數供應商會透過之前所提的代管或「即服務」遞送機制提供 OpenStack 雲端，該機制指在代管環境中隨需提供可用的 OpenStack 雲端。對於想要使用 OpenStack 而不想有額外作業和其他基礎架構的客戶而言，代管 OpenStack 雲端是很具吸引力的選項。

      代管雲端客戶會收到單一租戶的 OpenStack 雲端並享有所有優點，包括 API 存取而不需額外的部署和作業。如需詳盡的 OpenStack 代管雲端提供者清單，請參閱「OpenStack 市場」。下面是一些熱門的 OpenStack 雲端即服務供應商：

      IBM Bluemix Private Cloud：Blue Box 由 IBM 於 2015 年 6 月購買成為 IBM 雲端產品供應項目的一部分，現在稱為 IBM Bluemix Private Cloud。Blue Box 是最早提供代管 OpenStack 私有雲即服務的提供者之一，因此帶動許多專家進入 IBM Bluemix 團隊。 IBM Bluemix Private Cloud 團隊帶領 OpenStack 西雅圖使用者群組聚會，並對 OpenStack 社群積極提供貢獻，特別是對「OpenStack 操作員」工作團隊方面。他們的最新產品 IBM Bluemix Private Cloud（本端）可讓您將 OpenStack 私有雲硬體置於自己的資料中心，並完全交由 IBM Bluemix Private（本端）專家管理。

      Cisco Metapod：Cisco 是另一個提供 OpenStack 雲端即服務的供應商，之前是由 Metacloud 所開發。他們是從以 OpenStack 為基礎的「單箱式雲端」或「私有雲應用裝置」開始，之後開始提供代管的「私有雲」。他們自訂的 OpenStack 發行套件也以其優異的「使用者體驗 (UX)」著稱。

      Rackspace：在其資料中心內提供代管的專用私有雲環境。以良好的效能著稱。

      兩種代管環境類型：公有及私有雲

      公有雲：若要進一步瞭解 OpenStack 雲端的操作方式，可以試著聯絡其中一個以 OpenStack 為基礎的公有雲服務提供者。許多提供者可提供免費試用訂閱或便宜的先期利率。如需公有雲服務提供者清單，請參閱「OpenStack 市場」。

      私有雲：私有雲一般有兩種類型：代管雲端，表示雲端服務提供者會代表您操作位於遠端的專用雲端硬體和軟體，而本端雲端，表示您擁有自己的硬體並雇用他人替您執行。

      代管：代管 OpenStack 雲端產品提供更容易的選項，便於開始使用 OpenStack。例如，當您使用 IBM Bluemix Private Cloud 之類的提供者註冊時，代管的私有雲很可能幾小時內就能執行。

      本端雲端：本端雲端表示雲端的硬體位在您自己的資料中心，但由控管的提供者（例如 IBM Bluemix）代表您操作。在 2016 年進入市場的數個提供者都有新的「本端」雲端服務供應項目。如需提供者清單，請參閱「OpenStack 市場」。

      現在談談有趣的部分：為什麼 OpenStack 很適合您

      想檢查看看？您可以從 DevStack 開始。DevStack 是最簡單的方式，只要最小配置就可以安裝 OpenStack 測試雲端。如果您想要測試本端雲端環境，只要自行檢查即可，不需要服務的合約認同，就可以輕鬆地開始使用 DevStack。您可以從 OpenStack 網站下載它，設定全功能配置的程序非常容易。

      DevStack 主要是用作原型化環境。它提供單一全功能虛擬機器、單一全功能機器，以及多節點配置。然而，DevStack 並不適合用來執行正式作業雲端工作負載；例如，在機器重新啟動後，它不會回到完全運作狀態。

      您使用 DevStack 啟動的雲端不會受到監視或管理，所有除錯記載均預設為啟用。這非常有用，例如，升級之前，用來比較 OpenStack 版本以瞭解軟體是否能搭配新的 OpenStack 版本適當運作。目前適用於 Ubuntu、Fedora 及 CentOS/RHEL 平台。

      如果您在意您的雲端有特定需求，可以部署 OpenStack 專案，以便只符合您目前所具有的需求。 OpenStack 可以讓您在需要時繼續新增功能。舉例來說，假設您的網站是在自有的專用 Web 伺服器上執行，且您只執行 Tumblr 之類的服務，用來儲存部落格文章和影像。則您只需要一種方式來儲存大量資料。您可以只部署 Keystone 及 Swift 作為 OpenStack 配置，這可提供鑑別和儲存服務，而不需要任何計算！

      或者，假設您的雲端只需要計算基礎架構，因為您的網站只需要暫時儲存功能（重新開機時遺失）。則可以只使用 Keystone、Nova、Glance 專案來部署 OpenStack 雲端。在此情況下，Nova 中的網路作業功能可提供簡單的網路特性。如需更豐富的網路作業功能，請將 OpenStack 的 Neutron 元件新增至部署。您的計算基礎架構就會完美無缺。

      若要部署全功能的 OpenStack 雲端且具有完整的計算和儲存功能，請使用所有核心 OpenStack IaaS 模組，可能還要新增 Keystone 來提供鑑別服務。從這裡，您可以進一步自訂雲端，以確切符合您所需要的作業類型，例如，資料庫或資料分析叢集。您可以視需要繼續新增和升級 OpenStack 功能。

      一般 OpenStack 使用案例

      OpenStack 在許多垂直產業及各種使用案例中的運作都非常好。下面是一些最成功的 OpenStack 使用案例：

      雲端服務提供者：在這個使用案例中，OpenStack 平台提供大規模的虛擬基礎架構即服務 (IaaS)。一般此類型的堆疊會包含開放程式碼工具，例如 KVM (Hypervisor)、Ubuntu (OS)、OpenStack（雲端管理層）、Nagios（監視）、LogStash（記載）、Kibana（分析）、Chef（編配）、HAProxy（負載平衡器），以及自產元件。商品硬體通常是用來提供計算功能和某些類型的儲存空間；不過，可以部署多個硬體配置。「Rackspace 公有雲」使用案例是很好的例子。

      Telecom 服務提供者：「Telecom 服務提供者 (Telcos)」很早就採用 OpenStack 來滿足其私有雲需求。這些使用案例通常會提供「IT 即服務」給內部組織或夥伴組織。OpenStack 社群則持續處理 Telcos 特有的進階功能（例如「網路功能虛擬化」或 NFV）。

      應用程式開發雲端：OpenStack 的另一個常見的使用案例是在「應用程式開發」雲端啟用「持續整合/持續開發 (CI/CD)」。在此情況下，OpenStack 雲端可讓企業變得靈活，並增加客戶對於在 OpenStack 雲端上執行「事業線 (LOB)」應用程式的信任。這些 LOB 應用程式和工作負載通常會用到某些類型的原始碼儲存庫（例如 github.com）、測試控制工具（例如 Jenkins）、錯誤資料庫（例如 Atlassian），以及部署工具。

      使用協力廠商工具搭配 OpenStack

      OpenStack 可以妥善整合常用的協力廠商雲端作業、開發和管理工具。協力廠商工具可能包括 OpenStack API 的「軟體開發者套件 (SDK)」、編配工具、監視工具、記載工具和佈建工具。在實際的雲端部署中，您不可避免要使用一些其他開放程式碼工具。本節列出部分普遍與 OpenStack 搭配一起使用的工具類型。請注意，部分專利 OpenStack 發行套件可能不需要其中的任何工具，因為它們通常會含有自訂的對等工具。

      SDK：OpenStack 提供兩種方式來管理資料中心內的計算、儲存和網路資源：「儀表板」與「服務端點」。「OpenStack 服務端點」會透過可程式介面 (REST API) 顯示。使用者可以透過任何 REST 用戶端存取這些 API，例如在 OpenStack 專案中提供的「用戶端」工具。使用這些 API 的 SDK 也適用於各種程式設計語言。

      OpenStack 用戶端工具：OpenStack 專案含有用於各種專案的正式用戶端，包括「儀表板」、「計算」、「物件儲存空間」、「身分服務」及「映像檔服務」，而用於其他專案的其他用戶端則正在開發中。

      OpenStack SDK：提供正式的 OpenStack SDK for Python，而非正式的 SDK 可用於各種程式設計語言，包括 C、C++、Java .NET 及其他。

      編配工具：雖然 OpenStack 有自己的編配服務 (Heat)，許多開發還是會利用其他常見的編配工具，例如 Chef、Puppet 及 Ansible。

      Chef：Chef 是 Chef（之前稱為 Opscode）所提供的常用配置管理工具。普遍用於 OpenStack 部署。Chef 採用主從架構。使用已發佈的官方錦囊妙計，可協助您開始使用 Chef 部署 OpenStack。

      Puppet：Puppet 是 PuppetLabs 所提供的另一個常用配置管理工具。Puppet 也採用主從架構。您可以使用正式的 Puppet 模組來協助部署大規模的 OpenStack 雲端。

      Ansible：Ansible 軟體是一種無代理程式配置管理工具。雖然正式的 Ansible 操作手冊不適用於 OpenStack，但使用者還是可以從這些 Blue Box 操作手冊開始。

      監視工具：操作大規模的 OpenStack 雲端需要經常監視不同元件，包括計算資源、虛擬機器和虛擬網路。監視也可讓管理者主動發出提醒，例如在計算節點接近完全 CPU 使用率時。OpenStack 提供「遙測」服務 (Ceilometer)，此服務提供可啟用監視的事件型基礎架構，然而，它並未提供完整的監視功能。因此，OpenStack 雲端部署通常會利用其他開放程式碼監視工具，例如 Nagios。

      記載工具：若要提供進階日誌管理功能，可以使用開放程式碼工具，例如 LogStash。LogStash 提供執行記載、剖析及儲存日誌的集中化功能。與搜尋工具（例如 ElasticSearch）一起使用時，它會為您的 OpenStack 雲端提供功能強大、可調整的日誌管理解決方案。

      佈建：部署 OpenStack 雲端包含佈建裸機伺服器作為控制器節點或資源節點。用來佈建的常用工具包括 Chef、Puppet、Ubuntu MaaS 及 Crowbar。OpenStack 自有的裸機服務 Ironic 也是一個可用選項。

      雲端術語名詞解釋

      存取金鑰/秘密金鑰 (Access Key/Secret Key)：兩個成對且會自動產生加密金鑰，一起使用可以存取計算執行實例並與之通訊。秘密金鑰是用來以數位方式簽署每一個要求，而存取金鑰是用來驗證該簽章。

      可用性區域 (Availability Zone)：雲端中一組隔離的 Hypervisor。可用性區域提供虛擬機器容錯。OpenStack 支援容許邏輯分隔服務或建立服務叢集的可用性區域。通常這些區域的建立是基於方便存取電源或地理原因。您可以在啟動新執行實例時選取區域。

      區塊儲存空間 (Block Storage)：支援磁區、磁區 Snapshot 及磁區類型的儲存空間類型。區塊儲存空間磁區是未直接連接執行中執行實例的區塊裝置。它可以一次連接一個執行實例，如果執行實例本身故障或關閉，此裝置仍會存活。通常，OpenStack 磁區會連接現有的 SAN，因此您可以使用 API 來與其互動。

      容器 (Container)：在「物件儲存空間」中組織和儲存物件。在 OpenStack 中，容器包括協力廠商軟體，例如 Docker。

      固定/靜態 IP (Fixed/Static IP)：每次執行實例重新啟動時，都會與相同的虛擬機器執行實例相關聯的 IP 位址。在雲端中使用此類型的位址進行虛擬機器管理。客戶通常無法存取它。

      虛擬硬體範本 (Flavor)：說明虛擬機器映像檔的一組參數。參數包括 CPU、記憶體大小、儲存空間大小等等。

      浮動 IP (Floating IP)：與虛擬機器執行實例相關的 IP 位址，使該執行實例在每次重新啟動時都有相同的公用 IP 位址。浮動 IP 會建立一個公用、可連接的 IP 位址，然後將該位址的資料流量引導至配置的執行實例。此 IP 並不會真的當成執行實例的一部分建立；而是由雲端提供間接關係。首先，必須從提供的儲存區配置一個位址，然後將該位址連接至執行實例。

      Hypervisor：一種軟體，可裁決及控制虛擬機器對實際基礎硬體的存取。它代表在特定執行實例中佈建的實體計算節點。

      映像檔 (Image)：特定作業系統的檔案集合，使用者必須使用此集合來建立或重建伺服器。OpenStack 支援多重映像檔格式（AMI、VMDK、VHD、QEMU）。例如，伺服器映像檔可讓您建立預先配置的映像檔以供稍後使用。這些映像檔可用來啟動新伺服器執行實例，且會出現在列出的映像檔中。

      執行實例 (Instance)：虛擬機器。若要啟動（或「起轉」）執行實例，需要一組基本資訊：用作模型的磁碟映像檔、執行實例虛擬硬體範本（將它想成大小）、執行實例名稱（例如 MyInstance4032）。大部分 OpenStack 環境還需要您在啟動執行實例時提供網路裝置的參照。

      Meta 資料 (Metadata)：執行實例 meta 資料對於排序或編目執行實例非常有用。Meta 資料是每個執行實例的開放式鍵=值儲存。

      網址轉換 (Network Address Translation, NAT)：在雲端中，將某一類型的 IP 位址變更為另一類型的通訊協定。

      網路 (Network)：虛擬網路是第 2 層網路，提供實體（例如，雲端中的執行實例）之間的連線。

      物件 (Object)：物件是儲存在物件儲存空間中任何類型的資料，可以是任何格式：檔案、音樂、影片或二進位。

      物件儲存空間 (Object Storage)：一種儲存空間類型，支援最終一致、備援、非結構化的資料。OpenStack 物件儲存空間不是傳統的檔案系統，而是用於靜態資料的分散式儲存系統，例如虛擬機器映像檔、照片儲存、電子郵件儲存、備份及保存。它沒有「根」目錄、中央「大腦」或主要控制點，因此容許更大的可調整性、備援性及延續性。資料是以物件表示，而物件則儲存在容器中。

      專案/租戶 (Project/Tenant)：專案是一種邏輯使用者分組；也通稱為租戶。

      配額 (Quota)：按照每個專案設定的計算和儲存資源等資源限制。通常，在多租戶管理的雲端中，每個租戶的配額是依據其合約協定。配額可以按專案或類別基準設定，其目的在於限制資源的使用，例如 RAM、IP 位址、核心數、儲存磁區大小等。OpenStack 可強制資源使用的配額。

      角色 (Role)：角色包括一組權限和專用權。指派特定角色的使用者會繼承其相關的權限和專用權。

      安全群組 (Security Group)：套用於計算執行實例的一組過濾規則。安全群組類似系統群組的防火牆。您可以提供參照其他群組或 CIDR 區塊的存取規則。這些群組可以連接至執行實例，以存取該執行實例。

      伺服器 (Server)：參照虛擬機器 (VM) 執行實例的一般詞彙。另一個代表虛擬機器執行實例的一般詞彙是主機 (host)。

      服務 (Service)：提供端點的 OpenStack 服務，透過這些端點，使用者可以存取及控制資源。

      SSH 金鑰 (SSH key)：使用 SSH 金鑰可以安全連接（使用 ssh 指令）以 OpenStack 啟動的執行實例。OpenStack 的內建方法可以將 SSH 金鑰組的公用部分插入新啟動執行實例的授權金鑰檔。Nova 可以為您建立金鑰的私密部分，或者，您可以上傳現有金鑰組的公用部分。

      Snapshot：儲存磁區或映像檔的復原點副本。在功能上，Snapshot 與映像檔相同。

      使用者 (User)：使用者會耗用雲端資源。每一個使用者都是一個或多個專案的成員。在專案內，使用者可以新增和管理資源，例如執行實例和磁區。

      虛擬機器 (Virtual Machine, VM)：在 OpenStack 雲端內執行的執行實例；虛擬機器的另一個術語可以是伺服器或主機。

      磁區 (Volume)：以磁碟為基礎的資料儲存空間，可以是持續性的或暫時性的（重新啟動時遺失）。此儲存空間通常以 iSCSI 目標表示，其中含有支援延伸屬性的檔案系統。將磁區連接至伺服器時，可能會仰賴系統在執行實例中自動指派裝置 ID，以連接至新磁區。您也可以選擇指定。如果新磁區已連接，則必須將磁區格式化才能使用。後續使用該磁區將不需要格式化。
