      我們建議您的團隊考慮使用 OpenStack Orchestration 服務 (Heat)，每一個 IBM Bluemix Private Cloud 都已啟用此功能，這是最佳作法。您可以使用 Heat 在文字檔中有步驟地定義所有資源，並對多個雲端執行這個相同的檔案，以達到相同的結果（即所謂的堆疊）。

      您可以使用 Heat 來設定堆疊，它幾乎包含您想要的所有 OpenStack 資源。

      例如，如果您想設定一些具有相同安全群組定義的堆疊，該怎麼辦？如果您查看我在下面提供的 Script 範例的 82-110 行，可以看到安全群組在 Heat 中是如何定義的。此功能不能讓您「複製」安全群組本身，但可讓您在您想要的任意數目專案或不同雲端中建立相同的堆疊（例如，包含安全群組資源）。由於輸入文字檔相同，所產生的堆疊及所含的資源都將相同。您甚至可以在相同的專案下起動多個相同的堆疊。

      您可以在這裡和這裡找到 OpenStack 建立的簡介說明文件。

      您的 Heat 範本有多少資源全憑您的喜好，很容易進行。建構範本之後，您可以依喜好從中建立任意數目的堆疊，請使用如下的指令：

      openstack stack create -t <template file> <stack name>

openstack stack show <stack name>

      而且，您可以變更該檔案的內容，然後執行 update 指令，來「更新」現有的堆疊：

      openstack stack update -t <template file> <stack name>

      希望此資訊對您有用，這就是我嘗試及解決任何 多雲端問題的方式。

      Heat 堆疊使用者角色

      您可以在 OpenStack 內指派許多不同的角色給使用者。例如，一般使用者可能只獲指派一個叫作「成員」的角色。管理者的角色可能稱為 cloud_admin。

      這些角色的目的是要定義及限制對某些 API 呼叫的存取。例如，獲指派「成員」角色的使用者無法變更另一個使用者的密碼，但獲指派 cloud_admin 角色的使用者可以執行此動作。

      OpenStack Orchestration 服務 Heat 自己有支援另一組角色。

      在堆疊部署期間，Heat 服務會自動指派 heat_stack_user 角色給它建立的使用者。此角色對某些 API 呼叫具有受限存取權，因為 Heat 在某一個堆疊內建立的使用者專屬於該堆疊。

      如果您要讓使用者能夠管理不同的堆疊（例如，在許多執行中的堆疊內刪除及新增資源），則必須指派 heat_stack_owner 角色給這個使用者。heat_stack_owner 角色可以對 Heat 發出哪些 API 呼叫，其限制較少。

      可惜的是，heat_stack_user 和 heat_stack_owner 角色會彼此衝突，因此，如果您有一個使用者具有 heat_stack_owner 角色，則您不能將 heat_stack_user 角色指派給它，以避免發生此衝突。

      這些多重角色的目的是要協助使用「Keystone 信任」作為延遲鑑別方法。如果想要進一步瞭解此主題，您可以閱讀此部落格文章：http://hardysteven.blogspot.com/2014/04/heat-auth-model-updates-part-1-trusts.html
