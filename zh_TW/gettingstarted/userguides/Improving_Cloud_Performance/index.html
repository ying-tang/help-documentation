      淺談 OpenStack II -- 增進雲端效能

      • by Dustin Lundquist and Leslie Lundquist ● November 21, 2016

      • Tags:

        • openstack

        • openstack

        • sr-iov

        • virtio

        • vhost

      重新發佈來源：https://admin.blogs.prd.ibm.event.ibm.com/blogs/bluemix/2016/11/building-openstack-ii-improving-cloud-performance/

      本文是這一系列的第二篇，我們將說明更多雲端基本概念：更多 QEMU 相關資訊，以及有關如何增進雲端整體工作負載效能的更多資訊。

      基本上，增進雲端效能需要在存取網路和儲存裝置時改善 I/O 效率。透過將 I/O 虛擬化最佳化，訪客 VM 與主機之間的 I/O 裝置介面就會變得更有效率，進而能夠增進 I/O 的效能並提高彈性。因此，雲端效能會整體提升，因為工作負載可以使用的總處理器容量更多。

      透過虛擬化增進 I/O 效能是重點

      請記住，QEMU 是雲端 VM 的基本元素。是它讓工作執行！然而，因為 QEMU 是一種模擬器，所以當速度非常重要時，最好盡可能少用它。雖然 KVM 會改善 QEMU 的一些處理器和記憶體效能，但 I/O 效能還是有瓶頸。

      使用虛擬化來增進 I/O 效能的途徑有兩種：軟體途徑和硬體途徑。在此篇部落格文章中，我們將檢查每一種途徑。在每一種途徑中，我們所行經的每一個步驟都會讓 QEMU 進一步減少動作。為什麼？模擬愈少表示效能愈好。

      軟體途徑需要以更有效率的方式，將「傳統」裝置及其驅動程式虛擬化。硬體途徑則需要以更直接的方式將裝置連接至 VM，甚至可能使用實作「單一根 I/O 虛擬化 (SR-IOV)」功能的特殊化硬體。基本上，這是一種硬體協助途徑，可以將 VM 與可視需要實例化的虛擬化裝置快速配對。

      軟體途徑：QEMU、virtio 及 vhost

      如我們在前一篇部落格文章所述，QEMU 是設計當成模擬器操作，它甚至可以模擬已成為歷史的作業系統，例如 Windows NT 和 OS/2。它可以模擬作業系統預期要找到的任何硬體；因此，它所揭示的預設裝置是「一般」實體硬體裝置。但作業系統用來與這些裝置通訊的方法，不在虛擬化的設計範圍內！

      較舊的 I/O 裝置通常不是一般用途處理器，它們的速度較慢。因此，在使用舊「硬體」的時代，將 I/O 效能最佳化並沒有什麼好處。例如，實體磁碟需要大量的時間（4 毫秒）來存取磁碟上的某一位置，所以將驅動程式與磁碟控制器之間的通訊最佳化並沒有什麼好處。在現代的雲端中，使用 VM 作為伺服器，增進 I/O 效能可以獲得一堆好處，而且只要使用簡單易懂的方式就能完成。

      為什麼 I/O 虛擬化會有幫助？

      若要執行有效 I/O，您的雲端系統需要 (1) 訪客 VM 的驅動程式、(2) 主機軟體（例如 QEMU）中一些可以提供 I/O 裝置的其他後端，以及最後，(3) 主機軟體用來與基礎 I/O 裝置通訊的介面。

      如您在圖中所示，在 VM 內實行 I/O 要求時會用到數個元件。這些元件之間的每一次轉移都會對效能造成負面影響。為了示範 I/O 效能降低的速度有多快，下面會逐步分解其運作方式：

      • 首先，在 VM 中執行的程式會利用「一般」介面，向 VM 的作業系統發出 I/O 要求。（如果在 VM 中執行的軟體使用不同的方法，使用者就無法輕易瞭解此虛擬化。）

      • 接下來，VM 的作業系統會使用與 QEMU 所示裝置相對應的驅動程式。

      • 然後，該驅動程式會執行一些作業，這會由 KVM 設陷並交由 QEMU 實行。

      • 接著，QEMU 會使用類似的介面，向主機作業系統發出 I/O 要求。

      • 於是，主機作業系統會使用與實體硬體相對應的驅動程式來執行所要求的 I/O 作業。

      我們要如何提升此程序的速度？有兩個不錯的選項。第一，我們可以使用模擬任何特定裝置的方法，以更有效率的方式虛擬化。此選項通稱為「半虛擬化驅動程式」，最佳範例就是所謂的 virtio，它已納入 OpenStack Nova 之中。第二，若要進一步增進效能，我們可以從 VM 中完全移除裝置模擬，以消除另一個轉移作業。亦即，我們將 QEMU 的「裝置模擬」移出訪客 VM，並將它限制於主機作業系統。

      最佳化 1：半虛擬化驅動程式：virtio 方法

      Virtio 是 QEMU 在 VM 作業系統內所揭示的一系列虛擬化裝置，及其相對應的驅動程式。這些驅動程式會利用現代有效率的資料結構（例如，環式緩衝區），將訪客 VM 驅動程式與 QEMU 之間的介面最佳化，而不是盲目地模擬現有硬體裝置的介面。使用 virtio，VM 與 QEMU 的通訊變得更有效率，進而增進 I/O 效能。

      警告：virtio 需要訪客 VM 映像檔來包含這些較快速的 virtio 驅動程式，所以一些較舊的映像檔（例如 Windows 95 或 98）可能無法搭配 virtio 使用。還好，依預設，大部分作業系統現在都包含 virtio 驅動程式，大約自 2013 年之後就是這樣的作法。

      最佳化 2：Vhost（移除 QEMU 使用者空間裝置模型）

      下一個效能最佳化包括將 virtio 裝置模擬功能完全移出 VM，然後移入主機作業系統核心。此移動可避免因為從 KVM 轉移到 QEMU，再轉移回主機核心而產生的效能負面影響。

      硬體途徑：PCI 傳遞及 SR-IOV

      PCI 傳遞是可以套用於任何 PCI 裝置的一種通用機制。對於我們在前幾節所述的裝置和驅動程式的軟體虛擬化類型，它提供了替代方案。相對於 virtio 和 vhost，PCI 傳遞會直接向訪客 VM 揭示硬體 I/O 裝置，進而將 QEMU 與主機核心從 I/O 路徑中完全移除，以消除這些軟體模擬轉移所造成的效能負面影響。

      在許多使用案例中，PCI 傳遞是一個很好的增進效能替代方案。

      • 例如，有人在其影像卡上設定 PCI 傳遞，將此傳遞技術用於遊戲。請注意，在此情況下，不會使用虛擬化來增進遊戲效能，而是要在其他作業系統中執行遊戲。若沒有 PCI 傳遞，現代遊戲的虛擬化不一定可行。

      • 最常見的 PCI 傳遞裝置類型可能是網路卡。例如，您可以使用 vhost_net 執行網路卡傳遞，這會從網路資料路徑移除 QEMU，但 vhost_net 還是仰賴主機核心的完整網路堆疊。

      • 值得注意的是，PCI 傳遞通常對主機匯流排配接卡 (HBA) 和磁碟而言是沒有用的，因為大部分的伺服器只有一個磁碟控制器，而該控制器連接伺服器上的所有磁碟。因此，雲端中的 VM 無法共用那些磁碟的容量；所有磁碟都屬於單一 VM。下一節說明以 SR-IOV 啟用的 HBA 如何建立此難題的解決方案。

      • 您可以想像：對於 HBA 有一個 PCI 傳遞的機密應用程式，其中的雲端可以從 USB 啟動，然後所有磁碟儲存空間容量均配置給作為後端資料庫執行的單一 VM。

      請務必記住，PCI 傳遞需要每個訪客 VM 有專用的 I/O 裝置。此需求會讓每個 VM 的專用硬體減少因虛擬化而帶來的彈性，並大量增加成本。幸好，可以使用一些較新類型的硬體，以降低內建、硬體協助呈現虛擬化 PCI 裝置的成本。此技術稱為 SR-IOV。

      SR-IOV

      較新的高階網路介面和磁碟控制器均含有 SR-IOV（單一根 I/O 虛擬化）。基本上，這些硬體裝置會自行顯示為多個 PCI 裝置，且每一個裝置均連接至 VM。在一般部署中，會有一個主要裝置（即母項，或 PF），接著是一個或多個稱為虛擬功能裝置 (VF) 的邏輯裝置。透過配置 PF，可以建立其他 VF 裝置及其相關聯資源。然後，透過 PCI 傳遞的方式將這些 VF 連接至個別的 VM。

      可能的話，您可以對每個訪客 VM 中的兩個 I/O 裝置使用 PCI 傳遞：網路介面和磁碟控制器。例如，使用兩個 SR-IOV 裝置，您可以增進網路和儲存空間的 I/O 效能。

      摘要

      您可以使用我們在本文中說明的途徑（軟體途徑或硬體途徑），來增進雲端的工作負載效能。使用 virtio 和 vhost 是最佳化軟體途徑，而使用 PCI 傳遞和 SR-IOV 是硬體協助最佳化途徑。
