      IBM Blue Box 網路使用案例

      • by Leslie Lundquist and Dustin Lundquist ● September 7, 2016

      • Tags:

        • neutron

        • neutron

        • 使用案例

        • 網路

        • 白皮書

      最佳作法白皮書

      作者：Leslie Lundquist 和 Dustin Lundquist

      2016 年 7 月 30 日

      背景：IBM Bluemix Private Cloud 在 IBM Cloud 資料中心內執行。每一個 IBM Cloud 資料中心都建置為包含多個 Pod，它們形成子網路，這些子網路在全球所有的 IBM Cloud 資料中心之間共用一個大型 IP 位址空間。

      將網路分割成個別的 Pod 來限制任何單一網路的大小是很好的作法：它可限制第 2 層 (L2) 問題的範圍，並隔離其失敗網域。也就是說，透過使子網路變小，當它中斷時可能受到影響的項目就變少，因而更容易診斷。

      不過，因為這項設計是以 Pod 為基礎，如果客戶的主機最終到達不同的 Pod，則客戶需要利用第 3 層 (L3) 網路連線功能。Pod 會分割成各自含有 32 個位址的小區塊，因此當您的雲端變大時，IBM Cloud 無法保證您一定能讓另一個主機進入現有的 Pod。因此，客戶需要規劃多 Pod 案例成為常態。

      IBM Blue Box 雲端的客戶使用案例

      本節逐步執行 IBM Bluemix Private Cloud 客戶在 IBM Cloud 資料中心 IP 位址綱目的環境定義中遇到的數個客戶使用案例，並提供 IBM Bluemix Private Cloud 工程師已經實作的一般方法或解決方案。

      1. 私有網路之間的連線功能：

      在此情況下，客戶想要在兩個私有內部 OpenStack 網路之間進行通訊；因此，客戶建立兩個私有內部網路，並使用 OpenStack 路由器連接它們。

      註解：只要這兩個客戶網路是在個別的 OpenStack 路由器背後，您就可以與 Neutron 一樣嘗試容許並提供重疊的 IP 位址空間：您可以輕易地在個別路由器背後提供重疊的 IP 空間。不過，如果您有單一路由器，則直接連接至單一路由器背後的子網路不應該重疊。

      接下來的兩個使用案例是相關的：客戶的雲端與公用網際網路之間的入埠和出埠資料流量。這些可能是任何客戶或網站最常見的使用案例。

      2. 從公用網際網路的傳入（入埠）連線功能：

      在此情況下，客戶需要從公用網際網路存取其訪客 VM。若要建立解決方案，客戶需要將 IP 位址（最好是可公開遞送的 IPv6，或是浮動 IPv4）指派給每一個訪客 VM，而且客戶必須確定其安全群組是設定為容許其 VM 的適當入埠存取。

      3. 到公用網際網路的傳出（出埠）連線功能：

      在此情況下，客戶需要從其訪客 VM 存取網際網路，例如下載安裝套件。通常，訪客 VM 應該會在與其 Pod 相關聯的 Vyatta 上，透過 NAT 取得網際網路的出埠連線功能。

      註釋：使用案例 2 與 3 是參照廣域網際網路連線功能。我們在本文件其他地方說過，通常當您結合 IP 位址空間時會有一些重疊，在此情況下，您必須使用 NAT。這不是大家樂見的解決方案。替代方案是您可以建立自訂的網路解決方案；然而，這些解決方案通常很脆弱，就像所謂拜占庭式 (Byzantine) 建築，需要投入很多時間和精力來進行維護及維修。

      4. 存取 SoftLayer 服務：

      有時候，客戶必須能夠存取在 IBM Cloud 資料中心的私有網路上代管的服務。從安全觀點來看，只有絕對必要的訪客 VM 才應該存取其私有網路上的 IBM Cloud 代管服務。此存取權是由 Pod 的相關聯 Vyatta 上的 NAT 提供。 IBM Cloud 服務是在 10.0.0.0/14 網路上代管。

      註解：接下來的兩個使用案例 BYOIP 和 DirectLink，實際上是一體的兩面。它們只是使用 VPN 或 DirectLink（這可能是租用線路或 MPLS 電路）來提供其現有/遠端網路連線功能的機制。（MPLS 是 Multi-Protocol Label Switching 的縮寫，代表多重通訊協定標籤切換，是企業客戶常使用的一種 VPN 類型。）因此，它實際上就是 VPN。

      5. BYOIP 連線功能：

      在此情況下，客戶想要將他們自己的子網路帶到 IBM Bluemix Private Cloud 中，並從其內部部署的資料中心連接到其中。若要建立解決方案，我們可以建立套版 OpenStack 網路，其中含有客戶提供的子網路值。這些套版網路連接到已停用 SNAT 的 “default” OpenStack 路由器。我們將確保會為每一個客戶子網路將路由設定在相關聯的 Vyatta 中。此解決方案需要在 IBM Bluemix Private Cloud 與客戶的資料中心之間進行通道作業。

      6. DirectLink 連線功能：

      根據客戶所選取的 DirectLink 選項，我們可以在其 IBM Bluemix Private Cloud 內設定適當的網路。最常見的使用案例是設定網路作為另一個 BYOIP 網路。不過，在此情況下，我們需要為 DirectLink 使用不同的通道作業端點。

      註解：在這些情況下，客戶想要將他們的現有 IP 位址計劃和網路計劃延伸至其 IBM Bluemix Private Cloud。這是 hybrid（混合式），其中含有一個小寫 “h”，而不是大寫 “H”。它不是我們向大部分人銷售的 “Hybrid cloud”（混合式雲端），其後來意謂著在公有雲和私有雲之間建立連線功能。

      7. SoftLayer 私有網路連線功能：

      在此情況下，客戶需要在以兩個不同的 IBM Bluemix 帳戶執行的兩個雲端之間建立連線：一個是 IBM Bluemix Private Cloud（在 IBM Bluemix 資料中心）帳戶，另一個則是客戶自己的 SoftLayer 帳戶。此狀況需要在兩個 SoftLayer 帳戶之間啟用 VLAN 跨越（否則無關）。

      • a. 如果此客戶在這兩個雲端中都使用 IBM Cloud 資料中心可遞送的 10.x.x.x 子網路，則不需執行任何動作（假設這兩個帳戶之間的 VLAN 跨距已啟用的話）。
      • b. 如果此客戶是使用 BYOIP 網路，則 IBM Bluemix Private Cloud 與客戶以另一個 SoftLayer 帳戶執行的雲端之間需要進行通道作業。

      註解： 對案例 (a.) 而言，跨帳戶 VLAN 跨越是一項 IBM Cloud 資料中心功能，它實際上應該稱為「跨帳戶 VLAN 間遞送」，因為它不是在 L2 操作，而是在 L3。這個案例僅表示在 SoftLayer 現有的 IP 位址計劃內跨越兩個 IBM Cloud 帳戶。對案例 (b.) 而言，IBM Cloud 工程師必須判定及說明哪種類型的通道才可以運作。

      問題和解決方案聲明

      由於必須套用至那些 IPv4 使用案例（如本文件所述）的解決方案很複雜，建議客戶如需要配置較大的子網路時可慎重考慮採用 IPv6 位址，例如客戶想要指派大型專用雲端位址時想擁有的位址。儘可能轉換成 IPv6 很合理。現在我們建議客戶在接下來這 5 年要考慮完整移轉至 IPv6，這是最佳作法。

      結論

      本文件的建議 - 關於使用 IPv6 位址 - 讓全球每個人都能夠使用廣域 IP 儲存區進行工作。它解決許多（即使不是全部）這些客戶使用案例中所說明的難題，但就目前而言，必須由虛擬網路綱目（例如本文件所說明者）來解決。之所以能夠處理這些實作的複雜性，主要是因為 IBM 工程團隊非常熟悉所涉及的問題和利弊得失。

      廣域網際網路位址遞送表是目前可用的最大 IP 位址計劃，其連貫這一切。如果您使用該計劃來進行工作，並遵循規則，則您的位址將會是獨一無二的。這就是為什麼我們建議移轉至 IPv6。截至目前為止，本文件所說明的使用案例可讓您查看這些可運作的解決方案，IBM Bluemix Private Cloud 工程師建立它們來超越 IPv4 位址的廣域限制，以及必要的 IBM Cloud 資料中心的 Pod 結構。

      • 用語附註：遞送表是來自網路中某個特定點的 IP 位址空間的視圖。
