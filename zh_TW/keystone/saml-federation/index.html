      何謂 SAML 聯合

      「安全主張標記語言 (SAML)」通訊協定可以讓使用者使用「單一登入 (SSO)」搭配「身分提供者 (IDP)」和「服務提供者 (SP)」。使用者登入時，會將其認證提供給 IDP。IDP 會檢查其認證，然後使用該使用者的屬性建立「SAML 主張」。接著，將該主張傳遞給 SP，SP 會驗證 IDP 所傳送的主張。因為 SP 信任來自 IDP 的主張，所以使用者從不需將其認證傳送至 SP。

      Keystone 將用作 SP，並透過接收來自 IDP 的主張鑑別使用者。在本說明文件中，我們將使用稱為 Company XYZ IDP 的虛構 SAML IDP。

      請注意，此頁面的任何管理指令都需要使用者具有 cloud_admin 存取權。

      整合 IDP 與 Keystone

      Bluemix Private Cloud 操作員必須與 Company XYZ IDP 整合工程師一起合作，以設定兩個系統來一起運作。

      Bluemix Private Cloud 操作員需要下列資訊：

      • IDP Meta 資料 URL 端點（我們目前只支援透過 URL 而非檔案接收 meta 資料）
      • IDP meta 資料中的 IDP 實體 ID
      • 使用者屬性 - 這是由 IDP 所傳遞的 SAML 使用者屬性。您可以在 IDP 說明文件或 IDP 伺服器端日誌中找到此資訊。
      • 預設/起始對映 - 如需對映的說明文件，請造訪：http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination

      Company XYZ IDP 整合工程師可能需要 Bluemix Private Cloud 操作員提供下列資訊：

      • 服務提供者 meta 資料 URL
      • 服務提供者重新導向 URL（也稱為回呼 URL，這是 IDP 容許重新導向移至的白名單值）

      Bluemix Private Cloud 操作員應與整合工程師一起合作，以建立起始/預設對映。此對映可以讓 IDP 上的遠端使用者對映至 SP 上的本端群組。本文件的其餘部分可協助釐清何謂對映及其用法。整合工程師也需要 cloud_admin 存取權。這可以是 cloud_admin 使用者，或具有相同名稱之角色的使用者。

      應以對映名稱給定使用者屬性。例如，給定的屬性可以是：

      attributes:
- name: http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress
  id: email
- name: http://schemas.xmlsoap.org/claims/Group
  id: group

      給定的名稱是在 SAML 主張內的名稱。此 ID 是要用於 Keystone 對映的 ID。

      在 Bluemix Private Cloud 操作員完成整合 Company XYZ IDP 與 Keystone 之後，應會建立稱為 mapping-for-company-xyz-idp（或其中含有 IDP 名稱的類似對映名稱）的對映。

      管理對映的步驟如下：

      1. 管理群組（聯合使用者的存取權與群組相同） 2. 管理對映

      管理群組

      「Keystone 聯合」的運作方式：將「身分提供者」使用者聯合成「服務提供者」上的一個群組。將專案中的角色指派給該群組，藉以授與該群組存取權。可用的角色如下：

      • project_admin - 專案的 admin 角色。
      • cloud_admin - 雲端執行實例的 admin 角色。
      • _member_ - 專案的成員。
      • heat_stack_owner - Heat 堆疊的擁有者。

      若要列出群組，請執行下列指令：

      $ openstack group list

      若要建立群組，請執行下列指令：

      $ openstack group create --help
$ openstack group create <new group name here>

      若要列出角色，請執行下列指令：

      $ openstack role list

      若要列出角色指派，請執行下列指令：

      $ openstack role assignment list --name

      若要列出專案，請執行下列指令：

      $ openstack project list

      若在查看新增角色指派時需要說明，請執行下列指令：

      $ openstack role add --help

      若要新增群組的角色指派以取得專案中的某一角色，請執行下列指令：

      $ openstack role  add <role> --project <project> --group <group>

      聯合使用者會對映至「服務提供者」的群組。

      以下是授與群組存取權的作業順序：

      1. 首先，建立或編輯群組。 2. 接著，建立或編輯專案中群組的角色指派。

      範例 1. 建立 company_xyz_admin_group 群組。

      $ source cloud_adminrc # 您的 Openstack 認證檔
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      接著，建立角色指派（如果尚未存在的話）。 這些指令會新增 “demo” 專案中群組 cloud_admin 的 cloud_admin 角色。

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      執行 openstack role assignment list 指令之後，您應該會看到類似下面內容的輸出：

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      範例 2. 建立具有成員存取權的群組。

      $ source cloud_adminrc # 您的 Openstack 認證檔
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      輸入 openstack role assignment list 指令之後，您應該會看到類似下面內容的輸出：

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      管理對映

      在下列範例中，我們要使用 Company XYZ IDP，而且該 IDP 的對映已經存在。具有 cloud_admin 存取權（使用者或角色）的使用者可以編輯此對映。但此使用者不能建立或刪除對映。

      可以在 Horizon 上管理對映：使用「身分儀表板」->「聯合」->「對映」->「編輯」

      openstack mapping list 指令顯示對映清單。它應顯示 mapping-for-company-xyz-idp 對映。

      openstack mapping show mapping-for-company-xyz-idp -f json 指令會顯示給定對映的詳細資料。它應包含對映規則的 json 清單。

      openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json 指令會更新及設定對映。

      對映檔應遵循以下給定的格式：

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      規則檔應包含將遠端使用者對映到本端群組的規則清單。本端區段所含的規則可指定遠端使用者應對映的群組。需要有本端群組。

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      遠端區段所含的規則可指定應容許哪些遠端使用者進行存取。

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      可以在遠端規則中給定的可能使用者屬性類型如下：

      用來過濾遠端屬性的可能動作如下：

      empty
any_ony_of
not_any_of
blacklist
whitelist

      若要設定使用者名稱，可以在本端區段中使用 {0} 以表示使用遠端區段中的欄位。然後，我們可以新增其他資訊至新的聯合使用者名稱。我們可以藉由索引（”{i}”，其中 i 是遠端欄位的索引），在本端區段中參照遠端區段：

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "email"
      },
               ...
    ]
  }
]

      使用 Horizon

      「身分提供者」上的 OpenStack 儀表板應有標籤為使用 SSO 進行鑑別的下拉功能表。

      • 此下拉功能表應顯示 Company XYZ IDP 的標籤。請選取 IDP，然後按一下「連接」。
      • 然後，系統會將您重新導向到 IDP 登入頁面。
      • 順利使用 IDP 進行鑑別之後，您應被重新導向到 Horizon 頁面。

      使用 CLI

      附註：「指令行介面 (CLI)」用戶端只能與符合下列條件的 IDP 搭配使用： 1. 此 IDP 支援 SAML ECP 通訊協定 2. 此 IDP 支援 GET 的基本鑑別

      CLI 提供兩種類型的外掛程式： 1. v3unscopedsaml（抓取不限範圍的記號） 2. v3scopedsaml（使用記號來執行限定範圍的動作）

      1. 不限範圍的範例：

      我們要建立一個可以在環境中取用以使用 SAML 的檔案：

      # 此檔案稱為 companyxyz_saml_rc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=CompanyXYZPassword # 取代此值
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 # 取代此值
export OS_USERNAME=CompanyXYZUser # 取代此值
export OS_IDENTITY_PROVIDER=company-xyz-idp # 取代此值
export OS_PROTOCOL=saml2
export OS_AUTH_TYPE=v3unscopedsaml
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP # 取代此值

      建立檔案並編輯選項之後，我們可以執行下列指令以取得不限範圍的記號並記錄 ID：

      $ source companyxyz_saml_rc
$ openstack token issue

      2. 限定範圍的範例：

      現在，我們有了不限範圍的記號，所以可以將範圍設為專案，並列出該專案中的伺服器

      openstack server list --os-auth-type v3scopedsaml --os-token <os-token-id> --os-project-id <project-id>

      下列程式碼使用 SAML 鑑別外掛程式來透過 IDP 進行鑑別、將 SAML 主張傳送給 Keystone，以及接收不限範圍的記號。然後，使用不限範圍的記號來取得專案限定範圍的記號。接著，Script 會列出該專案中的伺服器。

      此檔案是 saml_list_servers.py：

      我們必須在執行 Script 之前取用檔案：

export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 # 取代此值

      其他注意事項

      目前，Keystone 中的聯合使用者及委託人/受託人功能有問題，造成聯合 Heat 使用者無法委派其 heat_stack_owner 角色。

      您必須將 heat_stack_owner 角色直接指派給使用者，以使用信任委派的暫行解決方法。這是在聯合登入期間所建立的使用者。使用者的網域已聯合，所以其值會是 None (Federation=None)。

      $ openstack user list

      聯合使用者的網域應為 None：openstack user show <federated user>

      我們在此需要使用者 ID：

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      另一個已知問題是在設定頁面中顯示的聯合使用者附有變更其密碼的選項。但使用者不能變更其密碼。
