      Concepts OpenStack liés au réseau

      • by Ying Tang ● 25 mai 2016

      • Tags:

        • OpenStack

        • OpenStack(OpenStack)

        • concepts(concepts)

      Le présent document propose une introduction basique à certains concepts communs liés au réseau. Vous rencontrerez ces concepts lors de l’utilisation de votre instance d’IBM Bluemix Private Cloud.

      Terminologie utilisée dans le présent document

      Concept Description
      L2 Deuxième couche (Layer-2) de l’architecture Internet ISO à 7 couches, où les paquets sont routés.
      VLAN Réseau local virtuel, un sous-réseau en réalité.
      IP Protocole IP (Internet Protocol), généralement utilisée sous forme d’abréviation pour une adresse IP, qui indique la destination d’un paquet sur le réseau.
      IPv4 Protocole IP (Internet Protocol) version 4. Adresse Internet au format xxx.xxx.xxx. Le suffixe /xx lui est généralement ajouté pour indiquer la taille du sous-réseau à l’aide d’un protocole appelé CIDR.
      CIDR Routage CIDR (Classless Inter-Domain Routing) ou mise en super réseau, qui permet de prolonger la durée de vie utile des adresses IPv4, qui peuvent se faire rares dans un pool global, et qui doivent donc être achetées en règle générale. Pour plus d’informations sur le routage CIDR, voir : http://searchnetworking.techtarget.com/definition/CIDR. Pour connaître le nombre d’IP que vous pourrez obtenir en dehors d’une plage spécifique, cliquez sur ce lien pour avoir accès à une calculatrice de taille de sous-réseau CIDR : http://www.subnet-calculator.com/cidr.php (notez que certaines de ces IPs seront pré-allouées et par conséquent non disponibles pour votre cloud).
      IPv6 Protocole IP (Internet Protocol) version 6. Protocole d’adressage Internet plus récent qui fournit un grand nombre d’adresses réseau mais qui peut rendre les architecture réseau plus complexes.
      NIC Carte d’interface réseau (Network Interface Card), composant matériel qui permet de connecter les serveurs à un réseau. Les cartes d’interface réseau virtuelles (composant logiciel qui fonctionne comme une carte d’interface réseau) sont fréquemment utilisées dans un cloud, et le terme “interface réseau” se réfère généralement à une carte d’interface réseau virtuelle, sauf si le contexte indique qu’elle se réfère à un matériel spécifique.
      MAC Identificateur individuel qui ressemble à un numéro de série matériel et qui identifie de manière unique chaque hôte figurant sur un réseau.

      Concepts de base liés au réseau

      Concept Description
      Réseau Segment Layer-2 (L2) isolé, similaire à un réseau local virtuel dans une mise en réseau standard. Un réseau peut avoir un ou plusieurs sous-réseaux.
      Sous-réseau Plage d’adresses IPv4.
      Port Point de connexion permettant de connecter un périphérique unique, comme la carte d’interface réseau d’un serveur virtuel, à un réseau virtuel. Le terme “port” décrit également la configuration réseau associée, comme l’identificateur MAC et les adresses IP à utiliser sur ce port.
      Adresse IP fixe Adresse IP affectée à une instance lors de l’amorçage, comme une adresse IP réseau affectée à une instance qui doit atteindre l’Internet public.
      Adresse IP flottante Adresse IP affectée à une instance à la demande, à partir d’un pool, souvent au format IPv6.
      Quota Un quota limite le nombre de ressources disponibles, telles que les allocations de type calcul ou disque, pour votre cloud.
      Groupes de sécurité Ensemble de règles de filtrage réseau, spécifique à votre projet, qui est appliqué aux interactions réseau d’une instance en spécifiant les IP qui peuvent ou ne peuvent pas envoyer de paquets à une autre IP ou un autre port.
      Règle de groupe de sécurité Liste de spécifications de protocole, port et adresse IP qui établissent la manière dont le trafic est autorisé à circuler vers/à partir de n’importe quelle instance donnée sur le réseau ; il s’agit des caractéristiques d’un groupe de sécurité.

      Type de réseau Cloud

      En règle générale, votre cloud possède un ou plusieurs réseaux internes et externes. Imaginez que les instances de votre cloud peuvent “se parler” entre elles sur les réseaux internes, et qu’elles “parlent” aux sources en dehors de votre cloud privé sur les réseaux externes.

      Plusieurs termes permettent de décrire les réseaux interne et externe. Par exemple, les réseaux internes sont parfois appelés “réseaux titulaires”. Les réseaux externes utilisés pour communiquer avec l’Internet public sont parfois appelés “réseaux fournisseurs”.

      Réseaux internes

      Un réseau interne “communique” au sein de votre cloud ; il établit une connectivité entre toutes les interfaces réseau virtuelles (NIC) qui y sont connectées. Chaque interface réseau interne est associée à une machine virtuelle. Une machine virtuelle unique peut être associée à plusieurs interfaces réseau, ce qui signifie qu’elle peut “communiquer” sur un grand nombre de réseaux.

      Vous pouvez établir des communications entre plusieurs machines virtuelles en les connectant au même réseau interne. Les réseaux internes sont associés à un projet unique (titulaire) et ne sont visibles et accessibles que pour ce projet.

      Réseaux externes

      Les réseaux externes peuvent “communiquer” au sein et en dehors de votre cloud. Il peut s’agir de l’Internet public, du réseau externe de votre cloud et du réseau IBM Cloud Data Center, qui contient le routeur IBM Cloud Data Center.

      • Le “réseau public” ou le “réseau externe” de votre cloud offre un accès à/depuis l’Internet public.
      • Le réseau privé SoftLayer est destiné à offrir un accès à d’autres services de SoftLayer, y compris les autres clouds avec lesquels vous souhaitez communiquer. (Pour plus d’informations sur l’accès au réseau privé SoftLayer, consultez les Notes sur l’édition)
      • Le réseau du routeur SoftLayer se connecte au routeur BCR de SoftLayer, qui vous permet d’atteindre des ressources en dehors de votre propre environnement de cloud privé. Ces ressources sont susceptibles de résider dans un centre de données sur votre site (Blue Box Local) ou dans un autre environnement de cloud privé que vous possédez, dans un centre de données SoftLayer ou à un autre emplacement (Blue Box Dedicated ou votre propre cloud existant).
      • Les réseaux publics et les réseaux privés SoftLayer sont des réseaux physiques.

      Connectivité entre machines virtuelles sur différents sous-réseaux privés ?

      Supposons que vous avez deux réseaux privés. Vous avez créé des machines virtuelles sur chaque sous-réseau et vous avez ajouté les groupes de sécurité nécessaires afin qu’ils puissent accepter le trafic entrant. Comment établir la connectivité entre les machines virtuelles sur différents sous-réseaux privés ? Supposons que vous avez un réseau nommé operations et un réseau nommé internal. Vous pouvez établir une communication entre ces deux réseaux de différentes manières :

      • Première méthode, connectez votre réseau operations au routeur par défaut. Cette connexion permet au trafic de circuler entre le réseau internal par défaut et le réseau operations. Vous pouvez maintenant déployer une instance sur le réseau internal, lui attribuer une adresse IP flottante publique, et utiliser cette instance pour vous connecter à l’instance cible.
      • Deuxième méthode assez similaire, créez un routeur interne entre le réseau internal et le réseau operations. Utilisez une instance avec une IP publique sur le réseau internal afin que puissiez accéder à votre machine virtuelle sur le réseau operations. En créant un routeur interne, vous garantissez que les machines virtuelles du réseau operations n’auront pas accès à Internet en externe.
      • Troisième méthode, créez une instance multihébergée. Créez une instance connectée aux réseaux internal et operations. Attribuez-lui une IP flottante publique. Utilisez cette machine virtuelle pour offrir un accès aux instances de votre réseau operations par l’intermédiaire d’une seconde interface privée.

      Quelles sont les précautions à prendre sur les machines en matière de sécurité avec les IP flottantes publiques ?

      Voici quelques bonnes pratiques :

      • Configurez sshd pour autoriser l’authentification par clé uniquement, et désactivez l’accès ssh root.
      • N’utilisez pas de noms d’utilisateur système standard ou par défaut pour SSH (ubuntu, deploy et similaire).
      • Utilisez un service système tel que fail2ban ou denyhosts pour bloquer les éventuelles attaques par force brute. Les deux outils sont fournis dans les distributions Linux les plus courantes.
      • Utilisez des règles de groupe de sécurité pour limiter l’accès SSH à un nombre limité d’adresses IP.
      • Mettez régulièrement à jour le système s’exécutant sur votre instance publique.
      • N’exécutez aucun service inutile sur votre instance publique (tel que http, ftp, smb, etc.)
