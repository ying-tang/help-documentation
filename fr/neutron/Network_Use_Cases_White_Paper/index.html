      Livre blanc sur les meilleures pratiques

      par Leslie Lundquist et Dustin Lundquist

      30 juillet 2016

      Arrière-plan : IBM Bluemix Private Cloud s’exécute dans les centres de données IBM Cloud. Chaque centre de données IBM Cloud est conçu pour contenir plusieurs capsules, formant des sous-réseaux, lesquels partagent un espace adresse IP étendu sur tous les centres de données IBM Cloud dans le Monde.

      Il est recommandé de diviser le réseau en plusieurs capsules distinctes afin de limiter la taille de n’importe quel réseau unique. Les problèmes liés à la portée de Layer 2 (L2) sont ainsi restreints et le domaine défaillant de celui-ci est isolé. En d’autres termes, en réduisant la taille du sous-réseau, certains éléments peuvent être affectés lorsqu’il tombe en panne, le diagnostic en est donc facilité.

      Cependant, en raison de cette conception basée sur des capsules, le client doit utiliser la connectivité du réseau Layer 3 (L3) si ses hôtes se retrouvent dans différentes capsules. Les capsules sont divisées en plus petits blocs de 32 adresses chacun. Ainsi, à mesure que votre cloud se développe, IBM Cloud ne peut pas garantir que vous pouvez toujours obtenir un autre hôte dans une capsule existante. Les clients doivent donc suivre la norme et prévoir d’utiliser plusieurs capsules.

      Scénarios d’utilisation client pour IBM Blue Box Cloud

      La présente section passe en revue différents scénarios d’utilisation client rencontrés par les clients d’IBM Bluemix Private Cloud dans le cadre du schéma d’adresse IP du centre de données IBM Cloud. Elle fournit également des approches ou des solutions d’ordre général en cours d’implémentation par les ingénieurs IBM Bluemix Private Cloud.

      1. Connectivité entre les réseaux privés :

      Dans ce scénario, un client souhaite communiquer entre deux réseaux OpenStack internes privés ; il crée donc deux réseaux internes privés et les connecte en utilisant un routeur OpenStack.

      Commentaire : tant que les deux réseaux client se trouvent derrière des routeurs OpenStack distincts, vous êtes au même niveau que Neutron, qui tente d’autoriser et de fournir un espace adresse IP de chevauchement : vous pouvez facilement fournir un espace IP de chevauchement derrière des routeurs distincts. Toutefois, si vous ne disposez que d’un seul routeur, les sous-réseaux directement connectés à l’arrière d’un routeur unique ne devraient pas se chevaucher.

      Les deux scénarios d’utilisation suivants sont liés, à savoir le trafic entrant et sortant entre le cloud du client et l’Internet public. Il s’agit probablement des scénarios d’utilisation les plus fréquents destinés aux clients ou aux sites.

      2. Connectivité entrante à partir de l’Internet public :

      Dans ce scénario, un client doit accéder à ses machines virtuelles invités à partir de l’Internet public. Pour créer une solution, le client doit affecter une adresse IP (de préférence IPv6 routable publiquement, sinon IPv4 flottante) à chaque machine virtuelle invité, puis vérifier que ses groupes de sécurité sont configurés de sorte à autoriser l’accès entrant approprié à ses machines virtuelles.

      3. Connectivité sortante vers l’Internet public :

      Dans ce scénario, un client doit accéder à Internet à partir de ses machines virtuelles invités, pour télécharger des packages d’installation par exemple. En règle générale, les machines virtuelles invités doivent obtenir la connectivité sortante vers Internet via le processus de conversion d’adresses réseau sur le routeur Vyatta associé à leur capsule.

      Commentaire : les scénarios d’utilisation 2 et 3 font référence à la connectivité Internet globale. En général, comme il a déjà été mentionné dans le présent article, lorsque vous combinez des espaces adresse IP, vous constatez des chevauchements, auquel cas vous devez utiliser le processus de conversion d’adresses réseau. Cette solution n’est pas souhaitable. Sinon, vous pouvez créer une solution de mise en réseau personnalisée, mais cette solution est souvent fragile, car complexe, et nécessite un investissement important en temps et en énergie dédié à la maintenance et à l’entretien.

      4. Accès aux services SoftLayer :

      Un client doit parfois accéder aux services hébergés sur les réseaux privés du centre de données IBM Cloud. D’un point de vue sécuritaire, seules les machines virtuelles invités strictement requises doivent avoir accès aux services hébergés IBM Cloud sur leur réseau privé. Cet accès est fourni par le processus NAT sur le routeur Vyatta associé à la capsule. Les services IBM Cloud sont hébergés sur le réseau 10.0.0.0/14.

      Commentaire : les deux scénarios d’utilisation suivants, BYOIP et DirectLink, constituent les deux facettes d’une même procédure. Ils décrivent le mécanisme permettant de fournir la connectivité à leurs réseaux existants/distants, à l’aide d’un réseau privé virtuel ou de DirectLink, qui peut être une liaison louée ou un circuit à commutation multiprotocole par étiquette (MPLS). (MPLS signifie “commutation multiprotocole par étiquette”. Il s’agit d’un type de réseau privé virtuel souvent utilisé par les clients d’entreprise). Ces deux scénarios d’utilisation reposent donc effectivement sur le réseau privé virtuel.

      5. Connectivité BYOIP :

      Dans ce scénario d’utilisation, un client souhaite importer ses propres sous-réseaux dans IBM Bluemix Private Cloud et s’y connecter à partir de son centre de données sur site. Pour créer une solution, vous pouvez créer des réseaux OpenStack de chevauchement avec les valeurs de sous-réseau fournies par le client. Ces réseaux superposés sont connectés à un routeur OpenStack “par défaut” pour lequel la conversion SNAT est désactivée. Vous devrez vérifier que les routes sont configurées dans le routeur Vyatta associé pour chacun de ces sous-réseaux client. Cette solution nécessite une tunnellisation entre IBM Bluemix Private Cloud et le centre de données du client.

      6. Connectivité DirectLink :

      Selon l’option DirectLink sélectionnée par le client, vous pouvez configurer les réseaux appropriés dans son instance d’IBM Bluemix Private cloud. Le scénario d’utilisation le plus fréquent consisterait à configurer un réseau en tant que réseau BYOIP. Ce processus exige toutefois d’utiliser des noeuds finaux de tunnellisation différents pour DirectLink.

      Commentaire : dans ces scénarios, les clients souhaitent étendre leur plan d’adresse IP et leur plan de réseau existants sur IBM Bluemix Private Cloud. Il s’agit d’une solution hybride avec un “h” minuscule et non un “H” majuscule. Il ne s’agit pas du “cloud Hybride” tel que nous le vendons, qui revient à créer une connectivité entre les clouds public et privé, pour la plupart des utilisateurs.

      7. Connectivité de réseau privé SoftLayer :

      Dans ce scénario, un client doit établir une connectivité entre deux clouds s’exécutant dans deux comptes IBM Bluemix distincts : le premier est un compte IBM Bluemix Private Cloud (dans le centre de données IBM Bluemix) et le second est le compte SoftLayer du client. Ce processus nécessite d’activer le chevauchement des réseaux locaux virtuels entre les deux comptes SoftLayer (sans quoi ils ne sont pas liés).

      • a. Si ce client utilise le sous-réseau 10.x.x.x routable d’un centre de données IBM Cloud dans les deux clouds, aucune action n’est requise en supposant que le chevauchement des réseaux locaux virtuels a déjà été activé entre ces deux comptes.
      • b. Si ce client utilise des réseaux BYOIP, la tunnellisation doit être activée entre IBM Bluemix Private Cloud et le cloud du client qui s’exécute dans un autre compte SoftLayer.

      Commentaire : pour le scénario (a.), le chevauchement des réseaux locaux virtuels entre les comptes est une fonctionnalité du centre de données IBM Cloud qui devrait s’appeler “routage entre les réseaux locaux virtuels entre les comptes”, car elle ne fonctionne pas au niveau L2, mais au niveau L3. Ce scénario décrit le chevauchement de deux comptes IBM Cloud au sein du plan d’adresse IP existant de SoftLayer. Pour le scénario (b.), les ingénieurs IBM Cloud doivent déterminer et expliquer le type de tunnel qui pourrait fonctionner.

      Informations sur les problèmes et leurs solutions

      Etant donné la complexité des solutions qui doivent être appliquées à ces scénarios d’utilisation IPv4 (telles que décrites dans le présent article), il est recommandé aux clients d’envisager sérieusement l’adoption d’adresses IPv6 lorsqu’ils doivent allouer des sous-réseaux de plus grande taille, pour l’affectation d’adresses à un cloud dédié de grande taille par exemple. La conversion en IPv6 est judicieuse dans la mesure du possible. Il est désormais recommandé aux clients d’envisager une migration intégrale vers IPv6 sur les 5 prochaines années.

      Conclusion

      Les recommandations présentées dans le présent article (pour l’utilisation d’adresses IPv6) permettent à tous les utilisateurs de travailler en dehors du pool d’IP global dans le Monde entier. Elles résolvent la plupart des difficultés, si ce n’est toutes, décrites dans ces scénarios d’utilisation client, qui doivent pour le moment être traités par des schémas de mise en réseau virtuels comme ceux décrit dans le présent article. La complexité de ces implémentations peut être gérée essentiellement grâce à l’équipe d’ingénieurs IBM qui connaît très bien les problèmes et les compromis impliqués.

      La table de routage d’adresse Internet globale est le plan d’adresse IP disponible le plus étendu, qui fusionne toutes ces recommandations. Si vous travaillez en dehors de ce plan et respectez les règles, votre adresse sera unique. C’est la raison pour laquelle il est recommandé de procéder à une migration vers IPv6. En attendant, les scénarios d’utilisation décrits dans le présent article offrent un aperçu des solutions possibles que les ingénieurs IBM Bluemix Private Cloud ont créé pour dépasser la limitation globale des adresses IPv4 et la structure de capsule des centres de données IBM Cloud.

      • Remarque sur la nomenclature : une table de routage est une vue d’un espace adresse IP à partir d’un point spécifique dans le réseau.
