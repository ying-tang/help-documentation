      Configuration du routeur multiniveau dans IBM Bluemix Private Cloud

      • by Ulysses Kanigel et Leslie Lundquist ● 31 août 2016

      • Tags:

        • neutron

        • neutron(neutron)

        • networking(mise en réseau)

        • router(routeur)

        • configuration(configuration)

        • multi-tier(multiniveau)

        • add_interface(add_interface)

      Le présent article décrit comment tester une configuration de routeur multiniveau et pourquoi l’exécuter.

      Vous souhaiterez peut-être procéder à une configuration de routeur multiniveau pour votre cloud dans l’un des scénarios d’utilisation suivants :

      1. Vous souhaitez empêcher les hôtes sur ces sous-réseaux (enfants) d’utiliser des IP flottantes.
      2. Vous souhaitez isoler ces réseaux (enfants) des autres réseaux de projet ou de titulaire au sein de votre cloud, en fournissant à chacun son propre routeur (enfant) sous le routeur principal.
      3. Chacun de ces réseaux de projet isolés doit avoir un point d’entrée et un point de sortie Internet, tout en restant isolé des autres réseaux au sein de votre cloud et en n’utilisant pas d’IP flottantes.

      La méthode de création d’un réseau multiniveau consiste dans un premier temps à établir des routes statiques du routeur principal vers chacun des routeurs “enfants”. L’objectif principal de ce test est de vérifier la persistance de ces routes statiques. Cet exemple illustre la configuration de deux réseaux enfants de ce type, appelés child-net1 et child-net2.

      Supposons que vous voulez tester une configuration de routeur multiniveau et confirmer que les routes configurées sont persistantes même après la reprise.

      La procédure de test est la suivante :

      
#Create networks and subnets

neutron net-create parent-net

neutron net-create child-net1

neutron net-create child-net2

neutron subnet-create parent-subnet 162.16.0.0/24

neutron subnet-create child-subnet1 162.16.1.0/24

neutron subnet-create child-subnet2 162.16.2.0/24

# Create routers

neutron router-create parent-router

neutron router-create child-router1

neutron router-create child-router2

# Set gateway of the parent router to the external network and add interface on the parent-router subnet

neutron router-gateway-set parent-router external

neutron router-interface-add parent-subnet b490f691-364f-4f7c-866c-1c45e7b8f8cf  #unique identifier for subnet

# Add interfaces for parent-router, child-router1 and child-router2 subnets

neutron port-create parent-router

neutron port-create child-router1

neutron port-create child-router2

neutron router-interface-add child-subnet1 port=c1b305a1-f634-4828-a6ce-d5a6a058824e  #unique identifier for port

neutron router-interface-add child-subnet1 2476ad51-e2cb-4d4e-bbd8-db18cc06ea88   #unique identifier for subnet

neutron router-interface-add child-subnet2 port=c1b467b1-f444-4325-a5dc-d4b3a04624d   #unique identifier for port

neutron router-interface-add child-subnet2 2378ad51-e4cc-2d3e-bcc5-db34cd05fa48    #unique identifier for subnet

# Set default route on child router(s) to parent-router subnet gateway address : they both talk to the parent router

neutron router-update child-router1 --routes type=dict list=true destination=0.0.0.0/0,nexthop=162.16.0.1

neutron router-update child-router2 --routes type=dict list=true destination=0.0.0.0/0,nexthop=162.16.0.1

      La mise à l’échelle de cette conception de sorte à inclure plusieurs sous-réseaux connectés à chaque routeur enfant ne devrait poser aucun problème, car le routage est effectué au niveau du routeur enfant.

      Remarque : n’oubliez pas que toute instance connectée à un routeur enfant ne sera associée à aucune IP flottante, car ce routeur enfant n’est pas directement connecté au réseau externe.

      Traitement des incidents : Error: Failed to add_interface:

      Il est possible que vous obteniez cette erreur si vous essayez de connecter une interface au routeur-titulaire-A pour le réseau-titulaire-B : Error: Failed to add_interface: Request Failed: internal server error while processing your request.

      Notre politique Neutron autorise effectivement l’ajout d’une interface de routeur aux utilisateurs dotés des rôles admin, cloud_admin ou project_admin. Cependant, le processus échoue car l’API Neutron est codée de manière explicite pour bloquer les opérations sur les ressources détenues par les titulaires autres que celui auprès duquel vous êtes authentifié, à moins que vous disposiez du rôle admin, comme illustré dans le code ci-dessous :

      https://github.com/openstack/neutron/blob/stable/mitaka/neutron/api/v2/attributes.py#L971

      Si vous avez besoin d’un réseau provenant d’un titulaire connecté à un routeur dans un autre titulaire, l’équipe de support devra effectuer cette opération à votre place. Saisissez la commande CLI que vous essayez d’exécuter et qui génère l’erreur pour que nous puissions l’exécuter à votre place.
