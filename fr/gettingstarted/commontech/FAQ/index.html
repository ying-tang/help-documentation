      Comment redimensionner une instance existante ?

      Bien que l’option Redimensionner l'instance apparaisse dans Horizon, cette fonctionnalité n’est actuellement pas prise en charge. Si vous souhaitez augmenter la taille d’une instance, il est recommandé de prendre une image instantanée de l’instance et de redéployer cette image dans une nouvelle version de plus grande taille.

      Comment démarrer une instance sur un hôte spécifique ?

      Cette fonctionnalité est prise en charge via l’API OpenStack. Pour plus de détails, voir Comment déployer une instance sur un noeud d’hyperviseur spécifique.

      Je dispose d’un nouveau cloud et deux projets sont déjà installés. A quoi correspondent-ils ?

      Les projets admin et service sont créés pour prendre en charge le fonctionnement sous-jacent d’IBM Bluemix Private Cloud. Ils doivent être conservés. Ne les modifiez et ne les déployez pas.

      Quels sont les ports/URL accessibles publiquement sur Bluemix Private Cloud et dans quel but ?

      Les ports ouverts par l’intermédiaire de Bluemix Private Cloud sont les suivants :

      80 - horizon (portail Web OpenStack)
443 - horizon (portail Web OpenStack)
5000 - keystone (API Authentication OpenStack)
6080 - nova-novnc (Console VNC OpenStack)
8000 - heat (API CloudFormation OpenStack)
8004 - heat (API Orchestration OpenStack)
8774 - nova (API Compute OpenStack)
8776 - cinder (API Block Storage OpenStack)
9292 - glance (API Image OpenStack)
9696 - neutron (API Network OpenStack)
35357 - keystone-admin (API Authentication Admin OpenStack)

      Trois ports supplémentaires sont activés si vous avez demandé à configurer Swift (OpenStack Object Storage) sur votre cluster : 6000, 6001, 6002

      Tous ces ports sont disponibles via votre URL Horizon personnalisée.

      Si vous devez également restreindre les ports egress au minimum, vous devrez au moins ouvrir les ports 67/68 pour DHCP et le port 80 pour les métadonnées.

      Comment mesurer le nombre d’opérations d’entrée-sortie par seconde sur l’unité d’une instance ou d’un volume ?

      Vous pouvez utiliser plusieurs outils pour mesurer les performances du disque.

      Un script Python facile d’utilisation, créé par Benjamin Schweizer, permettant de mesurer le nombre d’opérations d’entrée-sortie par seconde est disponible avec des instructions et des exemples de résultat :

      https://benjamin-schweizer.de/measuring-disk-io-performance.html

      La dernière version de ce script est disponible ici.

      Sa syntaxe est la suivante :

      iops [-n|--num_threads unités_exéc][-t|--time temps] <unité>

      Paramètres :

      unités_exéc : nombre d'unités d'exécution d'entrée-sortie concurrentes. Valeur par défaut = 1
temps    : temps en secondes. Valeur par défaut = 10
unité  : unité par bloc. Par exemple, /dev/sda ou \\\\.\\PhysicalDrive0

      Exemples :

      iops /dev/sda
iops -n 8 -t 2 /dev/disk0
iops --num_threads 1 --time 2 /dev/md1
iops --num_threads 16 --time 2 /dev/md1

      Lorsque j’utilise des groupes de sécurité, comment consigner les connexions TCP refusées à des ports interdits ?

      Si vous souhaitez consigner les connexions TCP refusées à des ports interdits, vous devrez ouvrir des groupes de sécurité et exécuter iptables en activant la consignation sur les instances.

      Quel est le nombre maximal d’unités virtuelles (de ports par ex.) pouvant être connectées à une machine virtuelle ?

      Le nombre maximal d’emplacements PCI virtuels disponibles est 20.

      Comment obtenir la liste de mes machines virtuelles et de leurs noms d’hôte ?

      Si vous êtes connecté en tant que cloud_admin, que vous avez besoin de la liste de toutes les instances sur un cloud et de l’hôte sur lequel elles résident, mais également des noms des machines virtuelles, vous pouvez exécuter la commande suivante :

      nova list --all-tenants --fields name,OS-EXT-SRV-ATTR:host

      La commande openstack host show disk name affiche-t-elle l’utilisation réelle du disque dur par l’hôte physique ?

      Non. Elle affiche des informations sur les versions des instances qui s’exécutent sur les hôtes. A ce stade, aucune information sur l’espace de permutation n’est incluse. La section qui suit décrit les informations incluses dans le cadre de l’édition Mitaka.

      Voir aussi https://derpops.bike/2016/10/07/openstack-nova-and-hypervisor-disk-consumption/

      La réponse à cette question se trouve dans la documentation OpenStack : http://docs.openstack.org/admin-guide/common/nova-show-usage-statistics-for-hosts-instances.html

      • La colonne disk_gb [dans la sortie openstack host show] affiche la somme des tailles de disque racine et éphémère (en Go) des instances qui s’exécutent sur l’hôte. [Cette valeur est] calculée en se basant sur les informations relatives aux versions des instances qui s’exécutent sur l’hôte. Cette commande n’exécute aucune requête portant sur l’utilisation du disque dur par l’hôte physique.
      • Le calcul est disponible sur la page : https://github.com/openstack/nova/blob/c521ba7fa679b6e9c790d84ea22f04fb999987db/nova/notifications/base.py#L448-L454 disk_gb = root_gb + ephemeral_gb
      • ephemeral_gb = indique la taille d’un disque de données éphémère secondaire. Il s’agit d’un disque non formaté vide qui n’existe que pour la durée de vie de l’instance. La valeur par défaut est 0.
      • root_gb = volume d’espace disque (en gigaoctets) utilisé pour la partition racine (/).
      • La valeur disk_gb pour Nova n’inclut actuellement PAS d’informations sur l’espace de permutation (c’est-à-dire l’allocation d’un espace de permutation facultatif pour l’instance. La valeur par défaut est 0.)

      L’inclusion de cet espace de permutation facultatif a été proposée pour l’édition Newton d’OpenStack, conformément à la spécification suivante (https://specs.openstack.org/openstack/nova-specs/specs/newton/implemented/resource-providers-allocations.html) :

      “Lorsque le noeud de calcul utilise la mémoire locale pour les disques de l’instance OU qu’il a été amorcé à partir du volume, la valeur utilisée doit correspondre à la somme des valeurs de zone root_gb, ephemeral_gb et swap de la version. La valeur de resource_provider_uuid doit correspondre à l’identificateur unique universel du noeud de calcul. Pour les instances qui ont été amorcées à partir du volume, la valeur de root_gb est définie sur 0.”

      Comment créer un volume à partir d’une image instantanée et supprimer les dépendances ?

      Lorsque vous créez un volume à partir d’une image instantanée, l’image instantanée d’origine peut contenir des (méta)données dépendantes dont le ou les volumes ont besoin. Vous pouvez exécuter openstack volume show $volumeid pour rechercher le snapshot_id parent. Lorsque vous essayez de supprimer le volume, l’erreur suivante peut se produire car vous devez commencer par supprimer l’image instantanée dépendante en premier :

      Failed to delete snapshot with name or ID ...

      Pour résoudre ce problème, créez une image à partir du volume, puis créez un volume à partir de cette image :

      Etape 1 : Créez une image temporaire à partir d’un volume :

      openstack image create --volume $volumeid $image_name

      Etape 2 : Créez un volume à partir de l’image temporaire :

      openstack volume create --image $imageID --size $x

      Vous pouvez ensuite supprimer l’image temporaire pour obtenir un volume indépendant.
