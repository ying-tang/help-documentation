      Migration d'une instance vers un nouveau cloud

      • by Jason Kennedy ● 5 octobre 2015

      • Tags:

        • nova

        • nova(nova)

        • migrate(migrer)

        • migration to new cloud(migration vers un nouveau cloud)

      En raison des mises à niveau, des transferts du centre de données ou d’autres événements inattendus, il devient parfois nécessaire de migrer vos instances d’un cloud OpenStack vers un autre.

      Dans cet exemple, nous allons prendre une image instantanée d’une instance (une instance simple qui exécute nginx), la télécharger sur notre machine locale, puis la retransférer vers le nouveau cloud. Commençons par examiner notre instance.

      Nous remarquons que nginx s’exécute à l’intérieur de l’instance ; vérifions à présent l’instance seule. (Remarque : l’instance se nomme snap_test, je peux donc exécuter l’utilitaire grep sur celle-ci) :

      A partir d’ici, nous pouvons observer l’IP externe de l’instance ; examinons-la de plus près afin de vérifier que nginx est actif et en cours d’exécution :

      A présent que nous savons que nginx est actif et en cours d’exécution, passons à notre nouveau cloud.

      Remarque sur le sourçage : vous devrez avoir sourcé les données d’identification de votre cloud existant pour pouvoir exécuter les commandes de cloud nécessaires à l’exécution de la fonction de prise d’images instantanées et de la fonction de transfert. Ici, j’ai installé python-openstackclient sur mon Mac et j’ai récupéré stackrc de mon cloud, en cliquant sur Télécharger le fichier RC OpenStack à droite, comme illustré dans l’image suivante :

      Assurez-vous de sourcer stackrc :

      Prenons maintenant une image instantanée de notre instance. Etant donné que nous connaissons son ID instance grâce à la commande Nova précédente, procédons comme suit :

      Vous pouvez suivre les étapes suivantes dans Horizon si vous le souhaitez ; la tâche de votre instance devrait passer à “prise d’image instantanée en cours”, puis à “transfert en cours”. Vous devriez ensuite pouvoir visualiser votre image instantanée dans image-list :

      Une fois cette étape terminée (elle peut prendre du temps, en fonction de la taille de l’image) et que l’image apparaît comme ACTIVE, téléchargez-la sur votre machine locale :

      Une fois le téléchargement terminé, l’image est sauvegardée en local. Transférons-la immédiatement dans notre nouveau cloud. Commencez par sourcer le fichier stackrc à partir de ce cloud afin de vérifier que vous interagissez avec le bon fichier, puis transférez l’image instantanée que vous avez créée :

      Remarque sur la commande image-create lorsque vous utilisez Glance. La commande précédente se décompose comme suit :

      --container-format : format de conteneur de l’image. Formats admis : ami, ari, aki, bare et ovf.

      --disk-format : format de disque de l’image. Formats admis : ami, ari, aki, vhd, vmdk, raw, qcow2, vdi et iso. (Notez qu’OpenStack utilise qcow2 pour toutes les images en interne.)

      --name : nom de votre image.

      --file : fichier local contenant une image de disque à transférer lors de la création. Sinon, les images peuvent être transmises au client via stdin.

      Une fois la commande exécutée, initialisez une instance à partir de l’image instantanée :

      Dans la commande précédente (nova boot --image snap_migration --nic net-id=ba0fdd03-72b5-41eb-bb67-fef437fd6cb4 --security-groups jksec --flavor m1.tiny snap_migration_test), j’ai utilisé --nic net-id que j’ai obtenu en exécutant nova net-list et en récupérant l’ID du réseau interne. De même, la variable --security-groups jksec provient de l’exécution de nova secgroup-list. Les deux commandes sont exécutées à l’aide de l’API. Vous trouverez également ces deux commandes dans le panneau Horizon.

      Une fois notre instance transférée, nous devons lui attribuer une IP externe pour que nginx puisse être accessible de l’extérieur. A ce propos, assurez-vous d’autoriser le point d’entrée port 80 sur le groupe de sécurité de votre nouveau cloud afin que votre instance soit accessible.

      Testons enfin notre instance transférée afin de vérifier que nginx est actif et en cours d’exécution sans problème (n’oubliez pas que nous ne nous sommes pas connecté à l’instance et qu’il s’agit uniquement d’une procédure de type glisser-déposer pour le moment) :
