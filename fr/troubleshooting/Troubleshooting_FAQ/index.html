      Q. Comment déterminer l’horodatage auquel ma machine virtuelle a été réamorcée pour la dernière fois ?

      R. Vous pouvez exécuter cette commande :

      zgrep BOOT_IMAGE /var/log/kern.log*

      pour afficher l’horodatage auquel l’hôte (machine virtuelle) a été amorcé pour la dernière fois, puis utiliser cet horodatage pour consulter les journaux à la recherche d’un éventuel indice qui pourrait expliquer vos problèmes.

      Q. Que faire si mon instance se fige et arrête de répondre ?

      R. Il est recommandé de vérifier la console de la machine virtuelle pour voir si elle est complètement figée, ou s’il existe une preuve de panique du noyau ou de toute autre erreur. Vous pouvez obtenir l’URL de la console en exécutant :

      nova get-vnc-console VM-UUID novnc

      Notez l’heure de la console, le cas échéant.

      Effectuez une capture d’écran de la console si elle est figée et qu’elle contient des informations utiles. Après avoir réamorcé, si vous avez activé les vidages sur incident sur la machine virtuelle, vous pouvez contrôler le fichier vmcore dans /var/crash (sur RedHat) ou les fichiers .crash (sur Ubuntu) et analyser le vidage. Vérifiez également les statistiques d’activité du système (sar) à l’heure approximative du blocage.

      Réamorcez la machine si vous devez la sauvegarder immédiatement. Encore mieux, lancez une nouvelle machine à partir de la même image. Si vous pensez que l’examen de l’infrastructure doit être approfondi, essayez de nous transmettre l’heure du blocage (Temps Universel Coordonné) avant réamorçage de la machine, ainsi que l’identificateur unique universel de l’instance, pour que nous puissions examiner les journaux.

      Si vous devez basculer en mode utilisateur unique de manière fiable, pour procéder à une vérification du système de fichiers ou pour corriger les droits sur le répertoire SSH par exemple, il est recommandé d’utiliser la console en plein écran dans le navigateur Safari. Vous devrez appuyer sur le bouton “send ctrl/alt/del” pour réamorcer la machine virtuelle tout en étant connecté à la console pour pouvoir l’intercepter à l’écran de chargement GRUB. Une fois dans le menu d’amorçage GRUB, appuyez sur e pour l’éditer. Voici un exemple pour le mode utilisateur unique dans CentOS :

      1. Faites défiler l’écran vers le bas et effacez une partie de la fin de la ligne (qui commence par console) ; ne conservez que console=tty.
      2. Après modification, la ligne devrait ressembler à ce qui suit : kernel /boot/vmlinuz-2.6.32-431.el6.x86_64 ro root=UUID=8a057bba-2787-46a7-9436-fd6dc8a3aa85 rd_NO_LUKS rd_NO_LVM LANG=en_US.UTF-8 rd_NO_MD crashkernel=auto rd_NO_DM SYSFONT=latarcyrheb-sun16 KEYBOARDTYPE=pc KEYTABLE=us console=tty notsc single
      3. Vous pouvez également essayer d’ajouter rw init=/bin/bash à la place de ro sur la ligne.
      4. Si le processus d’amorçage semble vouloir s’arrêter, appuyez sur Entrée plusieurs fois.

      Q. Que faire si je dois effectuer des opérations de récupération sur mon instance ?

      R.

      1. Obtenez une image ISO CD de récupération, à partir de http://www.system-rescue-cd.org par exemple : 
        wget https://dronedata.dl.sourceforge.net/project/systemrescuecd/sysresccd-x86/4.9.6/systemrescuecd-x86-4.9.6.iso
      2. Installez isohybrid à partir du package sysutils : 
        apt-get install syslinux [ou yum install syslinux]
      3. Ajoutez l’enregistrement de démarrage principal à l’ISO pour la convertir en disque dur amorçable. 
        isohybrid systemrescuecd-x86-4.9.6.iso
      4. Transférez le nouveau CD de récupération hybride à Glance : 
        openstack image create --file systemrescuecd-x86-4.9.6.iso --public sysrescuecd-4.9.6.iso
      5. Basculez l’instance qui doit être récupérée en mode récupération : 
        nova rescue brokeninstance --image sysrescuecd-4.9.6.iso
      6. Connectez-vous à l’instance via la console VNC d’Horizon et effectuez vos opérations de récupération. Vous devrez peut-être patienter 1 à 2 minutes avant que le CD de récupération soit visible dans la console. Rechargez l’écran de la console après avoir patienté ce délai.

      (/img/fr/systemrescuefromvnc.png)

      A partir d’ici, vous pouvez utiliser les outils suivants :

      blkid
fsck /dev/vdb1
mount -t ntfs /dev/xxx /mnt/windows -o ro
ntfs-3g /dev/sda1 /mnt/windows
      1. Une fois l’instance corrigée, quittez le mode récupération et redémarrez-la à partir du disque d’amorçage normal à l’aide de la commande suivante : 
        nova unrescue brokeninstance

      Q. Pourquoi ne puis-je pas redimensionner mon instance de m1.tiny en m1.medium ?

      Il est possible que l’instance ait été créée lorsque la version m1.tiny disposait d’un disque éphémère de plus grande taille. Le processus de redimensionnement vérifie cette supposition et le redimensionnement échoue s’il n’arrive pas à la vérifier.

      Vous devrez peut-être ouvrir un ticket de demande de service pour obtenir de l’aide afin de vérifier qu’il s’agit de la cause de votre incident. Si c’est le cas, l’équipe de support trouverait une erreur dans les journaux qui ressemble à ce qui suit :

      Exception during message handling: Resize error: Unable to resize disk down.

      Une alternative à l’ouverture d’un ticket consiste à définir la version m1.medium sur la taille de disque éphémère précédente, ou de suivre les étapes de traitement des incidents suivantes permettant de créer une instance à partir d’une image instantanée : /help-documentation/gettingstarted/commontech/Instance_Resize/

      Q. Pourquoi les ports sont-ils bloqués dans Neutron ?

      Les ports peuvent se bloquer dans Neutron lors de l’instanciation de machines virtuelles à l’aide de Heat. Ce problème semble être lié à un incident dans OpenStack Neutron (édition Mitaka) provoqué par une condition d’indétermination dans Neutron. Il est donc traité comme une limitation connue. Cette condition peut également provoquer un échec des métadonnées en plus de bloquer les ports.

      Il existe plusieurs solutions palliatives :

      Solution palliative #1. Les instances peuvent être déployées sans utiliser Heat, dans la mesure du possible. Le déploiement manuel semble contourner cette condition.

      Solution palliative #2. Un temps d’attente peut être défini dans Heat, en utilisant WaitCondition à partir de l’API. Cependant, WaitCondition peut ne pas convenir parfaitement à ce scénario.

      • WaitCondition fonctionne en combinaison avec WaitConditionHandler. Ce gestionnaire crée une URL/URI surveillée pour une chaîne de texte. La présence de cette chaîne indique la réussite d’une machine virtuelle déployée et son absence, au contraire, indique un échec. Une fois qu’une machine virtuelle est instanciée, elle peut recevoir l’instruction de remplir l’URL/URI avec la chaîne de texte appropriée via un appel API PUT.

      • Il n’existe actuellement aucun mécanisme dans notre déploiement OpenStack permettant à un routeur L3 de passer cet appel API. Notre équipe d’ingénieurs propose d’utiliser ‘depends_on’. (Voir la 3ème solution ci-dessous).

      Solution palliative #3. La meilleure solution consiste à utiliser depends_on. Ce paramètre crée un scénario dans Orchestration de telle sorte que le déploiement d’une machine virtuelle “dépende de” l’existence d’autres ressources définissables, comme un routeur L3 et un réseau par exemple.

      Voici un bref exemple d’utilisation de depends_on :

      add_router_interface:
   type: OS::Neutron::RouterInterface
   properties:
     router: { get_resource: router}
     subnet: { get_resource: subnet}


server1:
    type: OS::Nova::Server
    depends on: add_router_interface
    properties:
      image: { get_param: image }

      Pour plus de détails sur les dépendances explicites, cliquez sur le lien suivant :

      • https://wiki.openstack.org/wiki/Heat/Blueprints/hot-software-config-spec

      Q. Quels sont les ports à ouvrir pour OpenStack ?

      Vous devrez au moins ouvrir les ports 67/68 pour DHCP et le port 80 pour les métadonnées. De plus, si vous souhaitez que les machines virtuelles puissent accéder aux services d’API OpenStack, ajoutez les ports décrits ici :

      /help-documentation/gettingstarted/commontech/FAQ/#which-portsurls-etc-are-publicly-accessible-on-my-bluemix-private-cloud-and-for-what-purpose

      Si l’ensemble minimal de ports requis n’est pas ouvert, vous rencontrerez peut-être des problèmes avec cloud_init : vos instances ne pourront peut-être pas démarrer correctement.

      Q. Que signifie cette erreur : “Unable to retrieve volume limit information.” ?

      Fréquemment, lorsque je charge une page dans le tableau de bord Horizon, une fenêtre en incrustation rouge indiquant “Error: Unable to retrieve volume limit information.” apparaît. Qu’est-ce que cela signifie ?

      Il s’agit d’une erreur mineure qui se produit car votre déploiement ne contient pas de stockage par blocs Cinder.

      Cette erreur peut être résolue en supprimant le noeud final Cinder du service Keystone. Vous pourriez avoir à mettre à jour vos fichiers de configuration d’automatisation, auquel cas l’équipe de support devra procéder à une vérification. Ouvrez un ticket de demande de service si vous avez besoin d’aide pour résoudre cette erreur.

      Voici un exemple d’étapes que pourrait suivre le technicien de support :

      # openstack endpoint list | grep cinder 
| 19a90853883049ed92cf4af82a24c0db | RegionOne | cinderv2     | volumev2       | True    | name  | https://... | 
| 2c3e8d0a15304788a9dc1806917b61ee | RegionOne | cinderv2     | volumev2       | True    | other     | https://... | 
| 39434faf90484a208b61a517926042b2 | RegionOne | cinderv2     | volumev2       | True    | name2    | https://... | 

# openstack endpoint delete 19a90853883049ed92cf4af82a24c0db 
# openstack endpoint delete 2c3e8d0a15304788a9dc1806917b61ee 
# openstack endpoint delete 39434faf90484a208b61a517926042b2 

# openstack service list | grep cinder 
| ef7a982aaf224c62a09c6e813ac66141 | cinderv2   | volumev2       | 

# openstack service delete ef7a982aaf224c62a09c6e813ac66141

      Vérifiez que l’automatisation est correcte :

      $ grep -A2 ^cinder ../all.yml 
cinder:  
  enabled: False 
cinderv2:  
  enabled: False

      Q. Pourquoi l’accès au port 25 ne fonctionne-t-il pas pour mes e-mails sortants ?

      R. SoftLayer n’autorise plus de connexions sortantes via le port TCP 25 (SMTP) sur les nouveaux comptes. Pour plus d’informations, consultez l’article suivant :

      https://knowledgelayer.softlayer.com/procedure/outbound-email-port-25
