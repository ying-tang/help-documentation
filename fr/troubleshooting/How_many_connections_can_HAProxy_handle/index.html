      Combien de connexions simultanées peuvent être gérées par HAProxy ?

      • by Lampros Chaidas ● 21 octobre 2016

      • Tags:

        • troubleshooting

        • troubleshooting(traitement des incidents)

        • haproxy(haproxy)

        • maxconn(maxconn)

        • load testing(test de charge)

      Remarque : le présent article n’est PAS expressément lié à OpenStack Neutron LBaaS, mais uniquement à HAProxy qui s’exécute dans les instances.

      Je consultais récemment la configuration d’équilibrage de charge d’un client. Pendant que j’examinais la page d’état HAProxy avec le client, je me suis posé la question suivante : “Combien de connexions simultanées peuvent réellement être prises en charge ?”

      Le nombre maximal de connexions par serveur de back end est-il si important ? Le présent article décrit comment déterminer le nombre de connexions simultanées pouvant être gérées par HAProxy.

      Nous allons installer HAProxy sur un serveur VPS CentOS 6 et exécuter une série de tests pour répondre à la question posée. Nous allons configurer notre utilitaire de benchmarking sur un autre serveur VPS.

      Configuration requise :

      • 2 serveurs VPS CentOS 6.x

      Les spécifications des serveurs VPS utilisés dans ces tests sont les suivantes :

      • 2 Go de mémoire RAM
      • 2 coeurs
      • CentOS 6.7

      Installation de HAProxy sur le serveur VPS HAProxy :

      Etape 1. Installez le référentiel IUS (Inline with Upstream Stable) d’HAProxy pour CentOS 6 :

        yum -y install https://centos6.iuscommunity.org/ius-release.rpm

      Etape 2. Obtenez ensuite la version la plus récente d’HAProxy (au moment de la rédaction du présent article, il s’agit de 1.6.9).

        yum -y install haproxy16u && chkconfig haproxy on

      Etape 3. Déplacez votre configuration HAproxy par défaut à un autre emplacement, au cas où vous devriez la passer en revue ultérieurement :

        mv /etc/haproxy/haproxy.cfg /etc/haproxy/haproxy.cfg-default

      Etape 4. Configurez rsyslog de sorte à recevoir les journaux d’HAProxy :

        sed -i -- 's/#$ModLoad imudp/$ModLoad imudp/g' /etc/rsyslog.conf
  sed -i -- 's/#$UDPServerRun 514/$UDPServerRun 514/g' /etc/rsyslog.conf
  echo "\$UDPServerAddress 127.0.0.1" >> /etc/rsyslog.conf
  echo "local2.* -/var/log/haproxy.log" >> /etc/rsyslog.conf

      Les commandes ci-dessus définissent rsyslog de sorte à écouter sur le port UDP 514 et le limitent aux connexions localhost uniquement. La dernière ligne place les journaux HAProxy dans /var/log/haproxy.log.

      Etape 5. Redémarrez rsyslog de sorte à appliquer les modifications :

        service rsyslog restart

      Etape 6. Ajoutez une nouvelle configuration HAProxy - /etc/haproxy/haproxy.cfg (notez la ligne maxconn 20). Conformément à cette page de manuel, “les proxys n’accepteront plus de connexions une fois cette limite atteinte.” En d’autres termes, cette configuration n’autorise que 20 connexions simultanées au maximum.

      A quoi correspond la ligne fullconn 20 ? Sur cette page de manuel, elle est définie comme étant “le nombre de connexions sur le serveur de back end qui permettront aux serveurs d’utiliser le nombre maximal de connexions.”

      global
daemon
# we need the socket for hatop statistics
stats socket /var/run/haproxy.sock mode 600 level admin
log /dev/log local4
maxconn 20

defaults
log global
timeout connect 4000
timeout client 42000
timeout server 43000

listen lampros-pool-1
bind *:80
mode http
balance roundrobin
fullconn 20
server go_http_server_1 127.0.0.1:8080 cookie go_http_server_1 check inter 2000 rise 2 fall 3

      Installation du langage Go - Procédez comme suit sur les deux serveurs :

      Installez le langage Go avec l’utilitaire screen et git. Ce langage est requis sur les deux serveurs car, sur le serveur VPS HAProxy, nous allons compiler un serveur d’applications très basique. Sur le serveur VPS benchmarker, Go est requis pour pouvoir exécuter l’utilitaire benchmarker gobench.

      Sur le serveur HAProxy, nous allons configurer un serveur d’applications basique en utilisant le langage Go :

      Etape 1. Créez un répertoire nommé /root/slowserver

        mkdir -pv /root/slowserver

      Etape 2. Ajoutez dans ce répertoire un fichier nommé /root/slowserver/slowserver.go avec le contenu suivant :

      package main

import (
    "fmt"
    "net/http"
    "time"
)

func slow_handler(w http.ResponseWriter, r *http.Request) {
    time.Sleep(3000 * time.Millisecond)
    fmt.Fprintf(w, "I'm slooooow")
}

func fast_handler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "I'm FAST!!!!")
}

func main() {
    http.HandleFunc("/fast", fast_handler)
    http.HandleFunc("/slow", slow_handler)
    http.ListenAndServe(":8080", nil)
}

      Etape 3. Compilez ce fichier avec go build :

        cd /root/slowserver
  go build

      Etape 4. Démarrez slowserver dans une session screen détachée, de sorte qu’il reste en arrière-plan :

        screen -d -m -t go_slowserver /root/slowserver/slowserver

      Etape 5. Confirmez qu’il est en cours d’exécution à l’aide de netstat -ntlp | grep slowserver (le port sur lequel il écoute et l’ID processus devraient apparaître). Voici le résultat que j’obtiens :

        netstat -ntlp | grep slowserver
  tcp        0      0 :::8080                     :::*                        LISTEN      5204/slowserver

      Test de charge - Détermination du nombre de connexions simultanées pouvant être gérées par HAProxy

      Déterminons à présent le nombre de connexions simultanées pouvant être gérées par HAProxy. Avant de commencer, rappelons un détail basique à propos d’HAProxy : “Par défaut, le mode de fonctionnement d’HAProxy permet de maintenir les connexions permanentes actives : à chaque connexion, il traite chaque demande et chaque réponse, puis laisse la connexion en veille aux deux extrémités entre la fin d’une réponse et le début d’une nouvelle demande.”

      Cette déclaration signifie que les connexions à exécution longue auront un impact plus important sur le serveur que les connexions de courte durée. De ce fait, nous allons exécuter les tests sur la fonction “slow” de notre serveur d’applications.

      Conclusion : Test de charge - Détermination du nombre de connexions simultanées pouvant être gérées par HAProxy

      En conclusion, pour tester la capacité (nombre maximal de clients connectés à HAProxy), nous devons utiliser un serveur qui ouvre une connexion TCP, mais qui la maintient ouverte sur une période prolongée (3 secondes). Cette conclusion s’oppose au test de rapidité d’un serveur, où plus le trafic injecté est important, meilleurs sont les résultats obtenus.

      En seconde conclusion, votre nombre maximal de connexions (maxconn) doit être supérieur au taux de demande (par exemple, 100 clients) multiplié par la durée (3 secondes) pour pouvoir traiter chaque demande.

      Par exemple :

      maxconn=100*3 => maxconn=300
