      Définition de la fédération SAML

      Security Assertion Markup Language (SAML) est un protocole qui permet à un utilisateur d’utiliser la connexion unique (SSO) avec un fournisseur d’identité (IDP) et un fournisseur de services (SP). Lorsqu’un utilisateur se connecte, il fournit ses données d’identification au fournisseur d’identité. Ce dernier vérifie les données et une assertion SAML est créée avec les attributs de l’utilisateur. Cette assertion est ensuite transmise au fournisseur de services, qui vérifie qu’elle a été envoyée par le fournisseur d’identité. Etant donné que le fournisseur de services approuve les assertions provenant du fournisseur d’identité, l’utilisateur n’a jamais besoin d’envoyer ses données d’identification au fournisseur de services.

      Keystone agit en tant que fournisseur de services et authentifie les utilisateurs en recevant des assertions de la part d’un fournisseur d’identité. Dans la présente documentation, un fournisseur d’identité SAML fictif appelé Company XYZ IDP est utilisé.

      Remarque : toute commande d’administration sur cette page nécessite que l’utilisateur soit doté d’un accès cloud_admin.

      Intégration du fournisseur d’identité et de Keystone

      Un opérateur Bluemix Private Cloud doit travailler en collaboration avec un ingénieur d’intégration Company XYZ IDP pour configurer les deux systèmes afin qu’ils fonctionnent ensemble.

      L’opérateur Bluemix Private Cloud a besoin des informations suivantes :

      • Noeud final de l’URL de métadonnées du fournisseur d’identité (seule l’extraction des métadonnées via une URL et non un fichier est actuellement prise en charge)
      • ID d’entité du fournisseur d’identité issu des métadonnées du fournisseur d’identité
      • Attributs utilisateur - Attributs utilisateur SAML transmis par le fournisseur d’identité. Ils sont disponibles dans la documentation du fournisseur d’identité ou dans les journaux côté serveur du fournisseur d’identité.
      • Mappage par défaut/initial - La documentation sur les mappages est accessible ici : http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination

      L’ingénieur d’intégration Company XYZ IDP peut avoir besoin des informations suivantes de la part de l’opérateur Bluemix Private Cloud :

      • URL des métadonnées du fournisseur de services
      • URL de redirection du fournisseur de services (également appelée URL de rappel, il s’agit d’une valeur sur liste blanche vers laquelle le fournisseur d’identité autorisera la redirection)

      L’opérateur Bluemix Private Cloud devrait travailler en collaboration avec l’ingénieur d’intégration pour créer le mappage initial/par défaut. Le mappage représente la méthode par laquelle les utilisateurs distants du fournisseur d’identité sont mappés aux groupes locaux sur le fournisseur de services. Le reste du présent document fournit la définition d’un mappage ainsi que son fonctionnement. L’ingénieur d’intégration aura également besoin d’un accès cloud_admin. Il peut s’agir de l’utilisateur cloud_admin ou d’un utilisateur doté du rôle portant le même nom.

      Les attributs utilisateur doivent se voir attribuer le nom du mappage. Par exemple :

      attributs :
- nom : http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress
  ID : email
- nom : http://schemas.xmlsoap.org/claims/Group
  ID : group

      Le nom donné est le nom inclus dans l’assertion SAML. L’ID est celui qui sera utilisé dans les mappages Keystone.

      Une fois que l’opérateur Bluemix Private Cloud a terminé d’intégrer le fournisseur d'identité Company XYZ à Keystone, un mappage appelé mapping-for-company-xyz-idp doit être créé (ou un mappage portant un nom similaire contenant le nom du fournisseur d’identité).

      La procédure de gestion du mappage est la suivante :

      1. Gérer les groupes (les utilisateurs fédérés auront les mêmes droits d’accès que le groupe) 2. Gérer les mappages

      Gestion des groupes

      La fédération Keystone permet de fédérer les utilisateurs du fournisseur d’identité à un groupe sur le fournisseur de services. Le groupe se voit accorder un accès lorsqu’un rôle lui est affecté sur un projet. Les rôles disponibles sont les suivants :

      • project_admin - Rôle admin pour le projet.
      • cloud_admin - Rôle admin pour l’instance cloud.
      • _member_ - Membre d’un projet.
      • heat_stack_owner - Propriétaire des piles Heat.

      Pour répertorier des groupes :

      $ openstack group list

      Pour créer un groupe :

      $ openstack group create --help
$ openstack group create <nom du nouveau groupe>

      Pour répertorier des rôles :

      $ openstack role list

      Pour répertorier des affectations de rôle :

      $ openstack role assignment list --name

      Pour répertorier des projets :

      $ openstack project list

      Pour obtenir de l’aide sur l’ajout d’une affectation de rôle :

      $ openstack role add --help

      Pour ajouter une affectation de rôle afin qu’un groupe soit doté d’un rôle sur un projet :

      $ openstack role  add <rôle> --project <projet> --group <groupe>

      Les utilisateurs fédérés sont mappés dans des groupes sur le fournisseur de services.

      La procédure permettant d’accorder un accès aux groupes est la suivante :

      1. Commencez par créer ou éditer le groupe. 2. Créez ou éditez ensuite les affectations de rôle pour les groupes sur les projets.

      Exemple 1. Création d’un groupe company_xyz_admin_group.

      $ source cloud_adminrc # Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      Créez ensuite l’affectation de rôle si elle n’existe pas. Ces commandes permettent d’ajouter le rôle cloud_admin pour le groupe cloud_admin sur le projet “demo”.

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      Après avoir exécuté la commande openstack role assignment list, un résultat similaire à ce qui suit devrait apparaître :

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      Exemple 2. Création d’un groupe avec un accès member.

      $ source cloud_adminrc #Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      Après avoir exécuté la commande openstack role assignment list, un résultat similaire à ce qui suit devrait apparaître :

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      Gestion des mappages

      L’exemple du fournisseur d'identité Company XYZ utilisé ci-dessous est associé à un mappage qui existe déjà. Un utilisateur doté d’un accès cloud_admin (utilisateur ou rôle) pourra éditer le mappage. L’utilisateur ne pourra ni créer ni supprimer les mappages.

      Les mappages peuvent être gérés dans le tableau de bord Identité d’Horizon -> Fédération -> Mappages -> Editer

      La commande openstack mapping list affiche une liste de mappages. Elle devrait contenir le mappage mapping-for-company-xyz-idp.

      La commande openstack mapping show mapping-for-company-xyz-idp -f json affiche les détails du mappage donné. Ils devraient contenir une liste json répertoriant les règles du mappage.

      La commande openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json met à jour et définit le mappage.

      Le fichier de mappage devrait être au format suivant :

      [
   {
      "local": [
        <groupe>
      ],
      "remote": [
        <condition>
      ]
   }
]

      Le fichier de règles devrait contenir une liste de règles qui mappent les utilisateurs distants à un groupe local. La section local contient une règle qui spécifie le groupe auquel les utilisateurs distants devraient être mappés. Le groupe local est requis.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      La section remote contient des règles qui spécifient les utilisateurs distants auxquels un accès devrait être accordé.

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      Les actions admises pour filtrer les attributs distants sont les suivantes :

any_ony_of
not_any_of
blacklist
whitelist

      Pour définir le nom d’utilisateur, vous pouvez utiliser {0} dans la section local pour indiquer d’utiliser une zone issue de la section remote. Vous pouvez ensuite ajouter des informations supplémentaires sur le nouveau nom d’utilisateur fédéré. Vous pouvez faire référence à la section remote dans la section local par un index (“{i}”, où i est l’index de la zone remote) :

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "email"
      },
               ...
    ]
  }
]

      Utilisation d’Horizon

      Le tableau de bord OpenStack du fournisseur d’identité doit contenir un menu déroulant intitulé S’authentifier en SSO.

      • Le menu déroulant doit contenir la valeur Company XYZ IDP. Sélectionnez le fournisseur d’identité et cliquez sur Connexion.
      • Vous serez redirigé vers la page de connexion du fournisseur d’identité.
      • Après vous être authentifié avec succès auprès du fournisseur d’identité, vous devriez être redirigé vers la page Horizon.

      Utilisation de l’interface CLI

      Remarque : le client d’interface de ligne de commande fonctionne uniquement avec les fournisseurs d’identités qui remplissent les conditions suivantes : 1. Le fournisseur d’identité prend en charge le protocole ECP SAML 2. Le fournisseur d’identité prend en charge l’authentification de base sur une méthode GET

      L’interface CLI propose deux types de plug-in : 1. v3unscopedsaml (s’approprie un jeton non défini dans la portée) 2. v3scopedsaml (utilise un jeton pour effectuer des actions définies dans la portée)

      1. Exemple non défini dans la portée :

      Créez un fichier que vous pouvez sourcer dans votre environnement afin d’utiliser SAML :

      # This file is called companyxyz_saml_rc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=CompanyXYZPassword #replace this
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 #replace this
export OS_USERNAME=CompanyXYZUser #replace this
export OS_IDENTITY_PROVIDER=company-xyz-idp #replace this
export OS_PROTOCOL=saml2
export OS_AUTH_TYPE=v3unscopedsaml
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP #replace this

      Après avoir créé le fichier et édité les options, vous pouvez exécuter la commande suivante pour obtenir un jeton non défini dans la portée et enregistrer l’ID :

      $ source companyxyz_saml_rc
$ openstack token issue

      2. Exemple défini dans la portée :

      A présent que vous disposez d’un jeton non défini dans la portée, vous pouvez définir la portée d’un projet et répertorier les serveurs sur ce projet

      openstack server list --os-auth-type v3scopedsaml --os-token <ID-jeton-se> --os-project-id <ID-projet>

      Utilisation des bibliothèques d’API Python

      Remarque : les bibliothèques d’API Python fonctionnent uniquement avec les fournisseurs d’identités qui remplissent les conditions suivantes : 1. Le fournisseur d’identité prend en charge le protocole ECP SAML 2. Le fournisseur d’identité prend en charge l’authentification de base sur une méthode GET

      Le code ci-dessous utilise le plug-in auth SAML pour s’authentifier auprès du fournisseur d’identité, envoyer l’assertion SAML à Keystone et récupérer un jeton non défini dans la portée. Le jeton non défini dans la portée est ensuite utilisé pour obtenir un jeton défini dans la portée du projet. Le script répertorie ensuite les serveurs sur ce projet.

      Ce fichier se nomme saml_list_servers.py :

      Vous devrez sourcer un fichier avant d’exécuter le script :

      # This file is called companyxyz_saml_python_rc
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 #replace this
export OS_USERNAME=CompanyXYZUser #replace this
export OS_PASSWORD=CompanyXYZPassword #replace this
export OS_PROJECT_NAME=demo
export OS_IDENTITY_PROVIDER=company-xyz-idp #replace this
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP #replace this
export OS_PROTOCOL=saml2

      Vous pouvez ensuite exécuter le script python :

      source companyxyz_saml_python_rc
python saml_list_servers.py

      Remarques supplémentaires

      Il existe actuellement un problème lié aux utilisateurs fédérés et à la fonction donneur d’autorisation/administrateur dans Keystone, qui empêche l’utilisateur Heat fédéré de déléguer son rôle heat_stack_owner.

      Vous devrez utiliser une solution palliative pour la délégation de confiance en affectant directement le rôle heat_stack_owner à l’utilisateur. Il s’agit de l’utilisateur créé au cours de la connexion fédérée. Le domaine de l’utilisateur est fédéré et peut prendre la valeur None (Federation=None).

      $ openstack user list

      L’utilisateur fédéré devrait posséder un domaine None : openstack user show <utilisateur fédéré> Ici, l’ID utilisateur est requis :

      $ openstack role add heat_stack_owner --user <ID utilisateur fédéré> --project <projet>

      Il existe un autre problème connu où un utilisateur fédéré a la possibilité de changer son mot de passe dans la page des paramètres, mais ne parvient pas à le faire.
