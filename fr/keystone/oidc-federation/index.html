      Définition de la fédération Keystone

      La fédération Keystone permet d’utiliser les identités issues d’un fournisseur d’identité sur un fournisseur de services. Les différents protocoles et formats pris en charge sont OpenIDConnect (OIDC) et Security Assertion Markup Language (SAML).

      Le fournisseur d’identité stocke et gère les données d’identification de l’utilisateur et envoie des réclamations ou des assertions au fournisseur de services. Le client peut ainsi utiliser ses données d’identification d’entreprise pour s’authentifier auprès de Bluemix Private Cloud sans avoir à envoyer son mot de passe à Bluemix Private Cloud. En règle générale, Keystone est toujours le fournisseur de services étant donné qu’il utilise des identités issues de ressources externes.

      Définition d’OpenID Connect (OIDC)

      OpenID est un protocole d’authentification. OAuth est un protocole d’autorisation. OpenID Connect est une combinaison des deux.

      OpenID Connect est une couche d’identité simple au-dessus du protocole OAuth 2.0. Il permet aux clients de vérifier l’identité d’un utilisateur final en se basant sur l’authentification réalisée par un serveur d’autorisations. Il permet également aux clients d’obtenir des informations de profil de base sur l’utilisateur final, en mode interopérable et REST.

      Le présent document décrit un mécanisme permettant à une partie utilisatrice OpenID Connect de découvrir le fournisseur OpenID de l’utilisateur final et d’obtenir les informations lui permettant d’interagir avec lui, notamment ses emplacements de noeud final OAuth 2.0.

      Fonctionnement du protocole OpenID

      1. La partie utilisatrice envoie la demande au fournisseur OpenID pour authentifier l’utilisateur final.
      2. Le fournisseur OpenID authentifie l’utilisateur.
      3. Le fournisseur OpenID envoie le jeton d’ID et le jeton d’accès à la partie utilisatrice.
      4. La partie utilisatrice envoie une demande au noeud final d’informations sur l’utilisateur accompagnée du jeton d’accès reçu de la part du fournisseur OpenID.
      5. Le noeud final d’informations sur l’utilisateur renvoie les réclamations.

      Configuration d’OIDC

      Pour configurer OIDC, un opérateur Blue Box doit travailler en collaboration avec un ingénieur d’intégration de fournisseur d’identité employé par la société du client (par exemple, Company XYZ).

      L’opérateur Bluemix Private Cloud a besoin des informations suivantes :

      • URL de redirection/rappel autorisée
      • ID client et valeur confidentielle
      • Réclamation de l’utilisateur - Réclamation OIDC
      • L’URL connue/de configuration qui fournit toutes les informations permettant d’accéder au fournisseur
      • Mappage par défaut/initial - La documentation sur les mappages est accessible ici : http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination

      L’opérateur Bluemix Private Cloud devrait travailler en collaboration avec l’ingénieur d’intégration pour créer le mappage initial/par défaut. Le mappage représente la méthode par laquelle les utilisateurs distants du fournisseur d’identité sont mappés aux groupes locaux sur le fournisseur de services. Le reste du présent document fournit la définition d’un mappage ainsi que son fonctionnement.

      Une fois que l’opérateur Bluemix Private Cloud a terminé d’intégrer le fournisseur d'identité Company XYZ à Keystone, un mappage appelé mapping-for-company-xyz-idp doit être créé (ou un mappage portant un nom similaire contenant le nom du fournisseur d’identité).

      La procédure de gestion du mappage est la suivante :

      1. Gérer les groupes (les utilisateurs fédérés auront les mêmes droits d’accès que le groupe)

      2. Gérer les mappages

      Gestion des groupes

      La fédération Keystone permet de fédérer les utilisateurs du fournisseur d’identité à un groupe sur le fournisseur de services. Le groupe se voit accorder un accès lorsqu’un rôle lui est affecté sur un projet. Les rôles disponibles sont les suivants :

      • project_admin - Rôle admin pour le projet.
      • cloud_admin - Rôle admin pour l’instance cloud.
      • _member_ - Membre d’un projet.
      • heat_stack_owner - Propriétaire des piles Heat.

      Pour répertorier des groupes :

      $ openstack group list

      Pour créer un groupe :

      $ openstack group create --help
$ openstack group create <nom du nouveau groupe>

      Pour répertorier des rôles :

      $ openstack role list

      Pour répertorier des affectations de rôle :

      $ openstack role assignment list --name

      Pour répertorier des projets :

      $ openstack project list

      Pour obtenir de l’aide sur l’ajout d’une affectation de rôle :

      $ openstack role add --help

      Pour ajouter une affectation de rôle afin qu’un groupe soit doté d’un rôle sur un projet :

      $ openstack role  add <rôle> --project <projet> --group <groupe>

      Les utilisateurs fédérés sont mappés dans des groupes sur le fournisseur de services. La procédure permettant d’accorder un accès aux groupes est la suivante :

      1. Commencez par créer ou éditer le groupe.

      2. Créez ou éditez ensuite les affectations de rôle pour les groupes sur les projets.

      Exemple 1. Création d’un groupe company_xyz_admin_group.

      $ source cloud_adminrc # Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      Créez ensuite l’affectation de rôle si elle n’existe pas.

      Ces commandes permettent d’ajouter le rôle cloud_admin pour le groupe cloud_admin sur le projet “demo”.

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      Après avoir exécuté la commande openstack role assignment list, un résultat similaire à ce qui suit devrait apparaître :

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      Exemple 2. Création d’un groupe avec un accès member.

      $ source cloud_adminrc #Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      Après avoir exécuté la commande openstack role assignment list, un résultat similaire à ce qui suit devrait apparaître :

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      Gestion des mappages

      L’exemple du fournisseur d'identité Company XYZ utilisé ci-dessous suppose qu’un mappage existe déjà pour celui-ci. Un utilisateur doté d’un accès cloud_admin (utilisateur ou rôle) pourra éditer le mappage. L’utilisateur ne pourra ni créer ni supprimer les mappages.

      Les mappages peuvent être gérés dans le tableau de bord Identité d’Horizon -> Fédération -> Mappages -> Editer

      La commande openstack mapping list affiche une liste de mappages. Elle devrait contenir le mappage mapping-for-company-xyz-idp.

      La commande openstack mapping show mapping-for-company-xyz-idp -f json affiche les détails du mappage donné. Ils devraient contenir une liste json répertoriant les règles du mappage.

      La commande openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json met à jour et définit le mappage.

      Le fichier de mappage devrait être au format suivant :

      [
   {
      "local": [
        <groupe>
      ],
      "remote": [
        <condition>
      ]
   }
]

      Le fichier de règles devrait contenir une liste de règles qui mappent les utilisateurs distants à un groupe local. La section local contient une règle qui spécifie le groupe auquel les utilisateurs distants devraient être mappés. Le groupe local est requis.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      La section remote contient des règles qui spécifient les utilisateurs distants auxquels un accès devrait être accordé.

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      Les actions admises pour filtrer les attributs distants sont les suivantes :

      empty

not_any_of
blacklist
whitelist

      Pour définir le nom d’utilisateur, vous pouvez utiliser {0} dans la section local pour indiquer d’utiliser une zone issue de la section remote. Vous pouvez ensuite ajouter des informations supplémentaires sur le nouveau nom d’utilisateur fédéré. Vous pouvez faire référence à la section remote dans la section local par un index (“{i}”, où i est l’index de la zone remote) :

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {

      },
               ...
    ]
  }

      Utilisation d’Horizon

      Le tableau de bord OpenStack du fournisseur d’identité doit contenir un menu déroulant intitulé S’authentifier en SSO.

      1. Le menu déroulant doit contenir une valeur pour le fournisseur d’identité. Sélectionnez le fournisseur d’identité et cliquez sur Connexion.

      2. Vous serez redirigé vers la page de connexion du fournisseur d’identité.

      3. Après vous être authentifié avec succès auprès du fournisseur d’identité, vous devriez être redirigé vers la page Horizon.

      Utilisation de l’API OAuth

      Les données d’identification de l’utilisateur ne sont jamais envoyées à Keystone directement, mais sont envoyées aux noeuds finaux du fournisseur OAuth.

      Procédure d’exécution du code suivant :

      1. Prérequis d’installation : request-oauthlib, client Keystone et client Nova

      2. Fichier stackrc OIDC source

      3. Nom du fichier Python

      Exemple 3. Fichier stackrc OIDC

      export OS_AUTH_URL=https://<nom_domaine_complet>:5000/v3
export OS_IDENTITY_PROVIDER=<nom_fournisseur_identité>
export OS_PROTOCOL=<nom_protocole>
export OS_REDIRECT_URI=https://<nom_domaine_complet>:5000/v3/auth/OS-FEDERATION/websso/oidc/redirect
export OS_CLIENT_ID=<ID_client>
export OS_CLIENT_SECRET=<secret_client>
export OS_DISCOVERY_ENDPOINT=https://<NDC-OP>/.well-known/openid-configuration
export OS_TOKEN_ENDPOINT=https://<NDC-OP>/oauth/token
export OS_AUTHORIZATION_ENDPOINT=https://<NDC-OP>/authorize
export OS_PROJECT_ID=<ID_projet>

      Exemple 4. Utilisation de l’API pour le flux de codes d’autorisation

      Ce fichier se nomme oidc-authflow.py

      from keystoneauth1.identity.v3 import oidc
from keystoneauth1 import session
from requests_oauthlib import OAuth2Session
from keystoneauth1.identity.v3 import Token
from novaclient import client
import os

def main():

    auth_url = os.environ.get('OS_AUTH_URL')
    identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
    protocol = 'oidc'
    redirect_uri = os.environ.get('OS_REDIRECT_URI')
    scope = ['openid','profile','email']
    client_id = os.environ.get('OS_CLIENT_ID')
    client_secret = os.environ.get('OS_CLIENT_SECRET')
    discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')
    access_token_endpoint = os.environ.get('OS_TOKEN_ENDPOINT')
    authorization_endpoint = os.environ.get('OS_AUTHORIZATION_ENDPOINT')
    project_id = os.environ.get('OS_PROJECT_ID')

    oauth = OAuth2Session(client_id, redirect_uri=redirect_uri, scope=scope)
    authorization_url, state = oauth.authorization_url(authorization_endpoint,
                                                       access_type='offline',
                                                       approval_prompt='force')

      Ce fichier se nomme oidc-passflow.py

      from keystoneauth1 import session
from keystoneauth1.identity.v3 import oidc
from keystoneauth1.identity.v3 import Token
from keystoneclient.v3.client import Client

from novaclient import client
import os


auth_url = os.environ.get('OS_AUTH_URL')
identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
protocol = 'oidc'
redirect_uri = os.environ.get('OS_REDIRECT_URI')
scope = ['openid','profile','email']
client_id = os.environ.get('OS_CLIENT_ID')
client_secret = os.environ.get('OS_CLIENT_SECRET')
discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')
access_token_endpoint = os.environ.get('OS_TOKEN_ENDPOINT')
authorization_endpoint = os.environ.get('OS_AUTHORIZATION_ENDPOINT')
project_id = os.environ.get('OS_PROJECT_ID')

def get_project_and_federated_session():
    s = session.Session(verify=False)

    # get an access token
    payload = {'client_id': client_id, 'grant_type': 'password',
                'username': '<nom_utilisateur>',
                'password': '<mot_de_passe>', 'scope': 'openid',
                'connection':'Username-Password-Authentication'}

    response = s.post('https://<NDC-OP>/oauth/ro',
                            data=payload,
                            authenticated=False)
    print(response.json())
    access_token = response.json()['access_token']

    oidc_plugin = oidc.OidcAccessToken(auth_url, identity_provider, protocol, access_token)
    response = oidc_plugin.get_unscoped_auth_ref(s)
    scoped_token = Token(auth_url=auth_url,
                         token=response._auth_token,
                         project_id=project_id,
                         reauthenticate=False)
    federated_session = session.Session(auth=scoped_token)
    return  federated_session


def main():
    scoped_session = get_project_and_federated_session()
    nova = client.Client(version=2, session=scoped_session)
    for server in nova.servers.list():
        print server.name

if __name__ == '__main__':
    main()

      Utilisation de l’interface CLI OAuth

      Flux de codes d’autorisation pour la ligne de commande

      Requis : version de python-openstackclient > 3.0.0 Vérifiez la version en exécutant la commande suivante : pip freeze | grep openstackclient

      Procédure d’utilisation du client OpenStack avec le code d’autorisation :

      1. Créez un fichier appelé oidcrc

      2. Collez le contenu suivant dans le fichier. Mettez à jour les paramètres selon votre environnement.

        Fichier stackrc OIDC

         export OS_TOKEN_ENDPOINT=https://<remplacer>/oauth/token
 export OS_CLIENT_SECRET=<secret client>
 export OS_CLIENT_ID=<ID_client>
 export OS_DISCOVERY_ENDPOINT=https://<remplacer>/.well-known/openid-configuration
 export OS_IDENTITY_PROVIDER=auth0
 export OS_REDIRECT_URI=<URI_redirect>
 export OS_IDENTITY_API_VERSION=3
 export OS_DOMAIN=Default
 export OS_PROTOCOL=oidc
 export OS_AUTH_URL=https://<nom_domaine_complet>:5000/v3
 export OS_USERNAME=<nom_utilisateur>
 export OS_AUTHORIZATION_ENDPOINT=https://<remplacer>/authorize

      3. Obtenez un code d’autorisation auprès de votre gestionnaire d’identité.

      4. La commande suivante extrait un jeton Keystone que vous pouvez utiliser pour exécuter n’importe quelle commande jusqu’à son expiration :

          openstack --os-auth-type v3oidcauthcode --os-code <code-aut> --os-project-name <nom-projet> --os-project-domain-name Default token issue

      Exemple :

      root@allinone-multiple:~# openstack --os-auth-type v3oidcauthcode --os-code 6LxfzO7GvCjmmcRa --os-project-name admin --os-project-domain-name Default token issue


            +------------+----------------------------------+
      | Field      | Value                            |
      +------------+----------------------------------+
      | expires    | 2017-01-18 17:02:32.432419+00:00 |
      | id         | 7c43310f774f4401b5ec6abee8aa3d77 |
      | project_id | 2abdd5e144ad490294f20f9efb9968e7 |
      | user_id    | 6f81cf8c63fc49d1a12f8fd4f75cab82 |
      +------------+----------------------------------+

      Exécutez la commande user list à l’aide du jeton que vous venez de générer.

      openstack --os-auth-type v3token --os-token <ID jeton issu de la demande précédente> --os-project-name <nom-projet> --os-project-domain-name Default user list

      REMARQUE : l’ID client et la valeur confidentielle peuvent être transmis en tant que paramètres dans l’interface CLI OpenStack plutôt que de les sauvegarder dans le fichier rc à l’aide des paramètres –os-client-secret et –os-client-id

      Remarques supplémentaires

      Il existe actuellement un problème lié aux utilisateurs fédérés et à la fonction donneur d’autorisation/administrateur dans Keystone, qui empêche l’utilisateur Heat fédéré de déléguer son rôle heat_stack_owner.

      Vous devrez utiliser une solution palliative pour la délégation de confiance en affectant directement le rôle heat_stack_owner à l’utilisateur. Il s’agit de l’utilisateur créé au cours de la connexion fédérée. Le domaine de l’utilisateur est fédéré et peut prendre la valeur None (Federation=None).

      $ openstack user list

      L’utilisateur fédéré devrait posséder un domaine None : openstack user show *utilisateur fédéré*

      Ici, l’ID utilisateur est requis :

      $ openstack role add heat_stack_owner --user *federated user ID* --project *project*

      Il existe un autre problème connu où un utilisateur fédéré a la possibilité de changer son mot de passe dans la page des paramètres, mais ne parvient pas à le faire.
