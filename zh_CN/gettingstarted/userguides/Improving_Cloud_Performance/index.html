      加强 OpenStack II - 提高云性能

      • by Dustin Lundquist and Leslie Lundquist ● November 21, 2016

      • Tags:

        • openstack

        • openstack

        • sr-iov

        • virtio

        • vhost

      通过以下博客帖子重新发布：https://admin.blogs.prd.ibm.event.ibm.com/blogs/bluemix/2016/11/building-openstack-ii-improving-cloud-performance/

      在本文（系列中的第二部分）中，将介绍更多的云基础知识：有关 QEMU 的更多信息以及有关如何提高云总体工作负载性能的更多信息。

      基本上，提高云性能需要在访问网络和存储设备时提高 I/O 效率。通过优化 I/O 虚拟化来使访客 VM 与主机之间的 I/O 设备接口更加高效，可以提高 I/O 的性能和灵活性。因此，由于提升了处理工作负载的总体处理器能力，所以提高了云性能。

      通过虚拟化提高 I/O 性能

      请谨记，QEMU 是云 VM 的基本元素。由它来完成此任务！但是，由于 QEMU 是一个仿真器，因此，在速度对您很重要的情况下应尽可能少依赖 QEMU。虽然 KVM 改进了 QEMU 的处理器和内存性能，但 I/O 性能的瓶颈仍然存在。

      可使用以下两种途径来通过虚拟化提高 I/O 性能：软件途径和硬件途径。此博客中将分析这两种途径。使用这两种途径时，我们执行的每一步会逐渐让 QEMU 失去作用。为什么？仿真越少，意味着性能越好。

      软件途径涉及到更有效地虚拟化“传统”设备及其驱动程序。硬件途径涉及到将设备更直接地连接到 VM，甚至可以使用可实现“单根 IO 虚拟化”(SR-IOV) 功能的专用硬件。基本上，硬件辅助途径可以将 VM 与可根据需要进行实例化的虚拟设备进行快速配对。

      软件途径：QEMU、“virtio”和“vhost”

      如前面的博客中所述，QEMU 旨在作为仿真器运行，它甚至可以仿真过去的操作系统（如 Windows NT 和 OS/2）。它可以仿真操作系统期望找到的任何硬件；因此，它提供的默认设备都是“常见”的物理硬件设备。但是，操作系统与这些设备进行通信所使用的方法在设计的过程中并没有考虑到虚拟化。

      早期的 I/O 设备通常都不是通用处理器，而且其处理速度较慢。因此，在旧“硬件”时代，优化 I/O 性能不会带来明显的优势。例如，物理磁盘需要 4 毫秒这么长的时间来访问磁盘上的位置，所以说，优化驱动程序与磁盘控制器之间的通信将收效甚微。在现代云中，因为是使用 VM 作为服务器，所以提高 I/O 性能会带来明显的优势，并且可以使用易于理解的方式实现此目标。

      I/O 虚拟化为何可以提供帮助？

      要执行高效的 I/O，云系统需要 (1) 访客 VM 的驱动程序，(2) 负责提供 I/O 设备的主机软件（例如 QEMU）中的一些额外后端，以及 (3) 主机软件与底层 I/O 设备进行通信所使用的接口。

      如该图所示，在 VM 中完成 I/O 请求涉及到多个组件。这些组件之间的每一次移交都会导致性能下降。为演示 I/O 性能的下降速度，下面按步骤分解了它的工作方式：

      • 首先，VM 中运行的程序通过“普通”接口向 VM 的操作系统发出 I/O 请求。（如果 VM 中运行的软件使用其他方法，那么虚拟化对于用户不是透明的。）

      • 接下来，VM 的操作系统使用与 QEMU 提供的设备相对应的驱动程序。

      • 该驱动程序随后执行一些操作，KVM 将捕获这些操作并移交给 QEMU 来完成。

      • 然后，QEMU 使用类似的接口向主机操作系统发出 I/O 请求。

      • 主机操作系统随后使用与物理硬件相对应的驱动程序来执行请求的 I/O 操作。

      如何加速完成这个过程？有两个不错的选项。第一，可以更高效地虚拟化我们仿真任何特定设备的方式。该选项通常称为“半虚拟化驱动程序”，其最好的示例就是已合并到 OpenStack Nova 中的 virtio。第二，要进一步提高性能，可以通过从 VM 中完全移除设备仿真来消除另一次移交。也就是说，从访客 VM 中移走 QEMU 的“设备仿真”，并将其仅用于主机操作系统。

      优化 1：半虚拟化驱动程序：virtio 方法

      Virtio 由 VM 操作系统内 QEMU 提供的一系列虚拟化设备及其相应的驱动程序组成。这些驱动程序通过使用现代化的高效数据结构（如环形缓冲区）来优化访客 VM 驱动程序与 QEMU 之间的接口，而不是盲目地仿真现有硬件设备的接口。通过使用 virtio，VM 可以更高效地与 QEMU 进行通信，从而提高了 I/O 性能。

      附带说明一点，virtio 要求访客 VM 镜像包含速度更快的 virtio 驱动程序，因此一些较旧的镜像（例如 Windows 95 或 98）可能无法使用 virtio。幸运的是，现在，大多数操作系统默认情况下都包含 virtio 驱动程序，大约在 2013 年就开始这样做了。

      优化 2：Vhost（移除 QEMU 用户空间设备模型）

      下一项性能优化包括从 VM 中完全移走 virtio 设备仿真功能，并将此功能移入主机操作系统内核中。此移动操作可避免由从 KVM 移交到 QEMU 再移交回到主机内核引起的性能损失。

      硬件途径：PCI 直接分配和 SR-IOV

      PCI 直接分配是可应用于任何 PCI 设备的一般机制。它为前面部分中所述的设备和驱动程序的软件虚拟化类型提供一种替代方法。与 virtio 和 vhost 不同，PCI 直接分配将硬件 I/O 设备直接提供给访客 VM，从而完全从 I/O 路径中移除QEMU 和主机内核，并消除了由这些软件仿真移交操作引起的性能损失。

      PCI 直接分配是在许多情况下用于提高性能的理想选择。

      • 例如，某些用户通过为其视频卡设置 PCI 直接分配来将这种直接分配技术应用于游戏中。请注意，在这种情况下，虚拟化并不是用于提高游戏的性能，而是为其他操作系统运行这些游戏。如果不使用 PCI 直接分配，根本无法实现现代游戏的虚拟化。

      • 最常见的一种 PCI 直接分配设备或许就是网卡。例如，可以使用 vhost_net 进行网卡直接分配，以便从网络数据路径中消除 QEMU，但 vhost_net 仍然依赖于主机内核的完整网络堆栈。

      • 值得注意的是，PCI 直接分配对于主机总线适配器 (HBA) 和磁盘而言通常不实用，因为多数服务器都只有一个与服务器上所有磁盘相连的磁盘控制器。因此，无法在云中的 VM 之间共享这些磁盘的容量；所有磁盘都属于一个 VM。下一部分将讲述已启用 SR-IOV 的 HBA 如何创建用于解决此难题的解决方案。

      • 假设对 HBA 隐秘地应用了 PCI 直接分配，在此场景中，可从 USB 引导云，然后将所有磁盘存储容量分配给作为后端数据库运行的单个 VM。

      请谨记，PCI 直接分配对于每个访客 VM 都需要一个专用 I/O 设备。每个 VM 都需要专用硬件的这一要求降低了虚拟化的灵活性，而且显著增加了成本。幸运的是，出现了一些新型的硬件，这些硬件可通过内置的硬件辅助方式提供虚拟化 PCI 设备，从而降低了成本。这种技术称为 SR-IOV。

      SR-IOV

      较新的高端网络接口和磁盘控制器都包含 SR-IOV（单根 I/O 虚拟化）。基本上，这些硬件设备都可以将自身呈现为多个 PCI 设备，每个 PCI 设备都可以连接到一个 VM。在典型部署中，有一个主设备、一个父（或 PF）设备以及一个或多个称为虚拟功能单元 (VF) 的逻辑设备。通过配置 PF，可以创建其他 VF 设备及相关资源。然后，可以通过 PCI 直接分配将这些 VF 连接到单个 VM。

      可能会对每个访客 VM 的以下两个 I/O 设备使用 PCI 直接分配：网络接口和磁盘控制器。例如，如果使用两个 SR-IOV 设备，可以同时提高网络和存储器的 I/O 性能。

      摘要

      您可以使用本文中描述的软件途径和硬件途径中的任意一种来提高云的工作负载性能。优化的软件途径使用的是 virtio 和 vhost，优化的硬件辅助途径使用的是 PCI 直接分配和 SR-IOV。
