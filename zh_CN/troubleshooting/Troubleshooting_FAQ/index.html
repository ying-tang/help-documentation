      问：如何知道上次重新引导虚拟机的时间？

      答：可使用以下命令：

      zgrep BOOT_IMAGE /var/log/kern.log*

      以查看上次引导主机 (VM) 的时间，然后使用此时间戳记在日志中查找与可能要诊断的问题相关的任何线索。

      问：当云主机冻结且停止响应时该怎么办？

      答：建议检查虚拟机的控制台，查看其是否完全冻结或者是否存在内核严重错误或其他错误。可通过以下命令获取控制台的 URL：

      nova get-vnc-console VM-UUID novnc

      记录控制台上的时间（如果有）。

      捕获已冻结且包含有用信息的控制台的快照。重新引导后，如果在虚拟机上启用了崩溃转储，那么可查看 /var/crash 中的 vmcore（在 RedHat 上）或 .crash（在 Ubuntu 上）文件并从中分析转储。此外，检查冻结时前后的系统活动 (ar) 统计信息。

      如果需要尽快备份，请重新引导机器，或者如果要实现更好的结果，请基于相同镜像搭建一台新机器。如果需要对基础架构作进一步调查，请在重新引导之前，尝试让我们知道冻结的 UTC 时间和云主机的 UUID，以便我们了解能否在日志中发现一些问题。

      如果需要可靠地进入单用户方式（例如，运行文件系统检查或修正 SSH 目录许可权），可通过在 Safari 浏览器中使用全屏控制台来实现。您需要在登录到控制台时按“发送 ctrl/alt/del”按钮以重新引导 VM，以便能够在 GRUB 装入屏幕上捕获。进入 GRUB 引导菜单后，按 e 对其进行编辑。对于 CentOS 中的单用户方式，例如：

      1. 向下滚动并删除该行末尾的大部分内容（从 console 开始），仅保留 console=tty。
      2. 完成编辑后，该行应该类似于以下内容：kernel /boot/vmlinuz-2.6.32-431.el6.x86_64 ro root=UUID=8a057bba-2787-46a7-9436-fd6dc8a3aa85 rd_NO_LUKS rd_NO_LVM LANG=en_US.UTF-8 rd_NO_MD crashkernel=auto rd_NO_DM SYSFONT=latarcyrheb-sun16 KEYBOARDTYPE=pc KEYTABLE=us console=tty notsc single
      3. 您还可尝试添加 rw init=/bin/bash 以代替该行的 ro 部分。
      4. 如果引导过程似乎停止，请按 Enter 键几次。

      问：如果需要在云主机上执行拯救操作，该怎么做？

      答：

      1. 获取拯救 CD ISO（例如，从 http://www.system-rescue-cd.org 获取）： 
        wget https://dronedata.dl.sourceforge.net/project/systemrescuecd/sysresccd-x86/4.9.6/systemrescuecd-x86-4.9.6.iso
      2. 从 sysutils 软件包安装 isohybrid： 
        apt-get install syslinux [or yum install syslinux]
      3. 将主引导记录添加到 ISO 以使其成为可引导硬盘 
        isohybrid systemrescuecd-x86-4.9.6.iso
      4. 将新的混合拯救 CD 上传到 Glance： 
        openstack image create --file systemrescuecd-x86-4.9.6.iso --public sysrescuecd-4.9.6.iso
      5. 将需要拯救的云主机置于拯救方式： 
        nova rescue brokeninstance --image sysrescuecd-4.9.6.iso
      6. 通过 Horizon 中的 VNC 控制台登录到云主机，并执行所需的拯救操作。您可能需要等待 1 到 2 分钟才能在控制台中看到拯救 CD。在等待之后重新加载控制台屏幕。

      (/img/zh_CN/systemrescuefromvnc.png)

      在此处，您可以使用如下工具：

      blkid
fsck /dev/vdb1
mount -t ntfs /dev/xxx /mnt/windows -o ro
ntfs-3g /dev/sda1 /mnt/windows
      1. 如果您对云主机的修复情况感到满意，请使其退出拯救方式，然后使用以下命令从正常引导盘再次启动该云主机： 
        nova unrescue brokeninstance

      问：为何无法将云主机的大小从 m1.tiny 调整为 m1.medium？

      原因可能是实际是在 m1.tiny 云主机类型具有较大临时磁盘的情况下创建该云主机的。调整大小过程会检查这一情况，如果找不到该值，就无法调整大小。

      您可能需要开具支持问题单以确认这是否是失败的原因。支持团队将在日志中看到与下面类似的错误：

      Exception during message handling: Resize error: Unable to resize disk down.

      或者，如果不开具问题单就继续操作，可以将 m1.medium 云主机类型设置为与之前相同的临时磁盘大小，或者遵循以下地址中提供的故障诊断变通方法来使用快照创建新的云主机：/help-documentation/gettingstarted/commontech/Instance_Resize/

      问：哪些原因导致 Neutron 中出现僵化端口？

      使用 Heat 实例化 VM 时，Neutron 中可能出现僵化端口。此问题与 OpenStack Neutron（Mitaka 发行版）中出现竞争状态的缺陷有关。因此，我们将其视为已知限制。此情况可能还会导致元数据失败以及僵化端口。

      存在多种变通方法：

      变通方法 1：如果可能，请在不使用 Heat 的情况下部署云主机。手动部署似乎能够避免此情况。

      变通方法 2：另一个变通方法是使用 API 中的 WaitCondition 向 Heat 中引入延迟。但是，在此场景中，WaitCondition 可能不够理想。

      • WaitCondition 可与 WaitConditionHandler 结合使用。此处理程序将创建一个 URL/URI（监控此项中是否存在文本字符串）。存在此字符串指示已成功部署 VM，而不存在此字符串指示失败。实例化 VM 后，会指示该 VM 通过 PUT API 调用以使用相应文本字符串填充 URL/URI。

      • 目前，OpenStack 部署中不存在用于使 L3 路由器执行此 API 调用的机制。我们的工程团队建议使用“depends_on”。（请参阅后面的选项 3）。

      变通方法 3：最佳变通方法是使用 depends_on。它会在编排中创建一个场景，使 VM 部署“依赖于”是否存在其他可定义的资源（例如，L3 路由器和网络）。

      简单的 depends_on 示例：

      add_router_interface:
   type: OS::Neutron::RouterInterface
   properties:
     router: { get_resource: router}
     subnet: { get_resource: subnet}


server1:
    type: OS::Nova::Server
    depends on: add_router_interface
    properties:
      image: { get_param: image }

      以下链接提供了有关显式依赖关系的更多详细信息：

      • https://wiki.openstack.org/wiki/Heat/Blueprints/hot-software-config-spec

      问：需要针对 OpenStack 打开哪些端口？

      至少需要打开端口 67/68（针对 DHCP）和端口 80（针对元数据）。此外，如果您希望 VM 可以访问 OpenStack API 服务，请包含此处记录的端口：

      /help-documentation/gettingstarted/commontech/FAQ/#which-portsurls-etc-are-publicly-accessible-on-my-bluemix-private-cloud-and-for-what-purpose

      如果未打开所需的最小端口集，可能会遇到 cloud_init 问题，而这会导致云主机可能无法成功启动。

      问：此错误意味着什么？“无法检索云硬盘限制信息。”

      通常，在 Horizon 仪表板中装入页面时，会看到红色弹出框，指示“错误：无法检索云硬盘限制信息。”这意味着什么？

      此错误不会产生任何负面影响，但出现此错误的原因是在部署中没有包含任何 Cinder 块存储器。

      通过从 Keystone 服务中移除 Cinder 端点可以解决此错误。此云需要更新自动化配置文件，而且支持团队也需要进行检查。如果需要寻求帮助来解决此错误消息，请开具支持问题单。

| 39434faf90484a208b61a517926042b2 | RegionOne | cinderv2     | volumev2       | True    | name2    | https://... | 

# openstack endpoint delete 19a90853883049ed92cf4af82a24c0db 
# openstack endpoint delete 2c3e8d0a15304788a9dc1806917b61ee
