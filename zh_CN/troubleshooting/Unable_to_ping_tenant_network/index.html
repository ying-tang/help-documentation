      为何无法在租户网络中执行 ping 操作？

      网络故障诊断示例：无法在租户网络中执行 ping 操作

      问：我已创建了带有子网 (private_subnet) 的网络 (private_network)。我还创建了路由器 (my_router) 以将此网络连接到现有的内部网络。但是，连接到 private_network 的云主机无法针对内部网络中的云主机执行 ping 操作。它们可以针对网关 IP 地址 (10.0.0.1) 执行 ping 操作，也可以针对路由器上内部网络接口的 IP 地址 (192.168.1.113) 执行 ping 操作等等，但是无法针对内部网络中的任何云主机执行 ping 操作。我无法在另一个方向上执行 ping 操作，即从内部网络中的云主机到 private_network 网络中的云主机，但不仅仅是相反方向。

      下面是为了在 OpenStack 中创建相关组件而运行的命令（这些是我所能记住的命令）：

      1. neutron net-create private_network
      2. neutron subnet-create private_network --name private_subnet --gateway 10.0.0.1 10.0.0.0/24
      3. neutron router-create my_router
      4. neutron router-interface-add my_router private_subnet
      5. neutron port-create internal
      6. neutron router-interface-add my_router port=

      我需要在内部网络中手动创建一个额外端口来连接路由器，因为已向其连接了一个路由器。

      答：只添加 my_router 上的内部网络接口所导致的问题是：内部网络中的云主机具有默认网关 192.168.0.1（位于默认路由器上），而默认路由器不知道如何路由回 private_network。

      除了将相对子网中的端口添加到每个路由器中外，还需要添加路径以使其可在两个方向上进行路由…

      neutron router-update default --routes type=dict list=true destination=10.0.0.0/24,nexthop=10.0.0.1
neutron router-update my_router --routes type=dict list=true destination=192.168.0.0/22,nexthop=192.168.0.1

      问：尝试执行第一个命令失败，并返回了以下错误：

      Invalid format for routes: [{u'destination': u'10.0.0.0/24', u'nexthop': u'10.0.0.1'}], the nexthop is not connected with router

      我需要修改该命令以使其成功运行，如下所示，其中 192.168.1.119 是在内部网络中创建的用于连接 my_router 的端口。

      neutron router-update default --routes type=dict list=true destination=10.0.0.0/24,nexthop=192.168.1.119

      此配置足以让流量从连接到 private_network 的云主机流向连接到内部网络的云主机。但是，此配置好像存在一个额外的中继段来用于传输网络流量，即，流量首先路由到默认网关 (192.168.0.1)，然后重定向到 192.168.1.119。如果从内部网络中的云主机上对 private_network 中的云主机执行 ping 操作，可以看到以下内容：

      ubuntu@test:~$ ping 10.0.0.13
PING 10.0.0.13 (10.0.0.13) 56(84) bytes of data.
64 bytes from 10.0.0.13: icmp_seq=1 ttl=63 time=0.783 ms
From 10.0.0.13: icmp_seq=2 Redirect Host(New nexthop: 10.0.0.13)
64 bytes from 10.0.0.13: icmp_seq=2 ttl=63 time=0.624 ms
From 10.0.0.13: icmp_seq=3 Redirect Host(New nexthop: 10.0.0.13)

      一种更好的方法是只使用 ansible 脚本手动向内部网络中的每个主机添加路径，如下所示：

      route add -net 10.0.0.0/24 gw 192.168.1.119

      此配置的下一个问题是，private_network 中的云主机无法通过内部网络连接到公共因特网，例如，无法对 8.8.8.8 执行 ping 操作。您碰巧知道原因且知道我需要如何将其启用吗？

      糟糕，我忘记说明一点，在尝试对公共因特网中的主机执行 ping 操作时出现了错误消息 "destination net unreachable"，此错误指示请求在网关 (10.0.0.1) 范围内，但我不知道如何路由请求。

      答：您的观点是正确的，即通过 my_router 路由来访问内部网络的确会招致另一个网络中继段，但其产生的负面影响可以忽略不计。在每个云主机上添加静态路径可消除该中继段，但会增加很多的管理开销，也会引入一些故障点。建议首选在路由器级别维护路径。

      private_network 中的云主机为何无法访问因特网的原因是，路由器未设置外部网关（请参阅 external_gateway_info）…

      root@ds0006:~# neutron router-show the_router
+-----------------------+-------------------------------------------------------------+
| Field                 | Value                                                       |
+-----------------------+-------------------------------------------------------------+
| admin_state_up        | True                                                        |
| distributed           | False                                                       |
| external_gateway_info |                                                             |
| ha                    | True                                                        |
| id                    | af23ae89-3a13-43cd-a6cd-f919f7c03a64                        |
| name                  | my_router                                                 |
| routes                | {"destination": "192.168.0.0/16", "nexthop": "192.168.0.1"} |
| status                | ACTIVE                                                      |
| tenant_id             | 61471bdea08844478c1341ab39b4929a                            |
+-----------------------+-------------------------------------------------------------+

      如果运行 neutron router-gateway-set the_router external，那么会为路由器分配浮动 IP 并允许其访问外部网络。

      问：my_router 上是有意缺少外部网关的吗？我们将尽可能对外部网络“隐藏”private_network。我们希望可以通过配置了外部网关的默认路由器来路由目标为公共因特网的流量。这一点是否可行，如果可行，您能够告诉我们如何实现吗？

      答：可以将默认路由器添加到 my_router，如下所示：

      neutron router-update my_router --routes type=dict list=true destination=0.0.0.0/0,nexthop=192.168.0.1

      这可有效地提供与直接连接到外部网络的路由器相同的功能，不过无法分配浮动 IP。更灵活的做法是，通过向需要从因特网访问的云主机分配浮动 IP 来将路由器直接连接到外部网络并控制访问，同时使用安全组规则来避免各个服务器“区域”相互干扰。

      我已经证实，如果在未连接到外部网络的路由器上设置一个默认路径来指向默认路由器 IP，将允许路由到因特网。下面是在我们的实验室环境中实现的情况：

      讨论：（作为参考）要连接到 private_network 的云主机实际上是内部的，即，永远不会为其分配浮动 IP。我们的大多数云主机都是如此，我们的产品中大约只有 4 个云主机需要具有浮动 IP。

      其目的在于将内部网络视为 DMZ 等效项，并将需要浮动 IP 的云主机与其相连。所有其他云主机将连接到 private_network，并且只能使用已确定的配置通过默认路由器路由流量来访问公共因特网。
