      HAProxy 可处理多少个并发连接？

      • by Lampros Chaidas ● October 21, 2016

      • Tags:

        • troubleshooting

        • 故障诊断

        • haproxy

        • maxconn

        • 负载测试

      注：本文不是专门介绍 OpenStack Neutron LBaaS 的，只是介绍了云主机中运行的 HAProxy。

      最近，我正在查看一位客户的负载均衡配置。与该客户一同查看 HAProxy 状态页面时，我提出了下面的问题：“HAProxy 实际可支持多少个并发连接？”

      每个后端服务器的最大连接数是否真的有影响？本文介绍了一种方法来确定 HAProxy 可处理的并发连接数。

      我们将在 CentOS 6 VPS 上安装 HAProxy，并执行一系列测试来找出答案。在另一个 VPS 上，我们将设置基准测试实用程序。

      需求：

      • 2 个 CentOS 6.x VPS

      这些测试中使用的 VPS 的规格如下：

      • 2GB RAM
      • 2 个内核
      • CentOS 6.7

      在 HAProxy VPS 上安装 HAProxy：

      步骤 1：安装适用于 CentOS 6 的 HAProxy IUS (Inline with Upstream Stable) 存储库：

        yum -y install https://centos6.iuscommunity.org/ius-release.rpm

      步骤 2：获取 HAProxy 的最新版本（撰写本文时最新版本为 1.6.9）。

        yum -y install haproxy16u && chkconfig haproxy on

      步骤 3：将默认 HAproxy 配置移到其他位置以防以后需要查看：

        mv /etc/haproxy/haproxy.cfg /etc/haproxy/haproxy.cfg-default

      步骤 4：将 rsyslog 配置为接收来自 HAProxy 的日志：

        sed -i -- 's/#$ModLoad imudp/$ModLoad imudp/g' /etc/rsyslog.conf
  sed -i -- 's/#$UDPServerRun 514/$UDPServerRun 514/g' /etc/rsyslog.conf
  echo "\$UDPServerAddress 127.0.0.1" >> /etc/rsyslog.conf
  echo "local2.* -/var/log/haproxy.log" >> /etc/rsyslog.conf

      前几条命令将 rsyslog 设置为侦听 UDP 端口 514，并将其仅限于 localhost 连接。最后一行将 HAProxy 日志路径设置为 /var/log/haproxy.log。

      步骤 5：重新启动 rsyslog 以使更改生效：

        service rsyslog restart

      步骤 6：添加新的 HAProxy 配置 /etc/haproxy/haproxy.cfg，并记录 maxconn 20 行。联机帮助页中提及到“达到此限制时，代理将停止接受连接。”换言之，此配置仅允许最多有 20 个连接。

      fullconn 20 行有何含义？在联机帮助页中，这定义为“使服务器使用最大连接数的后端连接数。”

      global
daemon
# we need the socket for hatop statistics
stats socket /var/run/haproxy.sock mode 600 level admin
log /dev/log local4
maxconn 20

defaults
log global
timeout connect 4000
timeout client 42000
timeout server 43000

listen lampros-pool-1
bind *:80
mode http
balance roundrobin
fullconn 20
server go_http_server_1 127.0.0.1:8080 cookie go_http_server_1 check inter 2000 rise 2 fall 3

      在这两台服务器上安装 Go 语言：

      安装 Go 语言以及 screen 实用程序和 git。两台服务器上都需要此语言，因为在 HAProxy VPS 上将编译基础应用程序服务器。在基准程序 VPS 上，需要 Go 才能运行 gobench 基准程序实用程序。

      yum -y install git screen nc
cd /tmp
wget https://storage.googleapis.com/golang/go1.7.linux-amd64.tar.gz
tar -xvf go1.7.linux-amd64.tar.gz
mv go /usr/local
export GOROOT=/usr/local/go
mkdir /root/goapps
export GOPATH=/root/goapps
export PATH=$PATH:$GOPATH/bin:$GOROOT/bin
echo export "GOPATH=/root/goapps" >> ~/.bashrc
echo export "export PATH=$PATH:$GOPATH/bin:$GOROOT/bin" >> ~/.bashrc

      在 HAProxy 服务器上，将使用 Go 来设置基础应用程序服务器：

      步骤 1：创建名为 /root/slowserver 的目录

        mkdir -pv /root/slowserver

      步骤 2：在此目录中，添加一个包含以下内容的 /root/slowserver/slowserver.go 文件：

      package main

import (
    "fmt"
    "net/http"
    "time"
)

func slow_handler(w http.ResponseWriter, r *http.Request) {
    time.Sleep(3000 * time.Millisecond)
    fmt.Fprintf(w, "I'm slooooow")
}

func fast_handler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "I'm FAST!!!!")
}

func main() {
    http.HandleFunc("/fast", fast_handler)
    http.HandleFunc("/slow", slow_handler)
    http.ListenAndServe(":8080", nil)
}

      步骤 3：使用 go build 编译此文件：

        cd /root/slowserver
  go build

      步骤 4：在分离的屏幕会话中启动 slowserver，从而使其在后台运行：

        screen -d -m -t go_slowserver /root/slowserver/slowserver

      步骤 5：使用 netstat -ntlp | grep slowserver 确认其正在运行，您应该会看到其侦听的端口和进程 ID。下面是收到的输出：

        netstat -ntlp | grep slowserver
  tcp        0      0 :::8080                     :::*                        LISTEN      5204/slowserver

      在 HAProxy 服务器上设置 HATop

      步骤 1：下载并安装 HATop 以便可以监控 HAProxy：

        cd /tmp
  wget https://storage.googleapis.com/google-code-archive-downloads/v2/code.google.com/hatop/hatop-0.7.7.tar.gz
  tar -xzf hatop-0.7.7.tar.gz
  cd hatop-0.7.7
  install -m 755 bin/hatop /usr/local/bin
  install -m 644 man/hatop.1 /usr/local/share/man/man1
  gzip /usr/local/share/man/man1/hatop.1

      步骤 2：启动 HAProxy：

        service haproxy start

      步骤 3：启动 HAtop：

        hatop -s /var/run/haproxy.sock

      步骤 4：使用以下命令（一行）向 HAProxy 请求后端服务器 go_http_server_1 的状态：

        echo "show stat" | nc -U /var/run/haproxy.sock | awk -F, '/go_http_server/{print $18}'

      将显示以下输出：

        UP

        go get github.com/cmpxchg16/gobench

      负载测试 - 确定 HAProxy 可处理的并发连接数

      现在，让我们来查看 HAProxy 可处理多少个并发连接。在开始之前，请记住有关 HAProxy 的基本详细信息：“默认情况下，对于持久连接，HAProxy 以“保持活动”方式运行：在每个连接上，它会处理所有请求和响应，并在响应结束后到新请求开始前这段时间内使该连接保持空闲状态。”

      这句话表明，与短时间运行的连接相比，长时间运行的连接对服务器的影响更大。因此，我们将针对应用程序服务器的“缓慢”功能运行测试。

      结论：负载测试 - 确定 HAProxy 可处理的并发连接数

      从前面的试验中得出如下结论：为了测试容量（连接到 HAProxy 的最大客户端数），使用的服务器必须已打开 TCP 连接并使其保持打开状态一段时间（3 秒）。这与测试服务器运行速度有多快的情况正好相反，在该情况下，通过服务器的流量越多，获取的结果越佳。

      另一个结论是：最大连接数 (maxconn) 需要大于“请求率（例如，100 个客户端）* 时间（3 秒）”，才能为所有请求提供服务。

      例如：

      maxconn=100*3 => maxconn=300
