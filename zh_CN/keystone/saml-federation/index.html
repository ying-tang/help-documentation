      SAML 联合是什么

      安全性断言标记语言 (SAML) 是一种协议，允许用户结合使用单点登录 (SSO) 和身份提供者 (IDP) 与服务提供者 (SP)。当用户登录时，他们会将凭证提供给 IDP。IDP 将检查他们的凭证，并创建带有用户属性的 SAML 断言。然后，该断言将传递到 SP，SP 会验证断言是否由 IDP 发送。因为 SP 信任来自 IDP 的断言，所以用户不需要将其凭证发送给 SP。

      Keystone 将充当 SP，并通过接收来自 IDP 的断言对用户进行认证。在本文中，我们将使用名为 Company XYZ IDP 的虚构 SAML IDP。

      请注意，此页面上的任何管理命令都需要用户具有 cloud_admin 访问权。

      集成 IDP 和 Keystone

      Bluemix Private Cloud 操作员将需要与 Company XYZ IDP 集成工程师一起将两个系统设置为协同工作。

      Bluemix Private Cloud 操作员将需要以下信息：

      • IDP 元数据 URL 端点（目前，我们只支持通过 URL 而不是文件来检索元数据）
      • 来自 IDP 元数据的 IDP 实体 ID
      • 用户属性 - 这是通过 IDP 传递的 SAML 用户属性。该属性可以在 IDP 文档或 IDP 服务器端日志中找到。
      • 默认/初始映射 - 此处提供了有关映射的文档：http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination

      Company XYZ IDP 集成工程师可能需要 Bluemix Private Cloud 操作员提供以下信息：

      • 服务提供者元数据 URL
      • 服务提供者重定向 URL（也称为回调 URL，这是列入白名单的值，IDP 将允许重定向到此值）

      Bluemix Private Cloud 操作员应与集成工程师一起创建初始/默认映射。此映射是将 IDP 上的远程用户映射到 SP 上的本地组的方式。本文的其余部分可帮助您弄清映射是什么以及映射的使用方式。集成工程师也将需要 cloud_admin 访问权。这可以是 cloud_admin 用户或具有相同名称角色的用户。

      应随映射名称提供用户属性。例如，可以按照如下所示提供属性：

      attributes:
- name: http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress
  id: email
- name: http://schemas.xmlsoap.org/claims/Group
  id: group

      提供的名称是 SAML 断言内的名称。ID 是将用在 Keystone 映射中的 ID。

      Bluemix Private Cloud 操作员完成 Company XYZ IDP 与 Keystone 的集成之后，应会创建名为 mapping-for-company-xyz-idp 的映射（或其中包含 IDP 名称的类似映射名称）。

      下面是管理映射的步骤：

      1. 管理组（联合用户将具有与该组同样的访问权） 2. 管理映射

      管理组

      Keystone 联合的工作原理是将身份提供者用户联合到服务提供者上的组。通过为该组分配项目角色来授予组访问权。可用角色如下所示：

      • project_admin - 用于项目的 admin 角色。
      • cloud_admin - 用于云主机的 admin 角色。
      • _member_ - 项目的成员。
      • heat_stack_owner - heat 堆栈的所有者。

      列出组：

      $ openstack group list

      创建组：

      $ openstack group create --help
$ openstack group create <new group name here>

      列出角色：

      $ openstack role list

      列出角色分配：

      $ openstack role assignment list --name

      列出项目：

      $ openstack project list

      帮助查看添加角色分配：

      $ openstack role add --help

      为组添加角色分配以具有项目角色：

      $ openstack role  add <role> --project <project> --group <group>

      联合用户将映射到服务提供者上的组。

      下面是为组授予访问权的操作顺序：

      1. 首先，创建或编辑组。 2. 然后，为项目上的组创建或编辑角色分配。

      示例 1. 创建 company_xyz_admin_group 组。

      $ source cloud_adminrc # 您的 openstack 凭证文件
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      接下来，如果角色分配不存在，请创建角色分配。 以下命令可为“演示”项目的 cloud_admin 组添加 cloud_admin 角色。

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      执行 openstack role assignment list 命令后，您应该会看到类似于以下内容的输出：

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      示例 2. 创建带有成员访问权的组。

      $ source cloud_adminrc # 您的 openstack 凭证文件
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      输入 openstack role assignment list 命令后，您应该会看到类似于以下内容的输出：

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      管理映射

      下面，我们使用的是 Company XYZ IDP，并且已存在用于此示例的映射。具有 cloud_admin 访问权（用户或角色）的用户将能够编辑此映射。用户将无法创建或删除映射。

      在 Horizon 上，可以在“身份仪表板 -> 联合 -> 映射 -> 编辑”中管理映射

      openstack mapping list 命令可显示一组映射。应显示 mapping-for-company-xyz-idp 映射。

      openstack mapping show mapping-for-company-xyz-idp -f json 命令可显示指定映射的详细信息。其中应包含用于映射的规则的 json 列表。

      openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json 命令可更新和设置映射。

      映射文件应遵循下方给定格式：

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      规则文件应包含一组规则，这些规则可将远程用户映射到本地组。 本地部分中包含的规则指定了远程用户应映射到的组。本地组为必需项。

      {
  "group": {
    "domain": {  "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      远程部分中包含的规则指定了允许哪些远程用户进行访问。

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      下面是远程规则中可以提供的可能用户属性类型：

      以下是可能对远程属性进行过滤的操作：

      empty
any_ony_of
not_any_of
blacklist
whitelist

      要设置用户名，我们可以使用本地部分中的 {0} 来指示使用远程部分中的字段。然后，我们可以将信息添加到新联合的用户名。我们可以通过索引（”{i}”，其中 i 是远程字段的索引）引用本地部分中的远程部分：

      [
  {
    "local": [
      {
        "user": {  "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "email"
      },
               ...
    ]
  }
]

      使用 Horizon

      身份提供者上的 OpenStack 仪表板应该有一个标记为使用 SSO 进行认证的下拉菜单。

      • 下拉菜单应显示 Company XYZ IDP 的标签。选择 IDP 并单击“连接”。
      • 您将被重定向到 IDP 登录页面。
      • 成功向 IDP 进行认证后，应该会将您重定向到 Horizon 页面。

      使用 CLI

      注：命令行界面 (CLI) 客户端只能与满足以下条件的 IDP 配合使用： 1. IDP 支持 SAML ECP 协议 2. IDP 支持对 GET 进行基本认证

      CLI 提供了两种类型的插件： 1. v3unscopedsaml（抓取未限定作用域的令牌） 2. v3scopedsaml（使用令牌执行限定作用域的操作）

      1. 未限定作用域的示例：

      我们来创建一个文件，可以在环境中追踪此文件来源以使用 SAML：

      # 该文件名为 companyxyz_saml_rc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=CompanyXYZPassword # 替换此值
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 # 替换此值
export OS_USERNAME=CompanyXYZUser # 替换此值
export OS_IDENTITY_PROVIDER=company-xyz-idp # 替换此值
export OS_PROTOCOL=saml2
export OS_AUTH_TYPE=v3unscopedsaml
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP # 替换此值

      在创建文件并编辑选项后，我们可以运行以下命令来获取未限定作用域的令牌并记录此 ID：

      $ source companyxyz_saml_rc
$ openstack token issue

      2. 限定作用域的示例：

      现在，我们有一个未限定作用域的令牌，我们可以将作用域限定为一个项目，并列出该项目上的服务器

      openstack server list --os-auth-type v3scopedsaml --os-token <os-token-id> --os-project-id <project-id>

      使用 Python API 库

      注：Python API 库只能与满足以下条件的 IDP 配合使用： 1. IDP 支持 SAML ECP 协议 2. IDP 支持对 GET 进行基本认证

      以下代码使用 SAML auth 插件向 IDP 进行认证，向 Keystone 发送 SAML 断言并检索未限定作用域的令牌。然后，未限定作用域的令牌将用于获取一个项目作用域的令牌。接着，脚本将列出该项目上的服务器。

      该文件为 saml_list_servers.py：

      我们将需要在运行脚本之前追踪文件来源：

      # 该文件名为 companyxyz_saml_python_rc
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 # 替换此值
export OS_USERNAME=CompanyXYZUser # 替换此值
export OS_PASSWORD=CompanyXYZPassword # 替换此值
export OS_PROJECT_NAME=demo
export OS_IDENTITY_PROVIDER=company-xyz-idp # 替换此值
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP # 替换此值
export OS_PROTOCOL=saml2

      然后，我们可以运行 python 脚本：

      source companyxyz_saml_python_rc
python saml_list_servers.py

      其他说明

      目前，联合用户和 Keystone 中的委托人/受托人功能存在一个问题，该问题使联合的 Heat 用户无法委派其 heat_stack_owner 角色。

      您需要使用变通方法，通过直接向用户分配 heat_stack_owner 角色来进行信任委派。这是联合登录期间创建的用户。该用户的域已经联合，可以具有 None 值 (Federation=None)。

      $ openstack user list

      联合用户应具有 None 域：openstack user show <federated user>

      这里我们需要使用用户 ID：

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      另一个已知问题是会向联合用户呈现在设置页面中更改密码的选项。但他们无法更改其密码。
