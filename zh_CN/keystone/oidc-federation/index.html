      联合是什么？

      Keystone 联合支持将身份提供者 (IDP) 中的身份用于服务提供者 (SP)。受支持的不同协议和格式为 OpenIDConnect (OIDC) 和安全性断言标记语言 (SAML)。

      IDP 存储和管理用户的凭证并向 SP 发送声明或断言。这使得客户可以使用他们的企业凭证向 Bluemix Private Cloud 进行认证，无需将他们的密码发送至 Bluemix Private Cloud。Keystone 通常始终是 SP，因为它使用外部资源的身份。

      OpenID Connect (OIDC) 是什么？

      OpenID 是用于认证的协议。 OAuth 是用于授权的协议。 OpenID Connect 可以用于认证和授权。

      OpenID Connect 是 OAuth 2.0 协议顶部的简单身份层。可以让客户端根据授权服务器执行的认证验证最终用户的身份。同时能让客户端以可互操作和类似 REST 的方式获得有关最终用户的基本概要文件信息。

      本文描述了 OpenID Connect 依赖方用以发现最终用户的 OpenID 提供者，以及获取与其交互所需信息的机制，包括其 OAuth 2.0 端点位置。

      OpenID 协议如何工作？

      1. 依赖方将请求发送到 OpenID 提供者以对最终用户进行认证。
      2. OpenID 提供者对用户进行认证。
      3. OpenID 提供者将 ID 令牌和访问令牌发送到依赖方。
      4. 依赖方使用从 OpenID 提供者那里收到的访问令牌将请求发送到用户信息端点。
      5. 用户信息端点返回声明。

      设置 OIDC

      Blue Box 操作员将需要与来自客户公司（例如，Company XYZ）的 IDP 集成工程师一起设置 OIDC。

      Bluemix Private Cloud 操作员将需要以下信息：

      • 经授权的重定向/回调 URL
      • 客户端 ID 和客户端密钥
      • 用户声明 - OIDC 声明
      • 用于提供访问提供者所需的全部信息的熟知/配置 URL
      • 默认/初始映射 - 此处提供了有关映射的文档： http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination

      Bluemix Private Cloud 操作员应与集成工程师一起创建初始/默认映射。此映射为将 IDP 上的远程用户映射到 SP 上的本地组提供了相应的方式。本文的其余部分可帮助您弄清映射是什么以及映射的使用方式。

      Bluemix Private Cloud 操作员完成 Company XYZ IDP 与 Keystone 的集成之后，应会创建名为 mapping-for-company-xyz-idp 的映射（或其中包含 IDP 名称的类似映射名称）。

      下面是管理映射的步骤：

      1. 管理组（联合用户将具有与该组同样的访问权）

      2. 管理映射

      管理组

      Keystone 联合的工作原理是将身份提供者用户联合到服务提供者上的组。通过为该组分配项目角色来授予组访问权。可用角色如下所示：

      • project_admin - 用于项目的 admin 角色。
      • cloud_admin - 用于云主机的 admin 角色。
      • _member_ - 项目的成员。
      • heat_stack_owner - heat 堆栈的所有者。

      列出组：

      $ openstack group list

      创建组：

      $ openstack group create --help
$ openstack group create <new group name here>

      列出角色：

      $ openstack role list

      列出角色分配：

      $ openstack role assignment list --name

      列出项目：

      $ openstack project list

      帮助查看添加角色分配：

      $ openstack role add --help

      为组添加角色分配以具有项目角色：

      $ openstack role  add <role> --project <project> --group <group>

      联合用户将映射到服务提供者上的组。下面是为组授予访问权的操作顺序：

      1. 首先，创建或编辑组。

      2. 然后，为项目上的组创建或编辑角色分配。

      示例 1. 创建 company_xyz_admin_group 组。

      $ source cloud_adminrc # 您的 openstack 凭证文件
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      接下来，如果角色分配不存在，请创建角色分配。

      以下命令可为“演示”项目的 cloud_admin 组添加 cloud_admin 角色。

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      执行 openstack role assignment list 命令后，您应该会看到类似于以下内容的输出：

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      示例 2. 创建带有成员访问权的组。

      $ source cloud_adminrc # 您的 openstack 凭证文件
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      输入 openstack role assignment list 命令后，您应该会看到类似于以下内容的输出：

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      管理映射

      下面，我们使用的是 Company XYZ IDP 示例并假设已存在用于此示例的映射。具有 cloud_admin 访问权（用户或角色）的用户将能够编辑此映射。用户将无法创建或删除映射。

      在 Horizon 上，可以在“身份仪表板 -> 联合 -> 映射 -> 编辑”中管理映射

      openstack mapping list 命令可显示一组映射。应显示 mapping-for-company-xyz-idp 映射。

      openstack mapping show mapping-for-company-xyz-idp -f json 命令可显示指定映射的详细信息。其中应包含用于映射的规则的 json 列表。

      openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json 命令可更新和设置映射。

      映射文件应遵循下方给定格式：

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      规则文件应包含一组规则，这些规则可将远程用户映射到本地组。 本地部分中包含的规则指定了远程用户应映射到的组。本地组为必需项。

      {
  "group": {
    "domain": {  "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      远程部分中包含的规则指定了允许哪些远程用户进行访问。

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      下面是远程规则中可以提供的可能用户属性类型：

      以下是可能对远程属性进行过滤的操作：

      empty
any_ony_of
not_any_of
blacklist
whitelist

      要设置用户名，我们可以使用本地部分中的 {0} 来指示使用远程部分中的字段。然后，我们可以将信息添加到新联合的用户名。我们可以通过索引（”{i}”，其中 i 是远程字段的索引）引用本地部分中的远程部分：

      [
  {
    "local": [
      {
        "user": {  "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "HTTP_OIDC_CLAIM_EMAIL"
      },
               ...
    ]
  }
]

      使用 Horizon

      身份提供者上的 OpenStack 仪表板应该有一个标记为使用 SSO 进行认证的下拉菜单。

      1. 下拉菜单应显示 IdP 的标签。选择 IDP 并单击连接。

      2. 您将被重定向到 IDP 登录页面。

      3. 成功向 IDP 进行认证后，应该会将您重定向到 Horizon 页面。

      使用 OAuth API

      用户凭证从来不会直接发送到 Keystone，但是它们会发送到 OAuth 提供者端点。

      运行以下代码的步骤：

      1. 安装必备软件：request-oauthlib、Keystone 客户端和 Nova 客户端

      2. 追踪 OIDC stackrc 文件来源

      3. python 文件名

      示例 3. OIDC stackrc

      export OS_AUTH_URL=https://<fqdn>:5000/v3
export OS_IDENTITY_PROVIDER=<identity_provider name>
export OS_PROTOCOL=<protocol_name>
export OS_REDIRECT_URI=https://<fqdn>:5000/v3/auth/OS-FEDERATION/websso/oidc/redirect
export OS_CLIENT_ID=<client_ID>
export OS_CLIENT_SECRET=<client_secret>
export OS_DISCOVERY_ENDPOINT=https://<FQDN-OP>/.well-known/openid-configuration
export OS_TOKEN_ENDPOINT=https://<FQDN-OP>/oauth/token
export OS_AUTHORIZATION_ENDPOINT=https://<FQDN-OP>/authorize
export OS_PROJECT_ID=<project_id>

      示例 4. 为授权代码流程使用 API

      该文件为 oidc-authflow.py

      from keystoneauth1.identity.v3 import oidc
from keystoneauth1 import session
from requests_oauthlib import OAuth2Session
from keystoneauth1.identity.v3 import Token
from novaclient import client
import os

def main():

    auth_url = os.environ.get('OS_AUTH_URL')
    identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
    protocol = 'oidc'
    redirect_uri = os.environ.get('OS_REDIRECT_URI')
    scope = ['openid','profile','email']
    client_id = os.environ.get('OS_CLIENT_ID')
    client_secret = os.environ.get('OS_CLIENT_SECRET')
    discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')

      使用 OAuth CLI

      命令行的授权代码流程

      必需：python-openstackclient 的版本 > 3.0.0 通过运行以下命令查看版本： pip freeze | grep openstackclient

      将 OpenStack 客户端和授权代码结合使用的步骤：

      1. 创建名为 oidcrc 的文件

      2. 将以下内容粘贴到文件中。根据您的环境更新参数。

        OIDC stackrc

         export OS_TOKEN_ENDPOINT=https://<replace>/oauth/token
 export OS_CLIENT_SECRET=<client secret>
 export OS_CLIENT_ID=<client_id>
 export OS_DISCOVERY_ENDPOINT=https://<replace>/.well-known/openid-configuration
 export OS_IDENTITY_PROVIDER=auth0
 export OS_REDIRECT_URI=<redirect_uri>
 export OS_IDENTITY_API_VERSION=3
 export OS_DOMAIN=Default
 export OS_PROTOCOL=oidc
 export OS_AUTH_URL=https://<FQDN>:5000/v3
 export OS_USERNAME=<username>
 export OS_AUTHORIZATION_ENDPOINT=https://<replace>/authorize

      3. 从身份管理器获取授权代码。

      4. 以下命令用于访存可用来执行任何命令的 Keystone 令牌，直至令牌到期：

          openstack --os-auth-type v3oidcauthcode --os-code <auth-code> --os-project-name <project-name> --os-project-domain-name Default token issue

      示例：

      root@allinone-multiple:~# openstack --os-auth-type v3oidcauthcode --os-code 6LxfzO7GvCjmmcRa --os-project-name admin --os-project-domain-name Default token issue


            +------------+----------------------------------+

      使用刚刚生成的令牌执行 user list 命令。

      openstack --os-auth-type v3token --os-token <token id from previous request> --os-project-name <project-name> --os-project-domain-name Default user list

      注： 客户端 ID 和密钥可以作为参数在 OpenStack CLI 中传递，而不是使用 –os-client-secret 和 –os-client-id 参数将其保存在 rc 文件中。

      其他说明

      目前，联合用户和 Keystone 中的委托人/受托人功能存在一个问题，该问题使联合的 Heat 用户无法委派其 heat_stack_owner 角色。

      您需要使用变通方法，通过直接向用户分配 heat_stack_owner 角色来进行信任委派。这是联合登录期间创建的用户。该用户的域已经联合，可以具有 None 值 (Federation=None)。

      $ openstack user list

      联合用户应具有 None 域：openstack user show *federated user*

      这里我们需要使用用户 ID：

      $ openstack role add heat_stack_owner --user *federated user ID* --project *project*

      另一个已知问题是会向联合用户呈现在设置页面中更改密码的选项。但他们无法更改其密码。
