      K2K 联合是什么？

      Keystone to Keystone (K2K) 联合是一种使用一个 Keystone 云主机中的凭证登录多个 IBM Bluemix Private Cloud 云主机的方式。一个 Bluemix Private Cloud 云主机指定为身份提供者，其他云主机则指定为服务提供者。身份提供者存储用户凭证，并向服务提供者断言用户事实上是有效用户。用户的 ID 和密码从不会发送到服务提供者，而只会发送来自身份提供者的断言。

      K2K 如何工作？

      用户登录身份提供者，接收来自身份提供者的已签名 SAML 断言，然后使用该 SAML 断言向服务提供者进行认证。用于处理 SAML 流的登录流程由 K2K 认证插件处理（位于 keystoneauth1 中）。 凭证仅在身份提供者上管理，这些凭证由服务提供者通过断言来接受。

      如果已有多个 Bluemix Private Cloud 设置 K2K 功能，那么将有一个云主机指定为身份提供者，其他云主机则指定为服务提供者。服务提供者使用映射从身份提供者中选择用于映射到本地组的用户属性。用户使用 K2K 联合登录后，将向此用户授予分配给组的角色。

      默认情况下，将为身份提供者上的 cloud_admin 凭证创建名为 mapping-for-k2k-federation 的映射，以此将它们映射到名为 cloud_admin 的本地组。该组对演示项目具有 cloud_admin 角色访问权。

      K2K 联合的设置流程如下：

      1. 首先，管理服务提供者上的组（联合用户将与组拥有相同的访问权）

      2. 接下来，管理服务提供者上的映射（将远程身份提供者用户映射到本地服务提供者组）

      管理组

      K2K 联合的工作原理是将身份提供者用户联合到服务提供者上的组。通过为该组分配项目角色来授予组访问权。可用角色如下所示：

      • project_admin - 用于项目的 admin 角色。
      • cloud_admin - 用于云主机的 admin 角色。
      • _member_ - 项目的成员。
      • heat_stack_owner - heat 堆栈的所有者。

      要使用组命令，需要凭证源文件。您需要创建包含这些凭证的文件并使用相应的信息替换尖括号 < > 中显示的标记。

      # 该文件是 cloud_adminrc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=<cloud_admin password> # 更改此值
export OS_AUTH_URL=https://<service provider fqdn>:5000/v3
export OS_USERNAME=cloud_admin
export OS_TENANT_NAME=demo
export OS_CACERT=/opt/stack/ssl/openstack.crt
export OS_NO_CACHE=True

      创建凭证源文件后，您可以追踪此文件来源并使用下方任何 OpenStack 命令管理组和角色。 注：这些命令运行在服务提供者上。

      在环境中追踪凭证来源：

      $ source cloud_adminrc

      列出组：

      $ openstack group list

      创建组：

      $ openstack group create --help
$ openstack group create <new group name here>

      列出角色：

      $ openstack role list

      列出角色分配：

      $ openstack role assignment list --name

      列出项目：

      $ openstack project list

      帮助查看添加角色分配：

      $ openstack role add --help

      为组添加角色分配以具有项目角色：

      $ openstack role  add <role> --project <project> --group <group>

      联合用户将映射到服务提供者上的组。默认情况下，如果启用 K2K 联合，会存在名为 cloud_admin 的组。该组对“演示”项目具有 cloud_admin 角色。

      下面是为组授予访问权的操作顺序：

      1. 首先，创建或编辑组。
      2. 然后，为项目上的组创建或编辑角色分配。

      示例 1. 创建 cloud_admin 组。

      以下命令显示如何使用服务提供者 cloud_admin 用户的认证信息追踪文件来源。

      $ source cloud_adminrc
$ openstack group list
$ openstack group create cloud_admin --or-show
$ openstack role assignment list --name --group cloud_admin

      接下来，如果角色分配不存在，请创建角色分配。 以下命令可为“演示”项目的 cloud_admin 组添加 cloud_admin 角色。

      $ openstack role add cloud_admin --group cloud_admin --project demo
$ openstack role assignment list --name --group cloud_admin

      执行 openstack role assignment list 命令后，您应该会看到类似于以下内容的输出：

      # +-------------+------+---------------------+--------------+--------+-----------+
# | Role        | User | Group               | Project      | Domain | Inherited |
# +-------------+------+---------------------+--------------+--------+-----------+
# | cloud_admin |      | cloud_admin@Default | demo@Default |        | False     |
# +-------------+------+---------------------+--------------+--------+-----------+

      示例 2. 创建带有成员访问权的组。

      以下命令显示如何使用服务提供者 cloud_admin 用户的认证信息追踪文件来源。

      $ source cloud_adminrc
$ openstack group list
$ openstack group create demo_member_group --or-show
$ openstack role assignment list --name --group demo_member_group
$ openstack role add _member_ --group demo_member_group --project demo
$ openstack role assignment list --name --group demo_member_group

      输入 openstack role assignment list 命令后，您应该会看到类似于以下内容的输出：

      # +----------+------+---------------------------+--------------+--------+-----------+
# | Role     | User | Group                     | Project      | Domain | Inherited |
# +----------+------+---------------------------+--------------+--------+-----------+
# | _member_ |      | demo_member_group@Default | demo@Default |        | False     |
# +----------+------+---------------------------+--------------+--------+-----------+

      管理映射

      服务提供者上的 cloud_admin 用户有权读取和设置 K2K 联合的映射。请注意，cloud_admin 服务提供者用户不能删除现有映射和创建新映射。

      要使用映射命令，需要 v3 凭证源文件。您要创建包含这些凭证的文件并使用相应的信息替换尖括号 < > 中的标记。

      OpenStack 文档中包含有关管理映射的更多信息： http://docs.openstack.org/developer/keystone/mapping_combinations.html http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combinations

      用户使用他们的身份提供者帐户通过 K2K 联合登录服务提供者时，将在服务提供者本地创建用户帐户。请注意，联合期间创建的用户帐户是临时的。该帐户可能会在稍后删除，而不会发出警告。

      在默认映射中，将联合身份提供者上的 cloud_admin 用户，因此将在联合域中创建本地 cloud_admin 用户。该用户与“默认”域中的 cloud_admin 用户不同。联合域中临时的 cloud_admin 用户可能会在将来被删除。

      我们需要前一部分中创建的源文件，因此我们可以运行以下映射命令：

      $ source cloud_adminrc
$ openstack mapping list
$ openstack mapping show mapping-for-k2k-federation
$ openstack mapping show mapping-for-k2k-federation --format json
$ openstack mapping show --help
$ openstack mapping set mapping-for-k2k-federation --rules <mappings file>

      openstack mapping list 命令可显示一组映射。应显示 mapping-for-k2k-federation 映射。

      openstack mapping show mapping-for-k2k-federation 命令可显示给定映射的详细信息。其中应包含用于映射的规则的 json 列表。

      openstack mapping set mapping-for-k2k-federation --rules newmappings.json 命令显示了如何更新和设置 mapping-for-k2k-federation 映射。

      映射文件应遵循下方给定格式：

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      规则文件应包含一组规则，这些规则用于将远程用户映射到服务提供者上存在的组。 本地部分中包含的规则指定了远程用户应映射到的组。本地组规则为必需项。

      {
  "group": {
    "domain": {  "name": "Default"
    },
    "name": "cloud_admin"
  }
}

      远程部分中包含的规则指定了允许哪些远程用户进行访问。

      {
    "type": "openstack_user",
    "any_one_of": [
        "cloud_admin"]
}

      下面是远程规则中可以提供的可能用户属性类型：

      openstack_user
openstack_user_domain
openstack_roles
openstack_project
openstack_project_domain

      以下是可能对远程属性进行过滤的操作：

      empty
any_ony_of
not_any_of
blacklist
whitelist

      要设置用户名，我们可以使用本地部分中的 {0} 来指示使用远程部分中的字段。然后，我们可以将信息添加到新联合的用户名。对于 K2K 联合，需要添加某些信息以指示您正在使用的云，这一点非常重要。我们可以通过索引（”{i}”，其中 i 是远程字段的索引）引用本地部分中的远程部分：

      [
  {
    "local": [
      {
        "user": {  "name": "my_cloud/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
               ...
    ]
  }
]

      使用 Horizon

      身份提供者上的 OpenStack 仪表板应该有一个标记为使用 Keystone to Keystone 联合进行认证的下拉菜单。

      • 如果您选择身份提供者选项，将不会使用联合。用户将登录到身份提供者 OpenStack 云。
      • 如果选择服务提供者选项，将使用联合。身份提供者用户凭证将联合到服务提供者。
      • 要切换这两种提供者，用户需要注销并重新选择提供者。

      请注意：K2K for IBM Bluemix Private Cloud 3.0.0 的 Horizon 登录屏幕不是标准的 OpenStack 功能。将来的发行版中可能会发生更改。

      示例 1. Heat 用户

      目前，联合用户和 Keystone 中的委托人/受托人功能存在一个问题，该问题使联合的 Heat 用户无法委派其 heat_stack_owner 角色。

      创建 Heat 联合组

      使用服务提供者 cloud_admin 用户的认证信息追踪文件来源：source cloud_adminrc

      这里，我们使用角色 heat_stack_owner，而不是 heat_stack_user

      $ openstack group list
$ openstack group create heat_stack_owner_group --or-show
$ openstack role assignment list --name --group heat_stack_owner_group
$ openstack role add heat_stack_owner --group heat_stack_owner_group --project demo
$ openstack role assignment list --name --group heat_stack_owner_group
# +------------------+------+--------------------------------+--------------+--------+-----------+
# | Role             | User | Group                          | Project      | Domain | Inherited |
# +------------------+------+--------------------------------+--------------+--------+-----------+
# | heat_stack_owner |      | heat_stack_owner_group@Default | demo@Default |        | False     |
# +------------------+------+--------------------------------+--------------+--------+-----------+

      创建 Heat 联合映射

      $ cat <<EOF > federated_heat_stack_owner_mapping.json
[
  {
    "local": [
      {
        "user": {  "name": my_cloud/"{0}"
        }
      },
      {
        "group": {
          "domain": {  "name": "Default"
          },
          "name": "heat_stack_owner_group"
        }
      }
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
      {
        "type": "openstack_roles",
        "any_one_of": [
          "heat_stack_owner"
        ]
      }
    ]
  }
]
EOF
$ openstack mapping set mapping-for-k2k-federation --rules federated_heat_stack_owner_mapping.json

      您需要使用变通方法，通过直接向用户分配 heat_stack_owner 角色来进行信任委派。这是联合登录期间创建的用户。该用户的域已经联合，可以具有 None 值 (Federation=None)。

      $ openstack user list

      联合用户应具有 None 域：openstack user show <federated user>

      这里我们需要使用用户 ID：

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      目前有一个已知问题，即会向联合用户呈现在设置页面中更改密码的选项。但他们无法更改其密码。
