      IBM Blue Box 网络用例

      • by Leslie Lundquist and Dustin Lundquist ● September 7, 2016

      • Tags:

        • neutron

        • neutron

        • 用例

        • 网络

        • 白皮书

      最佳实践白皮书

      作者：Leslie Lundquist 和 Dustin Lundquist

      2016 年 7 月 30 日

      背景：IBM Bluemix Private Cloud 在 IBM 云数据中心内运行。每个 IBM 云数据中心都构建为包含多个 pod 以组成子网，这些子网在全球所有 IBM 云数据中心之间共享一个大型 IP 地址空间。

      将网络分割成独立的 pod 以限制任何单一网络的大小是一种有效的做法：它可限制第 2 层 (L2) 问题的范围并隔离其故障域。即，通过减小子网，使子网发生中断时影响范围更小，因而更易于诊断。

      但由于此设计基于 pod，如果客户主机最终位于不同 pod 中，客户就需要使用第 3 层 (L3) 网络连接。pod 被分隔为小型块，每个块含 32 个地址，因此随着云的增长，IBM 云无法保证您始终能够使其他主机进入现有 pod 中。因此，客户必须制定多 pod 案例规划并以此作为标准。

      IBM Blue Box 云的客户用例

      本部分逐步分析了 IBM Bluemix Private Cloud 客户在 IBM 云数据中心 IP 地址模式环境中遇到的多个客户用例，并提供了 IBM Bluemix Private Cloud 工程师已实施过的常规方法或解决方案。

      1. 专用网络之间的连接：

      在此案例中，客户希望在两个专用内部 OpenStack 网络之间进行通信；因此，客户创建两个专用内部网络，并使用 OpenStack 路由器将其连接起来。

      注释：只要这两个客户网络位于独立的 OpenStack 路由器之后，即表示您与 Neutron 一致，尝试允许并提供重叠的 IP 地址空间：您可以轻松在独立路由器后提供重叠的 IP 空间。但如果您使用单一路由器，那么直接连接到单一路由器背后的子网不应重叠。

      接下来两个用例彼此相关：客户的云与公共因特网之间的入站和出站流量。这些可能是任何客户或站点的最常见用例。

      2. 来自公共因特网的入口（入站）连接：

      在此案例中，客户需要从公共因特网访问其访客 VM。为创建解决方案，此客户需要向每个访客 VM 分配一个 IP 地址（最好为公共可路由的 IPv6，或者可使用浮动 IPv4），并且必须确保其安全组设置为允许对其 VM 的相应入站访问。

      3. 到公共因特网的出口（出站）连接：

      在此案例中，客户需要从其访客 VM 访问因特网，例如，下载安装包。通常，访客 VM 应获取通过与其 pod 关联的 Vyatta 上的 NAT 到因特网的出站连接。

      注释：用例 2 和 3 引用全局因特网连接。通常，正如我们在本文中其他地方所指明的，组合 IP 地址空间时将出现部分重叠，在此情况下，必须使用 NAT。这不是理想的解决方案。或者，您可以创建自定义的网络解决方案；但是，这些解决方案通常过于脆弱、复杂而死板，需要付出大量时间和精力进行维护。

      4. 访问 SoftLayer 服务：

      有时，客户需要访问 IBM 云数据中心的专用网络上托管的服务。从安全性角度来看，只有必需的访客 VM 才有权访问其专用网络上的 IBM 云托管服务。此访问权通过与 pod 关联的 Vyatta 上的 NAT 提供。IBM 云服务托管于 10.0.0.0/14 网络上。

      注释：接下来的两个用例 BYOIP 和 DirectLink 其实是同一枚硬币的正反两面。这两个用例只是使用 VPN 或 DirectLink（可能是租用线路或 MPLS 电路）提供到现有/远程网络的连接的机制。（MPLS 表示多协议标签交换，这是企业客户常用的一种 VPN 类型。）因此实际上都是 VPN。

      5. BYOIP 连接：

      在此案例中，客户想要将自己的网络子网带入 IBM Bluemix Private Cloud 中，并从自己的本地数据中心连接到这些子网。要创建解决方案，我们可以使用客户提供的子网值创建覆盖 OpenStack 网络。这些覆盖网络连接到禁用 SNAT 的“默认”OpenStack 路由器。我们将确保在每个客户子网的关联 Vyatta 中设置路由。此解决方案需要在 IBM Bluemix Private Cloud 与客户数据中心之间建立隧道连接。

      6. DirectLink 连接：

      基于客户所选 DirectLink 选项，我们可以在其 IBM Bluemix Private Cloud 中设置相应的网络。最常见的用例是将网络设置为另一个 BYOIP 网络。但在此情况下，我们需要针对 DirectLink 使用不同的隧道连接端点。

      注释：在这些案例中，客户想要将其现有 IP 地址计划和网络计划扩展至其 IBM Bluemix Private Cloud 中。它是一种混合形式的云，而不是我们所销售的“混合云”，这对于大部分人而言，表示在公共云和私有云之间创建连接。

      7. SoftLayer 专用网络连接：

      在此案例中，客户需要在两个独立 IBM Bluemix 帐户中运行的两个云之间创建连接：其中一个为 IBM Bluemix Private Cloud（位于 IBM Bluemix 数据中心内）帐户，另一个为客户自己的 SoftLayer 帐户。此情况需要在两个（原本不相关的）SoftLayer 帐户之间启用 VLAN 生成。

      • a. 如果此客户在两个云中均使用 IBM 云数据中心的可路由 10.x.x.x 子网，那么无需任何操作，假定在这两个帐户之间已启用 VLAN 生成。
      • b. 如果此客户使用 BYOIP 网络，那么在 IBM Bluemix Private Cloud 与客户的其他 SoftLayer 帐户中运行的云之间需要隧道连接。

      注释：对于案例 (a.)，跨帐户 VLAN 生成是一项 IBM 云数据中心功能，实际应称为“跨帐户 VLAN 间路由”，因为它并非在 L2 上运行，而是在 L3 上运行。此案例意味着只需在 SoftLayer 的现有 IP 地址计划内的两个 IBM 云帐户之间生成。对于案例 (b.)，IBM 云工程师必须确定并解释有效的隧道类型。

      问题和解决方案声明

      由于必须应用于这些 IPv4 用例的解决方案的复杂性（如本文中所述），建议客户在需要分配更大的子网时认真考虑采用 IPv6 地址，例如，为大型专用云分配地址时可能希望采用 IPv6 地址。转换至 IPv6 只要可行就是有意义的。作为最佳做法，我们现在建议客户考虑在接下来的 5 年内完全迁移至 IPv6。

      结论

      本文有关使用 IPv6 地址的建议使全球所有人都能够使用全球 IP 池开展工作。它解决了这些客户用例中描述的许多（如果不是全部）难题，这些难题目前必须通过虚拟联网模式来解决，例如，本文中所描述的模式。这些实施的复杂性在大部分情况下是可管理的，因为 IBM 工程团队非常熟悉问题和所涉及的权衡。

      全球因特网地址路由表是可用的最大 IP 地址计划，统一了所有这一切。如果您使用此计划开展工作，并且遵循相关规则，那么您的地址将成为唯一地址。因此，我们建议迁移至 IPv6。在此之前，本文中所描述的用例是为了便于您了解 IBM Bluemix Private Cloud 工程师为了跨越 IPv4 地址的全球限制以及必要的 IBM 云数据中心 pod 结构而创建的可行解决方案。

      • 术语注释：路由表是来自网络中特定点的 IP 地址空间的视图。
