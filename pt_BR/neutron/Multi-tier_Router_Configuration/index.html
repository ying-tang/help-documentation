      Configuração de roteador de várias camadas em seu IBM Bluemix Private Cloud

      • by Ulysses Kanigel and Leslie Lundquist ● August 31, 2016

      • Tags:

        • neutron

        • networking

        • router

        • configuration

        • multi-tier

        • add_interface

      Este artigo explica como testar uma Configuração do Roteador Multicamada e por que você pode querer um.

      Talvez você queira fazer uma configuração do roteador multicamada para a nuvem em um desses casos de uso:

      1. Você quer evitar que os hosts nessas sub-redes (filhas) usem IPs flutuantes.
      2. Você quer criar isolamento de rede para essas redes (filhas) de outro projeto ou redes de locatário em sua nuvem, dando a cada uma seu próprio roteador (filho) abaixo do roteador principal.
      3. Você precisa que cada uma dessas redes de projeto isoladas tenha ingresso e egresso da Internet, enquanto permanecem isoladas de outras redes em sua nuvem e também não usando IPs flutuantes.

      A maneira de estabelecer uma rede multicamada inicialmente é estabelecer rotas estáticas do roteador principal para cada um dos roteadores “filhos”. A ideia principal desse teste é assegurar que essas rotas estáticas sejam persistentes. Neste exemplo, nós configuramos duas redes filhas desse tipo, chamadas child-net1 e child-net2.

      Vamos dizer que você quer testar uma configuração do roteador multicamada e confirmar se as rotas que você configurou são persistentes no failover.

      Aqui estão as etapas que você pode precisar para executar o teste:

      
#Create networks and subnets

neutron net-create parent-net

neutron net-create child-net1

neutron net-create child-net2

neutron subnet-create parent-subnet 162.16.0.0/24

neutron subnet-create child-subnet1 162.16.1.0/24

neutron subnet-create child-subnet2 162.16.2.0/24

# Create routers

neutron router-create parent-router

neutron router-create child-router1

neutron router-create child-router2

# Set gateway of the parent router to the external network and add interface on the parent-router subnet

neutron router-gateway-set parent-router external

neutron router-interface-add parent-subnet b490f691-364f-4f7c-866c-1c45e7b8f8cf  #unique identifier for subnet

# Add interfaces for parent-router, child-router1 and child-router2 subnets

neutron port-create parent-router

neutron port-create child-router1

neutron port-create child-router2

neutron router-interface-add child-subnet1 port=c1b305a1-f634-4828-a6ce-d5a6a058824e  #unique identifier for port

neutron router-interface-add child-subnet1 2476ad51-e2cb-4d4e-bbd8-db18cc06ea88   #unique identifier for subnet

neutron router-interface-add child-subnet2 port=c1b467b1-f444-4325-a5dc-d4b3a04624d   #unique identifier for port

neutron router-interface-add child-subnet2 2378ad51-e4cc-2d3e-bcc5-db34cd05fa48    #unique identifier for subnet

# Set default route on child router(s) to parent-router subnet gateway address : they both talk to the parent router

neutron router-update child-router1 --routes type=dict list=true destination=0.0.0.0/0,nexthop=162.16.0.1

neutron router-update child-router2 --routes type=dict list=true destination=0.0.0.0/0,nexthop=162.16.0.1

      Não deve haver nenhum problema em ajustar a escala desse design para cima para incluir várias sub-redes conectadas a cada roteador filho, pois o roteamento é atingido no roteador filho.

      Nota: Lembre-se de que quaisquer instâncias conectadas a um roteador filho não conseguirão ter IPs flutuantes associados a elas, pois esse roteador filho não está diretamente conectado à rede externa.

      Resolução de problemas: erro: falha em add_interface:

      Você pode obter esse erro se tentar anexar uma interface ao roteador-A-do-locatário para uma rede-B-do-locatário: Erro: falha em add_interface: falha na solicitação: erro do servidor interno ao processar sua solicitação.

      Nossa política Neutron realmente permite incluir uma interface do roteador para usuários com funções admin, cloud_admin ou project_admin, mas falha porque a API Neutron é explicitamente codificada para não permitir operações em recursos de propriedade de locatários diferente daquele para o qual você está autenticado, a menos que você tenha a função admin, conforme refletido neste código:

      https://github.com/openstack/neutron/blob/stable/mitaka/neutron/api/v2/attributes.py#L971

      Se você precisar de uma rede de um locatário anexado a um roteador em um locatário diferente, a equipe de suporte precisará executar essa operação em seu nome. Forneça o comando da CLI que você está tentando executar que lança o erro e nós o executaremos para você.
