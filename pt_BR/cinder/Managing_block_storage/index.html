      Gerenciamento o armazenamento de bloco

      • by Ying Tang ● November 21, 2016

      • Tags:

        • cinder

        • block storage

        • publicURL

        • endpoint

        • not found

        • usage

      Este artigo foi adaptado da seguinte postagem do blog do Bluemix: Gerenciando volumes Cinder com o IBM Bluemix Private Cloud.

      O IBM Bluemix Private Cloud oferece vários modelos de armazenamento, para lidar com os requisitos exclusivos de nossos clientes. Se você tiver cargas de trabalho como sistemas de arquivos expansíveis e sistemas de gerenciamento de dados hospedados em seu Bluemix Private Cloud, o armazenamento de bloco pode ser adequado para o caso de uso específico de seu ambiente. Este artigo explica como introduzir nosso modelo de armazenamento de bloco.

      OpenStack Cinder

      O armazenamento de bloco é um armazenamento persistente independente que você pode anexar a e separar de seus servidores virtuais. Com o armazenamento de bloco, seus dados persistem além do ciclo de vida do servidor virtual. Por exemplo, se seu servidor virtual parar inesperadamente, seus dados armazenados no armazenamento de bloco estarão protegidos e permanecerão intactos.

      Como o Bluemix Private Cloud é desenvolvido com OpenStack, nós oferecemos o OpenStack Cinder como a opção de armazenamento de bloco aos nossos clientes. O Cinder dá aos desenvolvedores a capacidade de gerenciar seus volumes de blocos usando a UI do painel do OpenStack Horizon e as APIs padronizadas do OpenStack.

      Para clientes que estão executando o IBM Bluemix Private Cloud versão 3.1.0 e posterior, o Cinder suporta vários backends: o planejador consegue localizar os tipos de volumes certos, com flexibilidade. O armazenamento de bloco do Cinder é configurado para usar clusters do Ceph, por padrão.

      Como o Ceph funciona com o Bluemix Private Cloud

      O Ceph é uma solução de armazenamento distribuído com base no Reliable Autonomic Distributed Object Store (RADOS), que fornece aplicativos com objeto, bloco e armazenamento do sistema de arquivos em um cluster de armazenamento unificado, único.

      Um cluster do Ceph geralmente é composto de nós de monitoramento e nós de armazenamento (nós OSD). Os nós de monitor mantêm uma cópia do mapa do cluster para assegurar alta disponibilidade e acompanham o funcionamento do cluster.

      Com o Ceph, o Bluemix Private Cloud fornece uma opção de armazenamento de bloco com capacidade maior, desempenho excepcional e custo menor. De acordo com os requisitos específicos da carga de trabalho, é possível também customizar os seguintes tipos de nós para os clusters do Ceph:

      • Uma opção apenas SSD com discos OSD apenas SSD e Diário do Ceph co-localizado tem o desempenho de E/S ideal. Ela adequa a carga de trabalho de armazenamento esperando alto IOPS.

      • Uma opção híbrida com discos SATA OSD e Diário do Ceph baseado em SSD apresenta um bom desempenho de ES e custo menor, e se ajusta às cargas de trabalho médias do IOPS.

      • Uma opção multicamada com uma combinação de nós híbridos e apenas SSD é ideal para ambientes de grande escala.

      Depois que o armazenamento de bloco é configurado com as opções adequadas de multibackend e multicamada, você pode criar um volume Cinder, usando o painel do OpenStack Horizon ou o cliente do OpenStack Cinder.

      Criando um volume Cinder usando a API do OpenStack

      Para criar um volume usando o cliente do OpenStack, execute o seguinte comando do Cinder:

      $  cinder create <size> --display-name <name> --volume-type <type> \
--image-id <imageid> --availablity-zone <az>

      Os comandos a seguir mostram exemplos de como é possível customizar o volume recém-criado para o Bluemix Private Cloud.

      • <size>: o tamanho do volume em gibibytes (GiB).
      • <name>: o nome do volume.
      • <type>: CEPH_HYBRID ou CEPH_SSD. É possível usar o comando cinder type-list para obter uma lista de tipos de volume disponíveis.
      • <imageid>: como opção, especifique o ID da imagem que deseja usar para seu volume. É possível usar o comando nova image-list para listar as imagens existentes.
      • <az>: como opção, especifique a zona de disponibilidade na qual deseja criar o volume. É possível usar o comando cinder availability-zone-list para obter uma lista de zonas de disponibilidade.

      Para obter mais informações sobre este tópico, consulte o seguinte documento: Criando um volume com a linha de comandos.

      Criando um volume Cinder com o Horizon

      Para criar um volume de armazenamento de bloco usando o Horizon, o painel do OpenStack, você pode começar com as seguintes etapas:

      1. Efetue login no painel do Horizon.

      2. Na seção Projeto, selecione Calcular e, em seguida, clique em Volumes. Caso haja volumes existentes, é possível visualizar uma lista dos volumes e seus detalhes (como nome, descrição, tamanho e status), na UI do painel.

      Clique em Criar volume na parte superior direita do painel do Horizon. O diálogo Criar Volume é exibido.

      Forneça os seguintes detalhes sobre o volume:

      • Nome do volume: digite o nome do volume.
      • Descrição: digite uma descrição do volume.
      • Origem do volume: Nenhuma origem, volume nulo ou Imagem. Quando se escolhe um volume nulo, ele não contém um sistema de arquivos ou uma tabela de partição. Ao escolher a opção de imagem, um novo campo é exibido para Usar a imagem como origem e é possível selecionar a imagem na lista.
      • Tipo: Ceph_SSD ou Ceph_Hybrid.
      • Tamanho (GiB): selecione o tamanho do volume em gibibytes(GiB), por exemplo, 1.
      • Zona de disponibilidade: escolha a zona de disponibilidade, por exemplo, nova.

      Clique em Criar volume ao concluir. Agora, você consegue anexar o volume Cinder ao servidor virtual.

      Para obter mais informações sobre este tópico, consulte o seguinte documento: Criando um volume a partir do painel do Horizon.

      Depois de criar e anexar seus volumes Cinder a Servidores Virtuais, você pode facilmente ajustar e expandir os tamanhos de seus volumes conforme seus requisitos de capacidade crescem. Você também pode separar e reanexar os volumes de bloco a outro Servidor Virtual que requer armazenamento persistente, enquanto seus dados armazenados continuam protegidos.

      Resolução de problemas

      openstack volume list –name vs. cinder list –name

      Se você for um usuário com uma função que não seja cloud_admin, o comando openstack volume list --name não funcionará corretamente para você. Para contornar esse problema, você pode permitir um usuário com a função cloud_admin filtre os volumes por nome ou pode usar o comando cinder list --name, que funciona para qualquer usuário. Nós abrimos o defeito interno 374319 para esse problema e estamos trabalhando com nossa equipe de engenharia na resolução do problema.

      O terminal publicURL para o serviço volumev2 não localizou um erro ao executar os comandos de volume do cinder ou do openstack

      Se você obtiver esse erro, geralmente é porque você não tem nenhum nó de armazenamento de bloco (volume Cinder) em seu cluster atual. Se desejar solicitar o armazenamento de bloco, abra um chamado solicitando-o e nossa equipe de vendas irá ajudá-lo.

      Terminal publicURL para o serviço volumev3 … não localizado

      Este erro pode acontecer quando você tem uma versão anterior do serviço de volume em sua lista de terminais. Para resolver isso, inclua export OS_VOLUME_API_VERSION=2 em seu arquivo RC.

      Por que o tamanho do volume total de todos os volumes alocados no Cinder não corresponde ao total bruto do Ceph?

      Isso pode acontecer se você tiver volumes no Ceph que foram excluídos mas ainda estão no conjunto, e outros volumes foram criados a partir desses volumes antes da exclusão deles. A maneira como o Ceph funciona, volumes clonados são volumes filhos de cópia-na-gravação do volume do qual eles foram clonados, o que significa que o volume pai deve ser retido até que todos os seus filhos (e netos, bisnetos, etc.) sejam excluídos. Nesse caso, o uso não será igual à soma dos tamanhos dos volumes atualmente ativos.

      Mais informações sobre a camada de cópia-na-gravação do Ceph podem ser encontradas em: http://docs.ceph.com/docs/hammer/dev/rbd-layering/
