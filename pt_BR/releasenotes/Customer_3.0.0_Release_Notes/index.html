      Cliente do IBM Bluemix Private Cloud

      Notas sobre a liberação 3.0.0

      30 de junho de 2016

      Este documento oferece uma descrição técnica básica das ofertas mais recentes do IBM Bluemix Private Cloud, incluindo detalhes das ofertas Dedicada e Local. Em cada seção, é possível encontrar links para a documentação detalhada.

      Resumo da liberação mais recente

      A liberação 3.0.0 do IBM Bluemix Private Cloud inclui muitas atualizações para a plataforma subjacente do OpenStack na qual cada Bluemix Private Cloud é construído, e também upgrades para a infraestrutura e os recursos do Bluemix Private Cloud e do Bluemix Private Cloud Local. Mais notável é a movimentação para a liberação do OpenStack Mitaka, que ativa muitos recursos novos. Nós incluímos algumas melhorias gerais do produto, como avaliações de desempenho e um conjunto de testes de Rally atualizado, projetado para facilitar uma operação e implementação mais suaves. Esta liberação também inclui muitas melhorias na documentação.

      Cada IBM Bluemix Private Cloud, Versão 3.0.0, tem esses recursos novos:

      1. Suporte ao Openstack Mitaka.
      2. Identidade Federada Keystone-to-Keystone em várias nuvens do Bluemix Private.
      3. Website de documentação atualizado agora acessível por meio do Box Panel.
      4. Atualizações regulares para o IBM Bluemix Private Cloud Images.

      Recursos do IBM Bluemix Private Cloud 3.0.0 (nos Datacenters do IBM Cloud):

      1. Suporte para a ampliação de VLAN do Datacenter do IBM Cloud e conectividade com o hardware Bare Metal do cliente.
      2. Vyatta com rede de 10 G.

      Recursos do IBM Bluemix Private Cloud Local 3.0.0:

      1. Nós de armazenamento de bloco híbrido.
      2. Novos tipos de nós de cálculo.
      3. IBM Bluemix Private Cloud Local ativado por HIPAA.
      4. Suporte para o IBM Bluemix Private Cloud Local na plataforma Bluemix Local System (liberação Beta).

      Limitações conhecidas desta liberação

      Para obter documentação adicional do usuário, consulte a página principal da Ajuda do cliente, disponível on-line a qualquer momento em help.blueboxcloud.com.

      Uma Visão geral do produto para o IBM Bluemix Private Cloud versão 2.0 e posterior está disponível.

      Cada IBM Bluemix Private Cloud 3.0.0 tem esses novos recursos:

      Suporte ao Openstack Mitaka: Agora, vários recursos estão disponíveis com base em nossa mudança para o OpenStack Mitaka.

      • É possível configurar grupos de auto-scaling e LBaaS (v2) usando modelos do Heat e o painel do Horizon.
      • A liberação do Mitaka introduz o novo serviço de alerta aodh, que será instalado em qualquer ambiente que tenha o Ceilometer ativado.
      • Para obter mais informações, consulte as Notas sobre a liberação do OpenStack Mitaka.

      Serviços específicos do OpenStack ativados para cada IBM Bluemix Private Cloud, Versão 3.0.0, na liberação do Mitaka:

      • API de armazenamento de bloco & Extensões (Cinder v2.0)
      • API de cálculo & Extensões (Nova v2.1)
      • API do painel (Horizon) N/A
      • API de identidade & Extensões (Keystone v3.0)
      • API do serviço de imagem (Glance v2.2)
      • API de rede & Extensões (Neutron v2.0, Neutron LBaaS v2)
      • API de armazenamento de objetos & Extensões (Swift v1.0)
      • API de orquestração (Heat v1.0)
      • API de telemetria (Ceilometer v2.0) (mais aodh)

      Federação Keystone-to-Keystone em várias nuvens do IBM Bluemix Private: A federação Keystone-to-Keystone (K2K) permite efetuar login em várias nuvens usando suas credenciais de usuário armazenadas em um Provedor de Identidade do Keystone. Nós criamos um artigo de configuração de K2K para você.

      Website de documentação atualizado agora acessível por meio do Box Panel: Nós atualizamos e reprojetamos nosso website de documentação do cliente, agora disponível em help.blueboxcloud.com e facilmente acessível por meio da interface do Box Panel. Consulte o Guia do Usuário do Box Panel para obter mais informações.

      Atualizações regulares para o IBM Bluemix Private Cloud Images: Atualizações regulares para o IBM Bluemix Private Cloud Images suportado estão disponíveis por meio da interface do Box Panel. As notas sobre a liberação do cliente para todas as atualizações de imagens individuais são publicadas aqui. Nós também criamos um Guia do usuário para trabalhar com imagens de nuvem.

      IBM Bluemix Private Cloud 3.0.0 (nos datacenters do IBM Cloud):

      Suporte para ampliação de VLANs do datacenter do IBM Cloud e conexões de hardware Bare Metal gerenciadas pelo cliente: Esse recurso é projetado para qualquer cliente do IBM Bluemix Private Cloud para quem o desempenho de servidores Bare Metal é um requisito estrito, mas que também precisa de acesso à comunicação segura por meio da rede privada de datacenters do IBM Cloud, entre a infraestrutura Bare Metal e uma rede de instância de nuvem do OpenStack. Nota: Para usar esse recurso, o hardware Bare Metal deve estar em uma conta do SoftLayer separada, gerenciada e de propriedade do cliente.

      Vyatta com rede de 10 G: Este recurso é projetado para clientes com requisitos de largura de banda grande para egresso e ingresso da nuvem.

      IBM Bluemix Private Cloud Local 3.0.0:

      Novos tipos de nós de cálculo: Os nós de cálculo de expansão atendem aos seus requisitos de ES transitória mais rápida, mais RAM e um armazenamento transitório maior. Os nós de expansão permitem uma melhor cobertura de cargas de trabalho e tipos de instância. Com esta liberação, os tipos de cálculo são agrupados em Zonas de Disponibilidade para fácil direcionamento de cargas de trabalho para o tipo de cálculo apropriado. É possível selecionar a partir das extensões de nós de cálculo a seguir para incluir no IBM Bluemix Private Cloud Local:

      • É possível selecionar nós de cálculo de expansão especializados: ES ou HIGH STORAGE (Novo nesta liberação)
      • É possível selecionar nós de cálculo corporativos ou padrão adicionais: STANDARD COMPUTE ou ENTERPRISE COMPUTE (Anteriormente disponível)
      • É possível selecionar Nós de cálculo POWER (Habañero): esses nós suportam particularmente aplicativos IBM Watson e outros.

      Nós de armazenamento de bloco híbrido: Os nós de armazenamento de bloco híbrido fornecem um armazenamento de bloco de custo mais baixo e maior densidade. Nós incluímos um tipo de nó de armazenamento Ceph híbrido, que fornece um armazenamento de bloco de custo mais baixo e maior densidade. É 4x a capacidade para o mesmo ponto de preço que SSD, com uma pilha de software idêntica e arquitetura de referência. Observe que apenas um tipo de armazenamento de bloco pode ser usado por cluster: armazenamento de bloco SSD e Armazenamento de bloco híbrido não podem ser misturados no mesmo cluster.

      • É possível selecionar o Armazenamento de bloco: SSD ou HYBRID

      Nuvem ativada por HIPAA: Uma nuvem ativada por HIPAA tem as mesmas opções de configuração e expansão que qualquer outra oferta do IBM Bluemix Private Cloud, mas estamos ativando mais de 50 controles de segurança de informações adicionais que suportam nossos clientes que devem armazenar informações pessoais de funcionamento (PHI). Em razão desse aumento nos recursos de segurança de informação, uma nuvem ativada para HIPAA deve ser ativada para HIPAA desde sua concepção: não é possível fazer upgrade de uma nuvem existente para ativar o protocolo HIPAA. Além disso, não é possível fazer downgrade de um Bluemix Private Cloud Local ativado por HIPAA: a nuvem HIPAA existente deve ser limpa—por protocolo HIPAA—e você deve pedir uma nova nuvem Local com controladores não HIPAA.

      • É possível selecionar a Ativação por HIPAA para o IBM Bluemix Private Cloud Local.

      Suporte para Bluemix Private Cloud no Bluemix Local System (liberação Beta): Bluemix Local é um sistema IBM existente para gerenciamento de hardware (anteriormente chamado PureApp). Com sua liberação Beta, nós o estamos disponibilizando como uma plataforma para IBM Bluemix Private Cloud Local, além do hardware Cisco já disponível.

      Limitações conhecidas desta liberação

      Painel do Horizon e LBaaS v2:

      • O painel do Horizon LBaaS v2 não permitirá a criação de listeners https.
      • O compartilhamento de conjuntos não está disponível por meio da interface do Horizon.
      • Não é possível definir o peso de um membro para 0 por meio do Horizon.
      • A alternância de conteúdo L7 não está disponível com o driver padrão. (HaproxyOnHostPluginDriver)
      • A Terminação TLS não é suportada.

      Problemas em aberto do OpenStack Community para LBaaS v2:

      • LBaaS VIP é associado ao grupo de segurança Padrão por padrão. Uma solução alternativa é abrir as portas necessárias no grupo padrão. https://bugs.launchpad.net/neutron/+bug/1295424
      • Excluir LBaaSv2 não exclui o namespace LBaaS. https://bugs.launchpad.net/neutron/+bug/1495430
      • O status do monitor de funcionamento não reflete se o monitor de funcionamento foi bem-sucedido ou falhou. https://bugs.launchpad.net/neutron/+bug/1464229
      • O monitor PING envia solicitações de HTTP. Quando o monitor de funcionamento LBaaS v2 é configurado para Ping, ele envia solicitações de HTTP. https://bugs.launchpad.net/neutron/+bug/1463816
      • Não é possível excluir o monitor de funcionamento LBaaSv2 se seu listener tiver sido excluído. Uma solução alternativa é excluir o monitor de funcionamento órfão da linha de comandos. https://bugs.launchpad.net/neutron/+bug/1571097
      • É possível excluir lbaas-pool com lbaas-hm ainda anexado. Quando isso acontece, nem mesmo o admin consegue excluir o Monitor de Funcionamento restante, exceto indo diretamente para o banco de dados. https://bugs.launchpad.net/neutron/+bug/1526964

      Painel do Horizon e projetos:

      • No painel do Horizon, a página de edição de cotas não é vista por cloud_admin na janela modal Edição do Projeto. A modificação da cota por cloud_admin ainda é permitida, usando as APIs ou a CLI do OpenStack.
      • No Painel de Identidade do painel do Horizon, ao criar um novo projeto como a função cloud_admin, um menu pop-up de erro é exibido (Não é possível definir cotas do projeto). No entanto, o projeto é criado e recebe a cota padrão. Um usuário com uma função cloud_admin pode subsequentemente atualizar a cota, usando a API ou a CLI do OpenStack.
      • No Painel de Identidade do painel do Horizon, ao modificar um projeto existente como um usuário com a função cloud_admin, um menu pop-up de erro é exibido (Não é possível modificar o projeto). O projeto, na verdade, será atualizado. O menu pop-up é mostrado porque as cotas do projeto não podem ser definidas.

      Painel do Horizon e painel do Administrador:

      • No Painel do Administrador do painel do Horizon, ao clicar nos Hypervisors como um usuário com a função cloud_admin, um menu pop-up de erro é exibido: “Não é possível obter a lista de serviços de Nova.” Esse comportamento é esperado porque um usuário com as funções cloud_admin não tem permissão para realizar operações nesses itens.

      Outras limitações conhecidas:

      • Neutron RBAC-create: o comando da CLI neutron rbac-create retorna um código de status de erro “500” a menos que --target-tenant seja especificado no ponto de criação inicial.
      • Existe a limitação de desempenho de armazenamento de bloco, quando testado no ambiente de teste de desempenho de 30 nós com uma configuração Ceph híbrida (HDD + SSD bcache)
      • Federação Keystone-to-Keystone: existe atualmente um problema com a criação da delegação trust-for-trust por um usuário federado. Para encontrar uma solução alternativa para esse problema, a função heat_stack_owner precisa ser designada explicitamente para o usuário federado.
