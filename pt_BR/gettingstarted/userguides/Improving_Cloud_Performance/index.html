      Preparando-se para o OpenStack II - Melhorando o desempenho da nuvem

      • by Dustin Lundquist and Leslie Lundquist ● November 21, 2016

      • Tags:

        • openstack

        • sr-iov

        • virtio

        • vhost

      Republicado de https://admin.blogs.prd.ibm.event.ibm.com/blogs/bluemix/2016/11/building-openstack-ii-improving-cloud-performance/

      Neste artigo, segundo da série, vamos abranger mais fundamentos de nuvem: Mais sobre QEMU e muito mais sobre como melhorar o desempenho geral da carga de trabalho da nuvem.

      Essencialmente, melhorar o desempenho da nuvem requer criar melhorias de eficiência de E/S ao acessar dispositivos de rede e de armazenamento. Ao otimizar a virtualização de E/S para que as interfaces do dispositivo de E/S entre as MVs guest e o host sejam mais eficientes, é possível melhorar o desempenho e incluir flexibilidade para E/S. Assim, o desempenho da nuvem melhora no geral porque mais do total de recursos do processador está disponível para sua carga de trabalho.

      Trata-se de melhorar o desempenho de E/S por meio da virtualização

      Lembre-se de que QEMU é o elemento básico de uma MV de nuvem. Ele executa a tarefa! No entanto, como o QEMU é um emulador, faz sentido contar com ele o mínimo possível quando a velocidade for essencial. Embora o KVM inclua uma certa melhoria de desempenho do processador e da memória no QEMU, um gargalo permanece para o desempenho de E/S.

      Há dois caminhos para melhorar o desempenho de E/S com a virtualização: um caminho de software e um caminho de hardware. Vamos examinar cada um deles neste blog. Ao longo de cada caminho, cada etapa que percorremos deixa o QEMU mais fora de ação. Por quê? Menos emulação significa melhor desempenho.

      O caminho de software envolve virtualizar os dispositivos “tradicionais” e seus drivers de modo mais eficiente. O caminho de hardware envolve ligar os dispositivos mais diretamente às MVs e possivelmente mesmo usando hardware especializado que implementa um recurso chamado Virtualização de ES de Raiz Única (SR-IOV). Esse é essencialmente um caminho assistido por hardware para rapidamente emparelhar MVs com dispositivos virtualizados que podem ser instanciados conforme necessário.

      O caminho de software: QEMU, ‘virtio’ e ‘vhost’

      Como abordado em um blog anterior, o QEMU foi projetado para operar como um emulador, que pode até mesmo emular sistemas operacionais históricos como Windows NT e OS/2. Ele emula qualquer hardware que o sistema operacional espera localizar; portanto, os dispositivos padrão que ele expõe são dispositivos de hardware físico “comuns”. Mas o método que o sistema operacional usa para se comunicar com esses dispositivos não foi projetado com a virtualização em mente!

      Dispositivos de E/S mais antigos geralmente não eram processadores com finalidades gerais e eram mais lentos. Assim, nos dias de “hardwares” antigos, havia poucos benefícios para otimizar o desempenho de E/S. Por exemplo, os discos físicos levam um tempo (4 ms) significativo para acessar um local em um disco, portanto, pode haver pouco benefício em otimizar as comunicações entre o driver e o controlador de disco. Em uma nuvem moderna, com MVs como servidores, benefícios significativos surgem da melhoria do desempenho de E/S e isso pode ser feito de maneiras bem claras.

      Por que a virtualização de E/S ajuda?

      Para executar a E/S eficiente, o sistema em nuvem requer (1) um driver para a MV guest, (2) um backend adicional no software do host (por exemplo, QEMU) que fornece o dispositivo de E/S, e, finalmente, (3) as interfaces que o software do host usa para se comunicar com o dispositivo de E/S subjacente.

      Como pode ser visto na figura, vários componentes estão envolvidos no cumprimento de uma solicitação de E/S de dentro de uma MV. Cada transição entre esses componentes incorre em uma multa de desempenho. Para demonstrar com que velocidade o desempenho de E/S pode ser degradado, aqui está um detalhamento passo a passo de como ele funciona:

      • Primeiro, o programa que está em execução dentro da MV faz sua solicitação de E/S ao sistema operacional da MV, usando a interface “normal”. (Se o software em execução em uma MV tiver usado um método diferente, a virtualização não seria transparente para o usuário.)

      • Em seguida, o sistema operacional da MV usa o driver correspondente ao dispositivo exposto por QEMU.

      • O driver, então, executa uma certa operação, que é retida pelo KVM e transferida para o QEMU para preenchimento.

      • O QEMU usa uma interface semelhante para fazer uma solicitação de E/S para o sistema operacional do host.

      • O sistema operacional do host, então, usa o driver correspondente ao hardware físico para executar a operação de E/S solicitada.

      Como podemos acelerar este processo? Existem duas boas opções. Primeiro, podemos virtualizar de modo mais eficiente a maneira como emulamos qualquer dispositivo particular. Essa opção, geralmente, é referida como “drivers paravirtualizados” e o melhor exemplo disso é algo chamado virtio, que foi incorporado ao OpenStack Nova. Em segundo lugar, para mais melhorias de desempenho, podemos eliminar outra transição removendo a emulação de dispositivo de uma MV completamente. Ou seja, retiramos a “emulação de dispositivo” do QEMU da MV guest e a limitamos ao sistema operacional do host.

      Otimização 1: drivers paravirtualizados: a abordagem virtio

      Virtio é uma família de dispositivos virtualizados que são expostos pelo QEMU, juntamente com seus drivers correspondentes, em um sistema operacional da MV. Esses drivers otimizam a interface entre o driver da MV guest e o QEMU usando estruturas de dados modernas e eficientes, como ring buffers, em vez de emular de modo servil a interface de um dispositivo de hardware existente. Usando virtio, a MV se comunica de modo mais eficiente com o QEMU, melhorando, assim, o desempenho de E/S.

      Como advertência, virtio requer imagens da MV guest para incluir esses drivers virtio mais rápidos, então, algumas imagens mais antigas (por exemplo, Windows 95 ou 98) podem não funcionar com virtio. Felizmente, a maioria dos sistemas operacionais agora inclui drivers virtio por padrão e isso foi feito desde 2013.

      Otimização 2: Vhost (Removendo o modelo de dispositivo de espaço do usuário do QEMU)

      A próxima otimização de desempenho consiste em remover completamente o recurso de emulação de dispositivo virtio da MV e movê-lo para o kernel do sistema operacional do host. Essa movimentação evita a penalidade de desempenho associada às transições do KVM para o QEMU e de volta para o kernel do host.

      O caminho de hardware: passagem de PCI e SR-IOV

      A passagem de PCI é um mecanismo genérico que pode ser aplicado a qualquer dispositivo PCI. Ela oferece uma alternativa aos tipos de virtualização de software de dispositivos e drivers que descrevemos nas seções anteriores. Em contraste com virtio e vhost, a passagem de PCI expõe um dispositivo de E/S de hardware diretamente à MV guest, removendo, assim, o QEMU e o kernel do host do caminho de E/S completamente e eliminando as penalidades de desempenho dessas transições de emulação de software.

      A passagem de PCI é uma boa alternativa para a criação de um desempenho melhorado em muitos casos de uso.

      • Por exemplo, algumas pessoas usam essa técnica de passagem para jogos, configurando a passagem de PCI para sua placa de vídeo. Observe que, neste caso, a virtualização não é usada para melhorar o desempenho de jogos, mas para executar os jogos em outros sistemas operacionais. A virtualização de jogos modernos não é realmente viável sem a passagem de PCI.

      • Provavelmente, o tipo mais comum de dispositivo de passagem de PCI é a placa de rede. Por exemplo, você pode fazer a passagem da placa de rede com vhost_net, que se desfaz do QEMU do caminho de dados da rede, mas vhost_net ainda conta com a pilha de rede intacta do kernel do host.

      • É importante observar que a passagem de PCI geralmente é pouco prática para adaptadores de barramento de host (HBA) e discos, porque a maioria dos servidores tem apenas um único controlador de disco conectado a todos os discos no servidor. Assim, a capacidade desses discos não poderia ser compartilhada entre as MVs em uma nuvem; todos os discos pertenceriam a uma única MV. A próxima seção diz como HBAs ativados com SR-IOV criaram soluções para essa dificuldade.

      • Alguém pode imaginar uma aplicação esotérica da passagem de PCI para um HBA, em que uma nuvem pode ser inicializada de USB e, em seguida, toda a capacidade de armazenamento do disco alocada para uma única MV executada como um banco de dados de backend.

      É importante lembrar que a passagem de PCI requer um dispositivo de E/S dedicado para cada MV guest. Esse requisito de ter hardware dedicado para cada MV reduz a flexibilidade que vem da virtualização e aumenta significativamente o custo. Felizmente, alguns tipos mais novos de hardware estão disponíveis, que podem minimizar o custo por meio de uma apresentação integrada, assistida por hardware de dispositivos de PCI virtualizados. É uma técnica chamada SR-IOV.

      SR-IOV

      Interfaces de rede high-end, mais novas e controladores de disco incluem SR-IOV (single root I/O virtualization). Essencialmente, esses dispositivos de hardware podem se apresentar como vários dispositivos de PCI, sendo que cada um pode ser anexado a uma MV. Em uma implementação típica, pode haver uma principal, pai (ou PF) e, em seguida, um ou mais dispositivos lógicos conhecidos como unidades de função virtual (VFs). Ao configurar PF, os dispositivos VF adicionais podem ser criados juntamente com seus recursos associados. Então, esses VFs podem ser anexados a MVs individuais por meio da passagem de PCI.

      Potencialmente, você poderia usar a passagem de PCI para dois dispositivos de E/S por MV guest: uma interface de rede e um controlador de disco. Por exemplo, com dois dispositivos SR-IOV, você poderia obter uma melhoria de desempenho de E/S para rede e para armazenamento.

      Resumo

      Você pode melhorar o desempenho de carga de trabalho da nuvem usando qualquer um dos caminhos que descrevemos neste artigo, o caminho de software ou o caminho de hardware. Há um caminho de software otimizado com virtio e vhost, e um caminho otimizado assistido por hardware que usa a passagem de PCI e SR-IOV.
