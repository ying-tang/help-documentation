      Como posso redimensionar minhas instâncias?

      Recurso: Redimensionamento de instâncias (para cima)

      Descrição: Começando com a liberação 1.3 do Bluemix Private Cloud, nós suportamos o redimensionamento de instâncias (para cima). A operação de redimensionamento não é em tempo real: ela para e reinicia a instância. Um redimensionamento pode acontecer no mesmo host, o que é útil em ambientes pequenos.

      O redimensionamento de instâncias permite que uma instância mude seu tipo para um tipo maior. Uma instância pode ser redimensionada para um tipo maior apenas. Essa mudança incorre em uma pequena interrupção enquanto a instância é parada, migrada para um tipo de tamanho maior e iniciada novamente. Esse redimensionamento também pode realocar a instância para um novo hypervisor.

      NÃO atualize o tipo de destino (para o qual a instância será redimensionada) enquanto o processo de redimensionamento estiver em andamento, caso contrário a operação de redimensionamento poderá falhar e a instância poderá ser colocada em um estado de ERRO.

      Redimensionar instâncias grandes pode ser difícil, pois pode causar falha se não houver espaço suficiente em disco para a nova imagem do disco.

      Nota: Pode haver um pequeno atraso até que a rede seja retomada depois que uma instância tiver sido redimensionada ou migrada e estiver no estado VERIFY_RESIZE.

      Para obter mais informações, consulte http://docs.openstack.org/user-guide/enduser/cli_change_the_size_of_your_server.html para obter a documentação sobre como executar um redimensionamento.

      Resolução de problemas

      Se sua operação de redimensionamento demorar muito para instâncias grandes (por exemplo, alguns logs têm mostrado uma operação começando às 09h30 e o erro de autenticação aparecendo às 14h41), o token de autenticação usado pelo Nova para concluir essa operação pode realmente expirar durante esse período.

      É muito provável que, se você tentar iniciar um novo redimensionamento, o mesmo erro ocorra novamente. Nesse caso, sugerimos uma abordagem diferente. Se seu cluster tiver o armazenamento de bloco ativado, sugerimos as seguintes etapas:

      1. Encerre a MV de destino.
      2. Faça uma captura instantânea da MV de destino com nova image-create.
      3. Faça um volume a partir dessa captura instantânea usando cinder create --image-id <SNAPSHOT UUID>.
      4. Exclua a captura instantânea com image delete ou glance image-delete do OpenStack.
      5. Inicialize uma nova MV do volume com nova boot --boot-volume.

      Se desejar reter os mesmos endereços IP, é possível:

      1. Obter o UUID da porta do volume que inicializou a MV usando: neutron port-list | grep <IP of new VM> (a primeira coluna é o UUID da porta).

      2. Remova a porta da antiga MV usando nova interface-detach

      3. Atualize a porta na nova MV para usar o IP da antiga MV com: neutron port-update --fixed-ip ip_address=<old VM IP> <port UUID>

      Quando você estiver feliz com essa alternância, pode excluir a MV antiga com nova delete.

      Este processo é mais complicado do que você provavelmente gostaria. No entanto, alterne para o armazenamento de bloco já que o backend do guest vem com vários benefícios:

      • O redimensionamento do tipo acontecerá muito mais rapidamente porque o redimensionamento do disco é ignorado.
      • A migração para um novo hypervisor é muito rápida usando o armazenamento de bloco (o novo hypervisor simplesmente se conecta ao cluster de armazenamento de bloco).
      • Você pode ampliar o disco de sua MV rapidamente com o comando cinder extend.

      Se você não tiver o Armazenamento de bloco e quiser redimensionar para um tipo menor ou maior, poderá criar uma nova instância com o tipo de tamanho que desejar e ressincronizar os dados necessários da instância antiga para a nova.

      Por que o OpenStack não permite redimensionar uma instância para baixo?

      Você pode ver o motivo em https://blueprints.launchpad.net/nova/+spec/no-downward-resize - Essencialmente, permitir uma operação de redimensionamento para baixo pode impactar a integridade dos dados e, portanto, não é permitido. Como uma solução alternativa segura, é possível ainda criar uma nova MV com o tipo de tamanho menor e ressincronizar os dados necessários da instância maior para a instância menor.

      Quando eu redimensiono minha instância inicializada de um volume, o tipo muda, mas o tamanho permanece o mesmo. Por quê?

      Esse é o comportamento esperado. O tamanho do disco do tipo não é o tamanho do disco raiz ao inicializar a partir de um volume. É possível efetuar check-out desse erro inválido do OpenStack https://bugs.launchpad.net/nova/+bug/1412993 que explica isso melhor.

      Para redimensionar o disco, consulte https://docs.openstack.org/user-guide/dashboard-manage-volumes.html#edit-a-volume para redimensionar o volume. Se, por algum motivo, isso não funcionar, você também pode tentar as etapas listadas em https://mcwhirter.com.au/craige/blog/2014/Resizing_a_Root_Volume_for_an_Openstack_Instance/
