      IBM Blue Box のネットワークのユース・ケース

      • by Leslie Lundquist and Dustin Lundquist ● September 7, 2016

      • Tags:

        • neutron

        • ユース・ケース

        • ネットワーク

        • ホワイト・ペーパー

      ベストプラクティス・ホワイト・ペーパー

      著者: Leslie Lundquist、Dustin Lundquist

      2016 年 7 月 30 日

      背景: IBM Bluemix Private Cloud は、IBM クラウド・データ・センターで実行されます。各 IBM クラウド・データ・センターは、サブネットを形成して、複数のポッドが含まれるように構築されます。これらのサブネットは、世界中のすべての IBM クラウド・データ・センターで 1 つの大きな IP アドレス・スペースを共有します。

      ネットワークを別個のポッドに分割して、単一ネットワークのサイズを制限することをお勧めします。これにより、レイヤー 2 (L2) の問題の範囲が制限され、その障害ドメインが分離されます。つまり、サブネットを小さくすることで、その中断時に影響を受ける可能性があるものが減るので、診断が簡素化します。

      ただし、ポッドに基づいたこの設計のため、お客様は、ホストが別々のポッドに配置された場合に、レイヤー 3 (L3) ネットワーク接続を使用する必要があります。ポッドは、それぞれ 32 個のアドレスから成る小さなブロックに分割されます。そのため、クラウドが大きくなるにつれ、IBM クラウドでは、新たなホストを既存のポッドに常に配置できることを保証できなくなります。そのため、お客様は、標準として複数ポッドのケースを計画する必要があります。

      IBM Blue Box Cloud のお客様のユース・ケース

      このセクションでは、IBM クラウド・データ・センターの IP アドレス・スキーマのコンテキストで IBM Bluemix Private クラウドのお客様が直面したいくつかのお客様のユース・ケースを順に見ていき、IBM Bluemix Private Cloud エンジニアが実装してきた一般的なアプローチやソリューションを示します。

      1. プライベート・ネットワーク間の接続:

      この場合、お客様は、2 つのプライベート内部 OpenStack ネットワーク間で通信したいと考えます。そのため、お客様は、2 つのプライベート内部ネットワークを作成し、OpenStack ルーターを使用してそれらのネットワークを接続します。

      注釈: 2 つのお客様のネットワークが別個の OpenStack ルーターの背後にあれば、オーバーラップする IP アドレス・スペースを許可して提供しようとする際、Neutron が適しています。別個のルーターの背後にオーバーラップする IP スペースを容易に提供できるからです。ただし、ルーターが単一の場合、単一のルーターの背後に直接接続されたサブネットは、オーバーラップしてはなりません。

      次に挙げる 2 つのユース・ケース (お客様のクラウドと公衆インターネット間のインバウンド・トラフィックとアウトバウンド・トラフィック) は関連しています。恐らく、これらは、どのお客様やサイトでも、最も標準的なユース・ケースです。

      2. 公衆インターネットからの進入 (インバウンド) 接続:

      この場合、お客様は、公衆インターネットからゲスト VM にアクセスできる必要があります。ソリューションを作成するために、お客様は、IP アドレス (パブリック・ルーティング可能な IPv6 (望ましい) または浮動 IPv4) を各ゲスト VM に割り当てる必要があり、また VM への適切なインバウンド・アクセスが許可されるようにセキュリティー・グループをセットアップする必要があります。

      3. 公衆インターネットへの退出 (アウトバウンド) 接続:

      この場合、お客様は、例えば、インストール・パッケージをダウンロードするために、ゲスト VM からインターネットにアクセスできる必要があります。通常、ゲスト VM は、ポッドに関連付けられている Vyatta 上の NAT を介してインターネットへのアウトバウンド接続を取得する必要があります。

      注釈: ユース・ケース 2 および 3 は、グローバル・インターネット接続に関するものです。一般的に、この資料の別の箇所でも指摘したように、IP アドレス・スペースを結合した場合、オーバーラップが生じます。この場合、NAT を使用する必要があります。これは、望ましいソリューションではありません。あるいは、カスタマイズしたネットワーキング・ソリューションを作成することもできます。ただし、多くの場合、このようなソリューションは複雑であるため、脆弱です。また、保守および維持のための時間と労力に相当に投資する必要があります。

      4. SoftLayer サービスへのアクセス:

      場合によっては、お客様は、IBM クラウド・データ・センターのプライベート・ネットワーク上にホストされているサービスにアクセスできる必要があります。セキュリティーの観点から、プライベート・ネットワーク上の IBM クラウドにホストされたサービスにアクセスできる ゲスト VM は、厳密に必要なもののみにする必要があります。このアクセスは、ポッドに関連付けられている Vyatta 上の NAT によって提供されます。IBM クラウド・サービスは、10.0.0.0/14 ネットワーク上にホストされます。

      注釈: 次の 2 つのユース・ケース (BYOIP と DirectLink) は、本当に表裏一体です。これらは、VPN または DirectLink (専用回線または MPLS 回線) のいずれかを使用して、既存/リモートのネットワークへの接続を提供するためのメカニズムにすぎません。(MPLS は、Multi-Protocol Label Switching の略語であり、エンタープライズのお客様によってよく使用される VPN の 1 つのタイプです)。そのため、実際は VPN です。

      5. BYOIP 接続:

      この場合、お客様は、独自のネットワーク・サブネットを IBM Bluemix Private クラウドに持ち込み、オンプレミスのデータ・センターから接続したいと考えています。ソリューションを作成するために、お客様によって提供されたサブネット値を使用して、オーバーレイ OpenStack ネットワークを作成できます。こうしたオーバーレイ・ネットワークは、SNAT を無効にした「default」OpenStack ルーターに接続されます。このようなお客様のサブネットごとに、関連付けられている Vyatta で経路がセットアップされた状態にします。このソリューションでは、IBM Bluemix Private クラウドとお客様のデータ・センター間のトンネリングが必要です。

      6. DirectLink 接続:

      お客様によって選択された DirectLink オプションに基づいて、IBM Bluemix Private クラウド内に適切なネットワークをセットアップできます。最も一般的なユース・ケースとしては、ネットワークを別の BYOIP ネットワークとしてセットアップします。ただし、この場合、DirectLink 用に別のトンネリング・エンドポイントを使用する必要があります。

      注釈: この場合、お客様は、既存の IP アドレス・プランおよびネットワーク・プランを IBM Bluemix Private クラウドに拡張したいと考えています。これは (一般的な意味での) ハイブリッドです。「ハイブリッド・クラウド」として IBM が販売しているものとは異なります。「ハイブリッド・クラウド」は、ほとんどの人にとって、パブリック・クラウドとプライベート・クラウド間に接続を作成することを意味するようになっています。

      7. SoftLayer プライベート・ネットワーク接続:

      この場合、お客様は、2 つの別個の IBM Bluemix アカウントで実行されている 2 つのクラウド間に接続を作成する必要があります。1 つは IBM Bluemix Private Cloud (IBM Bluemix データ・センター内) アカウントで、もう 1 つはお客様の独自の SoftLayer アカウントです。この状況では、2 つの (何もしないと無関係の) SoftLayer アカウント間で VLAN スパンニングを有効にする必要があります。

      • a. このお客様が 両方のクラウドで IBM クラウド・データ・センターのルーティング可能な 10.x.x.x サブネットを使用している場合、これらの 2 つのアカウント間の VLAN スパンニングは既に有効になっていると想定すると、アクションは不要です。
      • b. このお客様が BYOIP ネットワークを使用している場合、IBM Bluemix Private Cloud と、別の SoftLayer アカウントで実行されているお客様のクラウド間にトンネリングが必要になります。

      注釈: ケース (a.) の場合、アカウント横断の VLAN スパンニングは、IBM クラウド・データ・センターの機能であり、L2 ではなく L3 で動作するため、本来は「アカウント横断の VLAN 間ルーティング」と呼ぶべきです。このケースは、単に SoftLayer の既存の IP アドレス・プラン内に 2 つの IBM クラウド・アカウントをスパンニングすることを意味します。ケース (b.) では、IBM クラウド・エンジニアは、機能するトンネルのタイプを判別して説明する必要があります。

      問題およびソリューションの記述

      (この資料で説明しているように) これらの IPv4 ユース・ケースに適用する必要があるソリューションは複雑であるため、大規模専用クラウド用にアドレスを割り当てる場合に必要になるサブネットなど、大きなサブネットを割り振る必要がある場合には、お客様が IPv6 アドレスの採用を真剣に検討することをお勧めします。IPv6 への移行は、可能な場合は常に妥当です。現在、ベスト・プラクティスとして、お客様が今後 5 年の間における IPv6 への完全移行を検討することをお勧めしています。

      まとめ

      この資料での IPv6 アドレス使用の推奨に従えば、全員が世界中のグローバル IP プールから作業できるようになります。これにより、お客様のユース・ケースで説明している問題のすべてと言わないまでも、多くが解決します。現在、こうした問題は、仮想ネットワーキング・スキーマ (本資料で説明しているものなど) によって解決する必要があります。そうした実装の複雑さが対応可能なレベルにとどまっているのは、主に、IBM エンジニアリング・チームが、関与する問題およびトレードオフについて熟知しているからです。

      グローバル IP アドレスのルーティング・テーブルは、使用可能な最大の IP アドレス・プランであり、それらのすべてを一元化したものです。このプランから作業し、ルールに従った場合、アドレスは固有のものになります。そのため、IPv6 への移行を推奨しています。 そのようなときが来るまでは、この資料で説明しているユース・ケースから、IPv4 アドレスのグローバル制限および必要な IBM クラウド・データ・センターのポッド構造を克服するために IBM Bluemix Private Cloud エンジニアが作成した、機能するソリューションを探します。

      • 用語の注: ルーティング・テーブルとは、ネットワーク内の特定のポイントからの IP アドレス・スペースのビューです。
