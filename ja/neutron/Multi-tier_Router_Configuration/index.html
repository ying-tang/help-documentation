      IBM Bluemix Private Cloud 内の多層ルーター構成

      • by Ulysses Kanigel and Leslie Lundquist ● August 31, 2016

      • Tags:

        • neutron

        • ネットワーキング

        • ルーター

        • 構成

        • 多層

        • add_interface

      この記事では、多層ルーター構成のテスト方法、およびこの構成が必要になる理由について説明します。

      以下のいずれかのユース・ケースで、ご使用のクラウドに多層ルーター構成をセットアップすることがあります。

      1. 当該 (子) サブネット上のホストで浮動 IP が使用されないようにする。
      2. メイン・ルーターの下で独自の (子) ルーターをそれぞれのネットワークに割り当てて、クラウド内の他のプロジェクトまたはテナント・ネットワークからの当該 (子) ネットワークのネットワーク分離を作成する。
      3. クラウド内の他のネットワークから分離された状態を保ち、浮動 IP も使用せずに、分離された当該プロジェクト・ネットワークのそれぞれにインターネットの入口と出口がある必要がある。

      最初に多層ネットワークを確立する方法としては、メイン・ルーターから各「子」ルーターへの静的経路を確立します。このテストの主な考え方は、確実にこのような静的経路を永続的なものにするということです。この例では、このタイプの 2 つの子ネットワーク child-net1 および child-net2 をセットアップします。

      多層ルーター構成をテストし、セットアップした経路がフェイルオーバー後にも永続的であることを確認するものとします。

      以下に、テストを実行するために行う必要があるステップを示します。

      
# ネットワークおよびサブネットを作成

neutron net-create parent-net

neutron net-create child-net1

neutron net-create child-net2

neutron subnet-create parent-subnet 162.16.0.0/24

neutron subnet-create child-subnet1 162.16.1.0/24

neutron subnet-create child-subnet2 162.16.2.0/24

# ルーターを作成

neutron router-create parent-router

neutron router-create child-router1

neutron router-create child-router2

# 外部ネットワークへの親ルーターのゲートウェイを設定し、parent-router サブネットでインターフェースを追加

neutron router-gateway-set parent-router external

neutron router-interface-add parent-subnet b490f691-364f-4f7c-866c-1c45e7b8f8cf  #サブネットの固有 ID

# parent-router、child-router1、および child-router2 サブネットのインターフェースを追加

neutron port-create parent-router

neutron port-create child-router1

neutron port-create child-router2

neutron router-interface-add child-subnet1 port=c1b305a1-f634-4828-a6ce-d5a6a058824e  #ポートの固有 ID

neutron router-interface-add child-subnet1 2476ad51-e2cb-4d4e-bbd8-db18cc06ea88   #サブネットの固有 ID

neutron router-interface-add child-subnet2 port=c1b467b1-f444-4325-a5dc-d4b3a04624d   #ポートの固有 ID

neutron router-interface-add child-subnet2 2378ad51-e4cc-2d3e-bcc5-db34cd05fa48    #サブネットの固有 ID

# 子ルーターのデフォルト経路を parent-router サブネット・ゲートウェイ・アドレスに設定: 両方が親ルーターに通信

neutron router-update child-router1 --routes type=dict list=true destination=0.0.0.0/0,nexthop=162.16.0.1

neutron router-update child-router2 --routes type=dict list=true destination=0.0.0.0/0,nexthop=162.16.0.1

      ルーティングが子ルーターで行われるため、この設計を拡張して、各子ルーターに接続された複数のサブネットを含めても、問題は生じないはずです。

      注: 当該子ルーターが外部ネットワークに直接接続されていないため、子ルーターに接続されたインスタンスに浮動 IP を関連付けることはできないので注意してください。

      トラブルシューティング: Error: Failed to add_interface:

      tenant-B-network の tenant-A-router にインターフェースを接続しようとした場合に以下のエラーが表示されることがあります。 Error: Failed to add_interface: Request Failed: internal server error while processing your request.

      Neutron ポリシーでは実際、admin、cloud_admin、または project_admin の役割のユーザーにルーター・インターフェースを追加することは許可されます。しかし、以下のコードに示されているように、admin 役割を持っていない限り、認証されているテナント以外のテナントによって所有されているリソースに対する操作を許可しないように Neutron API が明示的にコーディングされているため、失敗します。

      https://github.com/openstack/neutron/blob/stable/mitaka/neutron/api/v2/attributes.py#L971

      別のテナント内のルーターに接続されているテナントからのネットワークが必要な場合は、サポート・チームがお客様に代わってその操作を実行する必要があります。実行しようとしているが、エラーがスローされる CLI コマンドを提供してください。お客様に代わって IBM で実行いたします。
