      K2K フェデレーションについて

      Keystone to Keystone (K2K) フェデレーションは、1 つの Keystone インスタンスからの資格情報を使用して複数の IBM Bluemix Private Cloud インスタンスにログインする方法です。1 つの Bluemix Private Cloud インスタンスが ID プロバイダーに指定され、その他のクラウド・インスタンスはサービス・プロバイダーに指定されます。ID プロバイダーは、ユーザーの資格情報を保管し、ユーザーが実際に有効なユーザーであることをサービス・プロバイダーに対してアサートします。ユーザーの ID とパスワードはサービス・プロバイダーには送信されず、ID プロバイダーからのアサーションのみが送信されます。

      K2K の仕組み

      ユーザーは ID プロバイダーにサインインし、ID プロバイダーから署名済みの SAML アサーションを受け取ります。その後、ユーザーはその SAML アサーションを使用してサービス・プロバイダーで認証されます。SAML フローを処理するためのログイン・プロセスは、K2K 認証プラグイン (keystoneauth1 にあります) により処理されます。資格情報は ID プロバイダーでのみ管理され、サービス・プロバイダーはアサーションを通じてこれらの資格情報を受け入れます。

      複数の Bluemix Private Cloud に対して K2K 機能がセットアップされている場合、1 つのクラウド・インスタンスが ID プロバイダーに指定され、その他のクラウド・インスタンスはサービス・プロバイダーに指定されます。サービス・プロバイダーはマッピングを使用して、ID プロバイダー提供のユーザー属性を選択します。これらのユーザー属性は、ローカル・グループにマップするために使用されます。K2K フェデレーションを使用してユーザーがログインすると、グループに割り当てられた役割がユーザーに付与されます。

      デフォルトでは、mapping-for-k2k-federation という名前のマッピングが ID プロバイダーの cloud_admin 資格情報に対して作成されます。このマッピングによって、これらの資格情報は cloud_admin という名前のローカル・グループにマップされます。このグループには、デモ・プロジェクトにアクセスできる cloud_admin 役割があります。

      K2K フェデレーションをセットアップするためのフローを以下に示します。

      1. まず、サービス・プロバイダーでグループを管理します (フェデレーテッド・ユーザーには、グループと同じアクセス権限が付与されます)。

      2. 次に、サービス・プロバイダーでマッピングを管理します (ID プロバイダーのリモート・ユーザーをサービス・プロバイダーのローカル・グループにマップします)。

      グループの管理

      K2K フェデレーションは、ID プロバイダーのユーザーをサービス・プロバイダーのグループに統合することで機能します。プロジェクトで対象のグループに役割を割り当てると、そのグループにアクセス権が付与されます。使用できる役割は以下のとおりです。

      • project_admin - プロジェクトの admin 役割。
      • cloud_admin - クラウド・インスタンスの admin 役割。
      • _member_ - プロジェクトのメンバー。
      • heat_stack_owner - heat スタックの所有者。

      グループ・コマンドを使用するには、資格情報のソース・ファイルが必要です。資格情報を含むファイルを作成し、不等号括弧 < > 内のタグ情報を適切な情報に置き換える必要があります。

      #This file is cloud_adminrc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=<cloud_admin password> #Change this
export OS_AUTH_URL=https://<service provider fqdn>:5000/v3
export OS_USERNAME=cloud_admin
export OS_TENANT_NAME=demo
export OS_CACERT=/opt/stack/ssl/openstack.crt
export OS_NO_CACHE=True

      資格情報のソース・ファイルを作成した後、そのファイルをソーシングし、以下の OpenStack コマンドを使用してグループおよび役割を管理できます。注: これらのコマンドは、サービス・プロバイダーで実行されます。

      資格情報を環境にソーシングする:

      $ source cloud_adminrc

      グループをリストする:

      $ openstack group list

      グループを作成する:

      $ openstack group create --help
$ openstack group create <new group name here>

      役割をリストする:

      $ openstack role list

      役割の割り当てをリストする:

      $ openstack role assignment list --name

      リストをリストする:

      $ openstack project list

      役割の割り当てを追加する場合のヘルプを参照する:

      $ openstack role add --help

      グループに役割の割り当てを追加し、プロジェクトでの役割を持たせる:

      $ openstack role  add <role> --project <project> --group <group>

      フェデレーテッド・ユーザーは、サービス・プロバイダーのグループにマップされます。K2K フェデレーションが有効である場合、cloud_admin という名前のグループがデフォルトで存在します。このグループには、”demo” プロジェクトに対する cloud_admin 役割があります。

      グループにアクセス権限を付与するための操作手順は、以下のとおりです。

      1. まず、グループを作成または編集します。
      2. 次に、プロジェクトに対するグループの役割の割り当てを作成または編集します。

      例 1 cloud_admin グループの作成。

      以下のコマンドは、サービス・プロバイダーの cloud_admin ユーザーの認証情報を含むファイルをソーシングする方法を示しています。

      $ source cloud_adminrc
$ openstack group list
$ openstack group create cloud_admin --or-show
$ openstack role assignment list --name --group cloud_admin

      次に、まだない場合は、役割の割り当てを作成します。 これらのコマンドは、”demo” プロジェクトのグループ cloud_admin に役割 cloud_admin を追加します。

      $ openstack role add cloud_admin --group cloud_admin --project demo
$ openstack role assignment list --name --group cloud_admin

      openstack role assignment list コマンドを実行した後、以下のような出力が表示されます。

      # +-------------+------+---------------------+--------------+--------+-----------+
# | Role        | User | Group               | Project      | Domain | Inherited |
# +-------------+------+---------------------+--------------+--------+-----------+
# | cloud_admin |      | cloud_admin@Default | demo@Default |        | False     |
# +-------------+------+---------------------+--------------+--------+-----------+

      例 2 メンバー・アクセス権限を持つグループを作成します。

      以下のコマンドは、サービス・プロバイダーの cloud_admin ユーザーの認証情報を含むファイルをソーシングする方法を示しています。

      $ source cloud_adminrc
$ openstack group list
$ openstack group create demo_member_group --or-show
$ openstack role assignment list --name --group demo_member_group
$ openstack role add _member_ --group demo_member_group --project demo
$ openstack role assignment list --name --group demo_member_group

      openstack role assignment list コマンドを入力した後、以下のような出力が表示されます。

      # +----------+------+---------------------------+--------------+--------+-----------+
# | Role     | User | Group                     | Project      | Domain | Inherited |
# +----------+------+---------------------------+--------------+--------+-----------+
# | _member_ |      | demo_member_group@Default | demo@Default |        | False     |
# +----------+------+---------------------------+--------------+--------+-----------+

      マッピングの管理

      サービス・プロバイダーの cloud_admin ユーザーには、K2K フェデレーション用のマッピングを読み取り、設定するためのアクセス権限が付与されています。サービス・プロバイダーの cloud_admin ユーザーは、既存のマッピングの削除および新しいマッピングの作成はできないので注意してください。

      マッピング・コマンドを使用するには、v3 資格情報ソース・ファイルが必要です。資格情報を含むファイルを作成し、不等号括弧 < > 内のタグ情報を適切な情報に置き換えます。

      マッピングの管理の詳細については、次の OpenStack の資料を参照してください。 http://docs.openstack.org/developer/keystone/mapping_combinations.html http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combinations

      ユーザーが、ID プロバイダーのアカウントを使用して、K2K フェデレーションを通じてサービス・プロバイダーにログオンすると、サービス・プロバイダーでユーザー・アカウントがローカルに作成されます。フェデレーション時に作成されるユーザー・アカウントは一時アカウントなので注意してください。このアカウントは、警告なしで後日削除される場合があります。

      デフォルトのマッピングでは、ID プロバイダーの cloud_admin ユーザーが統合されるため、フェデレーテッド・ドメインでローカルの cloud_admin ユーザーが作成されます。このユーザーは、「デフォルト」ドメインの cloud_admin ユーザーとは異なります。フェデレーテッド・ドメインの一時 cloud_admin ユーザーは、後日削除される場合があります。

      次のマッピング・コマンドを実行するには、前のセクションで作成したソース・ファイルが必要です。

      $ source cloud_adminrc
$ openstack mapping list
$ openstack mapping show mapping-for-k2k-federation
$ openstack mapping show mapping-for-k2k-federation --format json
$ openstack mapping show --help
$ openstack mapping set mapping-for-k2k-federation --rules <mappings file>

      openstack mapping list コマンドは、マッピングのリストを表示します。mapping-for-k2k-federation マッピングを表示します。

      openstack mapping show mapping-for-k2k-federation コマンドは指定されたマッピングの詳細を表示します。マッピングのルールの json リストが組み込まれている必要があります。

      openstack mapping set mapping-for-k2k-federation --rules newmappings.json コマンドは、mapping-for-k2k-federation マッピングを更新および設定する方法を表示します。

      マッピング・ファイルは、以下に示すフォーマットに従う必要があります。

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      ルール・ファイルには、リモート・ユーザーをサービス・プロバイダーに存在するグループにマップするルールのリストが含まれます。ローカル・セクションには、リモート・ユーザーをマップするグループを指定するルールを組み込みます。ローカル・グループ・ルールは必須です。

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "cloud_admin"
  }
}

      リモート・セクションに、アクセスを許可すべきリモート・ユーザーを指定するルールを組み込みます。

      {
    "type": "openstack_user",
    "any_one_of": [
        "cloud_admin"]
}

      リモート・ルールに指定することが可能なユーザーの属性タイプは、以下のとおりです。

      openstack_user
openstack_user_domain
openstack_roles
openstack_project
openstack_project_domain

      リモート属性でフィルタリングが可能なアクションは、以下のとおりです。

      empty
any_ony_of
not_any_of
blacklist
whitelist

      username を設定するには、ローカル・セクションで {0} を使用して、リモート・セクションからフィールドを使用することを示すことができます。その後、新しいフェデレーテッド・ユーザー名に情報を追加できます。K2K フェデレーションでは、どのクラウドを使用しているのか示すための情報を追加することが重要です。インデックスにより、ローカル・セクションでリモート・セクションを参照できます (「{i}」。i はリモート・フィールドのインデックスです)。

      [
  {
    "local": [
      {
        "user": {
          "name": "my_cloud/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
               ...
    ]
  }
]

      Horizon の使用

      ID プロバイダーの OpenStack ダッシュボードには、「Keystone to Keystone フェデレーションを使用して認証 (Authenticate with Keystone to Keystone Federation)」というラベルのドロップダウン・メニューがあります。

      • 「ID プロバイダー」オプションを選択すると、フェデレーションは使用されません。ユーザーは、ID プロバイダーの OpenStack クラウドにサインインします。
      • 「サービス・プロバイダー」オプションを選択すると、フェデレーションが使用されます。ID プロバイダーのユーザーの資格情報がサービス・プロバイダーに統合されます。
      • プロバイダーを切り替える場合、ユーザーはログアウトし、プロバイダーを再選択する必要があります。

      注意事項: IBM Bluemix Private Cloud 3.0.0 の K2K 用の Horizon ログイン画面は、標準の OpenStack 機能ではありません。これは、今後のリリースで変更される可能性があります。

      Python API ライブラリーの使用

      次に、K2K 認証プラグインを使用した認証に使用するソース・ファイルを作成します。トークン OS_SERVICE_PROVIDER はサービス・プロバイダー ID を示します。これは、ID プロバイダーに対して次のコマンドを実行することで取得できます。

      $ openstack server provider list

      ここで、ID プロバイダー (IdP) からの cloud_admin 資格情報を使用して、サービス・プロバイダー (Sp) のインスタンスをリストするための Python スクリプトで使用されるソース・ファイルを作成できます。

      このソース・ファイル (federationrc) には、Idp にログインするための認証資格情報と、Sp ターゲット情報が含まれます。

      # This file is federationrc
export OS_USERNAME=cloud_admin
export OS_PASSWORD=<cloud admin password>
export OS_AUTH_URL=https://k2kf-idp.open-test.ibmcloud.com:5000/v3
export OS_PROJECT_NAME=demo
export OS_DOMAIN_NAME=Default
export OS_SERVICE_PROVIDER=<service provider id>

      ID プロバイダーで次のコマンドを実行することで、サービス・プロバイダーのリストを表示できます。

      $ openstack service provider list

      次のコードでは、Keystone-to-Keystone auth プラグインを使用して、スコープなしトークンを取得します。スコープなしトークンは、スコープ付きトークンを取得するために使用されます。スコープ付きトークンは、フェデレーテッド・ユーザーがアクセス権限を持ついずれかのプロジェクトにスコープされます。スコープ付きトークンを取得すると、サービス・プロバイダーのサーバー・インスタンスをリストできます。

      list_servers.py という名前のファイルを作成します。

      以下を行うことで、Python スクリプトを実行できます。

      $ source federationrc
$ chmod 700 list_servers.py
$ ./list_servers.py

      その他のマッピングおよびグループ

      例 1. Heat ユーザー

      現時点で、フェデレーテッド・ユーザーおよび Keystone の委託者/受託者機能には問題があり、これにより、フェデレーテッド Heat ユーザーは heat_stack_owner 役割を委任できません。

      Heat フェデレーション・グループの作成

      サービス・プロバイダーの cloud_admin ユーザーの認証情報を含むファイルをソーシングします: source cloud_adminrc

      ここでは、heat_stack_user 役割ではなく heat_stack_owner 役割を使用します。

      $ openstack group list
$ openstack group create heat_stack_owner_group --or-show
$ openstack role assignment list --name --group heat_stack_owner_group
$ openstack role add heat_stack_owner --group heat_stack_owner_group --project demo
$ openstack role assignment list --name --group heat_stack_owner_group
# +------------------+------+--------------------------------+--------------+--------+-----------+
# | Role             | User | Group                          | Project      | Domain | Inherited |
# +------------------+------+--------------------------------+--------------+--------+-----------+
# | heat_stack_owner |      | heat_stack_owner_group@Default | demo@Default |        | False     |
# +------------------+------+--------------------------------+--------------+--------+-----------+

      Heat フェデレーション・マッピングの作成

      $ cat <<EOF > federated_heat_stack_owner_mapping.json
[
  {
    "local": [
      {
        "user": {
          "name": my_cloud/"{0}"
        }
      },
      {
        "group": {
          "domain": {
            "name": "Default"
          },
          "name": "heat_stack_owner_group"
        }
      }
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
      {
        "type": "openstack_roles",
        "any_one_of": [
          "heat_stack_owner"
        ]
      }
    ]
  }
]
EOF
$ openstack mapping set mapping-for-k2k-federation --rules federated_heat_stack_owner_mapping.json

      ユーザーに heat_stack_owner 役割を直接割り当てることにより、信頼する代行に対する予備手段が必要になります。これは、フェデレーテッド・ログイン時に作成されるユーザーです。ユーザーのドメインは統合されており、値は None (Federation=None) である可能性があります。

      $ openstack user list

      フェデレーテッド・ユーザーはドメインなしにする必要があります: openstack user show <federated user> ここでユーザー ID が必要です。

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      設定ページでパスワードを変更するオプションがフェデレーテッド・ユーザーに提示されるという既知の問題があります。これらのユーザーは、自身のパスワードを変更することはできません。
