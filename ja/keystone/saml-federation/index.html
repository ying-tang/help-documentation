      SAML フェデレーションについて

      Security Assertion Markup Language (SAML) は、ID プロバイダー (IDP) およびサービス・プロバイダー (SP) でユーザーがシングル・サインオン (SSO) を使用できるようにするプロトコルです。ユーザーは、ログイン時に資格情報を IDP に提供します。IDP はそれらの資格情報をチェックし、ユーザーの属性を含む SAML アサーションが作成されます。このアサーションは SP に渡され、SP はそのアサーションが IDP から送信されたものであるか検証します。SP は IDP からのアサーションを信頼しているため、ユーザーは SP に資格情報を送信する必要ありません。

      Keystone は SP として動作し、IDP からのアサーションを受け取ることでユーザーを認証します。本資料では、例として XYZ 社の IDP という SAML IDP を使用します。

      ** このページの管理コマンドを使用する場合、cloud_admin アクセス権限が必要になるので注意してください。**

      IDP と Keystone の統合

      Bluemix Private Cloud オペレーターは、XYZ 社の IDP の統合エンジニアと協力して、両方のシステムが連携して動作するようセットアップする必要があります。

      Bluemix Private Cloud オペレーターには、以下が必要になります。

      • IDP メタデータの URL エンドポイント (現在、ファイルではなく、URL を通じたメタデータの取得のみがサポートされています)
      • IDP メタデータからの IDP エンティティー ID
      • ユーザー属性 - これは IDP によって渡される SAML ユーザー属性です。IDP の資料または IDP サーバー側のログで確認できます。
      • デフォルト/初期マッピング - マッピングに関する資料は、http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination で参照できます。

      XYZ 社の IDP の統合エンジニアは、Bluemix Private Cloud オペレーターから以下を入手する必要があります。

      • サービス・プロバイダーのメタデータの URL
      • サービス・プロバイダーのリダイレクト URL (コールバック URL とも呼ばれます。これは、IDP がリダイレクトできる先のホワイトリスト値です)。

      Bluemix Private Cloud オペレーターは、統合エンジニアと協力して初期/デフォルト・マッピングを作成する必要があります。マッピングにより、IDP のリモート・ユーザーが SP のローカル・グループにマップされます。本資料の残りの部分では、マッピングがどのようなもので、どのように使用されるのか分かりやすくご説明します。統合エンジニアには cloud_admin アクセス権限も必要です。これは、cloud_admin ユーザー、または同名の役割を持つユーザーである場合があります。

      ユーザー属性には、マッピング名を指定する必要があります。例えば、属性に以下のような名前を付けます。

      attributes:
- name: http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress
  id: email
- name: http://schemas.xmlsoap.org/claims/Group
  id: group

      指定された名前は、SAML アサーション内の名前を示します。id は、Keystone マッピングで使用される id です。

      Bluemix Private Cloud オペレーターが XYZ 社の IDP と Keystone の統合を完了した後、mapping-for-company-xyz-idp という名前 (または IDP 名を含む同様のマッピング名) のマッピングを作成する必要があります。

      マッピングを管理する手順は以下のとおりです。

      1. グループを管理します (フェデレーテッド・ユーザーには、グループと同じアクセス権限が付与されます)。 2. マッピングを管理します。

      グループの管理

      Keystone フェデレーションは、ID プロバイダーのユーザーをサービス・プロバイダーのグループに統合することで機能します。プロジェクトで対象のグループに役割を割り当てると、そのグループにアクセス権が付与されます。使用できる役割は以下のとおりです。

      • project_admin - プロジェクトの admin 役割。
      • cloud_admin - クラウド・インスタンスの admin 役割。
      • _member_ - プロジェクトのメンバー。
      • heat_stack_owner - heat スタックの所有者。

      グループをリストする:

      $ openstack group list

      グループを作成する:

      $ openstack group create --help
$ openstack group create <new group name here>

      役割をリストする:

      $ openstack role list

      役割の割り当てをリストする:

      $ openstack role assignment list --name

      リストをリストする:

      $ openstack project list

      役割の割り当てを追加する場合のヘルプを参照する:

      $ openstack role add --help

      グループに役割の割り当てを追加し、プロジェクトでの役割を持たせる:

      $ openstack role  add <role> --project <project> --group <group>

      フェデレーテッド・ユーザーは、サービス・プロバイダーのグループにマップされます。

      グループにアクセス権限を付与するための操作手順は、以下のとおりです。

      1. まず、グループを作成または編集します。 2. 次に、プロジェクトに対するグループの役割の割り当てを作成または編集します。

      例 1 company_xyz_admin_group グループの作成。

      $ source cloud_adminrc # Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      次に、まだない場合は、役割の割り当てを作成します。 これらのコマンドは、”demo” プロジェクトのグループ cloud_admin に役割 cloud_admin を追加します。

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      openstack role assignment list コマンドを実行した後、以下のような出力が表示されます。

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      例 2 メンバー・アクセス権限を持つグループを作成します。

      $ source cloud_adminrc #Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      openstack role assignment list コマンドを入力した後、以下のような出力が表示されます。

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      マッピングの管理

      以下では、XYZ 社の IDP を使用しています。この IDP のマッピングは既に存在しています。 cloud_admin アクセス権限を持つユーザー (ユーザーまたは役割) は、マッピングを編集できます。ユーザーはマッピングの作成や削除を行えません。

      マッピングは、Horizon で「ID ダッシュボード (Identity Dashboard)」->「フェデレーション (Federation)」->「マッピング (Mappings)」->「編集」を選択することで管理できます。

      openstack mapping list コマンドは、マッピングのリストを表示します。mapping-for-company-xyz-idp マッピングを表示します。

      openstack mapping show mapping-for-company-xyz-idp -f json コマンドは指定されたマッピングの詳細を表示します。マッピングのルールの json リストが組み込まれている必要があります。

      openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json コマンドはマッピングを更新して設定します。

      マッピング・ファイルは、以下に示すフォーマットに従う必要があります。

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      ルール・ファイルにはリモート・ユーザーをローカル・グループにマップするルールのリストが含まれます。ローカル・セクションには、リモート・ユーザーをマップするグループを指定するルールを組み込みます。ローカル・グループは必須です。

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      リモート・セクションに、アクセスを許可すべきリモート・ユーザーを指定するルールを組み込みます。

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      リモート・ルールに指定することが可能なユーザーの属性タイプは、以下のとおりです。

      リモート属性でフィルタリングが可能なアクションは、以下のとおりです。

      empty
any_ony_of
not_any_of
blacklist
whitelist

      username を設定するには、ローカル・セクションで {0} を使用して、リモート・セクションからフィールドを使用することを示すことができます。その後、新しいフェデレーテッド・ユーザー名に情報を追加できます。インデックスにより、ローカル・セクションでリモート・セクションを参照できます (「{i}」。i はリモート・フィールドのインデックスです)。

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "email"
      },
               ...
    ]
  }
]

      以下にサンプル・シナリオをいくつか示します。

      例 1. ここでは、XYZ 社の IDP のユーザーをマッピングし、admin_group のユーザーを許可します。ユーザーの E メール・アドレスは Keystone でユーザーを作成するために使用されます。ユーザーの E メールが test@test.ibmcloud.com である場合、結果として username は CompanyXYZ/test@test.ibmcloud.com となります。

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
      {
        "group": {
          "domain": {
            "name": "Default"
          },
          "name": "company_xyz_admin_group"
        }
      }
    ],
    "remote": [
      {
        "type": "email"
      },
      {
        "type": "group",
        "any_one_of": [
          "admin_group"
        ]
      }
    ]
  }
]

      例 2. 複数のグループ

      この例では、以下を使用します。

      Horizon の使用

      ID プロバイダーの OpenStack ダッシュボードに、ラベルが「SSO での認証 (Authenticate with SSO)」のドロップダウン・リストがあります。

      • このドロップダウンに、XYZ 社の IDP のラベルが表示されます。この IDP を選択し、「接続」をクリックします。
      • その後、IDP ログイン・ページにリダイレクトされます。
      • IDP で認証が成功すると、Horizon ページにリダイレクトされます。

      CLI の使用

      注: コマンド・ライン・インターフェース (CLI) クライアントは、次の条件を満たす IDP でのみ動作します。 1. IDP が SAML ECP プロトコルをサポートしている。 2. IDP が GET での基本認証をサポートしている。

      CLI は、次の 2 種類のプラグインを提供しています。 1. v3unscopedsaml (スコープなしトークンを取得します) 2. v3scopedsaml (トークンを使用してスコープ付きアクションを実行します)

      1. スコープなしの例:

      SAML を使用するために環境にソーシングできるファイルを作成します。

      # This file is called companyxyz_saml_rc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=CompanyXYZPassword #replace this
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 #replace this
export OS_USERNAME=CompanyXYZUser #replace this
export OS_IDENTITY_PROVIDER=company-xyz-idp #replace this
export OS_PROTOCOL=saml2
export OS_AUTH_TYPE=v3unscopedsaml
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP #replace this

      ファイルを作成しオプションを編集した後、次のコマンドを実行してスコープなしトークンを取得し、ID を記録できます。

      $ source companyxyz_saml_rc
$ openstack token issue

      2. スコープ付きの例:

      スコープなしトークンを取得したので、プロジェクトにスコープを適用して、そのプロジェクトのサーバーをリストできます。

      openstack server list --os-auth-type v3scopedsaml --os-token <os-token-id> --os-project-id <project-id>

      Python API ライブラリーの使用

      注: Python API ライブラリーは、次の条件を満たす IDP でのみ動作します。 1. IDP が SAML ECP プロトコルをサポートしている。 2. IDP が GET での基本認証をサポートしている。

      次のコードでは、SAML auth プラグインを使用して IDP で認証を受け、SAML アサーションを Keystone に送信し、スコープなしトークンを取得します。このスコープなしトークンを使用して、プロジェクトのスコープ付きトークンを取得します。その後、スクリプトにより、そのプロジェクトのサーバーがリストされます。

      このファイルは saml_list_servers.py です。

      スクリプトを実行する前にファイルをソーシングする必要があります。

      # This file is called companyxyz_saml_python_rc
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 #replace this
export OS_USERNAME=CompanyXYZUser #replace this
export OS_PASSWORD=CompanyXYZPassword #replace this
export OS_PROJECT_NAME=demo
export OS_IDENTITY_PROVIDER=company-xyz-idp #replace this
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP #replace this
export OS_PROTOCOL=saml2

      次に、Python スクリプトを実行できます。

      source companyxyz_saml_python_rc
python saml_list_servers.py

      その他の注記

      現時点で、フェデレーテッド・ユーザーおよび Keystone の委託者/受託者機能には問題があり、これにより、フェデレーテッド Heat ユーザーは heat_stack_owner 役割を委任できません。

      ユーザーに heat_stack_owner 役割を直接割り当てることにより、信頼する代行に対する予備手段が必要になります。これは、フェデレーテッド・ログイン時に作成されるユーザーです。ユーザーのドメインは統合されており、値は None (Federation=None) である可能性があります。

      $ openstack user list

      フェデレーテッド・ユーザーはドメインなしにする必要があります: openstack user show <federated user> ここでユーザー ID が必要です。

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      設定ページでパスワードを変更するオプションがフェデレーテッド・ユーザーに提示されるという既知の問題があります。これらのユーザーは、自身のパスワードを変更することはできません。
