      フェデレーションについて

      Keystone フェデレーションにより、ID プロバイダー (IDP) 提供の ID をサービス・プロバイダー (SP) で使用できます。サポートされる各種プロトコルおよびフォーマットは、OpenID Connect (OIDC) および Security Assertion Markup Language (SAML) です。

      IDP はユーザーの資格情報を保管および管理し、クレームまたはアサーションを SP に送信します。これにより、お客様は、パスワードを Bluemix Private Cloud に送信せずに、企業の資格情報を使用して Bluemix Private Cloud の認証を受けることができます。Keystone は、外部リソースから ID を取り込むため、通常は SP です。

      OpenID Connect (OIDC) について

      OpenID は認証用のプロトコルです。 OAuth は許可用のプロトコルです。 OpenID Connect は両方を行います。

      OpenID Connect は、OAuth 2.0 プロトコルの上部にある単純な ID 層です。これにより、クライアントは、許可サーバーが実行する認証に基づいて、エンド・ユーザーの身元を確認できます。また、クライアントは、相互運用可能な REST 型の方法でエンド・ユーザーに関する基本プロファイル情報を取得できます。

      本資料では、OpenID Connect リライング・パーティーが、エンド・ユーザーの OpenID プロバイダーを検出し、その OpenID プロバイダーと対話するために必要な情報 (OAuth 2.0 エンドポイントのロケーションなど) を取得するためのメカニズムについて説明します。

      OpenID プロトコルの仕組み

      1. リライング・パーティーは、OpenID プロバイダーにエンド・ユーザーを認証する要求を送信します。
      2. OpenID プロバイダーはユーザーを認証します。
      3. OpenID プロバイダーは ID トークンおよびアクセス・トークンをリライング・パーティーに送信します。
      4. リライング・パーティーは、OpenID プロバイダーから受け取ったアクセス・トークンを使用して、ユーザー情報エンドポイントに要求を送信します。
      5. ユーザー情報エンドポイントは、クレームを返します。

      OIDC のセットアップ

      Blue Box オペレーターは、お客様の会社 (例えば XYZ 社) の IDP 統合エンジニアと協力して OIDC をセットアップする必要があります。

      Bluemix Private Cloud オペレーターは、次の情報を入手する必要があります。

      • 許可されたリダイレクト/コールバック URL
      • クライアント ID とクライアント・シークレット
      • ユーザー・クレーム - OIDC クレーム
      • プロバイダーにアクセスするために必要なすべての情報を提供する既知/構成 URL
      • デフォルト/初期マッピング - マッピングに関する資料は、http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination で参照できます。

      Bluemix Private Cloud オペレーターは、統合エンジニアと協力して初期/デフォルト・マッピングを作成する必要があります。マッピングにより、IDP のリモート・ユーザーが SP のローカル・グループにマップされます。本資料の残りの部分では、マッピングがどのようなもので、どのように使用されるのか分かりやすくご説明します。

      Bluemix Private Cloud オペレーターが XYZ 社の IDP と Keystone の統合を完了した後、mapping-for-company-xyz-idp という名前 (または IDP 名を含む同様のマッピング名) のマッピングを作成する必要があります。

      マッピングを管理する手順は以下のとおりです。

      1. グループを管理します (フェデレーテッド・ユーザーには、グループと同じアクセス権限が付与されます)。

      2. マッピングを管理します。

      グループの管理

      Keystone フェデレーションは、ID プロバイダーのユーザーをサービス・プロバイダーのグループに統合することで機能します。プロジェクトで対象のグループに役割を割り当てると、そのグループにアクセス権が付与されます。使用できる役割は以下のとおりです。

      • project_admin - プロジェクトの admin 役割。
      • cloud_admin - クラウド・インスタンスの admin 役割。
      • _member_ - プロジェクトのメンバー。
      • heat_stack_owner - heat スタックの所有者。

      グループをリストする:

      $ openstack group list

      グループを作成する:

      $ openstack group create --help
$ openstack group create <new group name here>

      役割をリストする:

      $ openstack role list

      役割の割り当てをリストする:

      $ openstack role assignment list --name

      リストをリストする:

      $ openstack project list

      役割の割り当てを追加する場合のヘルプを参照する:

      $ openstack role add --help

      グループに役割の割り当てを追加し、プロジェクトでの役割を持たせる:

      $ openstack role  add <role> --project <project> --group <group>

      フェデレーテッド・ユーザーは、サービス・プロバイダーのグループにマップされます。グループにアクセス権限を付与するための操作手順は、以下のとおりです。

      1. まず、グループを作成または編集します。

      2. 次に、プロジェクトに対するグループの役割の割り当てを作成または編集します。

      例 1 company_xyz_admin_group グループの作成。

      $ source cloud_adminrc # Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      次に、まだない場合は、役割の割り当てを作成します。

      これらのコマンドは、”demo” プロジェクトのグループ cloud_admin に役割 cloud_admin を追加します。

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      openstack role assignment list コマンドを実行した後、以下のような出力が表示されます。

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      例 2 メンバー・アクセス権限を持つグループを作成します。

      $ source cloud_adminrc #Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      openstack role assignment list コマンドを入力した後、以下のような出力が表示されます。

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      マッピングの管理

      以下では、例として XYZ 社の IDP を使用し、この IDP のマッピングが既に存在していると仮定します。cloud_admin アクセス権限を持つユーザー (ユーザーまたは役割) は、マッピングを編集できます。ユーザーはマッピングの作成や削除を行えません。

      マッピングは、Horizon で「ID ダッシュボード (Identity Dashboard)」->「フェデレーション (Federation)」->「マッピング (Mappings)」->「編集」を選択することで管理できます。

      openstack mapping list コマンドは、マッピングのリストを表示します。mapping-for-company-xyz-idp マッピングを表示します。

      openstack mapping show mapping-for-company-xyz-idp -f json コマンドは指定されたマッピングの詳細を表示します。マッピングのルールの json リストが組み込まれている必要があります。

      openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json コマンドはマッピングを更新して設定します。

      マッピング・ファイルは、以下に示すフォーマットに従う必要があります。

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      ルール・ファイルにはリモート・ユーザーをローカル・グループにマップするルールのリストが含まれます。ローカル・セクションには、リモート・ユーザーをマップするグループを指定するルールを組み込みます。ローカル・グループは必須です。

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "company_xyz_admin_group"
  }
}

      リモート・セクションに、アクセスを許可すべきリモート・ユーザーを指定するルールを組み込みます。

      {
    "type": "group",
    "any_one_of": [
        "admin_group"]
}

      リモート・ルールに指定することが可能なユーザーの属性タイプは、以下のとおりです。

      リモート属性でフィルタリングが可能なアクションは、以下のとおりです。

      empty
any_ony_of
not_any_of
blacklist
whitelist

      username を設定するには、ローカル・セクションで {0} を使用して、リモート・セクションからフィールドを使用することを示すことができます。その後、新しいフェデレーテッド・ユーザー名に情報を追加できます。インデックスにより、ローカル・セクションでリモート・セクションを参照できます (「{i}」。i はリモート・フィールドのインデックスです)。

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "HTTP_OIDC_CLAIM_EMAIL"
      },
               ...
    ]
  }
]

      以下のセクションには、サンプル・シナリオがいくつか含まれます。

      例 1. E メール・アドレスを使用して Keystone でユーザー名を作成 この例では、XYZ 社の IDP のユーザーをマッピングし、admin_group のユーザーを許可します。ユーザーの E メール・アドレスは Keystone でユーザーを作成するために使用されます。ユーザーの E メールが test@test.ibmcloud.com である場合、結果として username は CompanyXYZ/test@test.ibmcloud.com となります。

      Horizon の使用

      ID プロバイダーの OpenStack ダッシュボードに、ラベルが「SSO での認証 (Authenticate with SSO)」のドロップダウン・リストがあります。

      1. ドロップダウン・メニューに IdP のラベルが表示されます。その IDP を選択し、「接続」をクリックします。

      2. その後、IDP ログイン・ページにリダイレクトされます。

      3. IDP で認証が成功すると、Horizon ページにリダイレクトされます。

      OAuth API の使用

      ユーザーの資格情報は Keystone には直接送信されず、OAuth プロバイダー・エンドポイントに送信されます。

      1. 前提条件 (request-oauthlib、Keystone クライアント、および Nova クライアント) をインストールします。

      2. OIDC stackrc ファイルをソーシングします。

      3. python ファイル名

      例 4. 許可コード・フローに対して API を使用

      このファイルは oidc-authflow.py です。

      from keystoneauth1.identity.v3 import oidc
from keystoneauth1 import session
from requests_oauthlib import OAuth2Session
from keystoneauth1.identity.v3 import Token
from novaclient import client
import os

def main():

    auth_url = os.environ.get('OS_AUTH_URL')
    identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
    protocol = 'oidc'
    redirect_uri = os.environ.get('OS_REDIRECT_URI')
    scope = ['openid','profile','email']
    client_id = os.environ.get('OS_CLIENT_ID')
    client_secret = os.environ.get('OS_CLIENT_SECRET')
    discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')
    access_token_endpoint = os.environ.get('OS_TOKEN_ENDPOINT')
    authorization_endpoint = os.environ.get('OS_AUTHORIZATION_ENDPOINT')
    project_id = os.environ.get('OS_PROJECT_ID')

    oauth = OAuth2Session(client_id, redirect_uri=redirect_uri, scope=scope)
    authorization_url, state = oauth.authorization_url(authorization_endpoint,
                                                       access_type='offline',
                                                       approval_prompt='force')
    print( 'Please go to %s and authorize access' % authorization_url)
    authorization_code = raw_input('Enter the authorization code here: ')

    s = session.Session(verify=False)

    oidc_plugin = oidc.OidcAuthorizationCode(auth_url, identity_provider,
                                             protocol, client_id=client_id,
                                             client_secret=client_secret,
                                             access_token_endpoint=access_token_endpoint,
                                             discovery_endpoint=discovery_endpoint,
                                             access_token_type='access_token',
                                             redirect_uri=redirect_uri,
                                             code=authorization_code)

    unscoped_auth_token = oidc_plugin.get_unscoped_auth_ref(s)
    scoped_token = Token(auth_url=auth_url,
                         token=unscoped_auth_token._auth_token,
                         project_id=project-id,
                         reauthenticate=False)
    federated_session = session.Session(auth=scoped_token)
    nova = client.Client(version=2, session=federated_session)
    for server in nova.servers.list():
        print server.name

if __name__ == '__main__':
    main()

      from keystoneauth1 import session
from keystoneauth1.identity.v3 import oidc
from keystoneauth1.identity.v3 import Token
from keystoneclient.v3.client import Client

from novaclient import client
import os


auth_url = os.environ.get('OS_AUTH_URL')
identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
protocol = 'oidc'
redirect_uri = os.environ.get('OS_REDIRECT_URI')
scope = ['openid','profile','email']
client_id = os.environ.get('OS_CLIENT_ID')
client_secret = os.environ.get('OS_CLIENT_SECRET')
discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')
access_token_endpoint = os.environ.get('OS_TOKEN_ENDPOINT')
authorization_endpoint = os.environ.get('OS_AUTHORIZATION_ENDPOINT')
project_id = os.environ.get('OS_PROJECT_ID')

def get_project_and_federated_session():
    s = session.Session(verify=False)

    # get an access token
    payload = {'client_id': client_id, 'grant_type': 'password',
                'username': '<username>',
                'password': '<password>', 'scope': 'openid',
                'connection':'Username-Password-Authentication'}

    response = s.post('https://<FQDN-OP>/oauth/ro',
                            data=payload,
                            authenticated=False)
    print(response.json())
    access_token = response.json()['access_token']

    oidc_plugin = oidc.OidcAccessToken(auth_url, identity_provider, protocol, access_token)
    response = oidc_plugin.get_unscoped_auth_ref(s)
    scoped_token = Token(auth_url=auth_url,
                         token=response._auth_token,
                         project_id=project_id,
                         reauthenticate=False)
    federated_session = session.Session(auth=scoped_token)
    return  federated_session


def main():
    scoped_session = get_project_and_federated_session()
    nova = client.Client(version=2, session=scoped_session)
    for server in nova.servers.list():
        print server.name

if __name__ == '__main__':
    main()

      OAuth CLI の使用

      コマンド・ラインの許可コード・フロー

      必須: python-openstackclient のバージョン > 3.0.0 次のコマンドを実行してバージョンを確認します。 pip freeze | grep openstackclient

      許可コードで OpenStack クライアントを使用する手順:

      1. oidcrc という名前のファイルを作成します。

      2. 次の内容をファイルに貼り付けます。ご使用の環境に基づきパラメーターを更新します。

        OIDC stackrc

         export OS_TOKEN_ENDPOINT=https://<replace>/oauth/token
 export OS_CLIENT_SECRET=<client secret>
 export OS_CLIENT_ID=<client_id>
 export OS_DISCOVERY_ENDPOINT=https://<replace>/.well-known/openid-configuration
 export OS_IDENTITY_PROVIDER=auth0
 export OS_REDIRECT_URI=<redirect_uri>
 export OS_IDENTITY_API_VERSION=3
 export OS_DOMAIN=Default
 export OS_PROTOCOL=oidc
 export OS_AUTH_URL=https://<FQDN>:5000/v3
 export OS_USERNAME=<username>
 export OS_AUTHORIZATION_ENDPOINT=https://<replace>/authorize

      3. ID マネージャーから許可コードを取得します。

      4. 期限切れになるまでコマンドを実行するために使用できる Keystone トークンを、次のコマンドで取り出します。

          openstack --os-auth-type v3oidcauthcode --os-code <auth-code> --os-project-name <project-name> --os-project-domain-name Default token issue

      例:

      root@allinone-multiple:~# openstack --os-auth-type v3oidcauthcode --os-code 6LxfzO7GvCjmmcRa --os-project-name admin --os-project-domain-name Default token issue


            +------------+----------------------------------+
      | Field      | Value                            |
      +------------+----------------------------------+
      | expires    | 2017-01-18 17:02:32.432419+00:00 |
      | id         | 7c43310f774f4401b5ec6abee8aa3d77 |
      | project_id | 2abdd5e144ad490294f20f9efb9968e7 |
      | user_id    | 6f81cf8c63fc49d1a12f8fd4f75cab82 |
      +------------+----------------------------------+

      上記で生成されたトークンを使用して user list コマンドを実行します。

      openstack --os-auth-type v3token --os-token <token id from previous request> --os-project-name <project-name> --os-project-domain-name Default user list

      注: –os-client-secret および –os-client-id パラメーターを使用することで、クライアントの ID およびシークレットを rc ファイルに保存するのではなく、OpenStack CLI でパラメーターとして渡すことができます。

      その他の注記

      現時点で、フェデレーテッド・ユーザーおよび Keystone の委託者/受託者機能には問題があり、これにより、フェデレーテッド Heat ユーザーは heat_stack_owner 役割を委任できません。

      ユーザーに heat_stack_owner 役割を直接割り当てることにより、信頼する代行に対する予備手段が必要になります。これは、フェデレーテッド・ログイン時に作成されるユーザーです。ユーザーのドメインは統合されており、値は None (Federation=None) である可能性があります。

      $ openstack user list

      フェデレーテッド・ユーザーはドメインなしにする必要があります: openstack user show *federated user*

      ここでユーザー ID が必要です。

      $ openstack role add heat_stack_owner --user *federated user ID* --project *project*

      設定ページでパスワードを変更するオプションがフェデレーテッド・ユーザーに提示されるという既知の問題があります。これらのユーザーは、自身のパスワードを変更することはできません。
