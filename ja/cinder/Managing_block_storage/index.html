      ブロック・ストレージの管理

      • by Ying Tang ● November 21, 2016

      • Tags:

        • cinder

        • ブロック・ストレージ

        • 公開 URL

        • エンドポイント

        • 見つかりません

        • 使用量

      この文書は、Bluemix ブログの投稿『Managing Cinder Volumes with IBM Bluemix Private Cloud』を基に作成されました。

      お客様の固有の要件に対処するために、IBM Bluemix Private Cloud は複数のストレージ・モデルを提供しています。Bluemix Private Cloud 上にホストされている拡張可能なファイル・システムやデータ管理システムなどのワークロードが存在する場合、ブロック・ストレージが環境固有のユース・ケースにうまく適合することがあります。この記事では、ブロック・ストレージ・モデルの使用を開始する方法を説明します。

      OpenStack Cinder

      ブロック・ストレージは仮想サーバーから接続したり切り離したりできるスタンドアロンの永続ストレージです。ブロック・ストレージを使用すれば、データは仮想サーバーのライフサイクルを越えて永続します。例えば、仮想サーバーが予期せずに停止した場合、ブロック・ストレージ内に保管されたデータは保護されそのまま残ります。

      Bluemix Private Cloud は OpenStack を使用しているため、OpenStack Cinder がブロック・ストレージ・オプションとしてクライアントに提供されます。Cinder を使用すると、開発者は OpenStack Horizon ダッシュボード UI と標準化された OpenStack API の両方を使用してブロック・ボリュームを管理できます。

      IBM Bluemix Private Cloud バージョン 3.1.0 以降を実行しているお客様の場合、Cinder は複数のバックエンドをサポートし、そのスケジューラーは適切なボリューム・タイプを柔軟に指定できます。Cinder ブロック・ストレージは、デフォルトで Ceph クラスターを使用するように構成されています。

      Ceph が Bluemix Private Cloud と連動する方法

      Ceph は、RADOS (Reliable Autonomic Distributed Object Store) に基づいた分散ストレージ・ソリューションで、単一の統合ストレージ・クラスター内のオブジェクト・ストレージ、ブロック・ストレージ、およびファイル・システム・ストレージをアプリケーションに提供します。

      Ceph クラスターは通常、モニター・ノードおよびストレージ・ノード (OSD ノード) で構成されています。モニター・ノードは、高可用性を確保するためにクラスター・マップのコピーを保持し、クラスターの正常性を追跡します。

      Ceph を使用することで、Bluemix Private Cloud では、より多くの容量と卓越したパフォーマンスを備えた、よりコストの低いブロック・ストレージ・オプションが提供されます。特定のワークロード要件に応じて、Ceph クラスターの以下のノード・タイプをカスタマイズすることもできます。

      • SSD のみの OSD ディスクおよび同一場所に設置された Ceph ジャーナルを使用した SSD のみのオプションは、最適な I/O パフォーマンスを提供します。これは IOPS の高いストレージ・ワークロードに適合します。

      • SATA OSD ディスクおよび SSD ベースの Ceph ジャーナルを使用したハイブリッド・オプションは、良好な IO パフォーマンスと低コストを実現し、IOPS が中程度のワークロードに適しています。

      • SSD のみのノードとハイブリッド・ノードが混在した多層オプションは、大規模な環境に理想的なオプションです。

      ブロック・ストレージが適切な複数バックエンド・オプションおよび多層オプションを使用して構成されたら、OpenStack Horizon ダッシュボードまたは OpenStack Cinder クライアントを使用して、Cinder ボリュームを作成できます。

      OpenStack API を使用した Cinder ボリュームの作成

      OpenStack クライアントを使用してボリュームを作成するには、次の Cinder コマンドを実行します。

      $  cinder create <size> --display-name <name> --volume-type <type> \
--image-id <imageid> --availablity-zone <az>

      以下のコマンドは、Bluemix Private Cloud 用に新しく作成されたボリュームをカスタマイズする方法の例を示しています。

      • <size>: ギビバイト (GiB) 単位のボリュームのサイズ。
      • <name>: ボリュームの名前。
      • <type>: CEPH_HYBRID または CEPH_SSD。cinder type-list コマンドを使用すると、有効なボリューム・タイプのリストを取得できます。
      • <imageid>: オプションで、ご使用のボリュームに使用するイメージの ID を指定します。nova image-list コマンドを使用すると、既存のイメージをリストできます。
      • <az>: オプションで、ボリュームを作成する可用性ゾーンを指定します。cinder availability-zone-list コマンドを使用すると、可用性ゾーンのリストを取得できます。

      このトピックに関する詳細情報については、コマンド・ラインを使用したボリュームの作成の資料を参照することができます。

      Horizon を使用した Cinder ボリュームの作成

      OpenStack ダッシュボードである Horizon を使用してブロック・ストレージ・ボリュームを作成するには、以下のステップを開始します。

      1. Horizon ダッシュボードにログインします。

      2. 「プロジェクト (Project)」セクションで、「コンピュート (Compute)」を選択し、次に「ボリューム (Volumes)」をクリックします。既存のボリュームがある場合、ボリュームとその詳細 (名前、説明、サイズ、状況など) のリストをダッシュボード UI から表示できます。

      Horizon ダッシュボードの右上部分で「ボリュームの作成 (Create Volume)」をクリックします。「ボリュームの作成 (Create Volume)」ダイアログが表示されます。

      ボリュームに関する以下の詳細情報を指定します。

      • ボリューム名: ボリュームの名前を入力します。
      • 説明: ボリュームの説明を入力します。
      • ボリューム・ソース: ソースなし、空のボリューム、またはイメージ。空のボリュームを選択すると、空のボリュームにファイル・システムまたはパーティション・テーブルは含まれません。イメージ・オプションを選択すると、新しく「ソースとしてイメージを使用 (Use image as a source)」フィールドが表示され、リストからイメージを選択できます。
      • タイプ: Ceph_SSD または Ceph_Hybrid。
      • サイズ (GiB): ギビバイト (GiB) 単位で、ボリュームのサイズを選択します。例: 1
      • 可用性ゾーン: 可用性ゾーンを選択します。例: nova

      完了したら「ボリュームの作成 (Create Volume)」をクリックします。これで Cinder ボリュームを仮想サーバーに接続できます。

      このトピックに関する詳細情報については、Horizon ダッシュボードからのボリュームの作成の資料を参照することができます。

      Cinder ボリュームを作成して仮想サーバーに接続した後は、容量要件が大きくなるにつれて、ボリューム・サイズを簡単に調整し、拡張することができます。また、保管されたデータを安全に保護しながら、ブロック・ボリュームを切り離して、永続ストレージが必要な別の仮想サーバーに再接続することもできます。

      トラブルシューティング

      openstack volume list –name vs. cinder list –name

      cloud_admin でない役割を持つユーザーの場合、openstack volume list --name コマンドが正常に機能しません。この問題を回避するには、cloud_admin 役割を持つユーザーに、ボリュームを名前別にフィルター処理させるか、お客様自身であらゆるユーザーに対して機能する cinder list --name コマンドを使用することもできます。この問題については社内障害 374319 を開設し、当社のエンジニアリング・チームと協力して問題の解決に取り組んでいます。

      cinder コマンドまたは openstack volume コマンドの実行中に「volumev2 サービス用の publicURL エンドポイントが見つかりません (publicURL endpoint for volumev2 service not found)」エラーが発生する。

      このエラーが発生するのは、通常、現行クラスター内にブロック・ストレージ (Cinder ボリューム) ノードが存在しないためです。ブロック・ストレージを注文する場合、ブロック・ストレージを要求するチケットをオープンしてください。営業チームが追加を支援します。

      volumev3 サービス用の publicURL エンドポイントが見つかりません (publicURL endpoint for volumev3 service … not found)

      このエラーは、エンドポイント・リスト内に以前のバージョンのボリューム・サービスがある場合に発生することがあります。これを解決するには、RC ファイルに export OS_VOLUME_API_VERSION=2 を追加してください。

      Cinder に割り振られたすべてのボリュームの合計ボリューム・サイズが、Ceph の未加工の合計サイズと一致しないのはなぜですか？

      この状態が発生するのは、削除されたがプール内に依然として存在するボリュームが Ceph 内にあり、削除される前にこれらのボリュームから別のボリュームが作成された場合です。Ceph の動作方法では、複製されるボリュームは複製元のボリュームのコピー・オン・ライト子ボリュームであり、これは、すべての子 (および孫、ひ孫など) が削除されるまでは親ボリュームを保持する必要があることを意味します。こうしたことから、使用量は現在アクティブなボリュームのサイズの合計と等しくなりません。

      Ceph のコピー・オン・ライトの階層化の詳細については、http://docs.ceph.com/docs/hammer/dev/rbd-layering/ を参照してください。
