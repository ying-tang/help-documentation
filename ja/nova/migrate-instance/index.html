      新しいクラウドへのインスタンスのマイグレーション

      • by Jason Kennedy ● October 5th, 2015

      • Tags:

        • nova

        • マイグレーション

        • 新しいクラウドへのマイグレーション

      ときどき、アップグレード、データ・センターの移動、その他の予想外のイベントが原因で、インスタンスを OpenStack クラウド間でマイグレーションする必要が生じます。

      この例では、インスタンス (nginx のみを実行しているシンプルなもの) のスナップショットを作成し、そのスナップショットをローカル・マシンにダウンロードしてから、新しいクラウドにアップロードします。まず、このインスタンスを見てみましょう。

      nginx がインスタンス内から実行されていることが分かります。では、インスタンス自体を確認してみましょう。 (注: インスタンスの名前は snap_test にしたため、以下のように明示的に grep できます):

      上記から、インスタンスの外部 IP が分かります。この IP にアクセスして、nginx が稼働中であることを確認してみましょう。

      これで稼働中であることが分かったので、次はそれを新しいクラウドに移動してみましょう。

      まず、source について注意があります。スナップショット作成および移動の実行に必要なクラウド・コマンドを実行するために、既存のクラウドの資格情報を source しておく必要があります。この場合、python-openstackclient を Mac にインストールし、クラウドから stackrc を取得しました。これを行うには、以下のイメージに示すように、右にある「Download OpenStack RC File」をクリックします。

      よろしければ、Horizon で追跡できます。そうした場合、インスタンスの「Task」がスナップショット作成、アップロード中の順に変更されることが分かります。その後、以下のように、image-list でスナップショットを確認できます。

      完了し (イメージ・サイズによっては少々時間がかかることがあります)、イメージが ACTIVE と表示されたら、以下のように、ローカル・マシンへのダウンロードに進みます。

      ダウンロードが完了したら、イメージがローカルに保存されています。では逆に、即時に新しいクラウドへアップロードしましょう。まず、クラウドから stackrc を source して、正しいものと確実に対話し、作成したスナップショットのアップロードに進みます。

      Glance を使用している場合の image-create コマンドに関する注意があります。前のコマンドでは、以下のようにしました。

      --container-format: イメージのコンテナー・フォーマット。許容されるフォーマットは、ami、ari、aki、bare、ovf です。

      --disk-format: イメージのディスク・フォーマット。許容されるフォーマットは、ami、ari、aki、vhd、vmdk、raw、qcow2、vdi、iso です。(なお、OpenStack では、内部的にはすべてのイメージに対して qcow2 を使用します)。

      --name イメージの名前。

      --file 作成時にアップロードするディスク・イメージが含まれたローカル・ファイル。あるいは、stdin を介してイメージをクライアントに渡すこともできます。

      完了したら、以下のように、スナップショットからインスタンスをブートしてみましょう。

      前のコマンド (nova boot --image snap_migration --nic net-id=ba0fdd03-72b5-41eb-bb67-fef437fd6cb4 --security-groups jksec --flavor m1.tiny snap_migration_test) では、nova net-list を実行して内部ネットワークの ID を取得することで入手した --nic net-id を使用しました。同様に、--security-groups の変数 jksec は、nova secgroup-list を実行して取得されたものです。いずれのコマンドも API を使用して実行されました。Horizon パネルでも両方のコマンドを確認できます。

      インスタンスが稼働したら、nginx に外部から到達できるようにするため、インスタンスに外部 IP を割り当てる必要があります。これについては、必ず、新しいクラウド・セキュリティー・グループで port 80 入口を許可して、インスタンスにその方向で到達できるようにしてください。

      最後に、移動後のインスタンスをテストして、問題なく nginx が稼働していることを確認してみましょう (まだインスタンスにログインさえしておらず、これまでドラッグ・アンド・ドロップの手順だったことに注意してください)。
