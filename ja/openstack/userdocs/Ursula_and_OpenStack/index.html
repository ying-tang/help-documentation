      Ursula と OpenStack

      OpenStack については大部分のお客様がよくご存じだと思われますが、Ursula についてはよく知らないというお客様もいらっしゃるでしょう。IBM Bluemix Private Cloud では、Ursula を使用して OpenStack の専用バージョンをインストールします。インストールを自動化することで、ユーザー・エラーを減らすことができるだけでなく、OpenStack のインストールをより迅速かつクリーンに行うことができ、カスタマイズの可能性も広がります。このブログは、お客様が Ursula を理解し、操作するためのスタート地点の役割を果たします。IBM が提供する Ursula の Playbook はすべてオープン・ソースで、GitHub から利用可能です。最終的には、これらの Playbook のうちいくつかを必要に応じてカスタマイズすることも可能です。

      Ursula とは

      • Blue Box が OpenStack クラウドのデプロイと管理を行うために作成した Ansible Playbook を集めたもの。

      Ursula は 2 とおりの方法でインストールできます。1 つは Vagrant を使用する方法、もう 1 つは手動デプロイによる方法です。この投稿では、手動デプロイによる方法に的を絞って説明します。

      Ursula には手動デプロイを行う方法についての説明が付属していますが、OpenStack 初心者にはやや難しい内容になっています。この投稿では、筆者自身が Ursula をインストールしたときの経験を共有し、そのときに行った方法について詳しく説明するとともに、問題が発生したときにそれを解決する方法についても、詳しく説明します。

      要件: 2 台の仮想プライベート・サーバー (VPS)

      • Ansible をセットアップするための VPS 1 台。筆者はこのために CentOS 6.x VPS を選びました。使用したのは 2GB、2 コアの VPS です。
      • デプロイ・ターゲットとして使用する VPS 1 台。このターゲットには、以下に示すより具体的な要件があります。
        • Ubuntu 14 LTS (Ubuntu 12 LTS は使用しないこと。)
        • 8GB の RAM (これより少なくてもデプロイできますが、処理速度が大幅に低下する可能性があります。)
        • 4 コア
        • 1 つの IP アドレス。この例では、VPS のパブリック IP は 69.87.123.456 です (これは明らかに有効な IP アドレスではありません。)

      CentOS 6.x VPS の準備

      この CentOS VPS は、OpenStack を Ubuntu VPS にインストールするために、Ansible によって使用されます。以下のステップを実行します。

      ステップ 1. IUS (Inline with Upstream Stable) リポジトリーをインストールします。これは Python 2.7 バイナリーを取得するために行います。

      ステップ 2. Python 2.7 パッケージといくつかの依存関係をインストールします。

      ステップ 3. Python 用の仮想環境を作成します。この環境の名前は ursula-python とします。

      ステップ 4. 仮想環境をアクティブ化します。

      ステップ 5. git を使用して Ursula リポジトリーを複製し、Python の要件をインストールします。

      ステップ 6. allinone の SSH 構成をセットアップします。CentOS 6.x VPS と Ubuntu ターゲットの間に SSH 鍵認証が既にセットアップされている必要があります。

      ~/.ssh/config を編集または作成して、以下の情報を追加します。

      上記の例では、ターゲット Ubuntu 14 VPS のユーザー名が root、IP が 69.87.123.456 であり、SSH 秘密鍵が /root/id_rsa に保管されているものと仮定しています。これらの内容は必要に応じて変更してください。

      envs/example/defaults-2.0.yml で取り込まれる SSL 証明書の修正

      ステップ 1. 証明書を抽出して調べ、その証明書が破損していることを確認します。

      ステップ 2. 証明書をファイル defaults2.0-cert.crt に抽出します。

      ステップ 3. そのファイルから詳細の取得を試みます。

      エラーを見るとわかるように、この証明書は明らかに破損しており、置き換える必要があります。

      ステップ 4. 新しい自己署名証明書と秘密鍵を生成します。共通名として偽のホスト名を使用します。この例では、共通名は openstack.chaidas.com です。

      ステップ 5. 破損した証明書を生成した証明書に置き換えます。

        sed -i -- 's!crt: |!crt: "{{ lookup(\x27'file\\x27',\x27'/etc/ssl/server.crt\\x27') }}"!g' defaults-2.0.yml

        sed -i -- 's!key: |!key: "{{ lookup(\x27'file\\x27',\x27'/etc/ssl/serverkey.pem\\x27') }}"!g' defaults-2.0.yml

      ステップ 6. FQDN を前の手順で共通名に使用したものに変更します (この例では openstack.chaidas.com)。このステップは省略しないでください。省略すると、手順を完了できません。

      デプロイの開始

      ステップ 1. 画面セッション内で allinone のインストールを実行します。8GB、4 Core VPS の場合、正常なデプロイにかかる時間は約 1 時間 26 分です。

      ステップ 2. このプロセスが終了したら、VPS の IP アドレス (この例では 69.87.123.456) にアクセスすることで Horizon にログインできるようになっているはずです。ユーザー名は「admin」、デフォルトのパスワードは「asdf」です。

      ステップ 3. これで終了です。
