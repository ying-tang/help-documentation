      OpenStack のネットワークの概念

      • by Ying Tang ● May 25, 2016

      • Tags:

        • OpenStack

        • 概念

      この資料では、一般的なネットワーキングの概念に関する基本的な概要を示します。これらの概念は、IBM Bluemix Private Cloud で作業している際に使用されます。

      この資料で使用している用語

      概念 説明
      L2 パケットがルーティングされる、ISO 7 レイヤー・インターネット・アーキテクチャーのレイヤー 2。
      VLAN 仮想ローカル・エリア・ネットワーク。基本的にサブネット。
      IP インターネット・プロトコル。通常、ネットワーク上のパケットの宛先を指定する IP アドレスの省略語として使用されます。
      IPv4 インターネット・プロトコルのバージョン 4。xxx.xxx.xxx という形式の IP アドレスです。現在通常、末尾に /xx を追加し、CIDR と呼ばれるプロトコルを使用してサブネット・サイズを示します。
      CIDR クラスレス・ドメイン間ルーティングまたはスーパーネッティング。IPv4 アドレスの有用寿命を延ばす手段です。IPv4 アドレスはグローバル・プールで不足する可能性があるため、通常、購入する必要があります。CIDR の詳細については、リンク http://searchnetworking.techtarget.com/definition/CIDR を参照してください。特定の範囲から得られる IP の数を把握するのに便利な CIDR サブネット・サイズ計算器については、リンク http://www.subnet-calculator.com/cidr.php にアクセスしてください (IP の一部は事前割り振りされていて、クラウド用に使用できないので注意してください)。
      IPv6 インターネット・プロトコルのバージョン 6。新しいインターネット・アドレッシング・プロトコルであり、大量のネットワーク・アドレスを指定しますが、ネットワーキング・アーキテクチャーが複雑化することがあります。
      NIC ネットワーク・インターフェース・カード。サーバーをネットワークに接続するハードウェア。また、仮想 NIC は、基本的に NIC と同様に機能するソフトウェアであり、クラウドで一般的に使用されます。用語「ネットワーク・インターフェース」は通常、コンテキストによって明示的にハードウェアを指すことが決定される場合を除き、仮想 NIC を指します。
      MAC ネットワーク上の各ホストを一意的に識別する、ハードウェア・シリアル番号のような個別 ID。

      基本的なネットワーキングの概念

      概念 説明
      ネットワーク 通常のネットワークにおける VLAN に似た、分離されたレイヤー 2 (L2) セグメント。ネットワークは 1 つ以上のサブネットを持つ場合があります。
      サブネット IPv4 アドレスの範囲。
      ポート 仮想サーバーの NIC など、ある 1 つのデバイスを仮想ネットワークに接続するための接続ポイント。用語「ポート」は、そのポートで使用される MAC および IP アドレスなど、関連するネットワーク構成も表します。
      固定 IP アドレス 公衆インターネットに到達する必要があるインスタンスのネットワーク IP アドレスなど、ブート時にインスタンスに割り当てられる IP アドレス。
      Floating IP アドレス 要求時にプールからインスタンスに割り当てられる IP アドレス。多くの場合、IPv6 形式です。
      クォータ 割り当て量は、コンピュートやディスクの割り振りなど、クラウドで使用可能なリソースの数量を制限します。
      セキュリティー・グループ プロジェクトに固有の一連のネットワーク・フィルター・ルール。どの IP がパケットを別の IP またはポートに送信できる、あるいは送信できないかを指定することで、インスタンスのネットワーク相互作用に適用されます。
      セキュリティー・グループ・ルール プロトコル、ポート、および IP アドレスの仕様のリストであり、ネットワーク上の任意の特定のインスタンスとの間でどのようにトラフィックのフローが許可されるのかを設定します。これは、セキュリティー・グループの明細です。

      クラウド・ネットワーク・タイプ

      一般的に、クラウドには、内部ネットワークおよび外部ネットワークがそれぞれ 1 つ以上含まれます。この概念は、クラウド内のインスタンスが、内部ネットワークで相互に通信し、外部ネットワークでプライベート・クラウドの外部にあるソースと通信するものとして捉えることができます。

      内部ネットワークおよび外部ネットワークを示すために、さまざまな用語が使用されることがあります。例えば、内部ネットワークは、「テナント・ネットワーク」と呼ばれることがあります。公衆インターネットとの通信に使用される外部ネットワークは、「プロバイダー・ネットワーク」と呼ばれることがあります。

      内部ネットワーク

      内部ネットワークはクラウド内で通信し、自身に接続されているすべての仮想ネットワーク・インターフェース (NIC) 間の接続を提供します。各内部ネットワーク・インターフェースは仮想マシン (VM) に関連付けられ、単一の VM に複数のネットワーク・インターフェースを関連付けることができます。つまり、各内部ネットワーク・インターフェースは、複数のネットワークで通信できます。

      VM 間の通信は、各 VM を同じ内部ネットワークに接続することによって可能になります。内部ネットワークは、単一のプロジェクト (テナント) に関連付けられ、そのプロジェクトに対してのみ表示され、アクセス可能になります。

      外部ネットワーク

      外部ネットワークは、クラウド内およびクラウド外で通信できます。公衆インターネット、クラウドの外部ネットワーク、IBM クラウド・データ・センター・ルーターが含まれた IBM クラウド・データ・センター・ネットワークが含まれることがあります。

      • クラウドの「パブリック・ネットワーク」(「外部ネットワーク」) は、公衆インターネットとの間のアクセスを提供します。
      • SoftLayer プライベート・ネットワークは、通信する他のクラウドを含め、SoftLayer の他のサービスへのアクセスを提供することを目的としています。(SoftLayer プライベート・ネットワークへのアクセスの詳細については、『リリース・ノート』を参照してください。
      • SoftLayer ルーター・ネットワークは SoftLayer Backend Core Router (BCR) に接続されます。この BCR により、独自のプライベート・クラウド環境の外部にあるリソース (ご使用のサイト (Blue Box Local) のデータ・センター内、所有している別のプライベート・クラウド環境内、SoftLayer データ・センター内、またはその他の場所 (Blue Box Dedicated や独自のレガシー・クラウド) にあるリソース) に到達できるようになります。
      • パブリック・ネットワークと SoftLayer プライベート・ネットワークは物理ネットワークです。

      異なるプライベート・サブネット上の仮想マシン間の接続

      2 つのプライベート・ネットワークがあるものとします。各ネットワーク・サブネット上に仮想マシンを作成し、着信トラフィックを受け入れることができるように必要なセキュリティー・グループを追加しました。異なるプライベート・サブネット上の VM 間の接続を実現するにはどのようにすればよいでしょうか? operations という名前の 1 つのネットワークと、internal という名前の 1 つのネットワークがあるものとします。以下の複数の方法で、両者間の通信を実現できます。

      • operations ネットワークをデフォルト・ルーターに接続します。この接続により、デフォルトの internal ネットワークと operations ネットワーク間でトラフィックがフローできるようになります。次に、インスタンスを internal ネットワークにデプロイし、パブリック浮動 IP アドレスを割り当て、そのインスタンスを使用してターゲット・インスタンスに接続できます。
      • あるいは、よく似たアプローチを使用し、internal ネットワークと operations ネットワークの間に新しい内部ルーターを作成することもできます。internal ネットワーク上でパブリック IP を持つインスタンスを使用して、operations ネットワーク上の VM にアクセスできます。内部ルーターを作成することで、operations ネットワーク上の VM がインターネットへの外部接続を確実に持たなくなります。
      • 別の可能性として、マルチホーム・インスタンスを作成できます。internal ネットワークおよび operations ネットワークに接続された 1 つのインスタンスを作成します。パブリック浮動 IP を割り当てます。その VM を使用して、2 つ目のプライベート・インターフェースを介して operations ネットワーク上のインスタンスにアクセスできます。

      パブリック浮動 IP を持つマシンではどのような種類のセキュリティー予防措置を取る必要がありますか?

      以下に、開始するための優れた慣例を示します。

      • 鍵認証のみを許可するように sshd を構成し、root ssh アクセスを無効にします。
      • SSH で標準またはデフォルトのシステム・ユーザー名 (ubuntu、deploy など) を使用してはなりません。
      • 潜在的なブルート・フォース・アタックをブロックするために fail2ban や denyhosts などのシステム・サービスを使用します。両ツールは、普及している Linux ディストリビューションでは既にパッケージされています。
      • セキュリティー・グループ・ルールを使用して、SSH アクセスを限られた数の IP アドレスに制限します。
      • 定期的にパブリック・インスタンス上で実行されているシステムを更新します。
      • パブリック・インスタンス上で不要なサービス (http、ftp、smb など) を実行してはなりません。
