      基本概念

      概念 説明
      Volume ボリュームは、永続的な保管を可能にするためにインスタンスに接続する、ブロック・ストレージ・デバイスです。永続ストレージは、インスタンスを終了しても削除されません。ボリュームはいつでも、実行中のインスタンスに接続したり、実行中のインスタンスから切り離して他のインスタンスに接続したりできます。また、ボリュームは削除することもできます。ボリュームのタイプは、管理ユーザーのみが作成できます。
      一時 インスタンスに関連付けられているディスクは一時ディスクであるため、インスタンスが終了すると、そのデータは削除されます。実行中のインスタンスから作成されたスナップショットは残りますが、最後のスナップショットの後で新たに一時ストレージに追加されたデータは、いずれも失われます。

      以下の表では、3 つの OpenStack ストレージの概念を比較します。

        一時ストレージ ブロック・ストレージ オブジェクト・ストレージ
      使用する目的: オペレーティング・システムの実行およびスクラッチ・スペース インスタンスへの新たな永続ストレージの追加 インスタンス・イメージも含めたデータの格納
      アクセスで経由するのは: ファイル・システム パーティション、フォーマット、およびマウントが可能なブロック・デバイス (/dev/vdc など) REST API
      アクセスできるのは: インスタンス内 インスタンス内 あらゆる場所
      管理するのは: OpenStack Compute (nova) OpenStack Block Storage (cinder) OpenStack Object Storage (swift)
      持続するのは: インスタンスが終了するまで ユーザーが削除するまで ユーザーが削除するまで
      サイジングを決定するのは: 管理者がサイズ設定、すなわちフレーバー を構成する ユーザーが初回の要求で指定する 利用可能な物理ストレージの容量
      標準的な使用例 最初のディスクは 10 GB、次のディスクが 30 GB 1 TB のディスク データ・セットの格納に数十 TB

      スワップ・スペース

      • スワップ・スペースは、スワップを使用している仮想マシンが実行されるのと同じコンピュート・ホスト・マシンでプロビジョンされます。
      • スワップ・スペースは、仮想マシンで、マウントされたデバイスとして (例えば、/dev/vdb 上、udev によって) 公開されます。

      openstack host show で物理ホストの実際のハード・ディスク使用量が表示されないのはどうしてですか?

      openstack host show では、ホスト上で実行されるインスタンスのフレーバーに関する情報のみが表示されます。あるお客様が、スワップ使用量が含まれるのかどうかについて質問されました。その当時、含まれていませんでした。http://docs.openstack.org/admin-guide/common/nova-show-usage-statistics-for-hosts-instances.html に従い、Mitaka リリース時点で含まれる内容を以下に示します。

      disk_gb 列 [openstack host show の出力内] では、ホストで実行されるインスタンスのルートと一時ディスク・サイズ (GB) の合計が表示されています。… [この値] は、ホストで実行されるインスタンスのフレーバーに関する情報を使用して計算されます。このコマンドでは、物理ホストのハード・ディスク使用量は照会されません。

      実際のフレーバーの計算は、https://github.com/openstack/nova/blob/c521ba7fa679b6e9c790d84ea22f04fb999987db/nova/notifications/base.py#L448-L454 で確認できます。

      disk_gb = root_gb + ephemeral_gb

      ephemeral_gb = セカンダリー一時データ・ディスクのサイズ。これは空の未フォーマット・ディスクであり、インスタンスの存続期間中にのみ存在します。デフォルト値は 0 です。 root_gb = ルート (/) パーティションで使用するディスク・スペース量 (G バイト)。

      現在、Nova の disk_gb には、スワップ (インスタンスに対するオプションのスワップ・スペース割り振りであり、デフォルト値は 0) は含まれていません。

      https://specs.openstack.org/openstack/nova-specs/specs/newton/implemented/resource-providers-allocations.html によると、このオプションのスワップ・スペースを含め始める作業が Newton に対して以下のように提案されています。”コンピュート・ノードがインスタンス・ディスクのローカル・ストレージを利用するか、ボリュームからブートされた場合、使用スペースは、フレーバーの root_gb、ephemeral_gb、および swap フィールド値の合計である必要があります。resource_provider_uuid は、コンピュート・ノードの UUID である必要があります。注: ボリュームからブートされたインスタンスは、root_gb 値は 0 になります。”

      https://derpops.bike/2016/10/07/openstack-nova-and-hypervisor-disk-consumption/ も参照してください。
