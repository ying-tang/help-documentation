      OpenStack を選ぶ理由

      • by Leslie Lundquist ● April 14, 2016

      • Tags:

        • OpenStack

        • 参照

      OpenStack™ は、Infrastructure-as-a-Service (IaaS) を提供するための、広範なオープン・ソースのソフトウェア・プラットフォームです。OpenStack の開発は、Rackspace と NASA の共同作業を通じて、2010 年に開始されました。OpenStack は、データ・センター内のコンピュート・リソース、ストレージ・リソース、およびネットワーキング・リソースのプールを、規模の大小にかかわらず管理できます。また、一元化されたダッシュボードを通じて、これらのリソースを使用可能にすることや、拡張を容易にすることができます。

      歴史と構想: OpenStack は、史上最も成長の早いオープン・ソース・プロジェクトで、現在では 2,000 万行を超えるコードが作成されています。OpenStack Foundation が管理しており、その開発、配布、および採用を促進しています。OpenStack プロジェクトは、34,000 人を超える個人貢献者と、550 社を超える参加企業によって構成され、これらの個人と企業はすべて同団体のメンバーとなっています。同団体の個人のメンバーシップは無料です。企業メンバーシップには 2 種類の支払いレベル (Platinum と Gold) があります。同団体の方針は、個人貢献者と参加企業の代表者で構成される役員会が決定します。

      ライセンス交付とリリース: OpenStack ソフトウェアは Apache ライセンスの下で利用可能で、リリース・スケジュールは年 2 回 (4 月/5 月、10 月/11 月) です。リリースのコードネームはアルファベット順に付けられます。最新リリースの名前は「Ocata」です。次回のリリースも同時に開発中です。これら年 2 回のリリースと並行して、OpenStack コミュニティーは OpenStack サミットで会合を行い、次回リリースの計画を策定するとともに、実績やユース・ケースの発表を行います。

      ハードウェア要件: OpenStack は、標準的なハードウェアや市販のハードウェアで機能します。専用のベンダー・ハードウェアは不要です。特定のエンタープライズ級ハードウェアを使用したいユーザーには、ドライバーが用意されています。これらのドライバーのリストは、OpenStack Web サイトの Marketplace で公開されています。

      ソフトウェア・プロジェクトの方法 (Big Tent): Liberty リリース以降、OpenStack コミュニティーは、OpenStack のソフトウェア開発のメタファーとして「Big Tent」を採用しています。プロジェクトはコミュニティーのメンバーによって提案され、「Big Tent」の下で保護されます。コミュニティーのメンバーが共同作業を決定すると、そのプロジェクトがリソースを獲得します。初期段階で正式に認可または拒否されるプロジェクトはありません。

      OpenStack コミュニティー: OpenStack の最大の強みは、精力的なコミュニティーです。このコミュニティーには、177 を超える国の 34,000 人を超える個人貢献者が参加しており、世界中に 70 を超えるユーザー・グループがあります。OpenStack のユーザー・グループは定期的に会合を行っています。これらのグループは、認識を広め、技術に関する議論を促進し、コミュニティー全体との仲間意識を示すための場として機能しています。OpenStack の初心者にとっては、これらのグループに参加して他のユーザーから知識を得ることが、OpenStack を使い始めるための最も簡単な方法です。

      市場のトレンドと OpenStack の現状: OpenStack は、プライベート・クラウドのデプロイ時に最も選ばれるプラットフォームとなりつつあります。市場収入は 2016 年までに 13 億ドルに達する見通しで、2018 年には 30 億ドルを超える見通しです。OpenStack の採用を促進する 3 大要素は、OpenStack IaaS プラットフォーム上ではイノベーションを迅速に進めることができること (DevOps)、クラウドのデプロイを自動化することで規模を簡単に拡大できること、および OpenStack がオープンなテクノロジーであることです。ソフトウェアの開発とテストが最も一般的に使用されるワークロードで、その次がインフラストラクチャー・サービスです。企業ワークロードは、特に OpenStack API 関連では増加傾向にあります。OpenStack API は、Infrastructure as a Service (IaaS) の標準となっています。さらに広い視野から見ると、OpenStack の採用は、学術研究部門、通信部門、金融部門、メディア部門などの市場部門全体で増加しています。

      OpenStack の資料: OpenStack を初めて扱うユーザー向けに、各種プラットフォーム (Ubuntu、RHEL、Debian、SUSE) に対応したインストール・ガイドが用意されています。これらのガイドはリリースごとに更新されます。また、OpenStack プロジェクトは、システム管理者、クラウド管理者、エンドユーザー、アーキテクト、およびクラウド開発者を対象とした包括的なオンライン資料を提供しています。セキュリティーや高可用性などの高度なトピックに関する資料も用意されています。これらの高度な資料は定期的に更新されますが、OpenStack のリリースに合わせて更新されるわけではありません。

      Ask OpenStack: OpenStack について具体的な質問がある場合は、Ask OpenStack で専門家に質問することができます。このサイトは、有名な StackOverflow サイトをモデルとしたクラウドソースの Q&A サイトです。

      OpenStack Marketplace: OpenStack のユーザーには、サービス・プロバイダー、ベンダー、システム・インテグレーター、ディストリビューション、トレーナー、コンサルタントなど、多くの選択肢があります。ユーザーが十分な情報に基づいて意思決定を行うことができるように、OpenStack のベンダー、製品、トレーニング、コンサルタント、およびその他のサービスに関する情報やレビューをまとめた中央リポジトリーが、OpenStack Marketplace に用意されています。

      OpenStack: コンポーネントの概要

      OpenStack プラットフォームは、複数のサービスの集まりであり、機能別のコンポーネントに分かれています。このうち、少数のコア・コンポーネントが連動して、コンピューティング、オブジェクト・ストレージとブロック・ストレージ、およびネットワーキングの各機能を、データ・センター内で提供します。また、これらすべてのリソースを一元的に管理するためのダッシュボードも用意されています。これらのコア・サービスは、本資料の以降のセクションで説明する一連のサポート・サービスを通じて統合されます。以下の図は、OpenStack のコンポーネント間の関係を示しています。

      図 4. OpenStack アーキテクチャーの概要 (コンポーネント別)

      OpenStack インフラストラクチャー (IaaS) サービス: コア・プロジェクト

      一般にコアと呼ばれるプロジェクトは、OpenStack コミュニティーが最初に策定したプロジェクトです。これらのプロジェクトは、長年にわたる実稼働環境での使用とテストを経ており、現在に至るまで最も広くデプロイされている OpenStack コンポーネントです。また、これらのコンポーネントは、同じオープン・プラットフォームと API 上で標準化されており、これがパブリック・クラウドとプライベート・クラウドのグローバル・ネットワークを強化しています。さらに、コア・プロジェクトは、従来のインフラストラクチャーに対する資本支出の戦略的かつ実行可能な代替策となります。

      OpenStack コア・インフラストラクチャー (IaaS) プロジェクトは以下のとおりです。

      • Nova (コンピュート)
      • Cinder (ブロック・ストレージ)
      • Swift (オブジェクト・ストレージ)
      • Neutron (ネットワーク)
      • Glance (イメージ)
      • Keystone (ID)

      実際には、これらのサービス (Swift を除く) は、もともと Nova の一環として開始されました。Nova は、Swift とともに最も初期に開始された OpenStack プロジェクトであるコンピュート・サービスです。各プロジェクトは、Nova から分離されたか (Glance および Cinder)、後から再実装されたもの (Neutron) です。Glance、Cinder、および Swift は、OpenStack のデータ処理およびストレージ機能を構成します。Neutron は、OpenStack のネットワーキング機能をデプロイしやすい形に統合します。Ironic は、自動化されたハードウェア・プロビジョニングを支援することで、IaaS の機能性と拡張性が十分に発揮されるようにします。

      Nova: Nova は OpenStack のコンピュート・サービスです。このサービスは、仮想マシンのオンデマンドのプロビジョニングと管理を提供します。KVM、XenServer®、VMware ESXi、Microsoft Hyper-V など複数のハイパーバイザーをサポートしています。また、LXC などの Linux コンテナーもサポートしています。

      Glance: Glance は、ディスク・イメージやサーバー・イメージの処理を目的とした、API からアクセス可能なサービス (ディスカバリー、登録、配布など) を提供します。Glance は、さまざまなバックエンド・セットアップ (OpenStack Swift など) にイメージを保管できます。Glance サービスは、Raw、VHD、VMDK、VDI など複数のイメージ形式に対応しています。

      Cinder: Cinder は、OpenStack のブロック・ストレージ・モジュールであり、仮想インスタンスの仮想ストレージを提供します。このサービスを使用すると、デバイスを作成して、そのデバイスを仮想サーバーに接続することや、その接続を解除することができます。Cinder は、OpenStack の Nova コンピュート・サービスと全面的に統合されており、Horizon ダッシュボードから管理可能です。また、Nimble®、Solidfire®、EMC® などのエンタープライズ・ストレージ・プラットフォームとの特定のドライバーを通じた統合がサポートされています。さらに、スナップショットもサポートされています。

      Swift: Swift は、分散した非構造化データを処理します。このサービスは、イメージ、ビデオ、仮想マシン・イメージ、およびアーカイブを保管するための優れたプラットフォームを提供します。Swift は、費用対効果が高く、API からアクセス可能で、冗長かつスケーラブルな分散型ストレージを提供し、バックアップ、アーカイブ、およびデータ保存に最適です。

      Neutron: Neutron は、ネットワークと IP アドレスを管理するための交換可能な API 駆動型プラットフォームを提供します。Neutron は、複数のネットワーク・モデル (フラット型、VLAN、VXLAN)、静的 IP、および DHCP をサポートしています。このサービスは、OpenFlow などの SDN プラットフォームを利用することにより、高度なネットワーキング機能を活用できます。

      Keystone: Keystone は、認証および許可の各機能を提供します。このサービスは、LDAP などの既存のバックエンド・ディレクトリーと統合します。Keystone は、使用権限を持つサービスとユーザーとのマッピングを指定する中央ディレクトリーを提供します。

      OpenStack を現実的な目的で使用する場合は、その目的が何であっても、Keystone はすべての OpenStack デプロイメントの必須コンポーネントです。すべての OpenStack デプロイメントには、何らかの形の認証サービスが必要であり、Keystone はその最も一般的なユース・ケースです。(特殊なテスト・ケースではノーオペレーション認証が使用されることもあります。) 例えば、Swift にはそれ自体に認証機能が備わっていますが、Keystone を使用できないテスト・デプロイメントを除いて、この認証機能が使用されることはあまりありません。

      OpenStack は、これらのモジュール (コンピュート、ストレージ、ネットワーキング、およびプロビジョニングの各コンポーネント) を相互に統合し、さらに追加のサポート・サービスを統合することで、Infrastructure as a Service (IaaS) の統合されたスケーラブルなユーザー・エクスペリエンスを提供します。また、上記の OpenStack サービスには、統一された RESTful API を通じてアクセスすることもできます。ユーザーは、必要に応じ、API インターフェースを使用して、独自のユーザー・エクスペリエンス (UX) を作成することができます。

      OpenStack サポート IaaS サービス

      OpenStack サポート・サービスは、OpenStack コア IaaS の機能を拡張します。これらのサービスには、セキュリティー・サービス、使用状況モニター、自動化デプロイ・サービスなどがあります。以下の項目では、これらの追加の OpenStack サービスのうち、前のセクションでリストした基本的な OpenStack IaaS サービスとともにデプロイされることが多いものを示します。

      サポート IaaS サービスには以下のようなものがあります。

      • Ceilometer (テレメトリー)
      • Heat (オーケストレーション)
      • Horizon (ダッシュボード)
      • Ironic (ベア・メタル・プロビジョニング)

      Ceilometer、Horizon、Heat を組み合わせて使用することで、すべての OpenStack コンポーネントの使用状況や機能に関する一般情報を統合できるため、OpenStack クラウド環境をより使いやすくすることができます。(注: OpenStack の最近のリリースでは、ライブラリー・コードである Oslo も必ず自動的にデプロイされます。これは、Oslo がこれらのサービスの共通ライブラリーとして必要なためです。)

      Ceilometer: Ceilometer は、複数の OpenStack サービスの使用状況とパフォーマンスのデータを集約します。

      Heat: Heat は、比較的複雑なインフラストラクチャーのデプロイを記述して自動化するためのテンプレート・ベースの機能を提供します。例えば、インスタンスがアプリケーション・サーバーとして実行され、データベースが Trove に保管される単純な Web アプリケーションとロード・バランサーをデプロイする必要がある場合は、アプリケーション・インスタンスの数を指定するパラメーターを含む Heat テンプレートを作成します。Heat を Ceilometer テレメトリー・サービスとともに使用すると、自動化スケーリング機能を使用できるようになります。上記の例では、この機能を使用することで、必要に応じてさらに多くのインスタンスをデプロイできます。

      Horizon: Horizon は、クラウド・リソースの表示、プロビジョン、および管理を一元的に行うことができるグラフィカル・ユーザー・インターフェースを提供します。Horizon ダッシュボードのインターフェースには、請求やモニターなどのサード・パーティー・サービスを簡単に統合できます。ただし、Horizon がなくても OpenStack は問題なく機能します。基底の OpenStack API を使用することで、同じ使用状況に関する情報にアクセスすることができます。

      Ironic: Ironic を使用すると、物理マシンを自動的にプロビジョンして管理できます。従来、企業はこのサービスを有料で購入または構築していましたが、このサービスはコンピュート・サービスの一種と考えることができます。

      OpenStack アプリケーション・サポート

      このセクションで取り上げる各 OpenStack プロジェクトは、特定のアプリケーションのラッパーとして機能します。これらのモジュールは、クラウドを特定の目的 (データベースへのアクセス、ビッグデータの分析、情報の共有など) に合わせて合理化するために最適な方法です。特に、これらの OpenStack モジュールは、ユーザーが必要とする特定のアプリケーションを既に実行している新規 Nova インスタンスの処理を高速化します。これらの OpenStack プロジェクトは、Docker コンテナーなどのコンテナー内で実行することで効率的に機能します。その主な理由は、多くの場合、ユーザーがこれらのインスタンスに直接アクセスできないためです。

      • Trove (データベース)
      • Sahara (データ処理 — Hadoop)
      • Manila (ファイル — CIFS)

      Trove: Trove は、OpenStack クラウド・インフラストラクチャー上で Database-as-a-Service (DBaaS) 機能を提供します。

      Sahara: Sahara は、スケーラブルなデータ処理スタックと、それに関連付けられた管理インターフェースを提供します。

      Manila: Manila は、共有ファイル・システム・サービスを提供します。Samba (Windows ファイル共有の Linux 実装) などの機能と連動します。

      継続的改善: Big Tent OpenStack サービスでは、上記以外のプロジェクトが次々と誕生し、実装されています。また、前述の広くデプロイされている OpenStack モジュール (統合 IaaS、サポート IaaS、および PaaS) によって既に確立されている品質基準をクリアすべく、絶え間なく改善されています。Big Tent の下で最近誕生したプロジェクトには、デプロイメント・サービス (TripleO)、DNS-as-a-Service (Designate)、鍵管理 (Barbican)、Load-Balancing-as-a-Service (Octavia)、Firewall-as-a-Service、Network Function Virtualization (NFV) などがあります。

      OpenStack クラウドの配布方法

      ここまで OpenStack の内容について説明してきましたが、実際に OpenStack クラウドを入手するにはどうすればよいでしょうか。OpenStack ソフトウェアは、現在、主に 3 とおりの方法でユーザーに配布されています。

      1. 一連のパッケージ (「ディストリビューション」と呼ばれる) として: OpenStack の Web サイトには、OpenStack のダウンロード可能な標準ディストリビューションが用意されています。このデフォルトの OpenStack ディストリビューション以外にも、多くのベンダーが、いくつかの付加価値サービスを含む専用ディストリビューションを提供しています。ディストリビューションを使用することで、OpenStack を使用する際に必要な手間のかかるインストールやアップグレードのプロセスがやりやすくなります。ただし、OpenStack クラウドの運用については、ディストリビューションを使用することで特に簡素化されるわけではありません。ディストリビューションを正しく使用できるのは、多くの場合、OpenStack の使用経験がかなり豊富なユーザーです。ディストリビューションを作成するベンダーは 25 社に及びます (Mirantis、Red Hat、HP、Platform 9 など)。OpenStack Marketplace には、OpenStack のデプロイに際して支援を求めることができるコンサルタントのリストともに、ディストリビューション・ベンダーのリストが公開されています。

      2. サービスとして (As a service): OpenStack as a service とは、経験豊富なクラウド運用企業 (IBM や Rackspace など) から、契約を通じて (プライベート) クラウド・サービスを購入することを指します。クラウド運用企業は、ユーザーが OpenStackを自由に「操作する」のではなく「使用する」ことができるように、ユーザーにサポートを提供します。これが OpenStack cloud as a service (IaaS) の主な利点です。これにより、ユーザーはクラウドの維持ではなく、自社のビジネス・アプリケーションに集中することができます。

      3. 標準の OpenStack ディストリビューション: 標準の OpenStack ディストリビューションには、OpenStack のサービスがダウンロード可能なパッケージとして組み込まれています。標準パッケージは、複数のプラットフォーム (Ubuntu®、CentOS、Debian など) 用に提供されています。OpenStack ソフトウェアは絶え間なく更新されており、これは一般にトランクと呼ばれる場所からダウンロードできます。継続的に行われるこれら更新は、動作が安定していません。動作が安定しているのは年 2 回のリリースだけです。動作が安定している最新のバージョンも常にダウンロードできるほか、作業中のバージョンもトランクから入手できます。

      ディストリビューション

      サード・パーティーのディストリビューション: OpenStack のカスタム・ディストリビューションが参加ベンダーによって提供されています。これらのディストリビューションは、そのベンダーが提供するいくつかの利点を備えています。カスタム・ディストリビューションには、付加価値サービスや専用サービスが含まれている場合があり、ベンダーの他のオファリングと特に統合しやすくなっているものもあります。OpenStack Marketplace には、サード・パーティー・ディストリビューションの総合リストが公開されています。最も利用者の多いサード・パーティー・ディストリビューションには、Mirantis OpenStack、Red Hat Enterprise Linux OpenStack Platform、Rackspace® Private Cloud Software、Ubuntu OpenStack などがあります。

      OpenStack Cloud As a Service

      少数のベンダーが、ホスト型の配布メカニズム、つまり前述の「as a service」型の配布メカニズムを通じて、OpenStack を提供しています。このメカニズムでは、ホスト環境内でオンデマンドで OpenStack クラウドを利用できます。インフラストラクチャーを追加することなく、運用によるオーバーヘッドも伴わずに OpenStack を使用したいユーザーにとっては、ホスト型の OpenStack クラウドは魅力的な選択肢です。

      ホスト型クラウドのユーザーは、OpenStack の利点を何ら損なうことなく、単一テナントの OpenStack クラウドを利用できます。デプロイや運用のオーバーヘッドを伴わずに API にアクセスすることも可能です。OpenStack Marketplace には、OpenStack ホスト型クラウドのプロバイダーの総合リストが公開されています。よく利用される OpenStack cloud as a service のベンダーの一部を以下に示します。

      IBM Bluemix Private Cloud: 2015 年 6 月に Blue Box が IBM によって買収され、IBM のクラウド製品オファリングに組み込まれました。この製品オファリングの現在の名称は IBM Bluemix Private Cloud です。Blue Box は、ホスト型の OpenStack private cloud as a service の提供を最も早く開始したプロバイダーであり、IBM Bluemix チームに多くの専門知識をもたらしました。IBM Bluemix Private Cloud チームは、OpenStack のシアトル・ユーザー・グループ会合のリーダーであり、OpenStack コミュニティー、特に OpenStack 運用企業のワークグループに積極的に貢献しています。その最新製品である IBM Bluemix Private Cloud Local では、ユーザーのデータ・センター内に OpenStack プライベート・クラウド・ハードウェアを設置することができ、その管理は IBM Bluemix Private Cloud の専門家がすべて担当します。

      Cisco Metapod: Cisco は、OpenStack cloud as a service を提供するもう 1 つのベンダーです。このサービスは、以前は Metacloud によって開発されていました。同社が最初に提供していたのは、OpenStack をベースとする Cloud-in-a-box、つまりプライベート・クラウド・アプライアンスであり、その後、ホスト型プライベート・クラウドの提供を開始しました。同社のカスタム OpenStack ディストリビューションも、優れたユーザー・エクスペリエンス (UX) で知られています。

      Rackspace: 自社のデータ・センターで管理するホスト型の専用プライベート・クラウド環境を提供します。優れたパフォーマンスで知られています。

      2 種類のホスト型環境: パブリック・クラウドとプライベート・クラウド

      パブリック・クラウド: OpenStack クラウドの運用のしくみを理解するための方法の 1 つは、OpenStack ベースのパブリック・クラウド・サービス・プロバイダーのうちいずれかを試してみることです。多くのプロバイダーが、無料のお試しサブスクリプションや廉価な導入料金を設定しています。OpenStack Marketplace には、パブリック・クラウド・サービスのプロバイダーのリストが公開されています。

      プライベート・クラウド: プライベート・クラウドは、2 つの異なる形式に分かれる傾向があります。1 つはホスト型クラウドで、専用ではあるがリモートのクラウド・ハードウェアとソフトウェアが、ユーザーの代わりにクラウド・サービス・プロバイダーによって運用されます。もう 1 つはローカル・クラウドで、ユーザーが自社のハードウェアを所有し、その運用を他社に委託するという形です。

      ホスト型: ホスト型 OpenStack クラウド製品では、OpenStack を比較的簡単な方法で使い始めることができます。例えば、IBM Bluemix Private Cloud などのプロバイダーと契約すると、ホスト型プライベート・クラウドの運用を数時間で開始できることがほとんどです。

      ローカル・クラウド: ローカル・クラウドとは、クラウド用のハードウェアを自社のデータ・センター内で所有するが、IBM Bluemix などのホスティング・プロバイダーにその運用を委託することを指します。2016 年には、複数のプロバイダーが、ローカル・クラウド・サービスの新オファリングで市場に参入しつつあります。OpenStack Marketplace にプロバイダーのリストが公開されています。

      お楽しみはここから: OpenStack が優れている理由

      では実際に OpenStack を使ってみましょう。まずは DevStack を試してみてください。DevStack は、OpenStack テスト・クラウドを最小限の構成でインストールするための最も簡単な方法です。サービスに契約せずに、あくまで個人的な確認のためにローカル・クラウド環境をテストしたい場合は、DevStack を使用して簡単に使い始めることができます。セットアップが簡単でオールインワン構成を備えた DevStack は、OpenStack Web サイトからダウンロードできます。

      DevStack は、プロトタイピング環境としての使用を想定しています。オールインワンの単一 VM、オールインワンの単一マシン、およびマルチノードの構成が組み込まれています。ただし、DevStack は実動クラウド・ワークロードの実行には適していません。例えば、マシンのリブート後は、完全に稼働可能な状態には戻りません。

      DevStack で起動するクラウドに対してモニターや管理を行うことはできず、すべてのデバッグ・ログはデフォルトで有効化されています。例えば、新しい OpenStack リリースにアップグレードする前に、そのリリースでソフトウェアが正常に機能するかどうかを確認するために、OpenStack の複数のバージョンを比較する場合には便利です。DevStack は、現在、Ubuntu、Fedora、および CentOS/RHEL の各プラットフォームで使用可能です。

      クラウドに対する具体的な要件が決まっている場合は、現時点で決まっている要件だけを満たす OpenStack プロジェクトをデプロイすることができます。OpenStack には、必要に応じて機能を追加していく方法が用意されています。例えば、自社の専用 Web サーバー上で実行されている Web サイトがあり、Tumblr などのサービス (ブログの投稿とイメージの保管) を実行しているだけであるとします。この場合、必要とされるのは多くのデータを保管する方法だけです。そこで、OpenStack 構成として Keystone と Swift だけをデプロイして、認証サービスとストレージ・サービスだけを提供すれば、コンピュートは必要ありません。

      別の例として、サイトで必要とされているのが一時ストレージ機能 (リブート時には失われる) だけであるため、クラウドにコンピュート・インフラストラクチャーだけが必要な場合を考えてみましょう。ここでは、Keystone、Nova、Glance の各プロジェクトだけを使用して OpenStack クラウドをデプロイできます。この場合、Nova に備わっているネットワーキング機能によって、簡単なネットワーク機能が実現します。より豊富なネットワーキング機能が必要な場合は、OpenStack の Neutron コンポーネントをデプロイメントに追加します。これで、すべての要件を満たすコンピュート・インフラストラクチャーが実現します。

      すべてのコンピュート機能とストレージ機能を備えた多機能な OpenStack クラウドをデプロイするには、すべてのコア OpenStack IaaS モジュールを使用します。また、多くの場合、認証サービスを提供するために、Keystone も追加します。この状態から、これから必要となる操作のタイプに合わせて、さらにクラウドをカスタマイズします。例えば、データベースやデータ分析クラスターを追加できます。それ以降も、必要に応じて OpenStack の機能を追加していくことや、アップグレードしていくことができます。

      OpenStack の一般的なユース・ケース

      OpenStack は、多くの業界のさまざまなユース・ケースで役立ちます。最もうまくいっている OpenStack のユース・ケースの一部をご紹介します。

      クラウド・サービス・プロバイダー: このユース・ケースでは、OpenStack プラットフォームがより大規模な仮想 infrastructure as a service (IaaS) を提供します。このタイプの標準的なスタックは、KVM (ハイパーバイザー)、Ubuntu (OS)、OpenStack (クラウド管理層)、Nagios (モニタリング)、LogStash (ロギング)、Kibana (分析)、Chef (オーケストレーション)、HAProxy (ロード・バランサー) などのオープン・ソース・ツールと、自社開発のコンポーネントで構成されます。コンピュート機能や特定の種類のストレージの提供には、多くの場合、市販のハードウェアが使用されます。ただし、複数のハードウェア構成をデプロイできます。Rackspace Public Cloud がこのユース・ケースのわかりやすい例です。

      通信サービス・プロバイダー: 通信サービス・プロバイダー (Telco) は、プライベート・クラウドの必要性を満たすために OpenStack を早期に採用した業界です。これらのユース・ケースでは、多くの場合、社内組織またはパートナー組織に「IT-as-a-Service」が提供されます。OpenStack コミュニティーは、Telco 固有の拡張機能を開発中です (Network Function Virtualization (NFV) など)。

      アプリケーション開発クラウド: OpenStack のもう 1 つの一般的なユース・ケースは、アプリケーション開発クラウドでの継続的インテグレーション/継続的開発 (CI/CD) の有効化です。この場合、OpenStack クラウドによってビジネスの俊敏性が実現し、OpenStack クラウドで基幹業務 (LOB) アプリケーションを実行することで、その信頼性が向上します。これらの LOB アプリケーションおよびワークロードには、多くの場合、いずれかの種類のソース・コード・リポジトリー (github.com など)、テスト・ハーネス (Jenkins など)、バグ・データベース (Atlassian など)、およびデプロイメント・ツールが含まれます。

      サード・パーティー・ツールと OpenStack の併用

      OpenStack は、クラウドの運用、開発、および管理を目的とした一般的なサード・パーティー・ツールとスムーズに統合できます。サード・パーティー・ツールには、OpenStack API 用のソフトウェア開発キット (SDK)、オーケストレーション・ツール、モニタリング・ツール、ロギング・ツール、プロビジョニング・ツールなどがあります。実際のクラウド・デプロイメントでは、いくつかの他のオープン・ソース・ツールを使用せざるを得ません。このセクションでは、OpenStack と併用されることが多いツールのタイプをいくつか示します。ただし、専用の OpenStack ディストリビューションの中には、このようなツールを必要としないものもあります。これは、多くの場合、そのディストリビューションが同等のカスタム機能を備えているためです。

      SDK: OpenStack には、データ・センター内のコンピュート、ストレージ、およびネットワーキングの各リソースを管理するための方法が 2 つあります。1 つはダッシュボード、もう 1 つはサービス・エンドポイントです。OpenStack のサービス・エンドポイントは、プログラマブル・インターフェース (REST API) を通じて公開されます。ユーザーは、いずれかの REST クライアント (OpenStack プロジェクトの一部として提供されるクライアント・ツールなど) を通じて、これらの API にアクセスできます。これらの API を操作するための SDK も、さまざまなプログラミング言語用に用意されています。

      OpenStack クライアント・ツール: OpenStack プロジェクトには、ダッシュボード、コンピュート、オブジェクト・ストレージ、ID サービス、イメージ・サービスなどのさまざまなプロジェクト用に、公式クライアントがあります。また、これ以外のプロジェクトについても、追加のクライアントを開発中です。

      OpenStack SDK: Python 用の公式の OpenStack SDK が提供されています。また、C、C++、Java .NET などのさまざまなプログラミング言語用に、非公式の SDK も提供されています。

      オーケストレーション・ツール: OpenStack には独自のオーケストレーション・サービス (Heat) がありますが、多くのデプロイメントでは、Chef、Puppet、Ansible など、他の一般的なオーケストレーション・ツールが使用されています。

      Chef: Chef は、Chef (旧称 Opscode) 社が提供する一般的な構成管理ツールです。OpenStack デプロイメントで多く使用されています。Chef ではクライアント/サーバー・アーキテクチャーが採用されています。OpenStack と Chef のデプロイを初めて行うユーザー向けに、公式の Cookbook が提供されています。

      Puppet: Puppet は、PuppetLabs 社が提供するもう 1 つの一般的な構成管理ツールです。Puppet でもクライアント/サーバー・アーキテクチャーが採用されています。大規模な OpenStack クラウドをデプロイする際に役立つ公式の Puppet モジュールが提供されています。

      Ansible: Ansible ソフトウェアは、エージェントレスの構成管理ツールです。OpenStack を対象とした公式の Ansible Playbook は提供されていませんが、これらの Blue Box Playbook を使用してデプロイを始めることができます。

      モニタリング・ツール: 大規模な OpenStack クラウドを運用する際には、さまざまなコンポーネント (コンピュート・リソース、仮想マシン、仮想ネットワークなど) を頻繁にモニターする必要があります。モニタリングを行うことで、コンピュート・ノードが最大 CPU 使用率に近付いたときなどに、管理者が事前にアラートを生成できるようになります。OpenStack には、イベント・ベースのインフラストラクチャーを提供することでモニタリングを可能にするテレメトリー・サービス (Ceilometer) が用意されていますが、このサービスは完全なモニタリング機能を提供するわけではありません。このため、OpenStack クラウド・デプロイメントでは、Nagios など他のオープン・ソースのモニタリング・ツールが使用されることが少なくありません。

      ロギング・ツール: 高度なログ管理機能を実現するために、LogStash などのオープン・ソース・ツールを採用することができます。LogStash には、ログの記録、解析、および保管を一元的に行うための機能が用意されています。このツールを ElasticSearch などの検索ツールと組み合わせて使用することで、OpenStack クラウドを対象とした強力かつスケーラブルなログ管理ソリューションを実現できます。

      プロビジョニング: OpenStack クラウドをデプロイするには、コントローラー・ノードまたはリソース・ノードとして、ベア・メタル・サーバーをプロビジョンする必要があります。プロビジョニングによく使用されるツールとしては、Chef、Puppet、Ubuntu MaaS、Crowbar などが挙げられます。OpenStack 独自のベア・メタル・サービスである Ironic も選択可能なオプションです。

      クラウドの用語集

      アクセス・キー/秘密鍵: ペアで使用される 2 つの自動生成暗号化鍵。コンピュート・インスタンスにアクセスして通信を行うために組み合わせて使用されます。秘密鍵は各要求に電子署名するために使用され、アクセス・キーはその署名を検証するために使用されます。

      アベイラビリティー・ゾーン: クラウド内の隔離された一連のハイパーバイザー。アベイラビリティー・ゾーンにより、仮想マシンのフォールト・トレランスが確保されます。OpenStack では、論理的分離またはサービスのクラスター化のいずれかが可能なアベイラビリティー・ゾーンがサポートされています。これらのゾーンが作成される一般的な理由は、電源へのアクセス確保や地理上の理由です。ユーザーは新規インスタンスの起動時にゾーンを選択できます。

      ブロック・ストレージ: ボリューム、ボリュームのスナップショット、およびボリュームのタイプをサポートするストレージのタイプ。ブロック・ストレージのボリュームは、実行中のインスタンスに直接接続されないブロック・デバイスです。一度に接続できるインスタンスは 1 つで、そのインスタンス自体が障害を起こした場合やシャットダウンした場合でも存続します。通常、OpenStack ボリュームは既存の SAN に接続されるため、API を使用してこれらのボリュームと対話できます。

      コンテナー: オブジェクトをオブジェクト・ストレージ内に整理して保管します。OpenStack では、Docker などのサード・パーティー・ソフトウェアがコンテナーとして使用されます。

      固定/静的 IP: VM インスタンスがリブートするたびに同じ VM インスタンスに関連付けられる IP アドレス。このタイプのアドレスは、クラウドでは VM 管理用に使用されます。通常はユーザーがアクセスすることはできません。

      フレーバー: 仮想マシン・イメージを表す一連のパラメーター。パラメーターには、CPU、メモリー・サイズ、ストレージ・サイズなどがあります。

      浮動 IP: VM インスタンスがリブートするたびに同じパブリック IP アドレスが割り当てられるように、その VM インスタンスに関連付けられる IP アドレス。浮動 IP は、アクセス可能なパブリック IP アドレスを作成し、トラフィックをそのアドレスから構成済みのインスタンスに誘導します。インスタンスの一部として IP が実際に作成されるわけではなく、クラウドが間接的な関係を提供します。まずアドレスが提供されたプールから割り振られる必要があり、そのアドレスをインスタンスに割り当てることができます。

      ハイパーバイザー: 実際の基盤となっているハードウェアに対する VM アクセスを調整し制御するソフトウェア。特定のインスタンスでプロビジョニングされる物理的なコンピュート・ノードを表します。

      イメージ: サーバーを作成または再構築する際に使用する必要のある、特定のオペレーティング・システム用のファイルのコレクション。OpenStack では複数のイメージ形式 (AMI、VMDK、VHD、QEMU) がサポートされています。例えば、サーバー・イメージを使用すると、後から使用できる構成済みのイメージを作成できます。これらのイメージを使用すると、新規サーバー・インスタンスを起動できます。これらのイメージは、イメージをリストすると表示されます。

      インスタンス: 仮想マシン。インスタンスを起動する (つまり「立ち上げる」) には、最低限の情報 (モデルとして機能するディスク・イメージ、インスタンスのフレーバー (つまりサイズ)、インスタンス名 (「MyInstance4032」など)) が必要です。 ほとんどの OpenStack 環境では、インスタンスの起動時にネットワーク・ドライブへの参照も指定する必要があります。

      メタデータ: インスタンスのメタデータは、インスタンスのソートやカタログ化を行う場合に便利です。メタデータは、インスタンスごとのフリー・フォームのキー/値ストアです。

      NAT (ネットワーク・アドレス変換): クラウド内であるタイプの IP アドレスを別のタイプの ID アドレスに変換するためのプロトコル。

      ネットワーク: 仮想ネットワークは、エンティティー間 (クラウド内のインスタンスなど) の接続を提供するレイヤー 2 ネットワークです。

      オブジェクト: オブジェクトは、オブジェクト・ストレージに保管される何らかのデータで、任意の形式 (ファイル、音楽、ビデオ、バイナリーなど) を取ることができます。

      オブジェクト・ストレージ: 結果的に整合性を取ることができる冗長な非構造化データをサポートするストレージのタイプ。OpenStack のオブジェクト・ストレージは、従来のファイル・システムとは異なり、仮想マシン・イメージ、フォト・ストレージ、E メール・ストレージ、バックアップ、アーカイブなどの静的データを対象とした分散型ストレージ・システムです。「ルート」ディレクトリー、中心的な「頭脳」、主要な制御点などは存在しないため、より優れたスケーラビリティー、冗長性、および耐久性を実現できます。データはオブジェクトによって表され、オブジェクトはコンテナーに保管されます。

      プロジェクト/テナント: プロジェクトはユーザーの論理グループで、一般にテナントとも呼ばれます。

      割り当て量: プロジェクトごとに設定される、コンピュート・リソースとストレージ・リソースに対するリソースの制限。通常、マルチテナント管理クラウドでは、各テナントの割り当て量がそれぞれの契約内容によって決められます。割り当て量はプロジェクトまたはクラスごとに設定可能で、RAM、IP アドレス、コア数、ストレージ・ボリュームのサイズなどのリソースの消費を制限することを目的としています。OpenStack ではリソースの消費に対して割り当て量を適用できます。

      役割: 役割は一連の権限と特権で構成されます。特定の役割を割り当てられたユーザーは、その役割に関連付けられた権限と特権を継承します。

      セキュリティー・グループ: コンピュート・インスタンスに適用される一連のフィルタリング・ルール。セキュリティー・グループは、システムのグループに対するファイアウォールのようなものです。他のグループまたは CIDR ブロックを参照するアクセス・ルールを指定できます。これらのグループは、インスタンスへのアクセス権限を付与するために、そのインスタンスに関連付けることができます。

      サーバー: 仮想マシン (VM) インスタンスを指す一般用語。仮想マシン・インスタンスを表すもう 1 つの一般用語はホストです。

      サービス: OpenStack サービスは、ユーザーがリソースにアクセスして制御するための経路となるエンドポイントを提供します。

      SSH 鍵: SSH 鍵は、OpenStack で起動されたインスタンスに (ssh コマンドを使用して) セキュアに接続するために使用されます。OpenStack には、SSH 鍵ペアの公開鍵を、新しくブートされたインスタンスの許可された鍵ファイルに挿入するための方法が組み込まれています。Nova がユーザーに代わって秘密鍵を作成することや、ユーザーが既存の鍵ペアの公開鍵をアップロードすることもできます。

      スナップショット: ストレージ・ボリュームまたはイメージの特定時点におけるコピー。スナップショットは、機能的にはイメージと同じです。

      ユーザー: ユーザーはクラウド・リソースを消費します。各ユーザーは 1 つまたは複数のプロジェクトのメンバーです。プロジェクトの中で、ユーザーはインスタンスやボリュームなどのリソースを作成および管理します。

      仮想マシン (VM): OpenStack クラウド内で実行されるインスタンス。VM を表す別の用語としてサーバーまたはホストがあります。

      ボリューム: 永続的または一時的 (リブート時に失われる) なディスク・ベースのデータ・ストレージ。このストレージは、多くの場合、拡張属性をサポートするファイル・システムを持つ iSCSI ターゲットとして表されます。ボリュームをサーバーに接続するときに、インスタンス内でシステムがデバイス ID を自動的に割り当てるようにすることで、新規ボリュームに接続することができます。ユーザーがデバイス ID を指定することもできます。新規ボリュームが接続されたら、ボリュームを使用するためにそのボリュームをフォーマットする必要があります。以降にそのボリュームを使用するときは、フォーマットを行う必要はありません。
