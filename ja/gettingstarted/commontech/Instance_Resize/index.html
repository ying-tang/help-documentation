      インスタンスのサイズ変更 (非ライブ)

      • by Jesse Keating, Sina Sadeghi ● May 23, 2016

      • Tags:

        • はじめに

        • インスタンス

        • サイズ変更

        • 移行

        • ボリューム

      インスタンスをサイズ変更するにはどうすればいいですか?

      機能: インスタンスのサイズ変更 (アップ)

      説明: Bluemix Private Cloud の 1.3 リリース以降では、インスタンスのサイズ変更 (アップ) がサポートされています。サイズ変更の操作は稼働中に実行できないため、インスタンスは停止し再起動されます。サイズ変更は同じホスト上で実行でき、小規模環境で役立ちます。

      インスタンスのサイズ変更では、インスタンスのフレーバーがより大きいフレーバーに変更されます。インスタンスは、より大きいフレーバーにのみサイズ変更できます。この変更では、インスタンスが停止し、大きいサイズのフレーバーに移行して再び開始される間に、わずかな中断が発生します。このサイズ変更では、インスタンスが新しいハイパーバイザーに再配置されることもあります。

      サイズ変更処理の進行中にターゲット・フレーバー (インスタンスのサイズ変更後のフレーバー) を更新しないでください。更新すると、サイズ変更操作が失敗してインスタンスがエラー状態になるおそれがあります。

      大きいインスタンスのサイズ変更は、新しいディスク・イメージ用に十分なディスク・スペースがない場合に障害を引き起こすおそれがあるため、注意が必要です。

      注: インスタンスがサイズ変更または移行されて VERIFY_RESIZE 状態になると、ネットワークが再開されるまでに短い遅延が発生することがあります。

      詳しくは、サイズ変更の実行方法に関する資料 (http://docs.openstack.org/user-guide/enduser/cli_change_the_size_of_your_server.html) を参照してください。

      トラブルシューティング

      インスタンスが大きく、サイズ変更操作に非常に長い時間がかかる場合 (例えば一部のログによると、09:30 に操作が開始して 14:41 に認証エラーが発生しています)、この間に、この操作を完了するために Nova によって使用される認証トークンが実際に期限切れになることがあります。

      高い確率で、新しいサイズ変更を開始しようとすると同じエラーが再び発生します。その場合は別のアプローチを推奨します。クラスターでブロック・ストレージが使用可能になっている場合は、以下のステップを推奨します。

      1. ターゲット VM をシャットダウンします。
      2. nova image-create を使用してターゲット VM のスナップショットを作成します。
      3. cinder create --image-id <SNAPSHOT UUID> を使用して、そのスナップショットからボリュームを作成します。
      4. OpenStack の image delete または glance image-delete を使用してスナップショットを削除します。
      5. nova boot --boot-volume を使用してボリュームから新規 VM をブートします。

      同じ IP アドレスを保持したい場合は、以下のようにします。

      1. neutron port-list | grep <IP of new VM> を使用して、ボリュームでブートされた VM の UUID を取得します (最初の列がポート UUID です)。

      2. nova interface-detach を使用して、古い VM からポートを削除します。

      3. neutron port-update --fixed-ip ip_address=<old VM IP> <port UUID> を使用して、古い VM IP を使用するように新しい VM 上でポートを更新します。

      この切り替えが正常に完了したら、nova delete を使用して古い VM をクリーンアップできます。

      このプロセスは、おそらくユーザーが好むプロセスに比べると複雑です。しかし、ゲスト・バックエンドとしてブロック・ストレージを切り替えることには以下の複数の利点があります。

      • ディスクのサイズ変更がスキップされるため、フレーバーのサイズ変更のスピードが大幅に向上します。
      • ブロック・ストレージを使用すると、新しいハイパーバイザーへの移行が非常に高速になります (新しいハイパーバイザーは単にブロック・ストレージ・クラスターに接続するだけです)。
      • cinder extend コマンドを使用して VM のディスクを停止せずに拡張できます。

      ブロック・ストレージがない場合により小さいフレーバーまたはより大きいフレーバーにサイズ変更したい場合、必要なサイズのフレーバーで新規インスタンスを作成し、古いインスタンスから新しいインスタンスに対して必要なデータの rsync を実行することができます。

      OpenStack のインスタンスのサイズ変更でサイズを下げることができないのはなぜですか?

      理由については、https://blueprints.launchpad.net/nova/+spec/no-downward-resize を参照してください。実質的には、サイズを下げる操作はデータ保全性に影響を及ぼすおそれがあるため、許可されません。安全な回避策として、より小さいサイズのフレーバーで新しい VM を作成し、大きいインスタンスから小さいインスタンスに対して必要なデータの rsync を実行することができます。

      ボリュームからブートされるインスタンスのサイズ変更を行ったとき、フレーバーは変更されますがサイズは同じままです。これはなぜですか?

      これは予期された動作です。フレーバーのディスク・サイズは、ボリュームからブートするときのルート・ディスクのディスク・サイズではありません。この無効な OpenStack バグは、https://bugs.launchpad.net/nova/+bug/1412993 で確認できます。ここでは、このバグについてより詳細に説明されています。

      ディスクのサイズ変更を行うには、https://docs.openstack.org/user-guide/dashboard-manage-volumes.html#edit-a-volume を参照してボリュームのサイズ変更を行ってください。何らかの理由でこれが機能しない場合、https://mcwhirter.com.au/craige/blog/2014/Resizing_a_Root_Volume_for_an_Openstack_Instance/ に記載されているステップを試すこともできます。
