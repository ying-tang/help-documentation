      既存のインスタンスのサイズをどのように変更できますか？

      Horizon には「インスタンスのサイズ変更 (Resize Instance)」オプションが表示されますが、この機能は現在サポートされていません。インスタンスのサイズを拡張する必要がある場合、推奨される経路は、インスタンスのスナップショットを作成し、そのスナップショットを新規の大きいフレーバーに再デプロイすることです。

      インスタンスを特定のホスト上でどのように開始できますか?

      この機能は OpenStack API を介してサポートされています。詳しくは、インスタンスを特定のハイパーバイザー・ノードにデプロイする方法を参照してください。

      新しいクラウドに 2 つのプロジェクトが既に存在します。これらは何ですか?

      admin プロジェクトおよび service プロジェクトは、IBM Bluemix Private Cloud の基盤となる操作をサポートするために作成されます。これらのプロジェクトはそのまま残しておく必要があります。それらを変更したりデプロイしたりしないでください。

      Bluemix Private Cloud で誰でもアクセス可能なポート/URL などは何ですか? また、それらの目的は何ですか?

      ファイアウォールを介して Bluemix Private Cloud からアクセス可能なポートは、以下のとおりです。

      80 - horizon (OpenStack Web ポータル)
443 - horizon (OpenStack Web ポータル)
5000 - keystone (OpenStack 認証 API)
6080 - nova-novnc (OpenStack VNC コンソール)
8000 - heat (OpenStack CloudFormation API)
8004 - heat (OpenStack オーケストレーション API)
8774 - nova (OpenStack コンピュート API)
8776 - cinder (OpenStack ブロック・ストレージ API)
9292 - glance (OpenStack イメージ API)
9696 - neutron (OpenStack ネットワーク API)
35357 - keystone-admin (OpenStack 認証管理 API)

      Swift (OpenStack オブジェクト・ストレージ) がクラスター上で構成されるように要求した場合、3 つの追加ポートが有効になります (6000、6001、6002)。

      これらのポートはすべて、カスタムの Horizon URL 経由で利用できます。

      また、出力ポートを必要最小限に制限する必要がある場合は、少なくとも DHCP に対してポート 67/68、メタデータに対してポート 80 を開く必要があります。

      インスタンス・ドライブまたはボリューム・ドライブの IOPS (IO Operations Per Second) をどのように測定できますか?

      ディスク・パフォーマンスの測定には、ツールをいくつ使用しても構いません。

      非常に使いやすい IOPS 測定用の Python スクリプトが Benjamin Schweizer 氏によって開発されています。手順と結果の例が以下に記載されています。

      https://benjamin-schweizer.de/measuring-disk-io-performance.html

      このスクリプトの最新バージョンは、現在ここから使用可能です。

      使用方法は以下のとおりです。

      iops [-n|--num_threads threads][-t|--time time] <device>

      パラメーター:

      threads := number of concurrent io threads, default 1
time    := time in seconds, default 10
device  := some block device, like /dev/sda or \\\\.\\PhysicalDrive0

      例:

      iops /dev/sda
iops -n 8 -t 2 /dev/disk0
iops --num_threads 1 --time 2 /dev/md1
iops --num_threads 16 --time 2 /dev/md1

      セキュリティー・グループを使用しているとき、禁止されたポートへの TCP 接続の拒否を記録するにはどうすればよいですか?

      禁止されたポートへの TCP 接続の拒否を記録するには、セキュリティー・グループを開き、インスタンスにログオンして iptables を実行する必要があります。

      仮想マシンに接続できる仮想デバイス (ポートなど) の最大数はいくつですか?

      使用可能なな仮想 PCI スロットの最大数は 20 です。

      すべての VM とそれらのホスト名のリストをどのように取得できますか?

      クラウド上のすべてのインスタンスとそれらを稼働するホストのリストが必要ですか? VM の名前も必要ですか? ユーザーが cloud_admin の場合、以下の 1 行の文を使用できます。

      nova list --all-tenants --fields name,OS-EXT-SRV-ATTR:host

      コマンド openstack host show disk name は物理ホストの実際のハード・ディスク使用量を表示しますか?

      いいえ。これはホストで実行されるインスタンスのフレーバーに関する情報を表示します。現時点では、スワップ使用量の情報は含まれていません。以下のセクションでは、Mitaka リリース時点で含まれる内容を示しています。

      https://derpops.bike/2016/10/07/openstack-nova-and-hypervisor-disk-consumption/ も参照してください。

      この質問に答えるには、OpenStack の資料 http://docs.openstack.org/admin-guide/common/nova-show-usage-statistics-for-hosts-instances.html を参照する必要があります。

      • [openstack host show 出力の] disk_gb 列には、ホスト上で実行されるインスタンスのルート・ディスクおよび一時ディスクのサイズの合計 (GB 単位) が表示されます。[この値] は、ホストで実行されるインスタンスのフレーバーに関する情報を使用して計算されます。このコマンドは、物理ホストの … ハード・ディスク使用量を照会しません。
      • 実際のフレーバー計算は、https://github.com/openstack/nova/blob/c521ba7fa679b6e9c790d84ea22f04fb999987db/nova/notifications/base.py#L448-L454 で確認できます。disk_gb = root_gb + ephemeral_gb
      • ephemeral_gb = セカンダリーの一時データ・ディスクのサイズを指定します。これはインスタンスの存続期間中のみ存在する、フォーマットされていない空のディスクです。デフォルト値は 0 です。
      • root_gb = root (/) パーティション用に使用するディスク・スペースの量 (ギガバイト単位)。
      • Nova の disk_gb にはスワップ情報は現在含まれていません。(つまり、インスタンスのスワップ・スペース割り振りはオプションです。デフォルト値は 0 です。)

      次の仕様書では、開始される OpenStack Newton リリースについての、このオプションのスワップ・スペースを含む作業が提案されています。https://specs.openstack.org/openstack/nova-specs/specs/newton/implemented/resource-providers-allocations.html:

      「コンピュート・ノードがインスタンス・ディスク用にローカル・ストレージを使用するか、ボリュームからブートされた場合、使用される値は、フレーバーの root_gb、ephemeral_gb、および swap フィールド値の合計である必要があります。resource_provider_uuid はコンピュート・ノードの UUID である必要があります。注: ボリュームからブートされたインスタンスは、root_gb 値は 0 になります。”

      スナップショットからボリュームを作成し、依存関係を削除するにはどうすればよいですか?

      スナップショットからボリュームを作成する際に、新しいボリュームで必要とされる依存データ (メタデータ) を、元のスナップショットが保持している可能性があります。openstack volume show $volumeid を実行することで、親の snapshot_id を見つけることができます。その後、そのボリュームを削除しようとすると、依存するスナップショットを先に削除する必要があることによる以下のエラーに遭遇する場合があります。

      次の名前または ID を持つスナップショットの削除に失敗しました: ... (Failed to delete snapshot with name or ID ...)

      これを回避するには、次のようにしてボリュームからイメージを作成し、そのイメージからボリュームを作成します。

      ステップ 1: 以下を実行してボリュームから一時イメージを作成します:

      openstack image create --volume $volumeid $image_name

      ステップ 2: 以下を実行して一時イメージからボリュームを作成します。

      openstack volume create --image $imageID --size $x

      その後一時イメージを削除して、独立したボリュームを手にすることができます。
