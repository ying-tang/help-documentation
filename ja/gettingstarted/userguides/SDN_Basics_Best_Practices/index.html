      クラウド・ネットワークとソフトウェア定義ネットワーク -- ベスト・プラクティス解説ホワイト・ペーパー

      • by Dustin Lundquist and Leslie Lundquist ● June 7, 2016

      • Tags:

        • sdn

        • ネットワーク

        • ベスト・プラクティス

        • vlan

        • vxlan

        • l2pop

        • arp

        • プロビジョニング

      概要:

      多くの「エンタープライズ」データ・センター・ネットワークや次世代ワイヤレス・プロバイダー・ネットワークにおいて、完全分散型ネットワーク制御から、ソフトウェア定義ネットワーク (SDN) を使用した集中ネットワーク制御のために設計されたネットワーク環境への移行が進んでいます。この移行は、ルーターやスイッチによる分散型ローカル学習と負荷の高いネットワーク・トラフィックが複合したことでネットワークの動作が複雑化したことに対応するためのものです。

      このような複雑化により、私たちが焦点を当てているクラウド・ネットワーク技術に影響が及んでいます。エンジニアは仮想的なオーバーレイ・ネットワークと格闘し、下位ハードウェア・ネットワークの実装の詳細に依存するため、実動クラウド環境での動作が予測できなくなっています。それでも、この問題にも解決策があります。

      ソフトウェア定義ネットワークの選択肢の中には、実現が容易に思えるものの、後から困難に直面する可能性があるものがあります。この記事では、その理由と背景について説明します。シンプルなネットワーク・ハードウェアおよびソフトウェアの既知の動作原理に基づいて、安定的かつ動作が予測可能なネットワーク環境を構築するための代替手段をいくつか紹介します。全体としては、SDN に最適な分野について検討した後、SDN との併用に適した技術を選択する方法について説明していきます。

      第 2 層の背景: イーサネット・スイッチ

      通信量の多いネットワーク環境が急激に複雑化してしまう背景について、まずイーサネット・スイッチの設計と動作原理から説明していきます。ここでは、スイッチの際立った特徴のみを取り上げます。最初のポイントは、イーサネット・スイッチが第 2 層 (L2) のプロトコル・レベルで動作すること、つまり IP アドレスではなく、MAC アドレスを処理することです。また、L2 ではデータグラムを「フレーム」というのに対して、第 3 層 (L3) では「パケット」といいます。説明を明確にするために、以後、一貫してこの用語を使用します。

      まず、スイッチがフレームを受信するたびに行われる処理を簡単に説明していきます。これにより、複雑化に伴って何が起こるかが分かります。フレームを受信すると、

      1. フレームのソース・アドレスが学習され、ポート情報が格納されます。

      2. 宛先アドレスが、既知のアドレスのリストと照合されます。

      3. 宛先がリストに見つかった場合は、関連するポートにフレームを送信します。

      4. 宛先アドレスが見つからない場合は、スイッチに認識されているすべてのポート (ただし進入ポートを除く) にフレームを送信します。この処理をフラッディングといいます。フレームは必要な回数だけ複製され、既知のすべてのポートにコピーが送信されます。

      この方法はごく基本的であり、多くの、それこそ何百台ものスイッチをツリー状に接続している場合であっても、ツリーにループがない限りは正常に機能します。「スパンニング・ツリー・プロトコル」というプロトコルはループを検出して予防し、フレームが際限なく循環することを防ぎます。(これらを BPDU (ブリッジ・プロトコル・データ・ユニット) フレームといいます。) これらのフレームは、輻輳 (ふくそう) やリンク・エラー (チェックサム無効など) のためにドロップされる場合があります。

      この処理の追加要素として、事前に設定された時間 (通常は 5 分から 4 時間) にわたってフレームを送受信しなかった場合、個々のスイッチにおいて既知のアドレスが「存続期間切れ」となります。新しいフレームの受信や古いアドレスの存続期間切れにより、すべてのスイッチが常に同じ状態にあるとは限らないことが分かります。既知のアドレスのリストはそれぞれ少しずつ異なります。これも設計の 1 つの側面です。

      輻輳がある場合は、L2 から複雑化が始まる可能性があります。原因は通常、大量の新規フレームの着信やフラッディングです。例として、1 つの新規フレームがツリー階層を上がっていくため、ツリーの他の部分のスイッチがその MAC アドレスを送信元として学習しない場合について考えてみます。同じ送信元からのフレームが別のスイッチに届くと、未知のアドレスであるため、フラッディングが発生します。フラッディング処理のために多くのスイッチがビジーになるため、別のフレームが着信した際に、その送信元 MAC アドレスをまだ処理したことがないツリー内の別のスイッチに着信することになります。結果として、さらにフラッディングが発生します。この連鎖は、ほとんど (またはすべて) のスイッチが新規フレームの送信元 MAC アドレスを学習 (または再学習) し、フラッディング・トラフィックが収束するまで続きます。

      その間、パフォーマンスが低下し、他の問題が発生する可能性があります。

      L3 の背景: ARP メッセージ

      アドレス解決プロトコル (ARP) は第 3 層 (L3) で機能し、IP アドレスを使用して、サブネット上の隣接するノードとの間でパケットをルーティングします。

      ARP が解決する課題は、隣接するノードにどのようにパケットを送信するかです。(つまり、同じサブネットにあるホストを識別して、そのホストに送信するための仕組みです。)

      脚注: この記事の趣旨から、サブネットとは、特定の IP アドレスのネットワーク・マスク部分 (XX.XX.XX.me) を共有するホストのグループを指すものとします。サブネット外のアドレスに到達するためには、ゲートウェイとして機能するいずれかのホストにパケットを送信します。このゲートウェイを通じて、より大きなネットワークに接続します。

      ARP は実に単純明快です。使用するメッセージは以下の例に示す 4 種類です。

      1. 192.0.2.10 はどこに割り当てられているか (要求)
      2. 192.0.2.10 はここに割り当てられている (応答)
      3. X には 192.0.2.10 が割り当てられている (プロキシーとしての応答)
      4. プローブ: このサブネット上のすべてのアドレスを報告せよ

      Gratuitous ARP メッセージはブロードキャストされる場合がありますが、通常、ARP 応答は要求側にユニキャストされます。プロキシー ARP はルーティングと併用します。プロキシーは、リモート・ホストへの送信が可能であるため、リモート・ホストのエントリーに代わって応答します。プロキシーは、そのホストの IP アドレスを持っています。

      ホストが自身で、または他のホストの代理で応答することができるほか、ARP プローブの場合もあるため、ARP メッセージは 4 種類あります。ARP 要求は、すべてイーサネットのブロードキャスト・アドレス (FF:FF:FF:FF:FF:FF。すべて F、2 桁が 6 グループ) に送信されます。イーサネット・サブネット上の任意のホストが、特定の IP アドレスを持っていることや、その関連ホストが特定の IP アドレスを持つことなどを応答できます。

      そのため、すでにお気付きかもしれませんが、ARP にセキュリティーは備わっていません。おそらく、サブネットの隣接するノードは信用できるでしょう。ホストのパケットがルーターへの最初のホップをセキュアに通過可能であることに関連し、一般にこの問題を「ファースト・ホップ・セキュリティー」といいます。

      前のセクションで説明したように、イーサネット・スイッチ・プロトコルでは、サブネット内でフレームを配信する際にホストの IP アドレス (L3) ではなく、MAC アドレス (L2) を使用します。そのため、各ホストには、サブネット上の通信先ローカル隣接ノードを対象とした、MAC アドレス (L2) から IP アドレス (L3) へのマッピング・テーブルが必要です。

      実際のマッピングは 2 レベルになっています。この様子を次の略図に示します。

      IP      o    o     o
          \ /      |
MAC        o       o
           |       |
ポート     o       o

      MAC アドレスはポートに 1 対 1 でマップされますが、MAC アドレスから IP アドレスへのマップは 1 対多になることがあります。(このマッピングの特徴はクラウド・デプロイメントでは一般的であり、VM が内部 IP や外部 IP などの情報を、接続先のネットワークごとに保持できます。)

      通信量の多いサブネットでは、多数の ARP 要求が出される可能性があります。各ホストが隣接ノードのアドレスを保持する時間は、最長でもわずか 5 分です。(つまり、L2 のイーサネット・スイッチで MAC アドレスの存続期間が切れるのと同様に、L3 ではサブネットの隣接ノードで存続期間が切れます。) 実際に、ARP とイーサネット・スイッチ・ポートで存続期間が同じ値に設定されていると、サブネットで奇妙な現象が発生します。一方の値を長くしなければ、最終的にはネットワーク・トラフィックがすべてブロードキャストになってしまうのです。

      L2 と L3 の連携: SDN を採用する理由

      これらのイーサネット・スイッチおよび ARP 要求のシステムは、ブロードキャスト・トラフィックが多すぎない場合であれば、つまりネットワークで対話を開始する MAC アドレスが多すぎてスイッチがすべてのアドレスをテーブルに収容できない事態でなければ、十分に機能します。(多くのスイッチで MAC アドレスのスロットが不足すると、フラッディングが始まります。) この制約に対処するために、多くのデータ・センターの設計者は、ローカル・ネットワークを小規模な L2 セグメントに分割し、それらをルーターで接続しています。つまり、問題を小規模な L2 ドメインに分割することで克服しています。

      この手法を階層型ルーティングといい、先ほどの問題を log N 個の小さな問題に分解することができます。

      それでも、多くのお客様はすべてのホストを同じ L2 ドメインに含めるように求めます。サブネットがお客様側のデータ・センターに存在する場合は、その構成が自然だからです。そのため、多くのクラウド・ソリューションでは、オーバーレイ・ネットワーク (プロバイダーの L3 インフラストラクチャー上にお客様の L2 ネットワークを構築するための技術) を利用してホストをサブネット・ドメインに結合しています。

      例えば、VxLAN はオーバーレイの一種であり、イーサネット・フレームを UDP パケットの中にカプセル化して伝送します。VXLAN オーバーレイ・ネットワークに参加する各ホストには、VTEP (VxLAN トンネル・エンドポイント) があります。VTEP は、UDP ポートと IP アドレスを持つリスニング UDP ソケットです。

      この状態は、SoftLayer データ・センターの多くの IBM Bluemix Private Cloud デプロイメントに当てはまります。SoftLayer はローカライズされたハードウェア「ポッド」にデータ・センターを編成するため、ソフトウェア定義オーバーレイ・ネットワーク (SDN) を導入して、同じハードウェア・ポッド内で、確立されていないお客様のクラウド・デプロイメントを接続する必要があります。IBM Bluemix Private Cloud 3.0 以降では、多くのソフトウェア定義ネットワーク・ソリューションを実装し、お客様がクラウドを相互に接続して多くのハイブリッド・クラウド・ソリューションを構築できるようになっています。(ここでのハイブリッド・クラウドは、複数のプライベートまたはパブリックのコンピューター・クラスターを相互に結合して、それらのクラスターが仮想マシンと関連ネットワークの集合を形成するように設計されていること、つまり複数のクラウド環境を結合することを指しています。)

      **プロビジョニング **

      一度にデプロイできる VLAN および VxLAN の数の限界について質問を受けることがあります。ここでは、プロビジョニングの基本について少し説明します。

      VLAN はスイッチ・ファブリック全体でプロビジョニングされるため、ファブリック内のすべてのスイッチ (おそらく数百台) が 4096 個の VLAN にそれぞれアクセスできます。ただし、4097 番目の VLAN が必要になった場合は、完全に新しいスイッチ・ファブリックでプロビジョニングする必要があります。これはイーサネットの基本的な制約であり、当社のプラットフォームの制約ではありません。

      この制約を取り払うには VxLAN を使用します。VxLAN では、イーサネットの 4096 個の各 VLAN に、さらに 65,000 個の仮想 VLAN をプロビジョニングできます。そのため、1 つのファブリックでの VxLAN の最大数は 268,435,456 になります。要するに、通常 VLAN が枯渇することはありません。

      IBM Bluemix Private Cloud の標準構成の場合は、1 つのファブリックで 3000 以上のお客様をサポートでき、それぞれに 65,000 個の VxLAN を割り当てられます。

      Box Panel では、関連付けられたスイッチ・ファブリックを反映する名前で VLAN を追跡できます。ファブリックで VLAN の限界を超えたことは一度もないため、名前は単に VL### (VL101 など) にしています。ファブリックを区別するために、fabric1-VL101 や fabric2-VL101 などの名前にしてもよいでしょう。

      脚注: ソフトウェア定義ネットワーク (SDN) は、中央データベースに一連のポリシーを定義し、計算によってそのポリシーを自動的に適用する手段として開発されました。現在では、この用語はネットワーク・オーバーレイの利用を指す場合に一般的に使用されるようになっています。
