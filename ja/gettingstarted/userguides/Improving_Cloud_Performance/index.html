      OpenStack の構築 II -- クラウド・パフォーマンスの向上

      • by Dustin Lundquist and Leslie Lundquist ● November 21, 2016

      • Tags:

        • openstack

        • sr-iov

        • virtio

        • vhost

      https://admin.blogs.prd.ibm.event.ibm.com/blogs/bluemix/2016/11/building-openstack-ii-improving-cloud-performance/ から転載

      この記事では、シリーズの 2 回目として、クラウドの基礎を掘り下げ、QEMU の詳細と、クラウド全体のワークロード・パフォーマンスを向上させる方法について説明します。

      基本的に、クラウドのパフォーマンスを向上させるには、ネットワーク・デバイスおよびストレージ・デバイスにアクセスする際の入出力効率を向上させる必要があります。入出力の仮想化を最適化してゲスト VM とホストの間の入出力デバイス・インターフェースを効率化することで、入出力のパフォーマンスと柔軟性を向上させることができます。これにより、ワークロードに振り向けられる総合的なプロセッサーの処理能力が増すため、クラウド全体のパフォーマンスが向上します。

      仮想化によって入出力パフォーマンスを向上させるとはそういうことです。

      重要なことは、QEMU がクラウド VM の基本要素であることです。このソフトウェアが大きな役割を果たします。ただし、QEMU はエミュレーターであるため、速度が重要な場合は、できるだけ QEMU に依存しないようにすると効果的です。KVM によって QEMU のプロセッサーおよびメモリー・パフォーマンスが多少向上しますが、入出力パフォーマンスのボトルネックは残ります。

      仮想化による入出力パフォーマンスの向上には、ソフトウェアによる手法とハードウェアによる手法の 2 つがあります。このブログではそれぞれの手法について検討していきます。それぞれの手法において、各ステップを通じて、QEMU の処理を軽減していきます。これはなぜですか?エミュレーションが少ないほどパフォーマンスが向上します。

      ソフトウェアによる手法では、「従来の」デバイスおよびドライバーの仮想化の効率を改善します。ハードウェアによる手法では、より直接的にデバイスを VM にリンクします。そのために、SR-IOV (Single Root IO Virtualization) という機能を実装する特殊なハードウェアを使用する場合もあります。これは、基本的には、必要に応じてインスタンス化できる仮想化デバイスで高速に VM をペア化するためのハードウェア支援型の手法です。

      ソフトウェアによる手法: QEMU、virtio、および vhost

      以前のブログで触れましたが、QEMU はエミュレーターとして動作するように設計されました。Windows NT や OS/2 など、過去のオペレーティング・システムであってもエミュレートできます。オペレーティング・システムによって検出されることを前提とするハードウェアがすべてエミュレートされるため、公開されるデフォルトのデバイスは「共通の」物理ハードウェア・デバイスです。ただし、オペレーティング・システムがそれらのデバイスとの通信に使用する手段は、仮想化を想定して設計されていませんでした。

      旧来の入出力デバイスは通常、汎用プロセッサーではなく、低速でした。そのため、かつてハードウェアに依存していた頃は、入出力パフォーマンスを最適化してもメリットはほとんどありませんでした。例えば、物理ディスクではディスク上の特定の位置にアクセスするためにかなりの時間 (4ms) がかかるため、ドライバーとディスク・コントローラーの間の通信を最適化してもほとんどメリットはありません。最新のクラウドでは、サーバーとして VM を使用するため、入出力パフォーマンスを改善すると大きなメリットが得られます。このような改善は、よく知られた方法で実現できます。

      入出力の仮想化が効果的な理由

      効率的な入出力のためにクラウド・システムに必要なものは、(1) ゲスト VM 用のドライバー、(2) 入出力デバイスを提供するホスト・ソフトウェア (QEMU など) の追加のバックエンド、および (3) ホスト・ソフトウェアが下位の入出力デバイスとの通信に使用するインターフェースです。

      この図に示されているように、VM 内からの入出力要求を満たすためにいくつかのコンポーネントが連携しています。これらのコンポーネントの間を遷移するごとにパフォーマンスが低下します。入出力のパフォーマンスはあっという間に低下してしまいます。この様子は、動作を各段階に分解すると分かります。

      • 最初に、VM 内で実行されているプログラムが、「通常の」インターフェースを使用して VM のオペレーティング・システムに入出力要求を出します。(VM で実行されているソフトウェアが別の方法を使用する場合、ユーザーに対して透過的な仮想化にはなりません。)

      • 次に、VM のオペレーティング・システムが、QEMU によって公開されるデバイスに対応するドライバーを使用します。

      • ドライバーがなんらかの操作を実行します。この操作は KVM によってトラップされ、QEMU に渡されて実行されます。

      • その後、QEMU が、同様のインターフェースを使用してホスト・オペレーティング・システムに対する入出力要求を実行します。

      • さらに、ホスト・オペレーティング・システムが、物理的なハードウェアに対応するドライバーを使用して、要求された入出力操作を実行します。

      この過程を高速化するには、どのような方法があるでしょうか。良い方法が 2 つあります。1 つは、特定のデバイスのエミュレート手段をより効率的に仮想化する方法です。この方法は通常「ドライバーの準仮想化」といいます。代表的な例として virtio があり、OpenStack Nova に導入されています。2 つ目の方法は、パフォーマンスを向上させるために、VM からデバイスのエミュレーションを完全に取り除いて遷移をなくすことです。 この場合は、QEMU の「デバイス・エミュレーション」をゲスト VM から取り除き、ホスト・オペレーティング・システムに制限します。

      最適化 1: ドライバーの準仮想化: virtio による方法

      virtio は、VM のオペレーティング・システム内で、対応するドライバーとともに QEMU によって公開される仮想化デバイスのファミリーです。これらのドライバーは、既存のハードウェア・デバイスのインターフェースをただ忠実にエミュレートするのではなく、リング・バッファーなどの最新の効率的なデータ構造を使用してゲスト VM ドライバーと QEMU の間のインターフェースを最適化します。virtio を使用すると、VM と QEMU との通信が効率化されるため、入出力のパフォーマンスが向上します。

      ただし、virtio ではこれらの高速な virtio ドライバーをゲスト VM のイメージに組み込む必要があるため、旧来のイメージ (Windows 95 や 98 など) では virtio を利用できません。幸い、2013 年以降はほとんどのオペレーティング・システムにデフォルトで virtio ドライバーが組み込まれています。

      最適化 2: vhost (QEMU ユーザー・スペース・デバイス・モデルの除去)

      次のパフォーマンス最適化では、virtio デバイス・エミュレーション機能をすべて VM からホスト・オペレーティング・システムのカーネルに移行します。この移行により、KVM から QEMU、さらにホスト・カーネルへの遷移に伴うパフォーマンスの低下を回避します。

      ハードウェアによる手法: PCI パススルーと SR-IOV

      PCI パススルーは、すべての PCI デバイスに適用できる汎用的な仕組みです。前のセクションで説明した、デバイスとドライバーのソフトウェア仮想化のほかに、ここで説明する方法も有力です。virtio と vhost とは対照的に、PCI パススルーは、直接ゲスト VM にハードウェア入出力デバイスを公開するため、入出力の過程から QEMU およびホスト・カーネルが完全に取り除かれ、それらのソフトウェア・エミュレーション遷移に伴うパフォーマンス低下が解消されます。

      PCI パススルーは、多くの状況でパフォーマンスを向上させる優れた代替手段です。

      • 例えば、ビデオカードの PCI パススルーをセットアップすることで、このパススルー手法をゲームに使用できます。この場合は、ゲームのパフォーマンスを向上させるためではなく、他のオペレーティング・システム用のゲームを実行するために仮想化を使用します。最新のゲームの仮想化は、PCI パススルーを使用しなければ実現できません。

      • 最も一般的な PCI パススルー・デバイスは、おそらくネットワーク・カードです。例えば、vhost_net でネットワーク・カード・パススルーを実現できます。この場合、ネットワーク・データのパスから QEMU を取り除く一方で、vhost_net は引き続きホスト・カーネルの元のままのネットワーク・スタックを使用します。

      • 大部分のサーバーにはディスク・コントローラーが 1 つしかなく、そのコントローラーがサーバーのすべてのディスクに接続されるため、一般にホスト・バス・アダプター (HBA) およびディスクに PCI パススルーを使用するのは非現実的です。そのため、それらのディスクの容量をクラウド内の VM で共有することはできず、すべてのディスクが 1 つの VM に属することになります。次のセクションでは、SR-IOV で実現された HBA でこの問題が解決される仕組みについて説明します。

      • HBA に対する PCI パススルーの高度な適用例として、クラウドを USB からブートした上で、バックエンド・データベースとして実行される 1 つの VM に全ディスク・ストレージ容量を割り振れるようにする場合が挙げられます。

      重要な点は、PCI パススルーではゲスト VM ごとに専用の入出力デバイスが必要なことです。VM ごとに専用のハードウェアが必要なため、仮想化による柔軟性が損なわれ、コストも大幅に上がってしまいます。幸い、新しいタイプのハードウェアが登場しており、ハードウェア支援による、組み込まれた仮想化 PCI デバイスを提示することで、コストを低減できます。この技術を SR-IOV といいます。

      SR-IOV

      新しいハイエンドのネットワーク・インターフェースやディスク・コントローラーには SR-IOV (single root I/O virtualization) が組み込まれています。基本的に、これらのハードウェア・デバイスは複数の PCI デバイスのように動作することができ、各デバイスを 1 つの VM に接続することができます。標準的な実装では、メイン 1 つ、親 (PF) 1 つ、1 つ以上の論理デバイス (仮想機能単位 (VF)) で構成します。PF を構成することで、関連するリソースとともに追加の VF デバイスを作成できます。その上で、それらの VF を PCI パススルーによって個々の VM に接続することができます。

      可能性としては、各ゲスト VM で 2 つの入出力デバイス (ネットワーク・インターフェースとディスク・コントローラー) に PCI パススルーを使用することもできます。例えば、SR-IOV デバイスが 2 つあると、ネットワークとストレージの両方で入出力のパフォーマンスを向上させられます。

      要約

      この記事で説明したどの手法を使用しても (つまりソフトウェアによる手法でもハードウェアによる手法でも) クラウドのワークロード・パフォーマンスを向上させることができます。ソフトウェアによる最適化手法には virtio や vhost があり、ハードウェア支援型の最適化手法では PCI パススルーや SR-IOV を使用します。
