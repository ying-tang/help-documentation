      OpenStack の構築 — QEMU とクラウド

      • by Dustin Lundquist and Leslie Lundquist ● November 21, 2016

      • Tags:

        • qemu

        • kvm

        • ハイパーバイザー

        • エミュレーター

      https://www.ibm.com/blogs/bluemix/2016/10/building-up-to-openstack/ から転載

       先日のことですが、クラウドのカスタマー・サポートが危機に直面しているときに、新入りの若いエンジニアが上級エンジニアに質問しました。「Quick…QEMU は何ですか? どうやって再起動するんですか?」この質問がきっかけとなってこの記事を書くことにしました。ここからは、OpenStack クラウドの核心、つまり QEMU (Quick Emulator) について深く掘り下げていきます。

      クラウドの他の機能部分に QEMU がどのように接続されるか説明しながら、OpenStack 全般について、中枢から表面まで、重要な実行インスタンスからユーザーに表示する Horizon ダッシュボードまで解説していきます。IBM Bluemix Private Cloud や OpenStack になじみがない場合は、この記事を参照することで、概要を把握できます。

      OpenStack は、リリース・サイクルを経るごとに導入が広がってきています。これは、幅広いユース・ケースや消費モデルに対応していることに起因します。しかし、開発者にとっては OpenStack を手放しで歓迎できるわけではありません。実際には、完全に理解するのが難しく、機能が不十分であり、パーツの相互関係がますます複雑になってきています。そのため、急速に変化する OpenStack について共同開発者が持つ知識は、各自まちまちだと考えるべきです。

      QEMU: オープン・ソース型マシン・エミュレーターおよび仮想化機能

      QEMU は Quick Emulator の略であり、最初はさまざまな CPU アーキテクチャーのための、ごく平凡なソフトウェア・エミュレーターでした。また、さまざまな CPU アーキテクチャー (ARM や SPARC など) のテストにも有用でした。当初 QEMU が提供していた入出力機能は限られており、ブロック・デバイス (ディスク) の場合はファイル、ネットワーク・デバイスの場合はタップ (3) インターフェースでした。

      クラウドの場合は、実行中の仮想マシン (VM) に割り振られたすべてのメモリーが QEMU プロセスによって使用されることを押さえておく必要があります。特に、純粋な QEMU や QEMU + KVM の場合はそうです (これから説明していきます)。このような理由から、QEMU は多くの面で実際には VM の実行インスタンスです。つまり、単に QEMU プロセスをストレージ・デバイスやネットワークに接続するだけの場合は、以下のように最小限のクラウド (基本的なマシン) を作成していました。

      最小限の QEMU = 基本的なマシン・エミュレーター

      ここから OpenStack が急激に複雑になっていきます。ただし、速度に関しては、QEMU 自体は期待するほど高速ではありません。QEMU は、純粋な状態では、それぞれの命令のメモリー範囲を変換して検証し、解釈します。命令をすべてエミュレートする必要があります。

      最小限の QEMU + KVM = パフォーマンス向上

      カーネル仮想マシン (KVM) は、新しい CPU 命令を利用して、この最小限のクラウドに高性能な CPU およびメモリーの仮想化を追加します。頻繁に呼び出される CPU 命令およびメモリー・アクセスの場合では、この方法が高速です。現在、KVM は x86、IBM POWER、および ARM をサポートしています。

      KVM は通常の命令を高速に処理するため、KVM を使用する場合は QEMU がヘルパーに位置付けられます。ただし一部の特権命令の場合は、QEMU によって解釈される手段にフォールバックするため、QEMU + KVM の速度が低下することがあります。

      一般に、KVM が QEMU の補助として機能する場合は、VM の実行が高速化されます。特権命令に到達するまで、下位のハードウェア CPU でネイティブ実行が進行します。特権命令に到達すると、仮想コンテキストでは、上位階層が呼び出されることになり、KVM がその命令への対処方法を QEMU に問い合わせる必要があります。

      図に KVM を追加すると、その性質上、仮想 CPU およびメモリーが QEMU (エミュレーション用) から取り除かれ、KVM (構成用) を使用してハードウェア CPU で実行を制御することで速度を追求することになります。

      まとめると、上の図に示したように、QEMU をインタープリター (CPU 命令を評価するソフトウェアの手段) と見なし、KVM を下位の命令を高速に実行する手段と見なします。

      最小限のマシンの分離: QEMU とストレージ・バックエンド

      qemu disk space help を見ると、QEMU が実際にさまざまなストレージ・バックエンドをサポートしていることが分かります。ストレージ関連の処理を QEMU から分離すると、過去に開発されたあらゆる技術が詰まっています。本質的には、1970 年代までに蓄積した仮想メモリーに関する知識を総合して、その知識をストレージ技術に適用したものです。何ページも使用するのではなく、4MB 程度を使用することで、以下の機能が実現されています。

      • コピー・オン・ライト
      • スナップショット
      • シン・プロビジョニング
      • 階層型ストレージ

      例えば、Ceph RBD (RADOS Block Device) などの最新のストレージ・ソリューションでは、豊富なストレージ機能をネットワーク・アクセシビリティーと組み合わせて速度、レジリエンシー、冗長性を最適化するとともに、リモート・アクセスを可能にしています。クラウド・ストレージ・ソリューションの最適化を推進するにあたっては、入出力パスから完全に QEMU を取り除くことで、ゲストから入出力までのパフォーマンスを改善できます。このトピックについては、後の記事で取り上げます。

      仮想化サーバーのパフォーマンスはベア・メタル・サーバーとほぼ同等

      KVM と QEMU、およびストレージやネットワークのバックエンド技術 (Ceph RBD など) を使用すると、数年前のベア・メタル・サーバーのパフォーマンスを超える CPU、メモリー、ディスク、およびネットワークのパフォーマンス・レベルを達成できます。また、新しいストレージ機能 (シン・プロビジョニングやスナップショットなど) によって柔軟に運用することも可能になります。現在流通しているハードウェアでは、これらのソリューションがさらに高速化されています。

      OpenStack の構築

      複雑な OpenStack を構築するにあたっては、QEMU プロセスが単一ユーザーしか想定していないことを念頭に置いておく必要があります。本質的には、OpenStack VM を使用することによって、マルチユーザー対応が可能になり、初期の VM 構成が容易になり、QEMU が基本的にインスタンス化した最小限の VM マシンのセキュリティーが向上するに過ぎません。最終的に、以下のように複雑な OpenStack の「マシン」が実現します。

      OpenStack Grizzly リリース (少し前のもの)

      OpenStack クラウドの核心まで理解しているからこそ、IBM Bluemix Private Cloud では異例の製品パフォーマンスとサービスをお客様に提供できるのです。詳細については、今後の記事もご参照ください。クラウド・コンピューティングの基本要素について引き続き掘り下げていきます。
