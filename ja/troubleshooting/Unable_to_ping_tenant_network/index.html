      テナント・ネットワーク間で ping できないのはどうしてですか?

      ネットワークのトラブルシューティングの例: テナント・ネットワーク間で ping できない

      Q. サブネット (private_subnet) を使用した独自のネットワーク (private_network) を作成しました。また、ルーター (my_router) を作成して、このネットワークを既存の内部ネットワークに接続しました。しかし、private_network に接続されたインスタンスが、内部ネットワーク上のインスタンスに ping できません。ゲートウェイ IP アドレス (10.0.0.1) に ping でき、またルーター上の内部ネットワークのインターフェースの IP アドレス (192.168.1.113) に ping できます。しかし、内部ネットワーク上のどのインスタンスにも ping できません。逆方向、つまり内部ネットワーク上のインスタンスから private_network 上のインスタンスへは ping できます。その反対方向でのみ ping できません。

      OpenStack で関連コンポーネントを作成するために実行したコマンドは、以下のとおりです (覚えている限りのものです)。

      1. neutron net-create private_network
      2. neutron subnet-create private_network --name private_subnet --gateway 10.0.0.1 10.0.0.0/24
      3. neutron router-create my_router
      4. neutron router-interface-add my_router private_subnet
      5. neutron port-create internal
      6. neutron router-interface-add my_router port=

      ルーターを接続する内部ネットワーク上の追加ポートを手動で作成する必要がありました。これは、ルーターが既に接続されていたためです。

      A. my_router 上の内部ネットワークからのインターフェースを追加するだけという方法の問題は、内部ネットワーク上のインスタンスにデフォルト・ゲートウェイ 192.168.0.1 が設定されていることにあります。これはデフォルト・ルーター上にあり、デフォルト・ルーターが private_network に再びルーティングする方法を認識していません。

      反対側のサブネットからのポートを各ルーターに追加することに加え、両方向でルーティングできるように経路を追加する必要があります。

      neutron router-update default --routes type=dict list=true destination=10.0.0.0/24,nexthop=10.0.0.1
neutron router-update my_router --routes type=dict list=true destination=192.168.0.0/22,nexthop=192.168.0.1

      Q. 最初のコマンドを実行する試行が失敗し、以下のエラーが表示されました。

      Invalid format for routes: [{u'destination': u'10.0.0.0/24', u'nexthop': u'10.0.0.1'}], the nexthop is not connected with router

      以下に示すように、正常に実行するためにコマンドを変更する必要がありました。ここで、192.168.1.119 は、my_router を接続するために内部ネットワーク上に作成したポートです。

      neutron router-update default --routes type=dict list=true destination=10.0.0.0/24,nexthop=192.168.1.119

      private_network に接続されたインスタンスから内部ネットワークに接続されたインスタンスにトラフィックがフローできるようにするには、これで十分であることが分かりました。ただし、この構成では、ネットワーク・トラフィックに追加のホップが生じているように思えます。つまり、トラフィックがまず、デフォルト・ゲートウェイ (192.168.0.1) にルーティングされてから、192.168.1.119 にリダイレクトされています。以下のように、内部ネットワーク上のインスタンスから private_network 上のインスタンスに ping すると、これが分かります。

      ubuntu@test:~$ ping 10.0.0.13
PING 10.0.0.13 (10.0.0.13) 56(84) bytes of data.
64 bytes from 10.0.0.13: icmp_seq=1 ttl=63 time=0.783 ms
From 10.0.0.13: icmp_seq=2 Redirect Host(New nexthop: 10.0.0.13)
64 bytes from 10.0.0.13: icmp_seq=2 ttl=63 time=0.624 ms
From 10.0.0.13: icmp_seq=3 Redirect Host(New nexthop: 10.0.0.13)

      より優れたアプローチは、以下のように、アンシブル・スクリプトを使用して内部ネットワーク上の各ホストへの経路を手動で追加するだけという方法のように思われます。

      route add -net 10.0.0.0/24 gw 192.168.1.119

      この構成で発生した次の問題は、private_network 上のインスタンスが内部ネットワークを介して公衆インターネットに接続できないというものです。例えば、8.8.8.8 に ping できません。この理由、およびそれを可能にするために行う必要があることを教えてください。

      公衆インターネット上のホストへの ping 試行時のエラー・メッセージが「destination net unreachable」であるということを述べるのを忘れていました。このエラー・メッセージは、ゲートウェイ (10.0.0.1) までは到達していますが、その後に要求をルーティングする方法が分からないということを示唆しています。

      A. my_router を介してルーティングして内部ネットワークに到達すると、追加のネットワーク・ホップが生じるということは正しいですが、影響は恐らく無視できるほどわずかです。すべてのインスタンスで静的経路を追加すると、そのようなホップはなくなりますが、相当の管理オーバーヘッドおよび多数の潜在的な単一障害点が追加されます。優劣をつければ、ルーター・レベルでの経路を維持することをお勧めします。

      private_network 上のインスタンスがインターネットに到達できない理由についてですが、これは、ルーターに外部ゲートウェイが設定されていないためです (external_gateway_info を参照)。

      root@ds0006:~# neutron router-show the_router
+-----------------------+-------------------------------------------------------------+
| フィールド            | 値                                                          |
+-----------------------+-------------------------------------------------------------+
| admin_state_up        | True                                                        |
| distributed           | False                                                       |
| external_gateway_info |                                                             |
| ha                    | True                                                        |
| id                    | af23ae89-3a13-43cd-a6cd-f919f7c03a64                        |
| name                  | my_router                                                 |
| routes                | {"destination": "192.168.0.0/16", "nexthop": "192.168.0.1"} |
| status                | ACTIVE                                                      |
| tenant_id             | 61471bdea08844478c1341ab39b4929a                            |

      neutron router-gateway-set the_router external を実行すると、浮動 IP がルーターに割り振られ、外部世界に到達できるようになります。

      Q. my_router 上に外部ゲートウェイがないのは意図的です。private_network を外部世界からできる限り隠蔽したいのです。外部ゲートウェイが既に構成されているデフォルト・ルーターを介して公衆インターネット宛てのトラフィックをルーティングできるようにしたかったのです。これが可能か分かりますか? 可能な場合、その実現方法を教えてください。

      A. 以下のように、デフォルト・ルーターを my_router に追加できる可能性があります。

      neutron router-update my_router --routes type=dict list=true destination=0.0.0.0/0,nexthop=192.168.0.1

      これにより、実質上、外部ネットワークに直接接続されたルーターと同じ機能が提供されます。ただし、浮動 IP を割り振ることはできません。ルーターを外部ネットワークに直接接続し、インターネットからアクセスする必要があるインスタンスにのみ浮動 IP を割り振ることでアクセスを制御し、セキュリティー・グループ・ルールを使用して各種サーバー「ゾーン」を相互から保護する方が、柔軟です。

      デフォルト・ルーターの IP を指すように、外部ネットワークに接続されていないルーター上のデフォルト経路を設定することで、インターネットにルーティングできることを確認しました。以下に、IBM のラボ環境でどのようになっているかを示します。

      ディスカッション: 参考情報として、private_network に接続しようとしたインスタンスは真に内部的なものです。つまり、浮動 IP が割り振られていません。実際、これがここで使用しているインスタンスの大多数であり、オファリングで浮動 IP が必要なインスタンスはおよそ 4 つのみです。

      意図としては、内部ネットワークを DMZ に相当するものとして扱い、浮動 IP を必要とするインスタンスをその内部ネットワークに接続します。他のすべてのインスタンスは private_network に接続され、お客様が指定した構成を使用して、デフォルト・ルーターを介してトラフィックをルーティングすることでのみ、公衆インターネットにアクセスできます。

      上記はネットワークを分離する妥当な方法だと思えたでしょうか。それでもなお、単一のネットワークを使用し、適切なセキュリティー・グループ・ルールを作成することを推奨しますか? イエスの場合は、この提案しているアプローチと比較してそのアプローチの長所は何でしょうか?
