      Q. 仮想マシンの最終リブート日時を判別するにはどのようにすればよいでしょうか?

      A. 以下のコマンドを使用できます。

      zgrep BOOT_IMAGE /var/log/kern.log*

      これにより、ホスト (VM) の最終ブート時が表示されます。その後、そのタイム・スタンプを使用して、ログを調べて、トラブルシューティングしている問題に関する手掛かりを確認できます。

      Q. インスタンスがフリーズして応答しなくなった場合、どうすればよいでしょうか?

      A. 仮想マシンのコンソールを調べて、完全にフリーズしているかどうか、またカーネル・パニックや別のエラーを示す形跡がないかを確認することをお勧めします。以下によって、コンソールの URL を取得できます。

      nova get-vnc-console VM-UUID novnc

      コンソールでの時刻をメモします (存在する場合)。

      フリーズしていて、有用な情報が含まれている場合は、コンソールのスクリーン・ショットをキャプチャーします。リブート後、仮想マシンで異常終了ダンプを有効にしていた場合、/var/crash を確認して vmcore ファイル (RedHat の場合) または .crash ファイル (Ubuntu の場合) を見つけ、そこからダンプを分析できます。また、フリーズ時刻前後のシステム・アクティビティー (sar) 統計も確認します。

      マシンをできるだけ早く稼働状態に戻す必要がある場合は、マシンをリブートします。できれば、同じイメージに基づいて新しいものをスピンアップします。インフラストラクチャーについてさらに調査を行う必要がある場合は、リブートする前のフリーズ時刻 (UTC) およびインスタンスの UUID をお知らせください。こちらで、ログに何かないかを確認いたします。

      例えばファイル・システム・チェックを実行する場合や SSH ディレクトリー許可を修正する場合など、確実にシングル・ユーザー・モードに入る必要がある場合は、Safari ブラウザーでフルスクリーン・コンソールを使用して成功した実績があります。GRUB ロード画面でキャッチできるように、コンソールにログインしている間に「send ctrl/alt/del」ボタンを押して、VM をリブートする必要があります。GRUB ブート・メニューが表示されたら、e を押して編集します。例えば、CentOS のシングル・ユーザー・モードの場合、以下のようにします。

      1. スクロールダウンし、(console から始めて) 行末のほとんどを消去し、そこに console=tty のみを残します。
      2. 編集後、行は以下のようになります。kernel /boot/vmlinuz-2.6.32-431.el6.x86_64 ro root=UUID=8a057bba-2787-46a7-9436-fd6dc8a3aa85 rd_NO_LUKS rd_NO_LVM LANG=en_US.UTF-8 rd_NO_MD crashkernel=auto rd_NO_DM SYSFONT=latarcyrheb-sun16 KEYBOARDTYPE=pc KEYTABLE=us console=tty notsc single
      3. 行の ro 部分の代わりに、rw init=/bin/bash を追加することもできます。
      4. ブート・プロセスが停止しているようであれば、数回 Enter キーを押します。

      Q. インスタンスでレスキュー操作の実行が必要になった場合は、どうすればよいでしょうか?

      **A. **

      1. 以下のようにして、レスキュー CD の ISO を http://www.system-rescue-cd.org などから取得します。 
        wget https://dronedata.dl.sourceforge.net/project/systemrescuecd/sysresccd-x86/4.9.6/systemrescuecd-x86-4.9.6.iso
      2. 以下のようにして、sysutils パッケージから isohybrid をインストールします。 
        apt-get install syslinux [または yum install syslinux]
      3. ISO にマスター・ブート・レコードを追加して、ブート可能なハード・ディスクにします 
        isohybrid systemrescuecd-x86-4.9.6.iso
      4. 以下のようにして、新しいハイブリッド・レスキュー CD を Glance にアップロードします。 
        openstack image create --file systemrescuecd-x86-4.9.6.iso --public sysrescuecd-4.9.6.iso
      5. 以下のようにして、レスキューが必要なインスタンスをレスキュー・モードに切り替えます。 
        nova rescue brokeninstance --image sysrescuecd-4.9.6.iso
      6. Horizon の VNC コンソール経由でインスタンスにログインし、実行する必要のあるレスキュー操作を行います。コンソールにレスキュー CD が表示されるまで、1、2 分の待機が必要な場合あります。待機した後にコンソール画面を再ロードします。

      (/img/ja/systemrescuefromvnc.png)

      ここからは、以下のようなツールを使用できます。

      blkid
fsck /dev/vdb1
mount -t ntfs /dev/xxx /mnt/windows -o ro
ntfs-3g /dev/sda1 /mnt/windows
      1. インスタンスが修正されたことを確認したら、レスキュー・モードを終了し、以下のコマンドを使用して再び通常のブート・ディスクから起動します。 
        nova unrescue brokeninstance

      Q. インスタンスを m1.tiny から m1.medium にサイズ変更できないのはどうしてですか?

      インスタンスは、まさに m1.tiny フレーバーがより大きな一時ディスクを備えているときに作成された可能性があります。サイズ変更プロセスはそのことを検査し、その値が見つからなかった場合、サイズ変更に失敗します。

      サポート・チケットをオープンして、これが失敗の原因であるのかを確認する支援を受ける必要が生じることがあります。サポート・チームがログで以下のようなエラーを確認します。

      Exception during message handling: Resize error: Unable to resize disk down.

      あるいは、チケットをオープンせずに進める優れたオプションとして、以前と同じより大きな一時ディスク・サイズに m1.medium フレーバーを設定するか、スナップショットから新規を作成するためのトラブルシューティングの回避策 (/help-documentation/gettingstarted/commontech/Instance_Resize/) に従うことができます。

      Q. Neutron のスタックしたポートの原因は何ですか?

      Heat を使用して VM をインスタンス化する場合に、スタックしたポートが Neutron で発生することがあります。この問題は、Neutron の競合状態に起因する OpenStack Neutron (Mitaka リリース) の欠陥に関連していると考えられています。したがって、これを既知の制限として扱います。この状態により、スタックしたポートとともに、メタデータ障害も発生することがあります。

      以下のように、いくつかの回避策が存在します。

      回避策 1. 可能であれば、Heat を使用せずにインスタンスをデプロイします。手動でデプロイすると、この状態が回避されるようです。

      回避策 2. 別の回避策としては、API から WaitCondition を使用して、Heat で遅延を導入します。ただし、このシナリオでは WaitCondition は最適でない可能性があります。

      • WaitCondition は、WaitConditionHandler と連動して機能します。このハンドラーにより、テキスト・ストリングについてモニターされる URL/URI が作成されます。このストリングが存在していることは、VM のデプロイが成功したことを示し、このストリングがないことは、失敗を示します。VM がインスタンス化されると、PUT API 呼び出しを介して、適切なテキスト・ストリングを URL/URI に取り込むように指示できます。

      • 現在、L3 ルーターでこの API 呼び出しを行うためのメカニズムがこの OpenStack デプロイメントには存在しません。エンジニアリング・チームでは、「depends_on」の使用を提案しています。(以下のオプション 3 を参照)。

      回避策 3. 最適な回避策としては、depends_on を使用します。これにより、VM のデプロイメントが他の定義可能なリソース (例えば、L3 ルーターやネットワーク) の存在に依存する (「depends on」) ように、調整されたシナリオが作成されます。

      depends_on の簡潔な例を以下に示します。

      add_router_interface:
   type: OS::Neutron::RouterInterface
   properties:
     router: { get_resource: router}
     subnet: { get_resource: subnet}


server1:
    type: OS::Nova::Server
    depends on: add_router_interface
    properties:
      image: { get_param: image }

      明示的な依存関係の詳細については、以下のリンクを参照してください。

      • https://wiki.openstack.org/wiki/Heat/Blueprints/hot-software-config-spec

      Q. OpenStack に対してどのポートをオープンにする必要がありますか?

      少なくとも、DHCP 用のポート 67/68 およびメタデータ用のポート 80 をオープンする必要があります。また、VM が OpenStack API サービスにアクセスできるようにする場合は、以下に記載されているポートを含めます。

      /help-documentation/gettingstarted/commontech/FAQ/#which-portsurls-etc-are-publicly-accessible-on-my-bluemix-private-cloud-and-for-what-purpose

      必須の最小ポート・セットがオープンされていない場合、cloud_init でトラブルが発生し、インスタンスが正常に開始できないことがあります。

      Q. 次のエラーは何を意味しますか? 「ボリューム制限情報を取得できません (Unable to retrieve volume limit information)」

      Horizon ダッシュボードでページをロードしているときによく、「エラー: ボリューム制限情報を取得できません (Unable to retrieve volume limit information)」という赤いポップアップが表示されます。これは何を意味するのでしょうか

      このエラーは安全性に問題はありませんが、デプロイメントに Cinder ブロック・ストレージがないために発生します。

      Keystone サービスから Cinder エンドポイントを削除するとエラーを解決できます。この場合、自動化構成ファイルの更新が必要となり、サポート・チームによる確認が必要となります。このエラー・メッセージの解決にヘルプが必要な場合、サポート・チケットをオープンしてください。

      サポート技術者が理解する必要があるステップの例は以下のとおりです。

      # openstack endpoint list | grep cinder
